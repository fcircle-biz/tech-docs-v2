サブエージェントと自動化

大きな作業を効率化するための考え方を学びます。サブエージェントへのタスク委譲、並列処理の活用、フック（hooks）による定型処理の自動化など、Claude Code をより賢く使いこなすための仕組みを理解します。

この章で学ぶこと

  • サブエージェントの概念と、大きなタスクを分割して委譲するメリットを理解する
  • 独立したタスクを並列実行することで作業を効率化できることを知る
  • フック（hooks）の概念と、定型処理を自動化するしくみを理解する
  • 自動化をいつ使うべきか、使いすぎないバランスを考えられる

12.1 サブエージェントとは

Claude Code を使っていると、「複数のファイルを調べながら同時に別の作業も進めたい」「大きな調査を分担して効率よく処理したい」という場面に出会います。そのような場面に対応するのがサブエージェントという考え方です。

サブエージェントとは、メインの Claude がタスクの一部を別のエージェント（サブエージェント）に委譲して処理させる仕組みです。メインのエージェントは全体の流れを管理しながら、個々の調査・分析・定型作業をサブエージェントに任せることができます。

この仕組みは特に次のような場面で力を発揮します。

広範な調査

コードベース全体を複数の観点から同時に調べる場合など、探索的・並行的な作業に向いています。

独立タスクの分担

互いに依存しない複数の作業を同時並行で進めることで、全体の処理時間を短縮できます。

役割の分離

設計・分析はメインが担当し、定型的な生成・変換作業はサブに任せるなど、役割を明確に分けられます。

補足: 「マルチエージェント」とも呼ばれる

複数のエージェントが連携して作業する仕組みは「マルチエージェント」とも表現されます。Claude Code の文脈では、メインのエージェントがオーケストレーター（指揮役）となり、サブエージェントが個別タスクを担当する構成が一般的です。

サブエージェント関連の機能

最近の Claude Code では、サブエージェントや並列作業を扱う具体的な機能が増えています。たとえば /agents（サブエージェントの管理）、/fork（会話の分岐）、/batch（一括処理）、バックグラウンドセッション、git の worktree を使った並行作業などがあります。本章では概念の理解を優先し、詳細は /help や公式ドキュメントで確認してください。

12.2 ファンアウト: メインから複数サブへの委譲

サブエージェントの典型的なパターンが「ファンアウト」です。メインのエージェントが大きなタスクを複数の小さなタスクに分割し、それぞれをサブエージェントに同時並行で委譲します。各サブが完了したら、メインがその結果を集約して最終的な成果物をまとめます。

サブエージェントのファンアウト構造

flowchart TD M["メインエージェント（オーケストレーター）
タスクを分析・分割・集約する"] M --> S1["サブエージェント A
調査・分析タスク"] M --> S2["サブエージェント B
ファイル生成タスク"] M --> S3["サブエージェント C
検証・チェックタスク"] S1 --> R["結果を集約"] S2 --> R S3 --> R R --> OUT["最終成果物・報告"]

このパターンは、たとえば「複数のモジュールのドキュメントを一括生成する」「コードベース内の複数ファイルを並列にレビューする」といった場面で使われます。独立して処理できる単位にタスクを分割することがポイントです。

並列化できる条件

ファンアウトで並列化できるのは、タスク同士が互いの結果に依存していない場合に限ります。「A が終わらないと B が始められない」という依存関係があるタスクは、逐次（順番に）実行する必要があります。

12.3 フック（hooks）による自動化

フック（hooks）とは、Claude Code の特定のタイミング（ツールの実行前後など）に、あらかじめ設定した処理を自動的に走らせる仕組みです。最もよく使うのはシェルコマンドの実行ですが、それだけでなく HTTP リクエストや MCP ツールの呼び出しなど、複数のタイプを設定できます。フックを使うと、「毎回手動で行っていた定型処理」を自動化できます。

フックは settings.json に設定します。どのタイミング（イベント）でどのコマンドを実行するかを記述する形式です。

タイミング（イベント） 説明 活用例
PreToolUse ツールが実行される直前 事前チェック、ログ記録、入力検証
PostToolUse ツールが実行された直後 コードフォーマット、通知送信、後処理
Stop Claude が応答を停止したとき 完了通知、ログの書き出し
Notification 通知イベントが発生したとき デスクトップ通知、Slackへの投稿

次のコードは、フックの設定イメージを示す概念的な例です。settings.json に記述します。

settings.json — フック設定の概念例 
{
  "hooks": {
    "PostToolUse": [
      {
        // ファイル編集後にコード整形を自動実行する例
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write ."
          }
        ]
      }
    ]
  }
}

フックの設定場所とタイプ

フックはユーザーレベル（~/.claude/settings.json）とプロジェクトレベル（.claude/settings.json）の両方で設定できます。プロジェクト固有の自動化はプロジェクトレベルに書くのが一般的です。上表は代表的なイベントの例で、ほかにも多くのイベントがあり、実行タイプもシェルコマンドだけでなく HTTP・MCP ツールなどから選べます。終了コードや JSON 出力で挙動を制御することもできます。詳細は公式ドキュメントの「Hooks」セクションを参照してください。

12.4 繰り返し作業を「仕組み化」する

フックとカスタムスラッシュコマンドを組み合わせると、繰り返し発生する作業を「仕組み」として定着させられます。毎回手順を思い出して手動で実行するのではなく、一度設定しておけば自動で動く状態を作ることができます。

カスタムスラッシュコマンド

よく使う指示や手順を .claude/commands/ にMarkdownファイルとして保存しておくと、/コマンド名 で呼び出せます。

例: /my-review で自分用のレビュー手順を自動実行

フックによる自動後処理

ファイルを編集するたびに整形（フォーマット）を走らせるなど、「毎回やるべきこと」をフックとして登録します。

例: PostToolUse で prettier を自動実行
手法 効果 設定場所
カスタムスラッシュコマンド 定型プロンプト・手順の再利用。呼び出しを短縮。 .claude/commands/
フック（PreToolUse） ツール実行前の自動チェックや入力検証 settings.json
フック（PostToolUse） 編集後の整形・テスト・通知の自動化 settings.json
フック（Stop） 作業完了時の通知やログ記録 settings.json

12.5 いつ使うか／使いすぎないバランス

サブエージェントや自動化は強力な手段ですが、すべての場面に適しているわけではありません。どんな状況で使うと効果的か、逆に向いていない場面はどこかを理解することが重要です。

有効な場面

  • 互いに独立した複数のタスクを同時に進めたい場合
  • 毎回同じ手順が発生する定型作業（整形・検査・通知など）
  • 大規模なコードベースの探索や、多くのファイルを横断する調査
  • 特定の判断は人間が行い、機械的な作業だけを委譲したい場合

注意が必要な場面

  • タスク間に依存関係があり、順番に実行する必要がある場合
  • サブエージェントに任せるコストが、分割による恩恵を上回る小さなタスク
  • フックの実行コマンドが誤動作した場合の影響が大きい処理
  • 自動化の設定・保守コストが実際の節約時間より大きい場合

フックの設定には慎重に

フックに設定したコマンドは自動的に実行されます。破壊的な操作（ファイル削除など）を誤って登録すると、意図しない被害が発生することがあります。フックを設定するときは、コマンドの動作を事前に十分確認してから登録しましょう。

やってみよう 12-1: 大きな調査をサブエージェント的に分割して依頼する

大きな調査依頼をサブエージェントのファンアウトのように分割して Claude Code に伝えることで、効率的な作業を体験します。

手順

  1. 1 自分のプロジェクトで「広範に調べたいこと」を1つ選ぶ（例: 「このコードベースにどんなエラーハンドリングのパターンが使われているか調べてほしい」）。
  2. 2 その調査を「独立した小タスク」に分割して書き出す（例: 「① src/ 配下を調べる」「② tests/ 配下を調べる」「③ 結果をまとめる」）。
  3. 3 Claude Code に「並行して調べて結果をまとめて」と伝える。たとえば「src/ と tests/ を並行して調査し、エラーハンドリングのパターンを比較してほしい」のように依頼する。
  4. 4 Claude Code がどのように作業を進めるかを観察し、分割して依頼した場合と一括で依頼した場合の違いを体感する。
観察のポイント: Claude Code が複数の方向から調べたり、構造を整理して報告するかどうかに注目しましょう。大きなタスクを明確に分割して依頼するほど、結果が整理されやすくなります。

理解度確認クイズ

  1. Q1. サブエージェントのファンアウトパターンが特に有効なのは、どのような条件のタスクに対してですか？

    回答を表示

    互いに依存しない（独立した）複数のタスクを同時並行で処理できる場合に有効です。タスク間に依存関係があり、順番に実行する必要がある場合はファンアウトには向きません。

  2. Q2. フック（hooks）は、どのファイルに設定しますか？また、代表的なタイミング（イベント）を1つ答えてください。

    回答を表示

    settings.json に設定します（ユーザーレベルは ~/.claude/settings.json、プロジェクトレベルは .claude/settings.json）。タイミングの例としては PostToolUse（ツール実行後）などがあります。

  3. Q3. 繰り返し使うプロンプトや手順を Claude Code に登録して呼び出せる仕組みを何と呼びますか？また、どこに保存しますか？

    回答を表示

    カスタムスラッシュコマンドと呼びます。.claude/commands/ ディレクトリに Markdown ファイルとして保存すると、/コマンド名 で呼び出せます。
