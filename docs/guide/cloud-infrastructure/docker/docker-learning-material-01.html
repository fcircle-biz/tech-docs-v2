Dockerとは何か

Docker（ドッカー）は、アプリを「必要なものを丸ごと詰めた箱」にまとめて、どのパソコンでも同じように動かせるようにする道具です。この章では、Dockerが何を解決するのか、なぜ便利なのかを、身近な例えを使ってつかみます。まだコマンドは打ちません。まずは「考え方」をしっかり理解しましょう。

この章で学ぶこと

  • 「自分のPCでは動くのに、他の人のPCでは動かない」問題とは何かを理解する
  • Dockerとコンテナの考え方を、身近な例えで説明できる
  • イメージとコンテナのざっくりした違いを理解する
  • Docker Hub とは何か、Dockerを使うと何が便利になるのかを知る

本教材の進め方について

本教材では、Dockerを動かすソフトとして Rancher Desktop（ランチャー デスクトップ）を使います。すでに Docker Desktop を使っている人でも、本教材で出てくる基本的な docker コマンドはほぼ同じなので、そのまま学習を進められます。環境の準備は第2章で行うので、この章では「Dockerとは何か」という考え方だけに集中しましょう。

1.1 「自分のPCでは動くのに、他の人のPCでは動かない」問題

プログラミングやチーム開発をしていると、よく起きるのが次のようなトラブルです。「自分のパソコンでは問題なく動いていたプログラムが、同僚のパソコンや本番のサーバーに持っていくと、エラーが出て動かない」——いわゆる「自分のPCでは動くのに」問題（英語で "It works on my machine" と呼ばれます）です。

なぜこんなことが起きるのでしょうか。それは、パソコンごとに環境（かんきょう）がバラバラだからです。ここでいう環境とは、たとえば次のようなものです。

  • ソフトのバージョン — 同じソフトでも 1.0 と 2.0 で動きが違うことがある
  • OS の違い — Windows と Mac で挙動が変わる
  • 設定や追加部品 — 必要なライブラリ（部品）が入っていない／設定が違う

つまり、「プログラム本体」だけを渡しても、それを動かすための周辺の環境がそろっていなければ、同じようには動かないのです。下の図は、同じプログラムでもパソコンごとに環境が違うと結果が変わってしまう様子を表しています。

図1: 従来の開発 — パソコンごとに環境が違う

flowchart TD P["同じプログラム"] --> A["開発者AのPC
バージョン18 ＋ 設定X"] P --> B["開発者BのPC
バージョン16 ＋ 設定Y"] P --> C["本番サーバー
必要な部品が不足"] A -->|環境が合う| OK["正常に動く"] B -->|環境が違う| NG1["エラーで動かない"] C -->|環境が違う| NG2["想定外の不具合"]

この「環境のばらつき」をなくし、誰のパソコンでも・どのサーバーでも同じように動かせるようにする道具こそが、これから学ぶ Docker です。

1.2 Dockerとは — 「引っ越しの段ボール箱」で考える

Docker を一言でいうと、アプリと、それを動かすために必要なものを丸ごと1つの箱に詰めて、どこでも同じように動かせるようにする道具です。この「箱」のイメージを、引っ越しでたとえてみましょう。

たとえ話: 引っ越しの段ボール箱

引っ越しのとき、コップだけを裸で運ぶと、相手の家に「ちょうど合う棚」がなければ置き場所に困ります。そこで私たちは、コップ・お皿・緩衝材をまとめて段ボール箱に詰めて運びます。箱ごと運べば、どの家に届けても中身はそのまま。すぐに同じ状態で取り出せます。Dockerの「箱」も同じで、アプリ本体＋必要なソフト＋設定をひとまとめにして運ぶので、届け先のパソコンの状態に左右されません。

この「箱」のことを、Dockerの世界では コンテナ（container） と呼びます。コンテナという言葉は、もともと貨物船で荷物を運ぶ「コンテナ」から来ています。中身が何であっても、決まったサイズの箱に入れて規格をそろえれば、同じやり方で運んだり積み替えたりできる——その考え方をソフトウェアに持ち込んだものです。

丸ごと詰める

アプリ本体だけでなく、動かすのに必要なソフトや設定もまとめて箱に入れます。

どこでも同じ

箱ごと動かすので、WindowsでもMacでもサーバーでも同じように動きます。

すぐ使える

箱を開ければすぐ動くので、面倒な環境構築をやり直す必要がありません。

Dockerを使うと、開発者全員が「同じ箱」を使うことになります。下の図のように、1つの設計図（イメージ）から、誰のパソコンでも同じコンテナが起動するため、「自分のPCでは動くのに」問題が起きにくくなります。

図2: Dockerを使った開発 — 同じイメージから同じコンテナが起動する

（アプリ ＋ 必要なものを封入した設計図）"] --> A["開発者AのPC"] IMG --> B["開発者BのPC"] IMG --> C["本番サーバー"] A --> RA["同じコンテナが起動"] B --> RB["同じコンテナが起動"] C --> RC["同じコンテナが起動"]

1.3 イメージとコンテナ — 「レシピと料理」で考える

Dockerを学ぶうえで、最初につまずきやすいのが イメージ と コンテナ の違いです。この2つは、「料理のレシピ」と「出来上がった料理」にたとえると、すっきり理解できます。

イメージ ＝ レシピ

イメージ（image）は、コンテナを作るための「設計図」や「レシピ」です。材料（必要なソフト）や手順（設定）が書かれた、動かす前のひな形です。レシピ自体は食べられませんが、これがあれば何度でも同じ料理を作れます。

コンテナ ＝ 料理

コンテナ（container）は、イメージ（レシピ）をもとに実際に作られ、動いている実体です。出来上がった料理にあたります。同じレシピから何皿でも作れるように、1つのイメージからいくつものコンテナを起動できます。

この関係を図にすると、次のようになります。インターネット上の保管庫（Docker Hub）からレシピ（イメージ）を取り寄せ、そこから料理（コンテナ）を作る、という流れです。

図3: イメージとコンテナの関係

flowchart LR HUB["Docker Hub
（イメージの保管庫）"] -->|ダウンロード| IMG["イメージ
（料理のレシピ）"] IMG -->|起動| C1["コンテナ1
（出来上がった料理）"] IMG -->|起動| C2["コンテナ2
（同じレシピの別の一皿）"]

覚え方のコツ

イメージは「静止した設計図」、コンテナは「動いている実体」と覚えましょう。1つのイメージから複数のコンテナを作れる、という点も重要です。この違いは第3章以降、実際にコマンドを動かすとさらにはっきり実感できます。

1.4 Docker Hub とは — イメージの保管庫

料理のレシピを一から自分で考えるのは大変です。便利なことに、世の中には有名な料理のレシピ集があります。Dockerにも同じように、世界中で使われているイメージ（レシピ）が集められたレシピ集の置き場があります。それが Docker Hub（ドッカー ハブ） です。

Docker Hub は、Dockerイメージを探したり・取り寄せたり（ダウンロード）・公開したりできるインターネット上のサービスです。たとえば、この教材で後ほど使う Webサーバーの Nginx（エンジンエックス） や、お試し用の hello-world といったイメージも、Docker Hub にあらかじめ用意されています。手元にイメージがなくても、Dockerが自動でDocker Hubから取り寄せてくれます（第3章で体験します）。

重要ポイント

Docker Hub のおかげで、私たちは有名なソフトの「すぐ動く箱」をゼロから作らずに利用できます。「Nginxを使いたい」と思ったら、自分でインストールや設定をしなくても、Docker Hub にあるNginxイメージを取り寄せるだけで使い始められます。これがDockerの大きな便利さの1つです。

1.5 Dockerを使うと何が便利になるのか

ここまでの内容をふまえ、Dockerを使うと具体的にどんな良いことがあるのかを整理しておきましょう。

  • 環境のばらつきがなくなる — 「自分のPCでは動くのに」問題が起きにくくなる
  • 環境構築が速い — 必要なソフトを1つずつ入れなくても、箱を取り寄せればすぐ使える
  • チームで共有しやすい — 同じイメージを配れば、全員がまったく同じ環境で開発できる
  • 後片付けがかんたん — コンテナを削除すれば、パソコンを汚さずに元に戻せる
  • 本番に近い状態で試せる — 開発でも本番でも同じ箱を使えるため、トラブルが減る

この章で出てきた重要用語の整理

例え話で理解した内容を、正式な用語で整理しておきましょう。これらの言葉は、これからの章で何度も出てきます。

用語 たとえ 意味
Docker 箱詰めの仕組み アプリと必要なものを箱（コンテナ）にまとめ、どこでも同じように動かす道具。
イメージ 料理のレシピ コンテナを作るための設計図・ひな形。動かす前の状態。
コンテナ 出来上がった料理 イメージをもとに実際に動いている実体。1つのイメージから複数作れる。
Docker Hub レシピ集の置き場 イメージを探す・取り寄せる・公開できるインターネット上の保管庫。
ホストマシン 箱を置く家 ホストマシン＝自分のPC（パソコン）のこと。Dockerやコンテナは、このホストマシンの上で動く。本教材では「ホスト」「ホストマシン」はどちらも自分のPCを指す。

手順

  2. 2 「イメージ」と「コンテナ」の違いを、レシピと料理の例えを使って一言で説明してみる。
  3. 3 第2章で環境を準備したら、実際にイメージを取り寄せてコンテナを動かすことを目標にする。

この章のまとめ

  • パソコンごとに環境が違うため、「自分のPCでは動くのに」問題が起きる。Dockerはこれを解決する道具。
  • コンテナは、アプリと必要なものを丸ごと詰めた「箱」。どこでも同じように動く。
  • イメージ＝レシピ、コンテナ＝出来上がった料理。1つのイメージから複数のコンテナを作れる。
  • Docker Hubはイメージの保管庫。ホストマシンはDockerを動かすパソコンやサーバー本体。

確認問題

  1. Q1.「自分のPCでは動くのに、他の人のPCでは動かない」問題は、なぜ起きるのでしょうか。

    回答を表示

    パソコンごとに「環境」（ソフトのバージョン・OS・設定・必要な部品など）がバラバラだからです。プログラム本体だけを渡しても、それを動かすための周辺環境がそろっていないと、同じようには動きません。

  2. Q2. イメージとコンテナの違いを、料理のたとえを使って説明してください。

    回答を表示

    イメージは「料理のレシピ（設計図）」で、動かす前のひな形です。コンテナは「そのレシピから実際に作られ、動いている料理（実体）」です。1つのイメージ（レシピ）から、複数のコンテナ（料理）を作れます。

  3. Q3. Docker Hub とは何をするためのものですか。

    回答を表示

    Dockerイメージを探したり、取り寄せたり（ダウンロード）、公開したりできるインターネット上の保管庫です。Nginxやhello-worldなど、有名なイメージがあらかじめ用意されており、手元になくてもDockerが自動で取り寄せてくれます。
