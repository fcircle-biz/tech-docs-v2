第6章 所要時間: 約45分

ファイルをコンテナに渡す：バインドマウントとボリューム

コンテナは使い捨ての「箱」です。コンテナを削除すると、中で変更したファイルも消えてしまいます。この章では、大事なファイルをホスト（自分のPC）側に置いたまま、コンテナへ渡す方法を2つ学びます。開発作業でとくに役立つバインドマウントと、データ保存に使うボリュームを理解しましょう。

この章で学ぶこと

  • コンテナ内の変更はコンテナ削除で消えることを理解する
  • バインドマウントでホストのファイルをコンテナに渡せる
  • ボリュームの考え方を知る
  • 開発中はバインドマウント・永続データはボリュームという使い分けを理解する

6.1 コンテナの中の変更は消える

これまでの章でコンテナを起動・停止してきました。ここで、重要な性質を確認しておきましょう。コンテナを削除すると、その中で変更したファイルも一緒に消えます。

たとえば、Nginxコンテナを起動して中にファイルを置いたとします。その後、docker rm でコンテナを削除すると、追加したファイルはなくなります。次に同じイメージからコンテナを作り直しても、ファイルは元通りには残っていません。

これはコンテナが「使い捨ての箱」として設計されているからです。毎回きれいな状態から始められる、という利点でもあります。ただし、消えてほしくないファイルはコンテナの外に置かなければなりません。その方法がこの章のテーマです。

コンテナ削除でファイルも消える

コンテナ内でファイルを作ったり書き換えたりしても、docker rm するとすべて消えます。大事なデータはホスト側に置いてコンテナに渡す習慣をつけましょう。

6.2 ファイルを渡す2つの方法

ホスト（自分のPC）とコンテナの間でファイルをやり取りする方法は、主に2つあります。バインドマウントとボリュームです。まずはざっくりとした違いを押さえましょう。

バインドマウント

ホスト（PC）上の特定のフォルダを、コンテナ内の特定フォルダに「つなげて見せる」方法です。PCのフォルダを編集するだけで、コンテナ内にも即座に反映されます。開発作業でとくに便利です。

ボリューム

Dockerが管理する保管領域にデータを保存する方法です。コンテナを削除してもデータが残ります。データベースなど消えてほしくないデータの保存に向いています。

この章では、まずバインドマウントを実際に試します。ボリュームは概念と基本コマンドを紹介します。使い分けについては6.5節でまとめます。

6.3 バインドマウントを試す

バインドマウントは、ホスト側のフォルダをコンテナの中から「そのまま見える」ようにする仕組みです。実際に手を動かして体験してみましょう。

ファイル構成を作る

まず、ホスト側（自分のPC）に次の構成でフォルダとファイルを作ります。作業ディレクトリの任意の場所に mount-sample フォルダを作り、その中に html フォルダ、さらにその中に index.html を置きます。

作業フォルダ構成 

mount-sample/
└── html/
    └── index.html    ← 自分で作るHTMLファイル

index.html の内容は、第5章で作ったものと同様のシンプルなHTMLで構いません。以下はサンプルです。

mount-sample/html/index.html 
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <title>マイページ</title>
</head>
<body>
  <h1>バインドマウントで表示中</h1>
  <p>このファイルはホスト側にあります。</p>
</body>
</html>

バインドマウントでコンテナを起動する

mount-sample フォルダの中（html フォルダの親）に移動してから、次のコマンドを実行します。-v オプションがバインドマウントの指定です。

ターミナル 
# PCのhtmlフォルダをコンテナのNginx配信フォルダにつないで起動する
docker run --name mounted-nginx -p 8080:80 -v ./html:/usr/share/nginx/html nginx
オプション 意味
--name mounted-nginx コンテナに mounted-nginx という名前をつける
-p 8080:80 PCの8080番ポートをコンテナの80番ポートに転送する
-v ./html:/usr/share/nginx/html PCの ./html フォルダを、コンテナ内の /usr/share/nginx/html（Nginxの配信フォルダ）につなぐ
nginx 使用するイメージ名

-v の書き方は「ホスト側のパス：コンテナ側のパス」と覚えてください。コロン（:）で区切って、左がPC側、右がコンテナ側です。

図1: バインドマウント — PC側のhtmlフォルダがコンテナ内に見えている

flowchart LR A["PC側: ./html
（index.html を格納）"] -->|"-v でマウント"| B["コンテナ内:
/usr/share/nginx/html"] B --> C["Nginx がファイルを配信"] C --> D["ブラウザで確認
http://localhost:8080"]

動作確認

  1. 1 ブラウザで http://localhost:8080 を開き、index.html の内容が表示されることを確認する
  2. 2 PC側の html/index.html をテキストエディタで書き換えて保存する（例: h1 タグの文字を変える）
  3. 3 ブラウザを再読み込みする。コンテナを作り直さなくても表示が変わることを確認する

これが開発中にバインドマウントが便利な理由です。ファイルを書き換えるたびにコンテナを作り直す必要がなく、すぐに結果を確認できます。

【スクリーンショット：ブラウザで http://localhost:8080 を開いた画面。自作の index.html の内容が表示されている様子】

コンテナを停止・削除する

確認が終わったら、コンテナを停止して削除します。コンテナを削除しても、PC側の html フォルダとファイルはそのまま残ります。

ターミナル 
# コンテナを停止する
docker stop mounted-nginx

# コンテナを削除する
docker rm mounted-nginx

6.4 ボリュームの紹介

ボリューム（volume）は、Dockerが管理する専用の保管領域です。バインドマウントがホスト側の特定フォルダを直接つなぐのに対し、ボリュームはDocker自身が場所を管理します。

コンテナを削除してもボリューム内のデータは残り続けます。このため、データベースのデータなど消えては困るデータの保存によく使われます。この章では実際のデータベース構築は行いませんが、基本的なコマンドを確認しておきましょう。

ターミナル（補足コマンド） 
# ボリュームの一覧を表示する
docker volume ls

# my-volume という名前のボリュームを作る
docker volume create my-volume

# my-volume ボリュームを削除する
docker volume rm my-volume

ボリュームをコンテナに渡す際も -v オプションを使います。書き方は「ボリューム名：コンテナ側のパス」です。バインドマウントとの違いは、ホスト側にパスを書く代わりにボリューム名を書く点だけです。

コンテナ削除後もデータが残る"]

6.5 使い分けのポイント

バインドマウントとボリューム、どちらを使えばよいのでしょうか。シンプルな目安があります。

開発中のソースファイル

→ バインドマウントが便利。ファイルを書き換えるだけで即座にコンテナに反映されるため、開発のサイクルが速くなります。

消えては困る永続データ

→ ボリュームが適している。データベースのデータやログなど、コンテナを削除しても残したいデータの保存に使います。

重要ポイント: 使い分けの基準

  • 開発中のファイル（HTML・コードなど） → バインドマウント（書き換えがすぐ反映される）
  • 消えてはいけないデータ（DBなど） → ボリューム（Docker管理で安全に保存）

よくあるエラーと対処法

エラー: マウント元のパスが見つからない

原因: -v ./html:... と書いたのに、現在のフォルダに html フォルダが存在しない。

対処: コマンドを実行する前に、ls（Macなど）または dir（Windows）で html フォルダが存在するか確認する。mount-sample フォルダの中でコマンドを実行しているかも確認しましょう。

注意: ブラウザで空のページが表示される

原因: ホスト側の html フォルダに index.html が存在しない。

対処: html/index.html を作成してから、ブラウザを再読み込みする。ファイル名のスペルミスにも注意。

エラー: ポート8080がすでに使用中

原因: 前の章で起動したコンテナや別のアプリがポート8080を使っている。

対処: docker ps で動いているコンテナを確認し、不要なコンテナを docker stop で停止してから再試行する。

この章のまとめ

  • コンテナを削除すると、その中で変更したファイルも消える。大事なファイルはホスト側に置いてコンテナに渡す。
  • バインドマウントは -v ホスト側パス:コンテナ側パス で指定する。PC側のフォルダ変更がコンテナにすぐ反映される。
  • ボリュームはDockerが管理する保管領域。コンテナを削除してもデータが残り続ける。
  • 開発中のソースファイルにはバインドマウント、消えてはいけない永続データにはボリュームを使うのが基本の使い分け。

確認問題

  1. Q1. コンテナを docker rm で削除すると、コンテナ内で変更したファイルはどうなりますか？

    回答を表示

    コンテナと一緒に削除されて、なくなります。コンテナは「使い捨ての箱」として設計されているため、中で変更したファイルもコンテナ削除で消えます。消えてほしくないファイルはホスト側に置いてコンテナに渡す必要があります。

  2. Q2. バインドマウントの -v オプションの書き方を答えてください。また、ホスト側の ./html フォルダをコンテナの /var/www/html にマウントする場合、どのように書きますか？

    回答を表示

    書き方は -v ホスト側のパス:コンテナ側のパス です。例の場合は -v ./html:/var/www/html と書きます。コロン（:）で区切り、左にホスト側、右にコンテナ側のパスを指定します。

  3. Q3. バインドマウントとボリュームはそれぞれどんな場面で使いますか？簡単に説明してください。

    回答を表示

    バインドマウントは開発中のソースファイル（HTMLやコードなど）を渡すときに便利です。ホスト側でファイルを書き換えるとコンテナにすぐ反映されます。ボリュームはデータベースのデータなど、コンテナを削除しても消えてほしくない永続データの保存に使います。
