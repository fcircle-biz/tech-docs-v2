GitHubとつなぐ：push / clone / pull

ローカルPCで記録したコミットを、インターネット上の GitHub へ送る（push）方法と、GitHub 上のリポジトリを手元に取得する（clone / pull）方法を学びます。GitHubとの接続には、第2章で準備した SSH 方式を使います。この章を終えると、自分の変更を GitHub 上に安全に保存できるようになります。

この章で学ぶこと

  • GitHub でリポジトリを作成する手順を説明できる
  • git remote add origin <URL> でローカルとリモートを接続できる
  • git push -u origin main で変更を GitHub にアップロードできる
  • git clone で GitHub のリポジトリを自分のPCに取得できる
  • git pull でリモートの最新変更をローカルに取り込める

4.1 ローカルとリモートのやり取りを図で理解する

第1章で「ローカル（自分のPC）」と「リモート（GitHub）」という概念を紹介しました。この章では、その2つを具体的なコマンドで結びつける方法を学びます。まず全体の流れを図で確認しておきましょう。

flowchart LR subgraph ローカル["ローカル（自分のPC）"] A["作業フォルダ
git リポジトリ"] end subgraph リモート["リモート（GitHub）"] B["共有リポジトリ"] end A -->|"git push: 変更を送る"| B B -->|"git clone: 丸ごと取得"| A B -->|"git pull: 最新を取り込む"| A

push（送る）

ローカルのコミットを GitHub へアップロードします。「自分の変更を共有する」操作です。

clone（丸ごと取得）

GitHub 上のリポジトリを初めてローカルに取得するときに使います。履歴ごとコピーされます。

pull（最新を取り込む）

clone 済みのリポジトリに、リモートの差分だけを取り込むときに使います。

4.2 GitHubでリポジトリを作成する

push の送り先となる「リモートリポジトリ」を GitHub 上に用意します。GitHub にログインし、右上の「＋」ボタン → New repository をクリックして作成画面を開きましょう。

項目 設定内容 補足
Repository name 例: my-first-repo 半角英数字・ハイフン・アンダースコアが使えます
Public / Private 初心者には Private がおすすめ Public＝誰でも閲覧可、Private＝自分だけ（招待制）。まずは Private が安心です
Add a README file 既存リポジトリを push する場合はチェックを外す チェックを入れると最初のコミットが自動生成されます
.gitignore / License 最初は None でOK 後から追加できます

注意: 初心者には Private がおすすめ

Public にすると、リポジトリの中身を世界中の誰でも見られる状態になります。個人情報、APIキーやパスワード、業務で扱うファイル、提出前の課題などを置いてしまうと、そのまま公開されてしまいます。学習中で迷ったら、まずは Private（自分だけが見られる）を選んでおくと安心です。Private で作っても、この先の push / clone / pull の手順は同じです。

README.md とは？

リポジトリの「説明書」ページです。GitHub のリポジトリページを開くと、README.md の内容が自動的に表示されます。プロジェクトの概要・使い方・セットアップ手順などを書くのが一般的です。

補足: 最初は README・.gitignore・license を付けずに作る

第3章で作ったリポジトリを後から push する場合は、GitHub 側でこれらを追加せず（空のまま）作るのがおすすめです。特に「Add a README file」にチェックを入れると、GitHub 側に最初のコミットが1つできてしまい、手元（ローカル）の履歴と GitHub 側の履歴がずれて、最初の push でつまずくことがあります。README などは、あとから手元で作って push すれば大丈夫です。

「Create repository」をクリックするとリポジトリが作成されます。次に、画面の緑色の「Code」ボタンを押して SSH タブを選ぶと、SSH形式のURL（例: git@github.com:ユーザー名/my-first-repo.git）が表示されます。この URL を次のステップで使います（ユーザー名 の部分は自分のGitHubユーザー名に置き換えてください）。

4.3 ローカルリポジトリとGitHubを接続する

ローカルの Git リポジトリは、作成しただけでは GitHub と無関係です。どの URL へ push すればよいかを教えるのが git remote add です。第2章で SSH を設定したので、ここでは SSH形式のURL（git@github.com:… で始まる形）を登録します。ユーザー名 の部分は自分のものに置き換えてください。

ターミナル 
# "origin" という名前でリモートURL（SSH形式）を登録する
git remote add origin git@github.com:ユーザー名/my-first-repo.git

# ブランチ名を "main" に統一する（必要な場合）
git branch -M main

# 登録できたか確認する（SSH形式のURLが表示されればOK）
git remote -v
flowchart LR A["ローカルリポジトリ"] -->|"git remote add origin URL"| B["origin という名前でURLを覚える"] B -->|"以降は 'origin' と書くだけ"| C["GitHub リポジトリ"]

origin とは？

リモートリポジトリに付ける「別名（エイリアス）」です。毎回長い URL を書く代わりに、慣例として origin という短い名前を使います。別の名前でも動作しますが、origin が世界標準の慣習です。

git branch -M main とは？

現在のブランチ名を main に変更（リネーム）するコマンドです。古い Git では既定のブランチ名が master になっていることがあり、GitHub の既定（main）と合わせるために使います。

4.4 git push — ローカルの変更をGitHubへ送る

接続が完了したら、いよいよ push です。初回は -u オプションを付けて実行します。

ターミナル（初回push） 
# 初回: -u で上流ブランチを記憶させる
git push -u origin main
実行結果の例 
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Writing objects: 100% (3/3), 230 bytes | 230.00 KiB/s, done.
To github.com:ユーザー名/my-first-repo.git
 * [new branch]      main -> main
Branch 'main' set up to track remote branch 'main' from 'origin'.

重要ポイント: -u オプションの意味

-u（--set-upstream）は「このローカルブランチの上流（送り先）は origin/main だ」と Git に記憶させる指示です。一度 -u を使えば、以降は git push だけで同じ場所へ送れるようになります。

実行結果の数値（オブジェクト数や転送量など）は環境によって多少違っても問題ありません。[new branch] main -> main のような行が出ていれば成功です。push 後に GitHub のリポジトリページをブラウザで開くと、ローカルで作成したファイルやコミット履歴が表示されます。README.md が存在する場合は、ページ下部にその内容が自動でレンダリングされて表示されます。

2回目以降のpush

-u を使った後は、新しいコミットを追加するたびに git push と打つだけで OK です。origin main の指定は不要になります。

4.5 git clone — GitHubのリポジトリを自分のPCに取得する

GitHub 上に存在するリポジトリを初めて手元に取得するときは git clone を使います。コミット履歴ごと丸ごとコピーされるため、取得直後から変更履歴を確認できます。

ターミナル 
# GitHubのURL（SSH形式）を指定してリポジトリを丸ごと取得する
git clone git@github.com:ユーザー名/my-first-repo.git

# 取得後はフォルダに入って作業する
cd my-first-repo

git clone を実行すると、リポジトリ名と同じフォルダが自動的に作成され、その中にファイルが展開されます。remote add の設定も自動で済んでいるため、すぐに git push や git pull を使えます。

「ZIP でダウンロード」との違い

GitHub の「Download ZIP」はファイルのみ取得でき、Git の履歴情報（コミット履歴）は含まれません。git clone は履歴ごと取得するため、あとから git log で歴史を確認したり、変更を push したりできます。

4.6 git pull — リモートの最新変更を取り込む

チームメンバーが変更を push した場合や、別のPCから push した場合など、リモートがローカルより新しくなることがあります。このとき git pull で差分を取り込みます。

ターミナル 
# リモート（origin）の main ブランチから最新を取り込む
git pull origin main

# 上流を設定済みなら短縮形でもOK
git pull

に新しいコミットが追加された"] --> B["git pull を実行"] B --> C["ローカルに差分が取り込まれる"] C --> D["ローカルとリモートが同じ状態に"]
コマンド 用途 使うタイミング
git clone <URL> 初回取得 まだローカルにない場合
git pull 差分取得 すでに clone 済みで最新を取り込みたい場合
git push 変更を送信 ローカルの新しいコミットをリモートへ送る

4.7 SSH接続とよくあるエラーの概要

GitHub はプッシュなどの書き込み操作に本人確認（認証）を求めます。このガイドでは、第2章で設定した SSH 方式でこの本人確認を行います。リモートURLが git@github.com:… の形（SSH形式）になっていれば、SSH方式で接続できます。

重要ポイント: この教材の標準手順はSSH方式

この教材の標準手順では、GitHubへの接続をすべて SSH 方式で行います。第2章でSSH鍵を作成・登録していれば、push / clone / pull のたびにパスワードを入力する必要はありません。

補足: HTTPS方式もあるが、この教材ではSSHで統一する

GitHub には SSH 方式のほかに、URL が https://github.com/… で始まる HTTPS 方式もあります。HTTPS方式では Personal Access Token（PAT）や Git Credential Manager（GCM）という仕組みで認証しますが、設定がやや複雑です。この教材では混乱を避けるため、GitHub接続はすべて SSH 方式で統一しますので、HTTPS・PAT・GCM の詳しい手順はここでは扱いません。

接続エラーが出たときの一般的な原因

  • Permission denied (publickey). と表示される — SSH鍵が GitHub に正しく登録されていない、またはリモートURLがSSH形式になっていない可能性があります。第7章のSSH接続エラーの手順で確認しましょう。
  • リモート URL が SSH 形式になっていない — git remote -v で登録済み URL を確認し、git@github.com:… の形になっているか確認しましょう。https://… になっている場合は、SSH形式に変更します（手順は第7章）。
  • remote が未設定 — git remote -v で何も表示されない場合は、git remote add origin git@github.com:ユーザー名/リポジトリ.git で登録してください。

エラーメッセージをそのままコピーして検索すると解決策が見つかることが多いので、慌てずに対処しましょう。

やってみよう 4-1: 第3章で作ったリポジトリをGitHubにpushする

第3章で作成してコミットしたローカルリポジトリを GitHub に push しましょう。

手順

  1. 1 GitHub にログインし、「New repository」で新しいリポジトリを作成する（Private を選んでOK）。README のチェックは外し、空の状態で作成する。
  2. 2 第3章のリポジトリフォルダをターミナルで開き、git remote add origin git@github.com:ユーザー名/リポジトリ名.git（SSH形式）を実行してリモートを登録する。ユーザー名 と リポジトリ名 は自分のものに置き換える。
  3. 3 git branch -M main を実行してブランチ名を統一する。
  4. 4 git push -u origin main を実行して push する。
  5. 5 ブラウザで GitHub のリポジトリページを開き、ファイルとコミット履歴が表示されることを確認する。README.md がある場合はページ下部に内容が表示されていることも確認する。

確認ポイント: GitHub のページに、第3章で作成したファイルが表示されていれば成功です！「Commits」リンクをクリックすると、コミット履歴も確認できます。

理解度確認クイズ

  1. Q1. git push -u origin main の -u は何を意味しますか？

    回答を表示

    -u（--set-upstream）は、ローカルの main ブランチの上流（push 先）を origin/main として記憶させるオプションです。一度使うと、次回から git push だけで送れるようになります。

  2. Q2. git clone と git pull の使い分けを説明してください。

    回答を表示

    git clone はリポジトリをローカルにまだ持っていないときに使い、履歴ごと丸ごと取得します。git pull はすでに clone 済みで、リモートの新しい変更（差分）だけを取り込むときに使います。

  3. Q3. push 時に Permission denied (publickey). のような接続エラーが出た場合、最初に確認すべきことは何ですか？

    回答を表示

    まず git remote -v でリモート URL が git@github.com:… の SSH 形式になっているかを確認します。次に、第2章で作成した SSH 公開鍵が GitHub に登録されているかを確認しましょう。詳しい手順は第7章のSSH接続エラーの対処を参照します。
