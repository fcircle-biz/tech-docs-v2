ブランチを使って作業する

本章では、Git の強力な機能である「ブランチ」を学びます。ブランチを使うと、本流（main）を壊さずに安全に新機能の開発や修正作業を進められます。チーム開発では、mainを安全に保つために作業用ブランチを作って進めることが多いです。コマンドの使い方から実際のブランチ作業の流れまで、一つひとつ体験しながら身につけましょう。

この章で学ぶこと

  • ブランチとは何か（作業を枝分かれさせる仕組み）を説明できる
  • mainブランチの役割（本流・公開版を置く場所）を理解する
  • git branch・git switch・git switch -c を使える
  • なぜブランチを使うのか（並行作業・main保護・レビュー単位）を説明できる
  • mainブランチから作業用ブランチを作成し、そのブランチで変更作業・pushができる

5.1 ブランチとは何か

ブランチ（branch） とは、「作業を枝分かれさせる仕組み」です。木の幹から枝が分かれるように、メインの作業ラインから別の作業ラインを作り出すことができます。枝（ブランチ）の上でどれだけ変更しても、幹（main）には影響しません。うまくいったら幹に合流させ、失敗したら枝を捨てればよいのです。

ブランチがない世界では、機能Aを開発しながら機能Bも触ろうとすると、変更が混ざり合ってしまいます。ブランチを使えば、機能Aの作業と機能Bの作業を完全に分離したまま並行して進められます。

ブランチのイメージ図

flowchart LR A["最初のコミット"] --> B["コミット2"] B --> C["コミット3
（mainの最新）"] B --> D["feature-loginを作成"] D --> E["ログイン機能を追加"] E --> F["バリデーションを修正"] C -->|"将来的に合流（merge）"| G["mainに取り込み"] F --> G

この図のように、feature-login ブランチは main から枝分かれして独立した流れで作業が進み、完成したら main に合流（マージ）します。合流するまで main は汚れません。

5.2 mainブランチとは何か

Git リポジトリを作ると、最初から main（または古い慣習では master）というブランチが存在します。mainブランチは「本流」です。公開されている・動作が確認されている・完成した状態のコードを置く場所というイメージです。

mainを守る

main は「完成・動作確認済み」の状態を保ちます。直接 main で作業するのは避け、ブランチを切って作業するのが基本です。

作業はブランチで

新機能・バグ修正・試験的な変更は作業用ブランチを作って進めるのが基本です。完成したら Pull Request を通じて main に合流させます。

重要ポイント: main は常に「動く状態」に保つ

チーム開発では main を「利用者にいつでも公開できる状態」「動くことを確認済みの状態」に保つルールを設けることが多いです。これを「いつでもデプロイできる状態」とも言います（デプロイ＝作ったものを利用者が使える場所に公開すること）。main に直接コミットを積むのではなく、ブランチで作業して Pull Request 経由で取り込む流れが標準的なスタイルです。

5.3 ブランチを確認する：git branch

現在どんなブランチがあるか、また今どのブランチにいるかを確認するコマンドが git branch です。

ターミナル 
# ローカルのブランチ一覧を表示
git branch
出力例 
  feature-login
* main

アスタリスク（*） が付いているブランチが、現在自分がいるブランチ（カレントブランチ）です。上の例では main にいることが分かります。

リモートブランチも含めて確認したいとき

git branch -a を使うと、ローカルとリモート両方のブランチ一覧が表示されます。リモートのブランチは remotes/origin/... という形で表示されます。

5.4 ブランチを切り替える：git switch

ブランチを切り替えるには git switch コマンドを使います。切り替えることで、そのブランチの最新の状態がフォルダに反映されます。

ターミナル 
# feature-login ブランチに切り替える
git switch feature-login

# main ブランチに戻る
git switch main

この教材では git switch を使います

git switch は Git 2.23（2019年リリース）以降で使える新しいコマンドです。この教材では git switch を使って説明します。とても古い Git（2.23より前）では使えないことがありますが、その場合は git checkout <ブランチ名> でも同じようにブランチを切り替えられます。古いドキュメントやネットの記事で checkout を見かけた場合も、ブランチの切り替えとしては同じ動きです。

切り替えた後に git branch を実行すると、アスタリスクが移動していることを確認できます。

出力例（feature-login に切り替えた後） 
* feature-login
  main

5.5 新しいブランチを作って切り替える：git switch -c

新しいブランチを作成して、同時にそのブランチに切り替えるには git switch -c <ブランチ名> を使います。-c は create（作成）の略です。「ブランチを作る」と「切り替える」を1コマンドでまとめて行うので、実際の開発ではこれを最も頻繁に使います。

ターミナル 
# feature-login という新しいブランチを作って切り替える
git switch -c feature-login

# 実行後は自動的に feature-login ブランチにいる
git branch  # → * feature-login と表示される
コマンド 動作 よく使う場面
git branch ブランチ一覧を確認 現在のブランチ状況を把握したいとき
git switch -c <名前> 作成＋切り替え 新しい作業を始めるとき（最頻出）
git switch <名前> 既存のブランチに切り替え 別のブランチに移動するとき

5.6 なぜブランチを使うのか

ブランチを使う理由は大きく3つあります。

複数人・複数作業を並行して進められる

Aさんがログイン機能、Bさんがプロフィール機能をそれぞれ別のブランチで同時に開発できます。お互いの変更が混ざることなく、独立して作業を進められます。

レビューの単位が明確になる

「この Pull Request はログイン機能の追加だけ」というように、変更の範囲を1つのブランチに絞ることで、レビューしやすい単位が生まれます。

mainを安全に保てる

作業中の未完成なコードを main に混入させずに済みます。万が一作業が失敗しても、main は影響を受けません。ブランチを削除すればなかったことにできます。

複数人・複数ブランチの並行作業イメージ

flowchart TD main["main（本流）"] --> A["feature-login
（Aさんがログイン機能を開発）"] main --> B["feature-profile
（Bさんがプロフィール機能を開発）"] main --> C["fix-typo
（誤字修正）"] A --> merge["mainに合流
（Pull Request経由）"] B --> merge C --> merge

5.7 作業用ブランチをリモートにpushする

ローカルで作成したブランチは、最初は自分のPCの中にしか存在しません。GitHub（リモート）に送って共有したり、Pull Request を作ったりするには、ブランチをpushする必要があります。

初回のpushには -u origin <ブランチ名> オプションが必要です。-u は上流（upstream）を設定するオプションで、「このローカルブランチは GitHub の同名ブランチと対応させる」と登録します。2回目以降は git push だけで送れます。

ターミナル 
# feature-login ブランチを初めて GitHub に送る
git push -u origin feature-login

# 2回目以降は短く書ける（-u が登録済みなので）
git push
出力例 
Enumerating objects: 4, done.
Counting objects: 100% (4/4), done.
To github.com:yourname/yourrepo.git
 * [new branch]      feature-login -> feature-login
Branch 'feature-login' set up to track remote branch 'feature-login' from 'origin'.

pushが完了すると GitHub 上にも同名のブランチが作られます。GitHub のリポジトリページを開くと「Compare ＆ pull request」ボタンが表示され、そのまま Pull Request を作れる状態になります（Pull Request は第6章で学びます）。

やってみよう 5-1: 作業ブランチを作ってpushする

第3・4章で作ったリポジトリを使い、作業ブランチを作成してコミット・pushするまでの流れを体験しましょう。

手順

  1. 1 main ブランチにいることを確認する（git branch）。
  2. 2 作業用ブランチを作って切り替える（git switch -c feature-test）。
  3. 3 適当なファイルを編集または新規作成する（例: test.txt を作り「テスト」と書く）。
  4. 4 ステージしてコミットする（git add . → git commit -m "テスト用ファイルを追加"）。
  5. 5 GitHub にpushする（git push -u origin feature-test）。
  6. 6 GitHub のリポジトリページを開き、feature-test ブランチが表示されていることを確認する。

コマンドまとめ

ターミナル 
git branch                          # 現在のブランチを確認
git switch -c feature-test          # ブランチ作成＋切り替え
# （ファイルを編集する）
git add .                           # 変更をステージ
git commit -m "テスト用ファイルを追加"  # コミット
git push -u origin feature-test     # GitHub にpush

ヒント: ブランチ名は「内容が分かる名前」にする

ブランチ名は後から見ても「何をしているブランチか」が一目で分かる名前にするのが良い習慣です。よく使われる命名パターンとして、プレフィックス（接頭語）/内容 の形でスラッシュ区切りにする書き方があります。

feature/add-profile

新機能の追加（feature/）— プロフィール追加

fix/typo-readme

バグ・誤字の修正（fix/）— READMEの誤字修正

docs/update-readme

ドキュメントの更新（docs/）

chore/update-deps

設定・依存関係の更新（chore/）

理解度確認クイズ

  1. Q1. ブランチを使う3つの理由を説明してください。

    回答を表示

    ①複数人・複数作業を並行して進められる（互いの変更が混ざらない）、②レビューの単位が明確になる（Pull Request 1つ＝1つの変更）、③mainブランチを安全に保てる（未完成コードが本流に混入しない）の3つです。

  2. Q2. 新しいブランチ feature-login を作って同時に切り替えるコマンドは何ですか？

    回答を表示

    git switch -c feature-login です。-c は create（作成）の略で、ブランチの作成と切り替えを1コマンドで行います。

  3. Q3. ローカルで作ったブランチを初めて GitHub にpushするコマンドを書いてください。

    回答を表示

    git push -u origin <ブランチ名> です（例: git push -u origin feature-login）。-u オプションで上流（upstream）を設定すると、2回目以降は git push だけで送れます。
