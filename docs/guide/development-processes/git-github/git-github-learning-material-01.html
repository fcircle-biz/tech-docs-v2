Git/GitHubで何ができるのか

最初の章では、コマンドを打つ前に「そもそも Git と GitHub は何のための道具なのか」を、専門用語をできるだけ増やさずにつかみます。ファイルを日付付きで管理する方法の限界から出発し、「変更を記録し・共有し・レビューして取り込む」という全体の流れを理解することがゴールです。

この章で学ぶこと

  • ファイル名に日付を付けて管理する方法の問題点を説明できる
  • Git は「変更履歴を記録する道具」、GitHub は「履歴を共有する場所」だと理解する
  • ローカルとリモートの違いを説明できる
  • コミット・リポジトリ・ブランチ・Pull Request のざっくりした意味をつかむ
  • 「記録 → 共有 → レビュー → 取り込み」という流れの全体像を理解する

1.1 ファイルを「日付付き」で管理する方法の限界

資料やプログラムを作るとき、こんなふうにファイルを保存した経験はないでしょうか。最初は便利に思えますが、続けていくと困った問題が次々に出てきます。

よくあるフォルダの中身 
企画書.docx
企画書_20240510.docx
企画書_修正版.docx
企画書_修正版2.docx
企画書_最終.docx
企画書_最終_田中修正.docx
企画書_本当に最終.docx        ← どれが本当の最新…？

この「日付やメモをファイル名に付ける」やり方には、次のような限界があります。

どれが最新か分からない

「最終」「本当に最終」が並び、結局どれを開けばよいか迷います。

誰が何を変えたか分からない

変更の理由や担当者がファイル名からは読み取れません。

過去にきれいに戻せない

「3日前の状態だけ戻したい」が手作業では困難です。

共同作業で上書き事故が起きる

2人が同時に直すと、片方の変更が消えてしまいます。

これらをまとめて解決してくれるのが Git と GitHub です。次の節から、それぞれが何をしてくれる道具なのかを見ていきます。

1.2 Git は「変更履歴を記録する道具」

Git（ギット） は、ファイルの変更の履歴を記録するための道具です。区切りのよいタイミングで「ここまでの状態」を記録しておくと、あとから「いつ・誰が・何を変えたか」をたどったり、過去の状態に戻したりできます。

この「ここまでの状態を1つ記録する」操作を コミット（commit） と呼びます。テレビゲームのセーブポイントをイメージすると分かりやすいでしょう。こまめにセーブしておけば、失敗しても直前のセーブからやり直せます。

履歴が残る

いつ・何を変えたかが、コミットの一覧として残ります。

過去に戻れる

好きなコミットの時点の状態を取り出せます。

ファイルは1つ

「最終版2」を増やさず、1つのファイルで管理できます。

Git を使うと、フォルダの中身は「企画書.docx」1つのまま、その変更の歴史だけが裏側に積み重なっていくイメージになります。図にすると次のようになります。

flowchart LR A["最初の作成
コミット1"] --> B["見出しを追加
コミット2"] B --> C["図を差し替え
コミット3"] C --> D["誤字を修正
コミット4（最新）"]

用語: バージョン管理システム

Git のように変更履歴を管理する道具を、一般にバージョン管理システム（VCS）と呼びます。Git は今もっとも広く使われているバージョン管理システムの一つで、多くの開発現場で使われています。

1.3 GitHub は「Git の履歴を共有する場所」

Git は自分のPCの中だけでも使えますが、それだけでは「自分のPCが壊れたら終わり」「他の人と共有できない」という弱点があります。そこで登場するのが GitHub（ギットハブ） です。GitHub は、Git の履歴をインターネット上に置いて、保存・共有・共同作業できるようにするサービスです。

自分のPCの中にある作業場所を ローカル、GitHub のようにインターネット上にある場所を リモート と呼びます。この2つの言葉はこの先ずっと出てくるので、ここで押さえておきましょう。

呼び方 場所 具体例
ローカル 自分のPCの中 手元で編集している作業フォルダ
リモート インターネット上 GitHub 上に置いたリポジトリ

ローカルとリモートの関係を図にすると、次のようになります。手元で記録した履歴を GitHub にアップロード（push）し、他の人はそれをダウンロード（clone / pull）して受け取ります。

flowchart LR subgraph ローカル["ローカル（自分のPC）"] A["作業フォルダ"] end subgraph リモート["リモート（GitHub）"] B["共有リポジトリ"] end A -->|"push: 送信"| B B -->|"clone / pull: 取得"| A

Git（ローカル）

自分のPCで変更履歴を記録する「道具」。GitHub が無くても単独で使えます。

GitHub（リモート）

Git の履歴を置いて共有する「場所」。チームでの共同作業やバックアップに使います。

重要ポイント: Git と GitHub は別物

Git は「道具（ソフト）」、GitHub は「場所（サービス）」です。名前が似ているので混同しがちですが、役割はまったく別。Git で記録した履歴を、GitHub という場所に置いて共有する——この関係をはっきりさせておくことが、最初のいちばん大事なポイントです。

  • Git：自分のPC上で変更履歴を管理する仕組み
  • GitHub：Gitの履歴をインターネット上で共有・共同作業するサービス

1.4 まず押さえる4つの言葉（ざっくり）

この先の章で何度も登場する基本用語を、いまはざっくりしたイメージだけつかんでおきましょう。正確な操作は後の章で1つずつ学ぶので、ここで完璧に覚える必要はありません。この章に出てくる用語は、いまは暗記しなくて大丈夫です。実際に手を動かす第3章以降で自然に身につきます。

リポジトリ（repository）

変更履歴を含めたプロジェクト一式を入れておく箱。「リポジトリを作る」＝「Git で管理する箱を用意する」という意味です。略して「リポジトリ」「レポ」とも呼びます。1つのプロジェクト専用のバインダー（関連する資料をまるごと1冊に綴じておくもの）をイメージすると分かりやすいでしょう。

コミット（commit）

「ここまでの変更」を1つ記録する操作、またはその記録1件。ゲームのセーブポイントにあたります。1コミットごとに短いメモ（メッセージ）を付けます。スマートフォンで写真を撮って残しておくように、その時点の状態を1枚パシャリと保存するイメージです。

ブランチ（branch）

作業を枝分かれさせる仕組み。本流（main）を残したまま、別の枝で安全に試せます。うまくいったら本流に合流させます。第5章でくわしく学びます。レポートの下書きを「別名でコピーして試し書き」するイメージで、元の本流に影響させずに新しい変更を試せます。

Pull Request（プルリクエスト）

「この変更を取り込んでください」とお願いし、レビューしてもらう仕組み。GitHub 上で行います。略して「PR」とも呼びます。第6章で体験します。「この変更を入れていいですか？」と先輩に見てもらって相談するようなイメージで、チームで内容を確認してから本流に取り込むことができます。

いまは「ぼんやり」で大丈夫

用語は、実際に手を動かすと一気に腑に落ちます。ここでは「そういう名前のものがあるんだな」くらいの理解で先に進みましょう。第3章以降で実際に使いながら身につけます。

1.5 全体の流れ「記録 → 共有 → レビュー → 取り込み」

最後に、このガイド全体で学ぶ大きな流れをつかんでおきましょう。Git/GitHub を使った開発は、おおまかに次の4ステップを繰り返します。各ステップが、このあとのどの章に対応するかも示します。

flowchart TD A["記録する
commit（第3章）"] --> B["共有する
push（第4章）"] B --> C["レビューする
Pull Request（第6章）"] C --> D["取り込む
merge（第6章）"] D -->|"次の作業へ"| A
  • 記録する — 手元で変更し、コミットとして履歴に残す（第3章）。安全に作業するためブランチを切ることも多い（第5章）。
  • 共有する — 記録した履歴を GitHub に push してアップロードする（第4章）。
  • レビューする — Pull Request を作り、内容を確認してもらう（第6章）。
  • 取り込む — 問題なければ本流（main）に取り込む＝マージする（第6章）。困ったときの対処は第7章。

いきなり全部を覚える必要はありません。次の第2章では、まずこの流れを実際に体験するための環境（Git・VS Code・GitHub アカウント）を整えていきます。

やってみよう 1-1: 自分の「最終版あるある」を振り返る

インストール前のこの段階では、まず「自分は今までどうファイルを管理してきたか」を振り返り、Git で何が解決できそうかをイメージしてみましょう。

手順

  1. 1 これまでに作った「◯◯_最終」「◯◯_修正版」のようなファイルを思い出して、2〜3個書き出してみる。
  2. 2 それぞれで「困ったこと」（最新が分からない・戻せない 等）を1つずつメモする。
  3. 3 その困りごとが、1.2〜1.3 の Git/GitHub のどの機能で解決できそうか、対応づけてみる。

理解度確認クイズ

  1. Q1. Git と GitHub の役割の違いを、ひとことで説明してください。

    回答を表示

    Git は変更履歴を記録するための「道具（ソフト）」で、自分のPCの中（ローカル）で動きます。GitHub はその履歴をインターネット上に置いて共有・共同作業できるようにする「場所（サービス）」です。

  2. Q2. 「ローカル」と「リモート」は、それぞれどこを指しますか？

    回答を表示

    ローカルは自分のPCの中にある作業場所、リモートはインターネット上（GitHub など）にある場所を指します。手元で記録した履歴を push でリモートへ送り、clone / pull でリモートから取得します。

  3. Q3. Git/GitHub を使った開発の大きな流れ「記録 → 共有 → ? → 取り込み」の「?」に入る言葉は何ですか？

    回答を表示

    「レビュー」です。記録（commit）して共有（push）した変更を、Pull Request でレビューしてもらい、問題なければ本流に取り込み（merge）ます。
