編集機能を作る

編集機能は「まず既存データを取得してフォームに表示し、送信された値で更新する」という2段階の流れです。第7章で学んだ登録（空のフォームを表示→送信→保存）との違いに注目しながら、GET で既存データを取得してフォームに表示し、POST で更新処理を行う一連の実装を学びます。

この章で学ぶこと

  • 既存データをフォームに表示する方法を理解する
  • 編集フォームから更新処理を行う流れを理解する
  • save メソッドで更新できることを理解する

8.1 登録と編集の違い

編集機能を作る前に、まず登録機能との違いを確認しましょう。第7章で作った登録機能では、「空のフォームを表示→ユーザーが入力→送信→新規保存」という流れでした。編集機能では最初のステップが違います。「既存のデータをデータベースから取得してフォームに表示する」というステップが加わるのです。

登録機能（第7章）

  1. 空のフォームを表示する
  2. ユーザーが入力して送信
  3. 新しいデータとして保存

編集機能（この章）

  1. 既存データを取得してフォームに表示する
  2. ユーザーが修正して送信
  3. 既存のデータを更新する

重要なのは、編集機能では「どのユーザーを編集するか」を特定するために ID が必要になる点です。URL に /users/1/edit のように ID を含めることで、「ID=1のユーザーを編集する」と Spring MVC に伝えます。

用語: パスパラメータ（Path Variable）

パスパラメータとは、URL の一部として埋め込む値のことです。/users/1/edit の「1」がパスパラメータです。@PathVariable アノテーションを使うと、この URL 内の値を Controller のメソッドで受け取れます。「どのデータを操作するか」を URL で指定する際によく使われます。

8.2 編集機能の全体の流れ

編集機能は GET（フォーム表示） と POST（更新処理） の2段階で実装します。まず GET リクエストで既存データを取得してフォームに表示し、ユーザーが修正して送信（POST）すると更新処理が実行されます。

この図のポイントは2段階の流れです。① GET でフォームを「既存データ入り」で表示する → ② ユーザーが修正して POST で送信する → ③ Controller が更新処理を呼び出す → ④ 一覧ページへリダイレクト。この「GET→POST→redirect」の流れは、第7章の登録機能と同じパターンです。

8.3 編集フォーム表示の Controller を作る

まず、編集フォームを表示するための GET メソッドを Controller に追加します。このメソッドのポイントは、フォームを表示する前に「編集対象のユーザーをデータベースから取得する」ことです。取得した情報を UserForm に詰め替えて Model に渡すと、Thymeleaf がフォームの初期値として表示してくれます。

UserController.java（編集フォーム表示） 
// GET /users/{id}/edit にアクセスされたときに呼ばれる
@GetMapping("/users/{id}/edit")
public String editForm(@PathVariable Long id, Model model) {
    // ① 既存ユーザーをIDで取得
    User user = userService.findById(id);

    // ② 取得したデータを UserForm に詰め替える
    UserForm userForm = new UserForm();
    userForm.setName(user.getName());
    userForm.setEmail(user.getEmail());

    // ③ フォームデータとIDをビューへ渡す
    model.addAttribute("userForm", userForm);
    model.addAttribute("userId", id);      // 送信先URLのIDに使う
    return "users/edit";                   // templates/users/edit.html を表示
}

コードのポイント

  • @PathVariable Long id: URL の {id} 部分を Java の変数として受け取るアノテーション。
  • findById(id): IDを使って既存データを取得するメソッド（第6章で実装済み）。
  • UserForm への詰め替え: 取得した User の値を UserForm にセットすることで、フォームに初期値が入った状態で表示される。
  • model.addAttribute("userId", id): フォームの送信先URL（/users/{id}/edit）を構築するために ID を別途渡す。

登録機能（第7章）では new UserForm() に何もセットせず空のままモデルに渡しました。編集機能では user.getName() などで取得した値をセットしてから渡す——これが登録と編集の大きな違いです。

8.4 編集フォームの HTML を作る

次に、templates/users/edit.html を作成します。登録フォーム（第7章）との違いは、フォームの 送信先 URL に ID が含まれる点と、th:field が Model から渡ってきた既存値を初期値として自動的に表示してくれる点です。

templates/users/edit.html 
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
    <meta charset="UTF-8">
    <title>ユーザー編集</title>
</head>
<body>
    <h1>ユーザー編集</h1>
    <!-- th:action でIDを含む送信先URLを生成 -->
    <form th:action="@{/users/{id}/edit(id=${userId})}"
          th:object="${userForm}" method="post">
        <div>
            <label>名前：</label>
            <!-- th:field で既存値を初期表示 &amp; POST で受け取る -->
            <input type="text" th:field="*{name}">
        </div>
        <div>
            <label>メール：</label>
            <input type="email" th:field="*{email}">
        </div>
        <button type="submit">更新</button>
    </form>
    <a th:href="@{/users}">一覧へ戻る</a>
</body>
</html>

Thymeleaf: @{/users/{id}/edit(id=${userId})} の読み方

@{...} はリンク式（URL生成）です。{id} の部分に括弧内の (id=${userId}) で渡した値が埋め込まれます。たとえば userId が 1 なら、送信先は /users/1/edit になります。th:field は Model の userForm から値を読み取り、input の value 属性に自動でセットしてくれます。

8.5 更新処理の Controller を作る

フォームから更新ボタンを押すと、POST /users/{id}/edit が送信されます。このリクエストを受け取る update メソッドを Controller に追加します。

UserController.java（更新処理） 
// POST /users/{id}/edit にフォーム送信されたときに呼ばれる
@PostMapping("/users/{id}/edit")
public String update(
        @PathVariable Long id,           // URLからIDを受け取る
        @ModelAttribute UserForm userForm // フォームの入力値を受け取る
) {
    // Serviceの更新メソッドを呼び出す
    userService.update(id, userForm.getName(), userForm.getEmail());

    // 更新後は一覧ページへリダイレクト
    return "redirect:/users";
}

コードのポイント

  • @PostMapping("/users/{id}/edit"): POST リクエストかつ URL に {id} を含むときに反応する。
  • @PathVariable Long id: URL の {id} を受け取る（どのユーザーを更新するか）。
  • @ModelAttribute UserForm: フォームの入力値（name・email）をまとめて受け取る（第7章と同じ）。
  • redirect:/users: 更新後は一覧へ戻す（第5章の PRG パターンと同じ）。

8.6 Service の更新処理と save メソッドの仕組み

Controller から呼ばれる Service の update メソッドを見てみましょう。ここで重要なポイントが1つあります。Repository の save メソッドは、登録にも更新にも使えるのです。

UserService.java（更新処理） 
// IDと新しい名前・メールを受け取って更新する
public User update(Long id, String name, String email) {
    // IDを指定してUserオブジェクトを作成
    User user = new User(id, name, email);
    // save に ID が入っていると「更新」として処理される
    return userRepository.save(user);
}

save メソッドは登録にも更新にも使える

Spring Data JDBC の save メソッドは内部で「ID の有無」を確認して動作を切り替えます。

ID なし（null）→ INSERT（登録）

新しいレコードを追加する。IDはDBが自動生成。

ID あり → UPDATE（更新）

そのIDのレコードを上書き更新する。

第7章の登録処理では new User(null, name, email) のように ID を null にして save しました。今回の編集処理では new User(id, name, email) のように既存の ID を入れて save します——これだけで「更新」として動作するのです。メソッドが1つで登録・更新両方できるため、コードをシンプルに保てます。

なぜ更新にも save が使えるの？

Spring Data JDBC の CrudRepository（または ListCrudRepository）が提供する save は、渡されたオブジェクトが「新規か既存か」を ID フィールドで判断します。ID が null なら INSERT、ID が入っていれば UPDATE の SQL を自動で発行します。この仕組みのおかげで、Repository に update メソッドを別途作らなくて済みます。

やってみよう 8-1: 編集機能を動かしてみる

この章で学んだコードをプロジェクトに追加して、実際に編集機能を動かしてみましょう。

手順

  1. 1 UserController に editForm メソッドと update メソッドを追加する。
  2. 2 UserService に update メソッドを追加する。
  3. 3 templates/users/edit.html を作成する。
  4. 4 一覧画面（第5章）の各行に「編集」リンクを追加する: th:href="@{/users/{id}/edit(id=${user.id})}"
  5. 5 アプリを起動して一覧から「編集」を押し、フォームに既存値が入っているか確認する。値を変更して「更新」を押し、一覧に反映されているか確認する。

理解度確認クイズ

  1. Q1. 編集フォームを表示する前に、なぜ既存データを取得しますか？

    回答を表示

    フォームに現在の値（名前・メールアドレスなど）を初期値として表示するためです。ユーザーが「どの値を変えるか」を確認しながら編集できます。空のフォームでは「現在の値が何か」が分からないため、既存データを取得して UserForm に詰め替えてから画面に渡します。

  2. Q2. 更新処理でIDが必要な理由は何ですか？

    回答を表示

    「どのユーザーを更新するか」を特定するためです。データベースには複数のユーザーが存在するため、IDがなければどのレコードを上書きすればよいか判断できません。また、Repository の save メソッドが「ID がある＝更新」と判断するためにも ID が必要です。

  3. Q3. save メソッドは登録以外に何に使えますか？

    回答を表示

    更新（UPDATE）にも使えます。save に渡すオブジェクトの ID が null なら INSERT（登録）、ID に値が入っていれば UPDATE（更新）として Spring Data JDBC が自動で処理を切り替えます。そのため、Repository に別途 update メソッドを定義しなくても更新処理が実現できます。
