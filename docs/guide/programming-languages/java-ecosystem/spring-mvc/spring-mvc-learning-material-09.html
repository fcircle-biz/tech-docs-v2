削除機能を作る

この章では、ユーザー管理アプリに「削除機能」を追加します。削除はデータを変える操作なので、リンクを踏むだけで実行されてしまう GET ではなく、必ずフォームを経由した POST で行う必要があります。その理由をしっかり理解しながら、Controller・Service・Repository がどのように連携して削除処理を実現するかを学びましょう。

この章で学ぶこと

  • 削除処理の全体的な流れ（一覧画面 → Controller → Service → Repository → リダイレクト）を理解する
  • 削除処理を GET ではなく POST で行う理由を説明できる
  • deleteById を使った Repository の削除処理を実装できる

9.1 削除処理の全体の流れ

削除機能も、これまで作ってきた一覧・詳細・登録・編集と同じく、Controller → Service → Repository の3層を通って処理されます。ただし削除の場合は「取得する」のではなく「消す」という操作なので、対象が何かを特定するためにユーザーの ID を受け取る必要があります。

処理の流れをまとめると次のとおりです。

flowchart TD A["一覧画面
削除ボタン（フォームPOST）"] -->|"/users/{id}/delete へPOST"| B["Controller
@PostMapping"] B --> C["Service
delete（Long id）"] C --> D["Repository
deleteById（id）"] D --> E[("Database
該当レコード削除")] B -->|"redirect:/users"| F["一覧画面へ戻る"]

ポイントは2つあります。① 削除する対象のユーザーを識別するために、URL にその ID を含めること。② フォームの送信先を POST にして、誤って削除されないようにすること（この理由は 9.2 でくわしく説明します）。削除が完了したら redirect:/users で一覧画面に戻ります。

9.2 なぜ削除は POST で行うのか

Web アプリのリクエスト方式には GET と POST の2種類があります。「どちらを使っても動くのでは？」と思うかもしれませんが、削除を GET で実装するのは非常に危険です。その理由を見てみましょう。

GET — 「見る」ためのリクエスト

ページを表示・取得するときに使います。ブラウザのアドレスバーにURLを入力したり、リンクをクリックしたりするのはすべて GET です。

  • 一覧表示
  • 詳細表示
  • フォームの表示

POST — 「変える」ためのリクエスト

データを変更・追加・削除するときに使います。フォームの送信ボタンを押したときに発火します。

  • データの登録
  • データの更新
  • データの削除

重要ポイント：GETで削除する危険性

もし削除処理を GET で実装すると、「/users/5/delete」のような URL にアクセスしただけで消えてしまいます。リンクをうっかりクリックするだけで、あるいは検索エンジンのクローラーがURLを巡回するだけで、大切なデータが消えてしまう可能性があります。データを変更する処理は、必ずフォームを通じた POST で行うのが Web アプリの鉄則です。

注意：GET は「副作用なし」が前提

GET はブラウザの「戻る」ボタン、ブックマーク、リンクのプリフェッチなど、ユーザーが意図しないタイミングで実行される場合があります。そのため GET には「何度実行しても状態が変わらない」という前提があります。削除・登録・更新のような「状態を変える」操作は POST を使い、GET には絶対に割り当てないようにしましょう。

9.3 一覧画面に削除ボタンを追加する

削除ボタンは HTML の <form> タグを使って実装します。<a> タグ（リンク）で実装すると GET になってしまうため、フォームを使って POST でリクエストを送ることが必須です。

Thymeleaf の th:action を使って URL を動的に生成します。@{/users/{id}/delete(id=${user.id})} と書くことで、それぞれのユーザーの ID が URL に埋め込まれます。

users/list.html（一覧画面の削除ボタン部分） 
<!-- 削除ボタン：formでPOSTリクエストを送る -->
<form th:action="@{/users/{id}/delete(id=${user.id})}" method="post">
    <button type="submit"
            class="btn btn-danger"
            onclick="return confirm('本当に削除しますか？')">
        削除
    </button>
</form>

ポイント：th:action でURLにIDを埋め込む

@{/users/{id}/delete(id=${user.id})} という書き方は Thymeleaf の URL 式です。{id} がプレースホルダーになっており、(id=${user.id}) でそこにユーザーの ID が入ります。例えばユーザーの ID が 5 なら、生成される URL は /users/5/delete になります。

また onclick="return confirm('本当に削除しますか？')" を付けると、ボタンを押したときに確認ダイアログが表示されます。「OK」を押した場合だけフォームが送信されるので、誤削除をさらに防ぐことができます。

9.4 Controller に削除メソッドを追加する

Controller には、POSTリクエストを受け取るメソッドを追加します。@PostMapping を使い、URL パターンを /users/{id}/delete に設定します。{id} の部分は @PathVariable アノテーションで受け取ります。

UserController.java（削除メソッド） 
// POST /users/{id}/delete を受け取る
@PostMapping("/users/{id}/delete")
public String delete(@PathVariable Long id) {
    // Serviceに削除処理を依頼する
    userService.delete(id);
    // 削除後は一覧画面へリダイレクト
    return "redirect:/users";
}

用語：@PathVariable

@PathVariable は URL の一部（{id}）をメソッドの引数として受け取るアノテーションです。例えば /users/5/delete へのリクエストが来たとき、id には 5L（Long 型の 5）が自動でセットされます。第8章の編集機能でも使いましたね。

削除処理が終わったら return "redirect:/users" で一覧画面へリダイレクトします。登録・編集と同様に、削除後も必ずリダイレクトすることで、ブラウザの「更新」ボタンによる二重削除を防ぐことができます。

9.5 Service・Repository の削除処理

Controller から呼ばれる Service の削除メソッドは非常にシンプルです。受け取った ID をそのまま Repository に渡して、削除を依頼するだけです。

UserService.java（削除処理） 
// IDを受け取り、Repositoryの削除処理を呼び出す
public void delete(Long id) {
    // Spring Data JDBC の deleteById でレコードを削除する
    userRepository.deleteById(id);
}

deleteById(id) は Spring Data JDBC（または Spring Data JPA）が自動で用意してくれるメソッドです。引数に削除したいレコードの ID を渡すだけで、対応するデータベースのレコードを削除してくれます。自分でSQL文を書く必要はありません。

重要ポイント：deleteById の内部動作

deleteById(id) は内部で DELETE FROM users WHERE id = ? というSQL文を実行しています。Spring Data JDBC / JPA がこの SQL を自動生成してくれるため、Repository インターフェースに deleteById を自分で定義する必要はありません。CrudRepository や JpaRepository を継承するだけで使えます。

削除処理は戻り値が不要なので、void で宣言している点にも注目してください。登録・更新では保存した結果のエンティティが返ってきましたが、削除は「消す」だけなので何も返す必要はありません。

やってみよう 9-1: 削除機能を実装する

ここまで学んだ内容をもとに、実際にユーザー管理アプリへ削除機能を追加してみましょう。

手順

  1. 1 list.html の各ユーザー行に、<form> タグを使った削除ボタンを追加する。th:action で URL に ID を埋め込み、method="post" を忘れずに指定する。
  2. 2 UserService に public void delete(Long id) メソッドを追加し、userRepository.deleteById(id) を呼び出す。
  3. 3 UserController に @PostMapping("/users/{id}/delete") メソッドを追加し、@PathVariable Long id で ID を受け取り、Service を呼んだあと redirect:/users を返す。
  4. 4 アプリを起動して、一覧画面からユーザーを削除できることを確認する。削除後に一覧へリダイレクトされ、対象ユーザーが消えていることを確かめる。

method="post" を付け忘れると GET になってしまいます。フォームを送信するとエラーになるか、意図しない動作になるので注意してください。

理解度確認クイズ

  1. Q1. 削除処理でIDが必要な理由は何ですか？

    回答を表示

    データベースには複数のユーザーが登録されているため、どのユーザーを削除するかを特定する必要があるからです。ID はレコードを一意に識別するキーなので、deleteById(id) に渡すことで正確に対象のレコードだけを削除できます。

  2. Q2. deleteByIdは何をするメソッドですか？

    回答を表示

    Spring Data JDBC / JPA が自動で用意してくれるメソッドで、引数に渡した ID に対応するデータベースのレコードを削除します。内部的には DELETE FROM テーブル名 WHERE id = ? というSQL を実行しています。自分でSQLを書く必要はありません。

  3. Q3. 削除処理をGETではなくPOSTで行う理由は何ですか？

    回答を表示

    GET はリンクをクリックするだけ、あるいは検索エンジンがURLを巡回するだけで実行されてしまうため、意図しないタイミングでデータが削除される危険があるからです。POST はフォームの送信ボタンを押したときにだけ実行されるので、誤削除のリスクを大幅に下げられます。データを変更する処理は必ず POST で行うのが原則です。
