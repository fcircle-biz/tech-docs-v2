Spring MVCとは

Spring MVC は、Spring で Web アプリケーションを作るための仕組みです。この章では、そもそも Web アプリケーションとは何か、ブラウザとサーバーはどうやり取りするのか、そして Spring MVC が「ブラウザからのアクセスを受け取り、Java の処理結果を HTML 画面として返す」流れの中で Model・View・Controller がどんな役割を担うのか、という全体像をつかみます。

この章で学ぶこと

  • Web アプリケーションの基本（ブラウザ・サーバー・リクエスト・レスポンス）を理解する
  • Spring MVC が何のための仕組みなのかを説明できる
  • ブラウザ→Controller→Service→Repository→DB→View→ブラウザという処理の流れをつかむ
  • MVC の Model・View・Controller の役割を説明できる

1.1 Web アプリケーションとは

Web アプリケーションとは、ブラウザ（Chrome や Edge など）を通して使うアプリのことです。ショッピングサイト・SNS・予約システムなど、あなたが普段ブラウザで使っているサービスの多くが Web アプリケーションです。これまでの教材で作ってきたプログラムは「自分のパソコンの中だけで動く」ものでしたが、Web アプリケーションはブラウザからアクセスして使う点が大きく違います。

Web アプリケーションでは、登場人物は大きく2つです。利用者が操作するブラウザ（クライアント）と、処理を行うサーバーです。ブラウザが「このページを見せて」とお願いし、サーバーがそれに応えて画面（HTML）を返す、というやり取りで成り立っています。このお願いをリクエスト（request）、その返事をレスポンス（response）と呼びます。

flowchart LR A["ブラウザ
(利用者)"] -->|リクエスト：このページを見せて| B["サーバー
(処理する側)"] B -->|レスポンス：HTML画面を返す| A

このリクエストとレスポンスのやり取りには、HTTPという「ブラウザとサーバーが会話するための共通ルール（約束ごと）」が使われます。難しく考える必要はありません。「ブラウザがお願い（リクエスト）を送ると、サーバーが返事（レスポンス）を返す」——この往復が Web アプリケーションの基本だと覚えておけば十分です。

用語: リクエストとレスポンス

リクエストは「ブラウザ→サーバー」へのお願い（例: 「/users のページを見せて」）。レスポンスは「サーバー→ブラウザ」への返事（例: ユーザー一覧の HTML）。本教材では、このリクエストを受け取り、レスポンス（HTML 画面）を返す部分を Spring MVC で作っていきます。リクエストには「GET」「POST」といった種類がありますが、それは第3章・第7章でくわしく学びます。

1.2 Spring MVC とは何か

Spring MVC は、Spring で Web アプリケーションを作るための仕組みです。初心者向けにひとことで言うと、「ブラウザからのアクセス（リクエスト）を受け取り、Java の処理結果を HTML 画面（レスポンス）として返す」ための仕組みだと考えてください。

Spring MVC の「MVC」とは、アプリの中身を Model（モデル）・View（ビュー）・Controller（コントローラー）という3つの役割に分けて作る考え方の名前です。役割をきちんと分けることで、「どこで何をしているか」が分かりやすくなり、修正や追加がしやすいアプリになります。それぞれの役割は 1.4 でくわしく見ていきます。

受け取る

ブラウザからのリクエスト（アクセス）を受け取ります。担当するのは Controller です。

処理する

Service が業務処理を行い、Repository がデータベースを読み書きします。

返す

View が結果を HTML 画面にして、ブラウザへ返します。

用語: 「Spring MVC」と「Spring Boot」

Spring MVC は Web アプリケーションを作るための仕組み（部品の集まり）です。一方 Spring Boot は、その Spring を「すぐ動かせる状態」で簡単に始められるようにしてくれる土台です。本教材では Spring Boot を使って Spring MVC のアプリを作っていきます。「Spring Boot で土台を用意し、その上で Spring MVC の機能を使う」とイメージしてください。

1.3 処理の流れ（全体像）

Spring MVC のアプリは、ブラウザからのリクエストを受け取ってから画面を返すまで、決まった流れで処理が進みます。この流れを最初に頭に入れておくと、第2章以降で「いま自分はどの部分を作っているのか」が分かりやすくなります。基本の流れは次のとおりです。

flowchart TD A["ブラウザ"] -->|リクエスト| B["Controller
(リクエストの受付係)"] B --> C["Service
(業務処理を担当)"] C --> D["Repository
(DBアクセスを担当)"] D --> E[("Database")] E -.データ.-> C C -.結果.-> B B -->|Modelにデータを入れる| F["View
(画面・HTML)"] F -->|HTMLを返す| A

順番に追ってみましょう。① ブラウザがリクエストを送る → ② Controller が受け取る → ③ Service に処理を依頼する → ④ Repository がデータベースを読み書きする → ⑤ 取得した結果が Controller まで戻る → ⑥ Controller が結果を Model に入れる → ⑦ View がその Model のデータを使って HTML を組み立てる → ⑧ HTML がブラウザに返って画面が表示される。この一連の流れを、本教材では機能（一覧・詳細・登録・編集・削除）ごとに作っていきます。

重要ポイント

Controller・Service・Repository は、前教材「Spring 基礎」「Spring Data JDBC」で学んだものと同じ役割分担です。本教材では、その入口に「ブラウザからのリクエストを受け取る Controller」と「画面を表示する View」が加わるだけ、と考えると理解がスムーズです。

1.4 MVC の役割 — Model・View・Controller

MVC は、アプリを Model・View・Controller の3つの役割に分ける考え方です。それぞれが「自分の担当だけに集中する」ことで、分かりやすく整理されたアプリになります。役割をまとめると次のとおりです。

要素 役割
Model（モデル） 画面に渡すデータ。Controller が用意し、View が表示に使う「データの箱」。
View（ビュー） 画面表示を担当。HTML を組み立ててブラウザに見せる。本教材では Thymeleaf を使う。
Controller（コントローラー） リクエストを受け取り、処理の流れを制御する。司令塔のような役割。

レストランにたとえると分かりやすいかもしれません。Controller は注文を受ける「ホール係」、Service や Repository は料理を作る「厨房」、Model は料理をのせる「お皿」、View はお客さんに見せる「盛り付け」です。ホール係（Controller）が注文を受け、厨房（Service・Repository）が料理を作り、お皿（Model）にのせて、きれいに盛り付けて（View）お客さん（ブラウザ）に出す——この流れがそのまま Spring MVC の処理に対応します。

用語: View と Thymeleaf

View は「画面表示の担当」という役割の名前です。その View を実際に作るための道具が Thymeleaf（タイムリーフ）で、HTML の中に Java 側のデータを埋め込んで表示できるようにしてくれます。Thymeleaf の使い方は第4章から少しずつ学ぶので、ここでは「View ＝ 画面、Thymeleaf ＝ その画面を作る道具」とだけ押さえておけば大丈夫です。

やってみよう 1-1: 身近な Web アプリの流れを考える

コードを書く前に、まず「Web アプリの中で何が起きているか」を自分の言葉でイメージしてみましょう。全体像を説明できると、以降の章がぐっと理解しやすくなります。

手順

  1. 1 普段ブラウザで使っているサービスを1つ選び、「どんなリクエストを送って、どんな画面（レスポンス）が返ってくるか」を3つ書き出す（例: 検索ボタンを押す→検索結果一覧が返る）。
  2. 2 1.3 の図を見ながら、その処理を「ブラウザ→Controller→Service→Repository→DB→View→ブラウザ」の流れに当てはめてみる。
  3. 3 Model・View・Controller がそれぞれ「お皿・盛り付け・ホール係」のどれに当たるか、レストランのたとえで説明してみる。

理解度確認クイズ

  1. Q1. Spring MVC は何のための仕組みですか？

    回答を表示

    Spring で Web アプリケーションを作るための仕組みです。ブラウザからのアクセス（リクエスト）を受け取り、Java の処理結果を HTML 画面（レスポンス）として返します。

  2. Q2. Controller は何を担当しますか？

    回答を表示

    ブラウザからのリクエストを受け取り、処理の流れを制御する「司令塔」の役割です。Service に処理を依頼し、その結果を Model に入れて View へ渡します。

  3. Q3. View は何を担当しますか？

    回答を表示

    画面表示を担当します。HTML を組み立ててブラウザに見せる役割で、本教材では Thymeleaf を使って作ります。

  4. Q4. Model は何を入れるものですか？

    回答を表示

    画面（View）に渡すデータを入れる「箱」です。Controller がデータを Model に入れ、View がそのデータを使って画面を表示します。
