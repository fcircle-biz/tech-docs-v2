同じ名前で違う動き - ポリモーフィズム

第10章で学んだ継承・オーバーライドをさらに発展させ、「同じメソッド名でも呼び出すオブジェクトによって違う動きをする」ポリモーフィズム（多態性）を学びます。抽象クラスとインターフェースも合わせて理解することで、オブジェクト指向設計の本質に迫ります。

この章で学ぶこと

11.1 ポリモーフィズム（多態性）とは

ポリモーフィズム（Polymorphism：多態性）とは、「同じ命令（メソッド呼び出し）を出しても、受け取るオブジェクトによって異なる動きをする」という仕組みです。ポリモーフィズムは「多くの形」を意味するギリシャ語が語源です。

たとえば、動物たちに「鳴け！」と命令したとします。犬は「ワン！」、猫は「ニャー！」、鳥は「チュン！」と反応します。命令は同じ「鳴け」でも、動物の種類によって結果が違います。これがポリモーフィズムです。Javaでは、同じメソッド名（例: sound()）を呼び出しても、オブジェクトの種類によって実行される処理が変わります。

同じ命令、違う動き

a.sound() という同じ呼び出しでも、a が Dog なら「ワン」、Cat なら「ニャー」という結果になります。

処理をまとめて書ける

Dog・Cat・Bird を別々に処理するif文を書かずに、「Animal 配列に入れてまとめてループ処理」できます。コードが短くなり、新しい動物を追加しても既存コードを変えなくてよくなります。

ポリモーフィズムの土台: オーバーライド

ポリモーフィズムを実現するには、第10章で学んだオーバーライドが必要です。親クラス Animal に sound() メソッドがあり、子クラス Dog と Cat がそれぞれ同じ名前で上書き（オーバーライド）することで、呼び出し時に違う動きが生まれます。

11.2 親クラス型の変数で子オブジェクトを扱う（アップキャスト）

ポリモーフィズムの核心は、「親クラス型の変数に子クラスのオブジェクトを代入できる」ことです。これをアップキャストと呼びます。Animal a = new Dog(); と書くと、変数 a は Animal 型ですが、中身は Dog オブジェクトです。a.sound() を呼ぶと、Dog のオーバーライドされたメソッドが実行されます。

PolymorphismDemo.java — アップキャストと多態性 
// 親クラス
class Animal {
    void sound() {
        System.out.println("...");  // 基本の動作（上書きされる想定）
    }
}
// 子クラス Dog: sound() をオーバーライド
class Dog extends Animal {
    @Override
    void sound() { System.out.println("ワン！"); }
}
// 子クラス Cat: sound() をオーバーライド
class Cat extends Animal {
    @Override
    void sound() { System.out.println("ニャー！"); }
}

アップキャストを使うと、Animal の配列に Dog や Cat をまとめて入れてループで一括処理できます。新しい動物クラスを追加しても、ループのコードは変更不要です。

Main.java — 配列でまとめて処理する 
public class Main {
    public static void main(String[] args) {
        // Animal 型配列に Dog と Cat を代入（アップキャスト）
        Animal[] animals = { new Dog(), new Cat(), new Dog() };

        // 同じ sound() 呼び出しでも、結果がオブジェクトにより変わる
        for (Animal a : animals) {
            a.sound();  // Dog→「ワン！」、Cat→「ニャー！」
        }
    }
}
terminal — 実行結果 
ワン！
ニャー！
ワン！

図で整理すると、親クラス Animal を型として使い、各子オブジェクトの実装が呼び出し時に決まるイメージです。

重要ポイント: 動的ディスパッチ

どのメソッドを実行するかは、プログラムを実行した瞬間（実行時）に変数の実際の型で決まります。これを「動的ディスパッチ」と呼びます。コードを書く時点ではなく、実行時に決まるため、非常に柔軟なプログラムが書けます。

11.3 抽象クラス（abstract class）と抽象メソッド

ポリモーフィズムをより強力にする仕組みが抽象クラスです。「Animal クラスそのものはインスタンス化しなくていい（Dog や Cat として使う）」「でも子クラスは必ず sound() を実装してほしい」という場面に使います。

abstract キーワードを付けると、中身のないメソッド（抽象メソッド）を宣言できます。子クラスは必ず抽象メソッドをオーバーライドする義務が生じます。抽象クラスは new でインスタンスを作れません。

Shape.java — 抽象クラスの例 
// abstract をつけると抽象クラスになる
abstract class Shape {
    String color;  // 具体的なフィールドは持てる

    Shape(String color) {
        this.color = color;
    }

    // 抽象メソッド: 中身なし。子クラスに実装を強制する
    abstract double area();

    // 通常メソッドも持てる
    void describe() {
        System.out.println("色: " + color + ", 面積: " + area());
    }
}
Circle.java — 抽象クラスを継承した具体クラス 
// Circle は Shape を継承し、area() を実装する義務がある
class Circle extends Shape {
    double radius;  // 半径

    Circle(String color, double radius) {
        super(color);
        this.radius = radius;
    }

    @Override
    double area() {
        return Math.PI * radius * radius;  // 円の面積 = πr²
    }
}

注意: 抽象クラスはインスタンス化できない

Shape s = new Shape("赤"); と書くとコンパイルエラーになります。抽象クラスは「設計図の骨格」であり、具体的な実装を持つ子クラスだけがインスタンス化できます。抽象メソッドを実装しない子クラスも、コンパイルエラーになります。

11.4 インターフェース（interface）と implements

インターフェースは「できることの約束（契約）」を定義する仕組みです。「このクラスはこのメソッドを必ず持っている」という保証を与えます。interface キーワードで定義し、クラスは implements キーワードで実装します。

インターフェースの大きな特徴は、1つのクラスが複数のインターフェースを実装できることです（Javaでは通常、クラスは1つしか継承できませんが、インターフェースは何個でも implements できます）。

Drawable.java — インターフェースの定義 
// interface: 「できることの約束」を定義する
interface Drawable {
    void draw();  // 自動的に public abstract になる
}

interface Resizable {
    void resize(double factor);  // 複数のインターフェースを持てる
}
Icon.java — 複数インターフェースを implements する例 
// 複数のインターフェースをカンマ区切りで実装できる
class Icon implements Drawable, Resizable {
    double size;

    Icon(double size) { this.size = size; }

    @Override
    public void draw() {
        System.out.println("アイコンを描画（サイズ: " + size + "）");
    }

    @Override
    public void resize(double factor) {
        size = size * factor;  // サイズを拡大・縮小する
        System.out.println("サイズ変更後: " + size);
    }
}
classDiagram class Drawable { <<interface>> +draw() void } class Resizable { <<interface>> +resize(factor) void } class Icon { +size double +draw() void +resize(factor) void } class Button { +label String +draw() void } Drawable <|.. Icon : implements Resizable <|.. Icon : implements Drawable <|.. Button : implements

Icon は Drawable と Resizable の2つのインターフェースを実装。Button は Drawable だけ実装

11.5 抽象クラスとインターフェースの違い・使い分け

抽象クラスとインターフェースは似ていますが、用途が異なります。どちらを使うかは「何を表現したいか」で決めます。

項目 抽象クラス（abstract class） インターフェース（interface）
フィールド 通常のフィールドを持てる 定数（public static final）のみ
継承・実装 1つしか継承できない（extends） 複数実装できる（implements）
具体的な処理 通常メソッド（処理あり）を持てる 原則メソッドの宣言のみ（Java 8以降は default メソッドも可）
向いている場面 「is-a 関係」: 共通の状態と処理を持つ親として使う（例: 動物・図形） 「can-do 関係」: 異なる種類のクラスに同じ能力を付与する（例: 描画できる・保存できる）

使い分けの目安

「〜は〜の一種だ（is-a）」と言えるなら抽象クラス。「〜は〜ができる（can-do）」と言えるならインターフェース。たとえば「犬は動物の一種 → 抽象クラス Animal」「犬は泳げる → インターフェース Swimmable」と整理できます。

やってみよう 11-1: 図形の面積を一括表示しよう

抽象クラス Shape を継承した Circle（円）と Rectangle（長方形）を作り、面積を計算するメソッドをそれぞれ実装します。Shape の配列に入れてポリモーフィズムで一括表示しましょう。

手順

  1. 1 抽象クラス Shape を定義し、抽象メソッド double area() を宣言する。通常メソッド printArea() も追加して「面積: ○○」と表示させる。
  2. 2 Circle クラスを作り、Shape を extends する。半径フィールドを持ち、area() に Math.PI * r * r を実装する。
  3. 3 Rectangle クラスを作り、Shape を extends する。幅・高さフィールドを持ち、area() に width * height を実装する。
  4. 4 main メソッドで Shape[] shapes = { new Circle(5), new Rectangle(4, 6), new Circle(3) }; と配列を作り、for ループで printArea() を呼び出す。ポリモーフィズムにより各図形の面積が表示されることを確認する。

期待される出力

面積: 78.53981633974483
面積: 24.0
面積: 28.274333882308138

理解度確認クイズ

  1. Q1. 次のコードを実行すると、何が出力されますか？理由も説明してください。

    Animal a = new Cat();
    a.sound();
    回答を表示

    「ニャー！」が出力されます。変数 a は Animal 型ですが、実際に代入されているのは Cat オブジェクトです。sound() を呼び出すと、実行時に実際の型（Cat）が確認され、Cat がオーバーライドした sound() が呼び出されます。これがポリモーフィズム（動的ディスパッチ）です。

  2. Q2. 抽象クラスを定義・利用するうえで正しい説明はどれですか？次の中から選んでください。

    • ア: 抽象クラスは new でインスタンスを作れる
    • イ: 子クラスは抽象メソッドを必ずオーバーライドしなければならない
    • ウ: 抽象クラスに通常のメソッドは書けない
    回答を表示

    正解はイです。ア: 抽象クラスはインスタンスを作れません（コンパイルエラー）。イ: 子クラスは abstract メソッドをすべてオーバーライドする義務があります（しない場合、子クラスも abstract にする必要があります）。ウ: 抽象クラスは通常のメソッドを持つことができます。

  3. Q3. インターフェースを使う理由として最も適切なものはどれですか？

    • ア: インスタンスフィールドをたくさん定義したいとき
    • イ: 関係のない複数のクラスに共通の「できること」の約束を与えたいとき
    • ウ: 1つのクラスの機能を上書きしたいとき
    回答を表示

    正解はイです。インターフェースは「できること（can-do）の約束」を複数のクラスに与えるために使います。たとえば、Dog・Car・Boat のような全く異なるクラスに「Swimmable（泳げる）」インターフェースを実装させることができます。ア: インターフェースにインスタンスフィールドは定義できません。ウ: メソッドの上書きはオーバーライドの概念であり、インターフェースの目的とは異なります。
