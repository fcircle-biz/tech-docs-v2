データを守る - カプセル化とアクセス制御

オブジェクトの中身を外部から守る「カプセル化」とアクセス修飾子（public / private）を学びます。ATMの仕組みをたとえに、なぜ情報を隠すことが大切なのか、そしてgetter / setterメソッドを使って安全にデータを操作する方法を理解していきましょう。

この章で学ぶこと

  • カプセル化とは何か、なぜデータを隠すことが重要なのかを説明できる
  • アクセス修飾子（public / private）の違いと使い分けを理解する
  • フィールドを private にして、不正な値の書き込みを防ぐ方法を身につける
  • getter / setter メソッドを使ってフィールドを安全に読み書きできる

9.1 カプセル化とは

カプセル化（Encapsulation）とは、オブジェクトが持つデータ（フィールド）を外部から直接触れないように隠し、決められた窓口（メソッド）だけを通してアクセスさせる考え方です。「大切なものをカプセルに入れて守る」イメージです。

たとえ話: ATMの仕組み

銀行のATMを思い浮かべてください。ATMの中には「金庫（残高データ）」があります。しかし私たちは金庫に直接手を伸ばすことはできません。「引き出す」「預け入れる」「残高照会」という決められた画面操作（メソッド）を通してしか、お金を動かせません。

  • 金庫の残高を直接書き換える → できない（private）
  • 画面操作で引き出す → できる（public メソッド）

プログラムでも同じです。もし残高（フィールド）を誰でも自由に書き換えられると、balance = -100000 のような不正な値が入ってしまいます。カプセル化はこういった事故や不正を防ぐための重要な仕組みです。

カプセル化の2大メリット

  • ① 不正な値から守る: setter でチェック処理を入れられる
  • ② 内部実装を自由に変えられる: 外部から見える窓口（メソッド）さえ変えなければ、中の仕組みを変えても影響が出ない

9.2 アクセス修飾子 — public と private

Javaでは、フィールドやメソッドの前にアクセス修飾子を付けることで「どこからアクセスできるか」を制御します。最初に覚えるべき2つは public と private です。

修飾子 アクセスできる範囲 主な用途
public どこからでもアクセス可 クラス・メソッドの公開窓口（getter/setter）
private 同じクラスの中だけ フィールド（データ）を隠す
protected 同じパッケージ＋子クラスから 継承（第10章）で使う
（省略/デフォルト） 同じパッケージ内のみ パッケージ内の共有

初心者の基本ルール

最初は フィールドは private・メソッドは public と覚えておけば大丈夫です。protected とデフォルトは継承やパッケージ設計を学ぶ段階で自然に身につきます。

9.3 フィールドを private にして直接アクセスを防ぐ

まず、フィールドを public にしてしまった場合の問題点を見てみましょう。

BankAccount.java — フィールドが public の危険な例 
public class BankAccount {
    // public にすると外から直接書き換えられてしまう（危険！）
    public int balance;
}

// 外部コードから不正な値を直接代入できてしまう
BankAccount account = new BankAccount();
account.balance = -100000; // マイナス残高を強制書き込み！

フィールドを private にすると、クラスの外からの直接アクセスがコンパイルエラーになります。

BankAccount.java — private でアクセスを遮断 
public class BankAccount {
    // private にするとクラスの外からは触れない
    private int balance;
}

// 外部コードから書き換えようとするとエラーになる
BankAccount account = new BankAccount();
account.balance = -100000; // コンパイルエラー！アクセス不可

よくある誤解: private にすると使えなくなる？

private にしたフィールドは外部から直接使えなくなるだけです。同じクラスの中のメソッドからは普通に使えます。次のセクションで学ぶ getter / setter が「窓口」となって、安全にアクセスできるようにします。

9.4 getter / setter メソッド

private フィールドを外部から読み書きするための窓口が getter（ゲッター） と setter（セッター） です。名前は慣習として「getフィールド名 / setフィールド名」のように書きます。

getter（読み取り）

フィールドの値を返すメソッド。外部はこのメソッドを通じて値を取得します。名前は getXxx()（戻り値あり）。

setter（書き込み）

フィールドに値をセットするメソッド。値チェックを入れられるのが最大の利点。名前は setXxx（値）（引数あり、戻り値なし）。

setter の中で値チェックを行うことで、不正な値（例: マイナスの残高）が入り込むのを防ぐことができます。

BankAccount.java — getter と setter 
public class BankAccount {
    private int balance; // 外部から直接触れない

    // getter: 残高を読み取る窓口
    public int getBalance() {
        return balance;
    }

    // setter: 値チェックをしてから書き込む窓口
    public void setBalance(int amount) {
        if (amount < 0) { // マイナスは拒否！
            System.out.println("エラー: 残高はマイナスにできません");
            return;
        }
        balance = amount; // 問題なければ代入
    }
}

getter だけ / setter だけ を作ることも OK

「外部に読ませたいが書かせたくない」場合は getter だけ作ります。反対に「書かせたいが読ませたくない（ログパスワードなど）」は setter だけ作ります。アクセスを細かく制御できるのがカプセル化の強みです。

9.5 カプセル化したクラスの完全な例

これまでの内容をまとめて、カプセル化された BankAccount クラスを完成させましょう。deposit（預け入れ）と withdraw（引き出し）という専用メソッドも追加します。

BankAccount.java — カプセル化の完全な例 
public class BankAccount {
    private int balance; // 残高（外部から直接触れない）

    // getter: 残高を読み取る
    public int getBalance() {
        return balance;
    }

    // 預け入れ（正の値のみ許可）
    public void deposit(int amount) {
        if (amount <= 0) {
            System.out.println("エラー: 預け入れ額は正の値にしてください");
            return;
        }
        balance += amount;
        System.out.println(amount + "円を預け入れました");
    }

    // 引き出し（残高不足チェックつき）
    public void withdraw(int amount) {
        if (amount <= 0) {
            System.out.println("エラー: 引き出し額は正の値にしてください");
            return;
        }
        if (amount > balance) {
            System.out.println("エラー: 残高が不足しています");
            return;
        }
        balance -= amount;
        System.out.println(amount + "円を引き出しました");
    }
}

このクラスを使う側のコードを見てみましょう。

Main.java — BankAccount を使う 
public class Main {
    public static void main(String[] args) {
        BankAccount account = new BankAccount();

        account.deposit(5000);   // 5000円預け入れ
        account.withdraw(2000);  // 2000円引き出し
        account.withdraw(9999);  // 残高不足でエラー

        // getter で残高を確認
        System.out.println("残高: " + account.getBalance() + "円");
    }
}
terminal — 実行結果

2000円を引き出しました
エラー: 残高が不足しています
残高: 3000円

まとめ: カプセル化の3ステップ

  1. フィールドを private にする
  2. 読み取り用の public getXxx()（getter）を作る
  3. 書き込み用の public setXxx（値）（setter）を作り、値チェックを入れる

やってみよう 9-1: 不正な値をはじくクラスを作ろう

Person（人物）クラスを作り、年齢フィールドを private にして setter で「0〜150の範囲外を拒否」するチェックを実装しましょう。

手順

  1. 1 Person クラスを作り、private int age フィールドを宣言する。
  2. 2 getAge() と setAge(int age) を実装する。setter には「0未満または150超の場合はエラーメッセージを表示して処理を中断する」チェックを入れる。
  3. 3 Main クラスで setAge(-5)・setAge(200)・setAge(25) を呼び、正常な値だけ設定されることを getAge() で確認する。
  4. 4 余力があれば、private String name を追加し、getName()/setName() も実装してみる（名前が空文字列なら拒否）。

理解度確認クイズ

  1. Q1. カプセル化とは何ですか？ATMのたとえを使って説明してください。

    回答を表示

    カプセル化とは、オブジェクトが持つデータ（フィールド）を外部から直接触れないように隠し、決められたメソッドだけを通してアクセスさせる考え方です。ATMのたとえでは「金庫（残高データ）に直接手を伸ばすことはできず、画面の操作（public メソッド）を通してしかお金を動かせない」状態がカプセル化に当たります。これにより不正な値の書き込みを防げます。

  2. Q2. private で宣言したフィールドは、同じクラスの中のメソッドから使えますか？

    回答を表示

    はい、使えます。private は「外部のクラスからアクセスできない」という制限です。同じクラスの中（getter / setter / depositなどのメソッド）からは普通にアクセスできます。 外部から直接書き換えられなくなるだけで、クラス内部での利用には一切制限がありません。

  3. Q3. setter メソッドを使う最大の利点は何ですか？

    回答を表示

    setter メソッドの最大の利点は、値を設定する前にチェック処理を挟めることです。たとえば「残高はマイナス禁止」「年齢は0〜150の範囲」といったバリデーション（検証）を setter の中に書くことで、不正な値がフィールドに入るのを構造的に防ぐことができます。フィールドを public にして直接代入を許す場合は、こうしたチェックを入れる場所がありません。
