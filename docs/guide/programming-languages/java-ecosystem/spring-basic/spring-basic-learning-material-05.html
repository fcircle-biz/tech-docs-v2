DIを使ったクラス分割

Spring アプリでは、役割ごとにクラスを分けて作るのが基本です。Controller・Service・Repository という3つの層に分け、それぞれに専門の仕事を担当させます。そして DI によって、これらのクラスを Spring が連携させてくれます。この章では、この基本構成を理解し、次教材（Spring MVC・Spring Data JDBC）へつながる考え方をしっかり身につけましょう。

この章で学ぶこと

  • Controller、Service、Repository の役割分担を理解する
  • DI によってクラスを連携させる流れを理解する
  • Spring MVC や Spring Data JDBC につながる基本構成を理解する

5.1 役割ごとにクラスを分ける

Spring アプリでは、1つのクラスにすべての処理を詰め込まず、役割ごとにクラスを分けるのが基本です。代表的な分け方が次の3層です。

層 役割
Controller 入力を受け取る
Service 業務処理を行う
Repository データアクセスを行う

処理の流れは「Controller → Service → Repository」の順になります。図にすると次のとおりです。

flowchart LR A["Controller
（入力を受け取る）"] -->|"呼び出す"| B["Service
（業務処理）"] B -->|"呼び出す"| C["Repository
（データアクセス）"]

それぞれのクラスは自分の役割だけに集中します。Controller は「誰が何を頼んできたか」を受け取るだけ、Service は「業務ロジック（どんな処理をするか）」だけ、Repository は「データをどう保存・取得するか」だけを知っていればよい、という設計です。

5.2 コード例（3クラス）

実際のコードで3つの層がどのように書かれるか見てみましょう。まずController（入力を受け取り、Service を呼び出す）です。

UserController.java 
@Component
public class UserController {
    // Service を DI で受け取る（自分で new しない）
    private final UserService userService;

    // コンストラクタで Spring から注入される
    public UserController(UserService userService) {
        this.userService = userService;
    }

    public void register() {
        // Service に処理を任せる
        userService.register("田中");
    }
}

次にService（業務処理を行い、Repository を呼び出す）です。

UserService.java 
@Service
public class UserService {
    // Repository を DI で受け取る
    private final UserRepository userRepository;

    // コンストラクタで Spring から注入される
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public void register(String name) {
        // データ保存は Repository に任せる
        userRepository.save(name);
    }
}

最後にRepository（データアクセスを担う。今はコンソール出力で代替）です。

UserRepository.java 
@Repository
public class UserRepository {
    public void save(String name) {
        // 実際は DB に保存するが、今はコンソールで確認
        System.out.println(name + "を保存しました");
    }
}

5.3 連携の流れ

上の3つのクラスが Spring によってどのように連携するか、ポイントを確認しましょう。

呼び出しの流れ

  • Controller は Service を呼び出す
  • Service は Repository を呼び出す
  • 各クラスは自分の役割に集中する

Spring の DI が行っていること

  • 必要な部品は Spring が用意して渡してくれる
  • コンストラクタで下の層の Bean を受け取っている
  • 自分で new しなくてよい

第3章・第4章で学んだ内容を思い出してください。コンストラクタで受け取っているのが DI（依存性注入）の正体です。UserController は UserService を、UserService は UserRepository を、それぞれ Spring（Bean コンテナ）から受け取っています。各クラスは Bean として登録されているから、Spring が自動で注入できるのです。

この「Controller → Service → Repository」という考え方は、Spring MVC や Spring Data JDBC でも変わりません。Web からのリクエストを Controller が受け取り、Service で処理し、Repository でデータベースを操作する、という流れがそのまま次教材へつながります。

5.4 なぜ分けるのか（メリット）

役割ごとにクラスを分けることには、具体的にどんなメリットがあるのでしょうか？

どこを直せばよいか分かりやすい

データの保存に問題があれば Repository だけ見ればよい。業務ロジックのバグなら Service だけを調べればよい。

一部だけ差し替え・テストしやすい

Repository を「本物の DB 接続」に差し替えたり、テスト用の偽物に入れ替えたりするのが簡単になる。

複数人で分担しやすい

Aさんが Controller、Bさんが Service、Cさんが Repository を担当、という分業がしやすくなる。

重要ポイント: 役割分担の本質

  • 各クラスは「1つの役割だけ」に責任を持つ
  • DI が各クラスを Spring コンテナから自動で連結する
  • この構成が Spring MVC・Spring Data JDBC の土台になる

次教材への橋渡し

本教材では、Repository の save() がコンソールへ文字を出力するだけになっています。これは動きを確認しやすくするためです。Web からの入力（HTTP リクエスト）は「Spring MVC」で、実際の DB への保存は「Spring Data JDBC」で、それぞれ次教材で本物にしていきます。今は「3層の役割分担」と「DI で連携する流れ」をしっかりつかんでおきましょう。

理解度確認クイズ

  1. Q1. Controller の役割は何ですか？

    回答を表示

    入力を受け取る役割です。Web アプリであれば HTTP リクエストを受け取り、Service に処理を依頼します。

  2. Q2. Service の役割は何ですか？

    回答を表示

    業務処理（ビジネスロジック）を行う役割です。「何をどう処理するか」というアプリの本質的なロジックをここに書きます。

  3. Q3. Repository の役割は何ですか？

    回答を表示

    データアクセス（データの保存・取得など）を行う役割です。実際の DB 操作はここに集約します。

  4. Q4. Controller → Service → Repository の順に分けるメリットは何ですか？

    回答を表示

    各クラスが役割に集中でき、修正やテスト・分担がしやすくなります。変更の影響範囲も分かりやすく、「どこを直せばよいか」がひと目で判断できます。
