AOPとは

AOP（アスペクト指向プログラミング）は、ログ出力・例外処理・認証といった「複数の場所で共通して必要になる処理」を、本来の業務処理から分離してまとめて差し込む仕組みです。この章では、その目的と使い方をログ出力を例に、ステップを踏んでやさしく学んでいきます。

この章で学ぶこと

  • AOP の目的を理解する
  • 共通処理を分離する考え方を理解する
  • ログ出力を例に AOP の使い方を理解する

7.1 AOP とは

AOP（エーオーピー）とは、Aspect Oriented Programming（アスペクト指向プログラミング）の略です。「アスペクト（Aspect）」は日本語で「側面・観点」という意味で、アプリケーションを「異なる側面・関心事」に分けて考える手法を指します。

アプリケーションには、業務処理（ユーザー登録・商品検索など）のほかに、ログ出力・例外処理・認証・トランザクション管理といった処理が出てきます。これらは「複数の場所で共通して必要になる処理」であり、横断的関心事（おうだんてき・かんしんじ）と呼ばれます。「横断的」とは、いろいろな処理に共通して横断的に出てくる、という意味です。

AOP のいちばん大切な考え方

AOPとは、いろいろな場所で必要になる共通処理を、まとめて管理するための仕組みです。共通処理を業務処理の中に混ぜずに、分離してあとから差し込むことで、コードをすっきりと保ちます。

AOP を使うと、共通処理を1か所にまとめて書いておくだけで、Spring が自動的に対象のメソッドに差し込んでくれます。業務処理クラスは本来やりたい処理だけを書けばよくなり、コードが読みやすくなります。

7.2 AOP を使わない例（問題点）

まず、AOP を使わない場合のコードを見てみましょう。ログ出力（「開始」「終了」の出力）を、業務処理の中に直接書いています。

UserService.java 
@Service
public class UserService {
    public void register(String name) {
        System.out.println("register開始");       // ログ出力
        System.out.println(name + "を登録しました"); // 業務処理
        System.out.println("register終了");       // ログ出力
    }
}

このコードには、次のような問題があります。

ログが業務処理に混ざる

ログ出力の行が業務処理の行と混在しており、本来やりたい処理（登録）が読みにくくなっています。

同じ処理が何度も登場する

メソッドが増えるたびに、同じようなログ出力のコードをコピー&ペーストして書く必要があります。

変更が大変

ログの書き方を変えたくなったとき、書いたすべての場所を直す必要があります。

7.3 AOP を使う例

次に、AOP を使った場合のコードを見てみましょう。ログ出力の処理を別クラス（Aspect）にまとめて書くことで、業務処理クラスはすっきりします。

LoggingAspect.java 
@Aspect      // AOPクラスであることを示す
@Component   // SpringのBeanとして管理される
public class LoggingAspect {

    // serviceパッケージ配下の全メソッド実行前に動く
    @Before("execution(* com.example.demo.service.*.*(..))")
    public void beforeMethod() {
        System.out.println("メソッド開始");
    }

    // serviceパッケージ配下の全メソッド実行後に動く
    @After("execution(* com.example.demo.service.*.*(..))")
    public void afterMethod() {
        System.out.println("メソッド終了");
    }
}

このコードのポイントを一つずつ見ていきましょう。

アノテーション・記述 意味
@Aspect このクラスが AOP 用クラス（共通処理をまとめた Aspect）であることを Spring に伝える
@Component Spring Bean として管理される。Spring がこのクラスを自動で認識して動かしてくれる
@Before 対象メソッドの実行前にこのメソッドを呼び出す
@After 対象メソッドの実行後にこのメソッドを呼び出す
execution（...） どのメソッドに処理を差し込むかを指定する条件。service パッケージ配下の全クラスの全メソッド、という意味

execution の書き方（ざっくりイメージ）

execution(* com.example.demo.service.*.*(..)) は「com.example.demo.service パッケージにあるすべてのクラスの、すべてのメソッドを対象にする」という意味です。* はワイルドカード（なんでも）、.. は「引数はなんでも」を表します。今は「service パッケージ配下のメソッドに差し込む」とざっくり理解しておけばOKです。

7.4 図解: AOP の動き

AOP を使うと、業務メソッドの実行前後に、Aspect のログ処理が自動で差し込まれます。その流れを図で確認しましょう。

青い部分（@Before・@After）が AOP によって自動で差し込まれる処理です。緑の部分（本来の処理）はログのことを気にせず、業務処理だけを書けばよくなります。

業務クラスはシンプルに

UserService はログ出力を一切書かなくても、Spring が自動的に Aspect の処理を差し込んでくれます。

共通処理は1か所に集約

ログの書き方を変えたいときは LoggingAspect だけを修正すれば、すべてのメソッドに自動で反映されます。

7.5 AOP の基本用語

AOP にはいくつかの専門用語があります。最初は覚えにくいですが、「何を指しているか」のイメージをざっくりつかんでおきましょう。

用語 意味
Aspect 共通処理をまとめたクラス（例: LoggingAspect）
Advice 実際に実行される共通処理（Before・After・Around などのメソッド本体）
Join Point 処理を差し込める場所（Spring AOP では主にメソッド呼び出しのタイミング）
Pointcut どこに処理を差し込むかの条件（execution(...) で指定する部分）
Before メソッド実行前に処理する（@Before）
After メソッド実行後に処理する（@After）
Around メソッド実行前後に処理する（@Around。Before＋After を1つにまとめたイメージ）

Advice の種類（ざっくり整理）

@Before・@After・@Around などをまとめて Advice（アドバイス） と呼びます。今の段階では「差し込む処理の種類」くらいのイメージで大丈夫です。深掘りは後の章や実践で行っていきます。

理解度確認クイズ

  1. Q1. AOP とは何ですか？

    回答を表示

    Aspect Oriented Programming（アスペクト指向プログラミング）の略です。ログ出力などの共通処理を本来の業務処理から分離し、まとめて管理・自動で差し込む仕組みです。

  2. Q2. AOP はどのような処理に向いていますか？

    回答を表示

    ログ出力・例外処理・認証・トランザクション管理など、複数の場所で共通して必要になる処理（横断的関心事）に向いています。

  3. Q3. @Aspect は何を表しますか？

    回答を表示

    そのクラスが AOP 用のクラス（共通処理をまとめた Aspect）であることを表します。Spring はこのアノテーションを見て、対象クラスを Aspect として認識します。

  4. Q4. @Before と @After の違いは何ですか？

    回答を表示

    @Before は対象メソッドの実行前に処理を行い、@After は対象メソッドの実行後に処理を行います。両方を使うことで、メソッドの「開始時」と「終了時」それぞれに共通処理を差し込めます。
