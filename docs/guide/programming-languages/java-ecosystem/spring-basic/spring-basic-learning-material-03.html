DIとは

DI（依存性注入）の意味を、new で自分でオブジェクトを作る場合と比べながら理解しましょう。「DIとは、必要な部品を自分で作るのではなく、Springに用意してもらう仕組み」という考え方を、コードと図を通じて丁寧に説明します。

この章で学ぶこと

  • DI の意味を理解する
  • new で直接オブジェクトを作る場合との違いを理解する
  • コンストラクタインジェクションの基本を理解する

3.1 DI とは

DI は Dependency Injection の略で、日本語では「依存性注入」と呼ばれます。あるクラスが別のクラスを必要とするとき、自分で new してオブジェクトを作るのではなく、外部（Spring）から渡してもらうという考え方です。

この章でいちばん大切な一文

DI とは、必要な部品を自分で作るのではなく、Spring に用意してもらう仕組みです。

「依存性注入」という言葉は難しく聞こえますが、2つに分けて考えるとシンプルです。

「依存」とは？

あるクラスが別のクラスを必要としている関係のことです。たとえば UserService が UserRepository を使う場合、「UserService は UserRepository に依存している」と言います。

「注入」とは？

必要な部品（オブジェクト）を外から渡し込むことです。自分で new するのではなく、Spring が作ったものを受け取るイメージです。

つまり DI とは、「あるクラスが必要とする別のクラス（依存）を、外部（Spring）から渡してもらう（注入）こと」です。次のセクションでは、DI を使わない場合と使う場合を具体的なコードで比べてみましょう。

3.2 DI を使わない例（問題点）

まずは DI を使わない場合のコードを見てみましょう。ユーザー登録を行う UserService が、データ保存を担当する UserRepository を使う例です。

UserService.java（DIを使わない場合） 
public class UserService {
    // UserRepository を自分で new して作っている
    private UserRepository userRepository = new UserRepository();

    public void register(String name) {
        userRepository.save(name); // 保存処理を呼び出す
    }
}

このコードには、いくつかの問題点があります。

このコードの問題点

  • UserService が UserRepository を自分で new している（作り方まで知ってしまっている）
  • クラス同士の結びつき（結合）が強くなる
  • UserRepository の仕様が変わると UserService も変更が必要になる
  • テスト時に UserRepository を差し替えにくく、テストがしにくい

「結合が強い」というのは、クラスが互いの詳細を知りすぎている状態のことです。このような設計では、一方を変更するともう一方にも影響が及びやすくなり、変更やテストのたびにコードを広範囲で修正しなければならないという問題が生じます。

3.3 DI を使う例

では、Spring の DI を使って同じ処理を書き直してみましょう。ポイントはコンストラクタで UserRepository を受け取る書き方です。

UserService.java（DIを使う場合） 
@Service
public class UserService {
    private final UserRepository userRepository;

    // コンストラクタで UserRepository を受け取る（Springが渡してくれる）
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public void register(String name) {
        userRepository.save(name); // 保存処理を呼び出す
    }
}
UserRepository.java 
@Repository
public class UserRepository {
    public void save(String name) {
        // コンソールに保存結果を出力する
        System.out.println(name + "を保存しました");
    }
}

DI を使った場合のポイント

  • UserService は UserRepository を自分で new していない
  • コンストラクタの引数で UserRepository を受け取るだけでよい
  • Spring が自動で UserRepository を用意してコンストラクタに渡してくれる
  • この書き方を「コンストラクタインジェクション」という

ヒント: なぜ final を付けるのか

フィールドに final を付けると、一度設定されたら変更できないことが保証されます。コンストラクタで受け取った部品がその後に差し替えられてしまうのを防ぐための、よい習慣です。コンストラクタインジェクションでは final を付けることが推奨されています。

ヒント: @Autowired は省略できる

Spring では、コンストラクタが1つだけのクラスは @Autowired アノテーションを省略できます。上のコードで @Autowired を書かなくても Spring が自動的にインジェクションを行います。コンストラクタが複数ある場合のみ @Autowired の指定が必要です。

3.4 図解: DI の仕組み

Spring を使わない場合と使う場合を、図で比べてみましょう。

Spring を使わない場合

UserService が自分で new UserRepository() を呼び出して作成します。

結合が強く、変更・テストがしにくい

Spring を使う場合（DI）

Spring が UserRepository を作り、コンストラクタを通じて UserService に渡してくれます。

結合が弱く、変更・テストがしやすい

「Spring を使う場合」では、Spring が中心に立って、UserRepository を作り UserService に渡してくれます。UserService 自身は「コンストラクタで受け取る」だけでよく、相手の作り方を知る必要がなくなります。これが DI の本質です。

理解度確認クイズ

  1. Q1. DI とは何ですか？

    回答を表示

    Dependency Injection（依存性注入）の略です。必要な部品（オブジェクト）を自分で new せず、Spring に用意してもらい外部から渡してもらう仕組みのことです。

  2. Q2. new で直接作る場合の問題点は何ですか？

    回答を表示

    クラスが相手の作り方まで知ることになり、クラス同士の結合が強くなります。その結果、変更のたびに関連するクラスも修正が必要になったり、テスト時に部品を差し替えにくくなったりします。

  3. Q3. コンストラクタインジェクションとは何ですか？

    回答を表示

    必要な部品をコンストラクタの引数で受け取り、Spring がそれを自動で渡してくれる書き方です。フィールドに final を付けることで、注入された部品が変更されないことを保証できます。
