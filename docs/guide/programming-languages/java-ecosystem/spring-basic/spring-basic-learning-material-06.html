設定ファイルとプロファイルの基本

Spring Boot アプリの設定を一手に担う application.properties の役割を学びます。アプリ名やポート番号などの設定値を Java コードの外に書き出す（外部化する）考え方と、開発・テスト・本番など環境ごとに設定をまとめて切り替える「プロファイル」の概要をつかみましょう。

この章で学ぶこと

  • application.properties の基本的な役割を理解する
  • 設定値を外部化する考え方を理解する
  • プロファイルの概要を理解する

6.1 application.properties の役割

Spring Boot のプロジェクトには、src/main/resources/application.properties というファイルが最初から用意されています。ここに「アプリの設定」を書いておくことで、Spring Boot が起動時に自動で読み込んでくれます。

設定ファイルとは？

アプリの動き方を決める値（ポート番号・アプリ名など）を、Java コードの外に書いておくファイルです。コードを書き直さなくてもここを変えるだけでアプリの挙動を変えられます。

たとえば、次のような設定を書くことができます。

application.properties 
# アプリケーションの名前
spring.application.name=spring-basic-demo

# サーバーの待ち受けポート番号
server.port=8080

各行の意味は次のとおりです。

spring.application.name

このアプリケーションの名前を設定します。ログや監視ツールでアプリを識別するために使われます。

server.port

サーバーが待ち受けるポート番号です。デフォルトは 8080 で、変更するとブラウザからアクセスする番号が変わります。

6.2 設定を外部化するメリット

設定値を Java コードの中に直接書く方法もありますが、なぜわざわざ application.properties に分けるのでしょうか。

外部化の最大のメリット

コードを書き換えずに、設定だけを変えることができます。たとえばポート番号を変えたくなったとき、Java ファイルを編集して再コンパイルする必要なく、application.properties の1行を書き直すだけで済みます。

外部化することで得られるメリットをまとめると、次のようになります。

コードと設定を分離できる

ビジネスロジックと設定が混在せず、コードがシンプルになります。

環境ごとに設定を変えやすい

開発・テスト・本番で異なる設定ファイルを用意するだけで切り替えられます。

イメージを図にすると次のようになります。同じプログラムを使いながら、設定ファイルを差し替えるだけで各環境に対応できます。

flowchart TD P["同じ Spring アプリ（プログラム）"] P --> D["開発環境用の設定ファイル
application-dev.properties"] P --> T["テスト環境用の設定ファイル
application-test.properties"] P --> PD["本番環境用の設定ファイル
application-prod.properties"] D --> R1["開発環境で起動"] T --> R2["テスト環境で起動"] PD --> R3["本番環境で起動"]

注意: パスワードなどの秘密情報の扱い

データベースのパスワードや API キーなどの機密情報を application.properties にそのまま書くと、ファイルをバージョン管理したときに外部に漏れる危険があります。実務では別途適切な管理方法が必要です。入門段階では深入りしませんが、頭の片隅に置いておきましょう。

なお、データベース接続の設定（URL・ユーザー名など）は Spring Data JDBC の教材でくわしく扱います。ここでは「設定をファイルに書き出せる」という考え方をつかんでおきましょう。

6.3 プロファイルの基本

6.2 で「環境ごとに設定を切り替えられる」と紹介しましたが、その切り替えを便利にする仕組みが プロファイル（Profile）です。環境の名前（例: dev・test・prod）を指定すると、その環境用にまとめた設定が自動で適用されます。

プロファイルのイメージ

「今日は開発モードで動かす」「本番モードで動かす」というように、モードを宣言するだけで設定がまとめて切り替わる仕組みです。制服を着替えるようなイメージです。

使いたいプロファイルは application.properties の中で次のように指定します。

application.properties 
# 使用するプロファイル名を指定する（dev=開発環境）
spring.profiles.active=dev

ここを dev（開発）から test（テスト）や prod（本番）に書き換えると、それぞれの環境用の設定が読み込まれます。

プロファイル名 主な用途 対応する設定ファイル例
dev 開発環境（ローカル PC） application-dev.properties
test テスト環境 application-test.properties
prod 本番環境 application-prod.properties

プロファイルの詳細な使い方（環境変数での切り替えや条件付き Bean 定義など）は、本教材では深入りしません。ここでは「環境の名前を指定するだけで設定をまとめて切り替えられる仕組みがある」というイメージをつかんでおきましょう。

理解度確認クイズ

  1. Q1. application.properties は何のために使いますか？

    回答を表示

    アプリ名・ポート番号・ログ設定などのアプリケーション設定を書くために使います。Spring Boot が起動時に自動で読み込んでくれるので、Java コードを変更せずに設定を変えることができます。

  2. Q2. 設定を Java コードから分離するメリットは何ですか？

    回答を表示

    コードを書き換えずに設定だけを変更でき、環境ごとに設定を切り替えやすくなります。開発・テスト・本番など異なる環境でも、同じプログラムを使いながら設定ファイルを差し替えるだけで対応できます。

  3. Q3. プロファイルはどのような場面で使いますか？

    回答を表示

    開発・テスト・本番など環境ごとに設定をまとめて切り替えたい場面で使います。spring.profiles.active にプロファイル名を指定するだけで、その環境用の設定ファイルが自動的に読み込まれます。
