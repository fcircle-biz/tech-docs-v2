ベストプラクティスとトラブルシューティング

JDBC を「正しく・安全に・速く」使い続けるための知恵を学ぶ最終章です。try-with-resources でリソース解放を確実にし、コネクションプーリングで接続コストを削減する仕組みを理解します。よくあるエラーの読み方・対処法とパフォーマンス向上のヒントも合わせて整理し、これまでの学習を総まとめします。

この章で学ぶこと

  • try-with-resources 文でリソースを確実に閉じる方法を理解する
  • コネクションプーリングの仕組みと HikariCP の基本を知る
  • よくある SQLException やエラーの読み方・対処法を押さえる
  • SELECT・PreparedStatement・バッチ処理のパフォーマンスヒントを活かせる

8.1 try-with-resources で確実に閉じる

第6章では try-catch-finally を使って、Connection や PreparedStatement を確実に閉じる方法を学びました。Java 7 以降では、try-with-resources 文を使うとこれをもっとシンプルに書けます。AutoCloseable インターフェースを実装したオブジェクト（Connection・Statement・ResultSet はすべて対象）は、try の丸括弧の中で宣言するだけで、ブロックを抜けるときに自動で close() が呼ばれます。

finally ブロックで手動 close を書く必要がなくなるため、「閉じ忘れ」や「例外が起きたときに close が実行されない」というバグを構造的に防げます。

観点 finally 版（第6章） try-with-resources 版
コード量 close を finally に手で書く（行が増える） 宣言だけで自動 close（行が減る）
閉じ忘れリスク 書き忘れの可能性がある 構造的にゼロ
複数リソース null チェックが必要で煩雑 セミコロン区切りで並べるだけ
TryWithResourcesExample.java 
// try の丸括弧内でリソースを宣言 → 自動 close される
try (Connection conn = DriverManager.getConnection(URL, USER, PASS);
     PreparedStatement ps = conn.prepareStatement(
             "SELECT id, name FROM users WHERE id = ?")) {

    ps.setInt(1, 1); // パラメータをセット
    try (ResultSet rs = ps.executeQuery()) { // ResultSet も宣言
        if (rs.next()) {
            System.out.println(rs.getString("name")); // 結果を表示
        }
    } // ← rs.close() が自動実行される
} // ← ps.close() → conn.close() の順に自動実行される

重要ポイント: try-with-resources を常に使おう

try の丸括弧内に宣言したリソースは、正常終了でも例外が起きても必ず close されます。複数リソースはセミコロン区切りで書き、宣言した逆順（後→前）に close されます。今後 JDBC コードを書くときは、常に try-with-resources を第一選択にしましょう。

8.2 コネクションプーリング

DriverManager.getConnection() は、その呼び出しのたびにデータベースへの新しい接続を作成します。接続の作成はネットワーク通信や認証処理を伴うため、比較的コストの高い処理です。リクエストのたびに接続を作って閉じていると、性能上のボトルネックになりやすい問題があります。

この問題を解決するのがコネクションプーリングです。あらかじめ一定数の接続を作っておき、アプリからの要求があれば「プール」から既存の接続を貸し出し、使い終わったら閉じずにプールへ返却します。接続の作成コストを大幅に削減できます。

flowchart LR A["Javaアプリ
（リクエスト）"] -->|"接続を要求"| B["コネクションプール
（接続を管理）"] B -->|"接続を貸し出し"| A A -->|"使い終わったら返却"| B B -->|"接続を維持・再利用"| C[("データベース")] C -.->|"通信"| B

Java でよく使われるコネクションプーリングの実装として HikariCP（ヒカリシーピー）と Apache DBCP があります。これらは javax.sql.DataSource インターフェースを実装しており、DataSource#getConnection() を呼ぶことでプールから接続を取得できます。アプリ側のコードは DriverManager を DataSource に置き換えるだけで、接続の「実体」は内部で使い回されます。

HikariCP 設定の例（概要） 
// HikariCP を使ったプール設定のイメージ
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:h2:mem:testdb"); // 接続先 URL
config.setUsername("sa");               // ユーザー名
config.setMaximumPoolSize(10);          // プール内の最大接続数

// DataSource からプール経由の接続を取得する
HikariDataSource ds = new HikariDataSource(config);
try (Connection conn = ds.getConnection()) { // プールから貸し出し
    // SQL 操作 …
} // ← ここで接続はプールへ返却される（実際には close されない）

HikariCP は Java 標準ライブラリではなく外部ライブラリです。H2 ドライバと同じように、依存関係に追加する必要があります（Maven の例）。

pom.xml（HikariCP の依存・抜粋） 
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
    <version>5.1.0</version>
</dependency>

2種類の「close」を区別する

接続の conn.close(): プールから借りた接続を返却するだけで、実際には切断されません。だからこそ、借りた接続は使い終わったら必ず close()（返却）してください。返し忘れるとプールが枯渇します。try-with-resources で借りれば自動で返却されます。

プール本体の ds.close(): HikariDataSource 自体もリソースです。こちらはアプリ終了時に一度だけ閉じ、保持している全接続を破棄します。プールは通常アプリ全体で1つだけ作り、使い回します。

用語: DataSource と DriverManager の違い

DriverManager は毎回新しい接続を作る「単純な窓口」。DataSource は接続プールを管理する「高機能な窓口」です。Spring Boot などの実用フレームワークでは、HikariCP が自動で DataSource として設定されます。本ガイドでは学習のため DriverManager を使ってきましたが、実務では DataSource を使うのが標準です。

8.3 よくあるエラーと対処

JDBC のコードを書き始めると、必ずと言っていいほど遭遇するエラーがあります。エラーメッセージを読み解けると、修正に要する時間が大幅に短くなります。代表的なものを以下に整理します。

エラー・例外 よくある原因 対処法
ClassNotFoundException ドライバの JAR がクラスパスにない pom.xml / build.gradle にドライバ依存を追加する
No suitable driver found 接続 URL の書き式が誤っている、またはドライバが未ロード URL の先頭（jdbc:h2: 等）を確認。ドライバ依存を再確認する
Connection refused DB サーバーが起動していない、またはホスト・ポートが誤り DB の起動確認。接続 URL のホスト・ポートを見直す
Access denied / Login failed ユーザー名・パスワードが誤っている 認証情報を確認。DB 側でユーザー権限を確認する
ResultSet is closed Statement や Connection を閉じた後に ResultSet を参照している ResultSet の操作を Statement/Connection の close より前に完了させる
ORA-00942 / Table not found テーブル名のスペルミス、または接続先 DB が違う テーブル名と接続 URL を再確認する

SQLException の読み方: catch (SQLException e) でキャッチした例外は、e.getMessage()（エラー内容）と e.getSQLState()（エラーの種類を示す5桁コード）を確認します。e.printStackTrace() でスタックトレースを表示すると、どのコードで起きたかも把握できます。

SQLException の詳細を確認する 
try (Connection conn = DriverManager.getConnection(URL, USER, PASS)) {
    // … SQL 操作 …
} catch (SQLException e) {
    // エラー内容・SQLState を確認してデバッグに役立てる
    System.err.println("エラーメッセージ: " + e.getMessage());
    System.err.println("SQLState: " + e.getSQLState());
    e.printStackTrace(); // スタックトレースを出力
}

8.4 パフォーマンス向上のヒント

JDBC のパフォーマンスは、少し気をつけるだけで大きく改善できます。代表的なヒントをまとめます。

必要な列だけ SELECT する

SELECT * は全列を転送するため無駄が多い。SELECT id, name のように必要な列だけを指定する。

PreparedStatement を使い回す

SQL 解析はコストがかかる。同じ SQL を繰り返す場合、PreparedStatement を一度作って使い回すとオーバーヘッドを減らせる。

バッチ処理で大量 INSERT を高速化

addBatch() で SQL をまとめ、executeBatch() で一括送信。1件ずつ実行するより DB とのやり取りが激減する。

接続を必要最小限の時間だけ保持する

接続を長時間開きっぱなしにするとプールが枯渇する。SQL を実行する直前に取得し、すぐ close（返却）する習慣をつける。

バッチ処理のコード例を見てみましょう。第6章で学んだトランザクションと組み合わせ、失敗時は rollback() で全件取り消すのが完成形です。

バッチ INSERT の例（rollback 付き） 
String sql = "INSERT INTO users (name, email) VALUES (?, ?)";
try (Connection conn = DriverManager.getConnection(URL, USER, PASS);
     PreparedStatement ps = conn.prepareStatement(sql)) {

    conn.setAutoCommit(false); // バッチはトランザクションと組み合わせる
    try {
        for (String[] row : dataList) { // 登録したいデータを順に処理
            ps.setString(1, row[0]);    // name をセット
            ps.setString(2, row[1]);    // email をセット
            ps.addBatch();              // SQL をバッチに追加
        }
        ps.executeBatch(); // まとめて送信（1件ずつより高速）
        conn.commit();     // 全件成功したらコミット
    } catch (SQLException e) {
        conn.rollback();   // 1件でも失敗したら全件取り消し
        throw e;           // 例外は握りつぶさず呼び出し元へ伝える
    }
}

重要ポイント: バッチも「全部成功か全部取り消し」

setAutoCommit(false) でまとめたバッチは、executeBatch() の途中で1件でも失敗すると例外になります。その際 rollback() を呼ばないと、それまでに送った分が中途半端に残るおそれがあります。try-catch で rollback() し、例外を握りつぶさず呼び出し元へ伝えましょう。

DB 側のチューニング: インデックスを活用する

WHERE 句で頻繁に絞り込む列には、DB 側でインデックスを張ると検索速度が劇的に上がります。JDBC プログラムの改善だけでなく、テーブル設計・インデックス設計も性能に大きく影響します。SQL や DB 設計の深掘りは「次のステップ」（8.5）で紹介するリソースで学べます。

やってみよう 8-1: コードを try-with-resources に書き換える

これまでに書いた JDBC コードを振り返り、finally ブロックで手動 close していた部分を try-with-resources に書き換えてみましょう。リソース解放をより安全・簡潔にする練習です。

手順

  1. 1 第3〜7章で作成したサンプルコードを1つ選ぶ。finally ブロックで close() を呼んでいるコードが対象。
  2. 2 try (Connection conn = ...; PreparedStatement ps = ...) { ... } の形式に書き換える。finally ブロックと手動 close を削除する。
  3. 3 書き換え前後でプログラムの動作が変わらないことを確認する。ResultSet も try-with-resources にネストして書けるか試してみよう。
  4. 4 （発展）8.4 のバッチ処理コードを参考に、addBatch() / executeBatch() を使って複数件の INSERT をまとめて実行してみよう。

理解度確認クイズ

  1. Q1. try-with-resources を使うと、finally ブロックで手動 close するコードと比べてどんな利点がありますか？

    回答を表示

    try の丸括弧内で宣言したリソースが正常終了でも例外が発生しても必ず自動 close されます。「閉じ忘れ」や「例外時に close が実行されない」バグを構造的に防げるため、コードが短くなり安全性が高まります。

  2. Q2. コネクションプーリングを使う主な理由は何ですか？また、Java での代表的な実装を1つ挙げてください。

    回答を表示

    データベース接続の作成はコストが高いため、あらかじめ複数の接続を作っておき使い回す（貸し出し・返却）ことで、接続コストを削減しパフォーマンスを向上させます。代表的な実装は HikariCP（または Apache DBCP）です。

  3. Q3. 「No suitable driver found」というエラーが発生しました。考えられる原因と対処法を述べてください。

    回答を表示

    主な原因は、① JDBC ドライバの JAR がクラスパスに存在しない、または② 接続 URL の書式（jdbc:h2: 等のプレフィックス）が誤っている、のいずれかです。pom.xml / build.gradle のドライバ依存設定と接続 URL を再確認することで対処できます。
