実践的なDAO（Data Access Object）パターン

データベース処理を専用クラスに整理する DAO パターンの設計と実装を学びます。SQL とビジネスロジックを分離することで、変更に強く・再利用しやすく・テストしやすいコードを書く方法を実践します。

この章で学ぶこと

  • DAO パターンの目的と「関心の分離」のメリットを説明できる
  • テーブル1行を表すエンティティクラス（User）を作成できる
  • DAO インターフェースで「できること」を先に定義する設計ができる
  • JDBC（PreparedStatement）を使って DAO を実装できる

7.1 DAO パターンとは

これまでの章では、main メソッドや処理の流れの中に SQL を直接書いてきました。小さなプログラムではそれで問題ありませんが、アプリが大きくなると「どこで SQL を実行しているか分からない」「同じ SQL を何度も書く」「テストしづらい」という課題が出てきます。

DAO（Data Access Object）とは、「データベースへのアクセス処理を専用クラスにまとめる」設計パターンです。呼び出し側（サービス層・main）は「取得したい」「保存したい」とお願いするだけで、SQL の詳細は DAO クラスが引き受けます。

flowchart LR A["呼び出し側
（サービス/main）"] --> B["UserDao
（DAOインターフェース）"] B --> C["UserDaoImpl
（DAO実装クラス）"] C --> D[("データベース")]

この図のように、「呼び出し側 → DAO → データベース」の3層に分けることで、次のメリットが生まれます。

変更に強い

SQL を変えたいときは DAO クラスだけ直せばよく、呼び出し側を触る必要がありません。

再利用しやすい

同じ DB 処理を複数の画面や機能から呼ぶとき、DAO のメソッドを使い回せます。

テストしやすい

DAO をモックに差し替えることで、DB なしでロジックだけをテストできます。

用語: 関心の分離（Separation of Concerns）

「関心の分離」とは、ひとつのクラス・メソッドはひとつのことだけに責任を持つべき、という設計の考え方です。DAO パターンは「DB アクセス」という関心をひとつのクラスに分離することで、コードの見通しをよくします。

7.2 エンティティクラス

DAO パターンでは、テーブルの1行を表す エンティティクラス（値をまとめる入れ物）を用意します。users テーブルであれば User クラスです。テーブルの列（カラム）がそのままクラスのフィールドになります。

「1行 = 1オブジェクト」の対応関係を常に意識してください。ResultSet から取り出したデータを User オブジェクトに詰めることで、Java らしくデータを扱えます。

User.java（フィールドとコンストラクタ） 
public class User {
    // テーブルの列（カラム）がフィールドになる
    private int id;       // id 列
    private String name;  // name 列
    private String email; // email 列

    // 全フィールドを受け取るコンストラクタ
    public User(int id, String name, String email) {
        this.id    = id;
        this.name  = name;
        this.email = email;
    }
}
User.java（ゲッター） 
    // ゲッター: フィールドの値を外部から取得する
    public int getId()       { return id; }
    public String getName()  { return name; }
    public String getEmail() { return email; }

    @Override
    public String toString() {
        // 確認用: "User{id=1, name=佐藤, email=...}"
        return "User{id=" + id + ", name=" + name
             + ", email=" + email + "}";
    }

ヒント: セッタ―も必要なケース

INSERT 後に DB が自動採番した id を後からセットしたい場合などに setId() のようなセッターを追加します。必要な分だけ作ればよく、すべてのフィールドに必須ではありません。

7.3 DAO インターフェース

DAO では、実装クラスを作る前に インターフェースで「何ができるか」を先に宣言します。「何ができるか（契約）」と「どう実装するか（詳細）」を分けることで、将来実装を差し替えても呼び出し側を変えずに済みます。

UserDao インターフェースには、基本的な CRUD（作成・読取・更新・削除）操作を5つのメソッドとして宣言します。

UserDao.java（インターフェース） 
import java.util.List;

public interface UserDao {
    // 全件取得
    List<User> findAll();
    // 1件取得（id 指定）
    User findById(int id);
    // 新規登録
    void insert(User user);
    // 更新
    void update(User user);
    // 削除（id 指定）
    void delete(int id);
}

ポイント: 「何ができるか」を先に定義する

インターフェースを先に作ると、チームの別メンバーが実装する前でも「このメソッドを呼べばデータが取れる」という約束が決まります。呼び出し側とDB処理の実装を並行して進められるため、大規模な開発でも効率が上がります。

7.4 DAO の実装

UserDaoImpl クラスが UserDao インターフェースを implements して、JDBC（PreparedStatement）で各メソッドを実装します。ここでは代表として findById と insert の2つを詳しく見ていきます。

UserDaoImpl.java（クラス宣言と mapRow） 
import java.sql.*;

public class UserDaoImpl implements UserDao {
    // 接続 URL を保持する（コンストラクタなどで受け取ってもよい）
    private final String url;

    public UserDaoImpl(String url) { this.url = url; }

    // ResultSet の1行を User オブジェクトに変換する共通処理
    private User mapRow(ResultSet rs) throws SQLException {
        return new User(
            rs.getInt("id"),       // id 列 → int
            rs.getString("name"),  // name 列 → String
            rs.getString("email")  // email 列 → String
        );
    }
}

mapRow メソッドは ResultSet の1行を User に変換する処理です。同じロジックを各メソッドで書かずに済むよう、共通の private メソッドにまとめるのがコツです。

UserDaoImpl.java（findById の実装） 
    @Override
    public User findById(int id) {
        String sql = "SELECT id, name, email FROM users WHERE id = ?";
        try (Connection conn = DriverManager.getConnection(url);
             PreparedStatement ps = conn.prepareStatement(sql)) {
            ps.setInt(1, id);   // ? に id をバインド
            try (ResultSet rs = ps.executeQuery()) {
                if (rs.next()) {
                    return mapRow(rs); // 1行 → User に変換
                }
            }
        } catch (SQLException e) {
            throw new RuntimeException("findById 失敗: id=" + id, e);
        }
        return null; // 該当なしの場合
    }

findById() の戻り値の方針

findById() は該当データがない場合 null を返します。そのため呼び出し側では if (user != null) で確認してから利用しましょう。確認せずに使うと NullPointerException の原因になります。

発展として、「見つからないかもしれない」ことを型で表現する Optional<User> を戻り値にする設計もあります（return Optional.ofNullable(...)）。呼び出し側で null チェックの漏れを防ぎやすくなります。

UserDaoImpl.java（insert の実装） 
    @Override
    public void insert(User user) {
        String sql = "INSERT INTO users (name, email) VALUES (?, ?)";
        try (Connection conn = DriverManager.getConnection(url);
             PreparedStatement ps = conn.prepareStatement(sql)) {
            ps.setString(1, user.getName());  // ? にname をバインド
            ps.setString(2, user.getEmail()); // ? にemail をバインド
            ps.executeUpdate();               // INSERT を実行
        } catch (SQLException e) {
            throw new RuntimeException("insert 失敗", e);
        }
    }

ポイント: try-with-resources で確実にリソース解放

try (Connection conn = ... の形は try-with-resources といい、ブロックを抜けると自動で close() が呼ばれます。第6章で学んだとおり、DAO の実装でも必ず使うようにしましょう。

実務での発展: URL ではなく DataSource を渡す

本章では学習のため、DAO に接続 URL（文字列）を渡して毎回 DriverManager.getConnection(url) しています。実務では、コンストラクタで javax.sql.DataSource を受け取り、dataSource.getConnection() で接続を得る設計が一般的です。こうするとコネクションプール（第8章）を活かせ、接続先の設定も DAO の外側に追い出せます。

7.5 呼び出し側がすっきりする

DAO を使うと、main やサービス層からは SQL を一切書かずにデータベース操作ができます。呼び出し側は「何をしたいか」だけを書けばよく、コードの意図が一目で分かるようになります。

Main.java（DAO を使った呼び出し例） 
public class Main {
    public static void main(String[] args) {
        String url = "jdbc:h2:~/testdb";
        // DAO はインターフェース型で受け取る
        UserDao dao = new UserDaoImpl(url);

        // 1件取得 — SQL を意識しなくてよい
        User user = dao.findById(1);
        // findById は該当データがないと null を返すので確認してから使う
        if (user != null) {
            System.out.println(user); // User{id=1, name=佐藤, ...}
        } else {
            System.out.println("id=1 のユーザーは見つかりませんでした");
        }

        // 新規登録
        dao.insert(new User(0, "田中", "tanaka@example.com"));
    }
}

重要ポイント: SQL は DAO に閉じ込める

呼び出し側（main・サービス層）には SQL を一切書かないのが DAO パターンの鉄則です。SQL の変更や修正が必要になったとき、DAO クラスだけ直せばよい状態を保ちましょう。「SQL は DAO の中」と覚えてください。

flowchart LR A["Main/サービス層
（SQLを書かない）"] -->|"dao.findById(1)"| B["UserDao
（インターフェース）"] B -->|"implements"| C["UserDaoImpl
（SQLはここだけ）"] C -->|"PreparedStatement"| D[("データベース")] D -.結果.-> C C -.Userオブジェクト.-> A

やってみよう 7-1: findAll() を自分で実装する

UserDaoImpl に findAll() メソッドを追加して、users テーブルの全件を取得して表示してみましょう。

手順

  1. 1 UserDaoImpl に findAll() を追加する。SQL は SELECT id, name, email FROM users。
  2. 2 ResultSet を while (rs.next()) でループし、mapRow(rs) で User を作成して List に追加する。
  3. 3 Main.java から dao.findAll() を呼び出し、ループで全件を System.out.println() で表示する。
  4. 4 事前に insert() で数件登録しておき、全件が表示されることを確認する。

実装のヒント

List<User> list = new ArrayList<>(); を作り、ループで list.add(mapRow(rs)); してから return list; するのが基本の書き方です。

理解度確認クイズ

  1. Q1. DAO パターンの目的は何ですか？ひとことで説明してください。

    回答を表示

    データベースへのアクセス処理を専用クラス（DAO）にまとめ、ビジネスロジックと SQL を分離することで、変更に強く・再利用しやすく・テストしやすいコードにすることです。

  2. Q2. エンティティクラスとは何ですか？また、どのような対応関係がありますか？

    回答を表示

    テーブルの1行（レコード）を Java オブジェクトとして表すクラスです。テーブルの列（カラム）がフィールドに対応し、「1行 = 1オブジェクト」の関係になります。users テーブルなら User クラスが対応します。

  3. Q3. DAO クラスに SQL を閉じ込める（呼び出し側に SQL を書かない）利点は何ですか？

    回答を表示

    SQL を変更したいときに DAO クラスだけを直せばよく、呼び出し側（main・サービス層）には一切変更が不要です。また、SQL が一箇所に集まるので管理しやすく、同じ処理を複数の場所から再利用できます。
