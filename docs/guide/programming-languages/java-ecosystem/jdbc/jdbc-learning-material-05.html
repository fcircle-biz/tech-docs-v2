プリペアドステートメント（PreparedStatement）の活用

SQL インジェクション対策と処理速度向上のための PreparedStatement の使い方を詳しく学びます。プレースホルダ（?）の使い方、パラメータのバインド方法、Statement との違いを理解して、より安全で効率的なデータベースプログラミングを身につけましょう。

この章で学ぶこと

  • Statement で文字列連結をする危険な書き方と、その問題点を理解する
  • SQL インジェクション攻撃の仕組みと与える被害を理解する
  • PreparedStatement のプレースホルダ（?）とパラメータバインドの基本を習得する
  • PreparedStatement が SQL インジェクションを防ぐ仕組みを説明できる
  • Statement と PreparedStatement の違いを整理し、使い分けの基準を理解する

5.1 Statement の問題点 — 文字列連結の危険な書き方

第3章・第4章で学んだ Statement を使った書き方では、ユーザーが入力した値を SQL 文に「文字列として貼り付ける」ことがよくあります。たとえば、名前で会員を検索するプログラムを次のように書いたとします。

危険な書き方（文字列連結） 
// ユーザーが入力した名前を受け取る
String input = getInputFromUser(); // 例: "佐藤 太郎"

// ★危険: ユーザー入力をそのまま SQL 文字列に連結している
String sql = "SELECT * FROM users WHERE name = '" + input + "'";

Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(sql); // 危険な SQL を実行してしまう

入力が正しい名前であれば問題なく動作します。しかし、ユーザーが意図的に悪意ある文字列を入力したら、どうなるでしょうか？ SQL の文字列連結は、入力内容をそのまま SQL 文の「命令の一部」として解釈してしまいます。これが、次節で学ぶ「SQL インジェクション」の根本的な原因です。

用語: 文字列連結

Java の + 演算子などで複数の文字列をつなぎ合わせることを「文字列連結」といいます。"... WHERE name = '" + input + "'" のように、変数の値を SQL 文字列の途中に埋め込む書き方です。直感的に書けますが、セキュリティ上の大きなリスクがあります。

5.2 SQL インジェクションとは

SQL インジェクション（SQL Injection）とは、悪意ある入力を SQL 文の「命令の一部」として紛れ込ませる攻撃です。「インジェクション（injection）」は「注入」という意味で、文字列連結を利用して、本来は「データ」として扱われるべき入力を、SQL の「命令」として注入します。

たとえば、先ほどの危険な書き方で、ユーザーが名前として ' OR '1'='1 という文字列を入力したとします。連結後の SQL はどうなるでしょうか？

SQL インジェクションの例 
-- 入力: ' OR '1'='1 を連結すると…
-- SELECT * FROM users WHERE name = '' OR '1'='1'
--
-- '1'='1' は常に真（TRUE）なので、
-- WHERE 句全体が常に TRUE になり全件取得されてしまう！

危険: SQL インジェクションで起こりうる被害

  • 全件データ漏洩 — 認証をすり抜けてすべてのユーザー情報を取得される
  • データ改ざん・削除 — UPDATE や DELETE を注入してデータを書き換えられる
  • ログインバイパス — パスワードを知らずに管理者としてログインされる

SQL インジェクションはウェブアプリの攻撃手法の中でも特に危険度が高いものです。必ず PreparedStatement で対策しましょう。

SQL インジェクションの仕組みを図で確認しましょう。「入力値」が「データ」ではなく「SQL の命令」として解釈される流れがポイントです。

flowchart TD A["ユーザー入力
' OR '1'='1"] --> B["文字列連結
WHERE name = '' OR '1'='1'"] B --> C{"SQL として解釈される"} C -->|"条件が常に TRUE"| D["全件取得される
（データ漏洩）"] C -->|"DROP TABLE など"| E["データ破壊
（最悪のケース）"]

5.3 PreparedStatement の基本 — プレースホルダ（?）の使い方

PreparedStatement を使うと、SQL 文の「値を埋め込む場所」を ?（プレースホルダ）として先に確定させ、あとから値だけを「データ」として安全にバインド（結びつける）できます。ユーザーの入力が SQL の「命令」として解釈されることはありません。

PreparedStatement の基本的な使い方 
// ユーザー入力（悪意ある文字列が来ても安全）
String input = getInputFromUser();

// 1. SQL は ? でプレースホルダを使って先に確定させる
String sql = "SELECT * FROM users WHERE name = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);

// 2. ? に値をバインドする（第1引数: ? のインデックス、第2引数: 値）
pstmt.setString(1, input); // 1番目の ? に input を「データ」として渡す

// 3. 実行する（executeQuery に SQL 文字列を渡さない点に注目）
ResultSet rs = pstmt.executeQuery();

? のインデックスは 1 始まり

Java の配列は 0 始まりですが、PreparedStatement の setXxx() メソッドで指定する ? の番号（インデックス）は 1 始まりです。SQL 文の左から何番目の ? かを数えて指定します。? が複数ある場合は setString(1, ...)・setInt(2, ...) のように順に指定します。

主なバインドメソッドは次のとおりです。Java の型に合わせて使い分けます。

メソッド Java の型 対応する SQL 型の例
setString(1, value) String VARCHAR, TEXT など
setInt(1, value) int INT, INTEGER など
setLong(1, value) long BIGINT など
setDouble(1, value) double DOUBLE, FLOAT など
setBoolean(1, value) boolean BOOLEAN など
setNull(1, Types.VARCHAR) null 値 NULL を明示的にセット

INSERT など、複数のパラメータが必要な場合も同様に ? を複数使えます。

複数のプレースホルダの例（INSERT） 
// 複数の ? には左から順にインデックス 1, 2, 3 ... を割り当てる
String sql = "INSERT INTO users (name, email, age) VALUES (?, ?, ?)";
PreparedStatement pstmt = conn.prepareStatement(sql);

pstmt.setString(1, "田中 次郎");      // 1番目 ? = name
pstmt.setString(2, "jiro@example.com"); // 2番目 ? = email
pstmt.setInt(3, 25);                   // 3番目 ? = age

int count = pstmt.executeUpdate(); // 挿入された行数が返る

5.4 なぜ安全なのか — 入力は常に「データ」として扱われる

PreparedStatement が安全な理由は、SQL 文の「構造」と「値」を完全に分離する仕組みにあります。SQL 文は prepareStatement() の時点でデータベースにあらかじめ送られ、解析・コンパイルされます。あとから setString() などで渡す値は、「データ（文字列そのもの）」として扱われるだけで、SQL 命令として解釈されることは絶対にありません。

flowchart TD A["prepareStatement()
SQL の構造を先に確定"] --> B["データベースが
SQL をコンパイル・解析"] B --> C["setString（1, input）
値を「データ」として渡す"] C --> D{"input が
' OR '1'='1 でも…"} D --> E["「name = ' OR '1'='1'」という
文字列データとして検索される"] E --> F["SQL 命令としては
解釈されない
→ 攻撃が成立しない"]

重要ポイント: なぜ PreparedStatement は安全か

  • SQL の構造はコンパイル時に確定 — prepareStatement() の段階でデータベースが SQL を解析します。この後から変えられるのは「値（データ）」だけです。
  • 値は純粋な「データ」として扱われる — setString() で渡した文字列に ' や SQL キーワードが含まれていても、SQL 命令としては絶対に解釈されません。
  • エスケープ処理を自動で行う — JDBC ドライバが値を適切にエスケープするため、プログラマが手動でエスケープする必要がありません。

5.5 速度面の利点と Statement との使い分け

PreparedStatement はセキュリティ面だけでなく、処理速度の面でも有利です。同じ SQL 文を繰り返し実行する場合、PreparedStatement は最初の prepareStatement() 呼び出し時に SQL を「プリコンパイル（事前解析）」し、実行計画をデータベース側でキャッシュします。2回目以降は SQL のコンパイルをスキップするため、高速に実行できます。

flowchart LR subgraph S["Statement（毎回コンパイル）"] direction TB S1["SQL文字列を連結"] --> S2["SQLを送信"] S2 --> S3["DB がコンパイル"] S3 --> S4["実行"] end subgraph P["PreparedStatement（プリコンパイル）"] direction TB P1["prepareStatement()
でコンパイル済み"] --> P2["値をバインドして送信"] P2 --> P3["DB はキャッシュを使う"] P3 --> P4["高速実行"] end

特に、ループの中で同じ INSERT を何百回も繰り返す処理などでは、PreparedStatement の速度優位が顕著に現れます。Statement と PreparedStatement の違いをまとめると次のとおりです。

観点 Statement PreparedStatement
SQL の指定方法 実行時に SQL 文字列を渡す 事前に SQL を確定し、? で値をバインド
セキュリティ 文字列連結で SQL インジェクションの危険あり SQL インジェクションを防げる
速度（繰り返し実行） 毎回 SQL をコンパイルするため遅い プリコンパイルで高速（実行計画を再利用）
使い分けの結論 ユーザー入力を扱わない・使い捨ての場合のみ 基本は常にこちらを使う

結論: 基本は常に PreparedStatement を使う

Statement が有効なのは、ユーザー入力を一切含まない固定の SQL を一度だけ実行する場合に限られます。ユーザー入力・外部から来る値・動的に変わる値が絡む場合は、迷わず PreparedStatement を使いましょう。 セキュリティと速度の両方でメリットがあります。

やってみよう 5-1: 危険な書き方と安全な書き方を比べる

文字列連結の危険な版と PreparedStatement の安全な版を実際に書き比べ、悪意ある入力を与えたときの挙動の違いを体感しましょう。

手順

  1. 1 H2 データベースに users テーブル（id, name, email 列）を作成し、数件のテストデータを INSERT しておく。
  2. 2 【危険な版】Statement と文字列連結で名前検索するプログラムを書く。入力として ' OR '1'='1 を与えて実行し、何件取得されるか確認する。
  3. 3 【安全な版】同じ検索を PreparedStatement と setString() で書き直す。同じ入力 ' OR '1'='1 を与えて実行し、0件（または一致する件数のみ）になることを確認する。
  4. 4 なぜ PreparedStatement では攻撃が成立しないのか、自分の言葉で一言まとめてみる。

ヒント

危険な版で全件取得されてしまった場合、そのまま次のステップへ進みましょう（本番サービスでは絶対に使わないこと）。安全な版で 0 件になれば、PreparedStatement が正しく機能している証拠です。

理解度確認クイズ

  1. Q1. PreparedStatement で使う ?（プレースホルダ）の役割を説明してください。

    回答を表示

    SQL 文の中で「あとから値を入れる場所（プレースホルダ）」を表す記号です。SQL の構造をあらかじめ確定させておくことで、後から渡す値が SQL 命令として解釈されるのを防ぎます。setString() などのメソッドでバインドした値は、純粋な「データ」として扱われます。

  2. Q2. pstmt.setString(2, "test@example.com") の第1引数「2」は何を意味しますか？

    回答を表示

    SQL 文の左から数えて「2番目の ?」を指定するインデックスです。PreparedStatement のインデックスは 1 始まりであることに注意してください。この場合、2番目の ? に文字列 "test@example.com" をデータとしてバインドします。

  3. Q3. PreparedStatement を使うと SQL インジェクションを防げるのはなぜですか？

    回答を表示

    SQL の「構造（命令の形）」は prepareStatement() の時点でデータベースにプリコンパイルされ、確定しているためです。あとから setString() などで渡す値は「データ（文字列そのもの）」としてのみ扱われ、どんな文字列が渡されても SQL 命令として解釈されることはありません。そのため、' OR '1'='1 のような悪意ある文字列が入力されても、それは「名前データ」として検索されるだけで攻撃は成立しません。
