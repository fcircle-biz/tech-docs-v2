トランザクション処理と例外処理

データベース操作では「途中で失敗したら全部なかったことにしたい」という場面が必ず出てきます。この章では、複数の処理をまとめて成功か失敗かを制御するトランザクション、自動コミットの止め方、commit／rollback の使い方、そして JDBC で欠かせない例外処理とリソース管理の基礎を学びます。

この章で学ぶこと

  • トランザクションの概念と「全部成功か全部失敗」の意味を説明できる
  • setAutoCommit(false) で自動コミットを止める方法を知る
  • commit() と rollback() を使ったトランザクション制御コードを書ける
  • SQLException を catch で受け取り、原因を確認する方法を知る
  • finally でリソースを確実に閉じる書き方を理解する

6.1 トランザクションとは

トランザクション（Transaction）とは、「複数の処理をひとまとめにして、全部成功するか、全部なかったことにするか、どちらかにする」仕組みです。途中で失敗した場合は、すでに実行した処理もすべて取り消します。

代表的な例が銀行の振込です。A さんの口座から 10,000 円を引いて（① 残高を減らす）、B さんの口座に 10,000 円を足す（② 残高を増やす）という2つの操作をセットで行います。もし①が成功して②が失敗したら、A さんのお金は消えてしまいます。これを防ぐために、①と②を1つのトランザクションにまとめ、「どちらかが失敗したら①も取り消す」と決めておくのです。

flowchart TD A(["トランザクション開始"]) --> B["① Aの残高を引く
（UPDATE）"] B --> C["② Bの残高を足す
（UPDATE）"] C --> D{"全部成功?"} D -->|"はい"| E["commit（確定）"] D -->|"いいえ"| F["rollback（全部取り消し）"] E --> G(["完了"]) F --> G

このように、トランザクションは「途中でやめられない一連の操作をひとまとまりにする枠組み」です。データベースの世界では、このトランザクションの性質を ACID 特性（原子性・一貫性・独立性・永続性）と呼びますが、まずは「全部成功か全部取り消しか」というイメージをしっかりつかんでおきましょう。

用語: ACID 特性

Atomicity（原子性）: 全部成功か全部取り消し。Consistency（一貫性）: データが矛盾しない状態を保つ。Isolation（独立性）: 複数のトランザクションが互いに干渉しない。Durability（永続性）: 確定したデータは障害後も消えない。本章では特に原子性（A）を中心に学びます。

6.2 自動コミットを止める

JDBC はデフォルトで自動コミット（auto-commit）モードが有効になっています。自動コミットとは、SQL を1文実行するたびに自動的に commit（確定）が行われる設定です。1件の INSERT や UPDATE だけを実行する場合はこれで問題ありませんが、「複数の SQL をまとめて1つの処理にしたい」という場面では困ります。

複数の SQL をトランザクションで制御するには、Connection の setAutoCommit(false) を呼んで、自動コミットをオフにする必要があります。こうすると、commit() を呼ぶまで変更は確定されず、rollback() を呼べば全部取り消せます。

設定 挙動 用途
setAutoCommit(true) 1文ごとに自動 commit（JDBC 初期値） 1件だけ更新する場合
setAutoCommit(false) 明示的に commit か rollback するまで確定しない 複数の SQL をまとめるとき（強調行）
自動コミットをオフにする 
Connection conn = DriverManager.getConnection(url, user, pass);

// 自動コミットをオフにする（デフォルトは true）
conn.setAutoCommit(false);

// この後の SQL は commit() を呼ぶまで確定されない

6.3 commit と rollback

自動コミットをオフにしたら、処理の結果に応じて commit() か rollback() を呼びます。両方とも Connection のメソッドです。

conn.commit()

全部成功したときに呼ぶ。変更をデータベースに確定します。

conn.rollback()

例外が起きたときに呼ぶ。変更をすべて取り消して元に戻します。

以下は、銀行振込を模した最小のトランザクションコードです。2件の UPDATE をまとめ、両方成功したら commit、例外が起きたら catch で rollback します。

TransferExample.java（抜粋） 
conn.setAutoCommit(false); // 自動コミットをオフ
try {
    // ① A の残高を減らす
    pstmt1.executeUpdate();
    // ② B の残高を増やす
    pstmt2.executeUpdate();
    conn.commit(); // 全部成功 → 確定
} catch (SQLException e) {
    conn.rollback(); // 失敗 → 全部取り消し
}

重要ポイント

rollback() は catch ブロック内で呼びます。commit 前に例外が発生したのに rollback を呼ばないと、トランザクションが未完了のまま残ります。その間は変更しようとした行にロックがかかったままになり、ほかの処理が待たされたり、その接続を占有し続けたりします。さらに、接続が閉じられるとき DB 側が暗黙的に rollback して、意図しないタイミングで変更が取り消されることもあります。必ず catch で明示的に rollback するクセをつけましょう。

6.4 例外処理（SQLException）

JDBC のすべての操作は SQLException（チェック例外）を投げる可能性があります。チェック例外とは、コンパイラが「ちゃんと対処しなさい」と強制する例外で、try-catch で囲むか、メソッドに throws を宣言しないとコンパイルが通りません。

SQLException をキャッチすると、失敗の原因を詳しく調べられます。代表的なメソッドを覚えておきましょう。

メソッド 返す内容 使いどころ
getMessage() エラーメッセージの文字列 ログ・デバッグ表示に使う
getSQLState() SQL状態コード（5桁文字列） エラー種別の特定に使う
getErrorCode() DBMS固有のエラー番号 DB特有の詳細原因の確認
SQLException のキャッチと情報取得 
try {
    // データベース操作
    pstmt.executeUpdate();
} catch (SQLException e) {
    // エラーの詳細を確認する
    System.out.println("エラー: " + e.getMessage());
    System.out.println("SQLState: " + e.getSQLState());
    conn.rollback(); // 変更を取り消す
}

SQLState とは？

SQLState は SQL 標準で定められた5桁のエラー分類コードです。先頭2文字がカテゴリを示し、たとえば 23 から始まれば「整合性制約違反（重複キーなど）」、08 なら「接続失敗」を意味します。ログに残しておくとトラブルシューティングに役立ちます。

6.5 リソース管理と finally

Connection・Statement・ResultSet は、使い終わったら必ず close() でリソースを解放しなければなりません。try ブロック内で例外が起きても確実に閉じるには、finally ブロックに close 処理を書くのが伝統的なパターンです。

finally でリソースを閉じる（伝統的な書き方） 
Connection conn = null;
try {
    conn = DriverManager.getConnection(url, user, pass);
    conn.setAutoCommit(false);
    // ... SQL 処理 ...
    conn.commit();
} catch (SQLException e) {
    if (conn != null) conn.rollback(); // 失敗時は取り消し
} finally {
    if (conn != null) conn.close();   // 必ず閉じる
}

finally ブロックは、try が正常終了しても例外で終了しても必ず実行されます。そのため、リソースの解放処理を書く場所として適しています。

注意: rollback() や close() 自体も SQLException を投げる

conn.rollback() や conn.close() もチェック例外 SQLException を投げます。上のコードは catch／finally の中でこれらを呼んでいるため、その例外を外側で受け止める必要があります。このまま main に書くと、メソッドに throws SQLException が無い場合はコンパイルエラーになります。

対処は2つです。① メソッド宣言に throws SQLException を付ける、② close() などをさらに try-catch で囲む（下の例）。次章で学ぶ try-with-resources を使えば、この煩雑さの大半を避けられます。

close() の例外を finally 内で処理する例 
} finally {
    if (conn != null) {
        try {
            conn.close();            // close() も SQLException を投げる
        } catch (SQLException ex) {
            ex.printStackTrace();    // finally 内でさらに try-catch して処理する
        }
    }
}

重要ポイント: 閉じ忘れ＝接続リーク

Connection を閉じ忘れると、データベースとの接続が残り続けます（接続リーク）。データベースが受け付けられる接続数には上限があるため、リークが積み重なると「これ以上接続できない」状態になりアプリが停止します。必ず finally または後の章で学ぶ try-with-resources で閉じること。

次のステップへ: try-with-resources

上記の finally を使った書き方はやや冗長です。Java 7 以降では try-with-resources という構文を使うと、close() を自動で呼んでくれるためより簡潔に書けます。この書き方は第8章「ベストプラクティスとトラブルシューティング」で詳しく学びます。

動作確認: ロールバックを確かめる

トランザクションが正しく機能しているか確認するには、わざと途中で例外を発生させてみましょう。ロールバックが成功していれば、1件目の変更も取り消されてデータが元のまま残ります。

コンソール出力例（ロールバック成功時） 
① A の残高を引く → 成功
② B の残高を足す → 例外発生（意図的）
エラー: 意図的な例外
rollback を実行しました
確認: A の残高は元のまま 10000 円

「A の残高は元のまま」と表示されれば、ロールバックによって①も取り消されたことが確認できます。

やってみよう 6-1: 2件更新をトランザクションにまとめる

2件の UPDATE を1つのトランザクションにまとめ、2件目でわざと例外を起こしてロールバックを確かめましょう。1件目も取り消されることがポイントです。

手順

  1. 1 conn.setAutoCommit(false) を呼んで自動コミットをオフにする。
  2. 2 1件目の UPDATE（例: id=1 の name を変更）を executeUpdate() で実行する。
  3. 3 2件目の UPDATE の直前に throw new SQLException("意図的な例外") を挿入して例外を起こす。
  4. 4 catch ブロックで conn.rollback() を呼ぶ。
  5. 5 ロールバック後に SELECT で id=1 のデータを確認し、name が変更前のままであることを確かめる。

確認ポイント: 1件目の UPDATE が「なかったことになっている」ならトランザクションの取り消しは成功です。もし変更が残っている場合は、setAutoCommit(false) が呼ばれているか、rollback が catch 内で呼ばれているかを見直しましょう。

理解度確認クイズ

  1. Q1. setAutoCommit(false) を呼ぶ理由は何ですか？

    回答を表示

    JDBC はデフォルトで1文ごとに自動 commit されます。複数の SQL をまとめて「全部成功か全部取り消し」にするには、自動コミットをオフにして明示的に commit() か rollback() を呼ぶ必要があるためです。

  2. Q2. rollback() の役割を説明してください。

    回答を表示

    トランザクション開始後にコミット前に実行した SQL による変更をすべて取り消し、データをトランザクション開始前の状態に戻す役割です。例外が発生した catch ブロック内で呼び、データの不整合を防ぎます。

  3. Q3. JDBC で SQLException をチェック例外として扱うのはなぜですか？また、getMessage() と getSQLState() の違いは何ですか？

    回答を表示

    データベース操作はネットワーク断・ロック競合など予測できないエラーが起こりやすいため、コンパイラが「必ず対処しなさい」と強制するチェック例外として設計されています。getMessage() は人が読める自然言語のエラーメッセージを返し、getSQLState() は SQL 標準で定義された5桁のエラー分類コードを返します。ログにはどちらも残しておくとトラブルシューティングに役立ちます。
