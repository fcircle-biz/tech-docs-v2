Repositoryインターフェース

前章でエンティティクラス（User）を作りました。エンティティはデータの「形」を表すクラスでしたが、実際にデータベースへ保存・取得するには「入口」となる仕組みが必要です。その入口が Repository（リポジトリ） です。この章では、Spring Data JDBC の中心的な仕組みである CrudRepository を使って、データベースとやり取りする Repository を作る方法を学びます。

この章で学ぶこと

  • Repository の役割を理解する
  • CrudRepository の基本を理解する
  • 基本的な CRUD メソッドを理解する

5.1 Repository とは

Repository（リポジトリ） は、データベースとやり取りするための入口です。「repository」は英語で「倉庫・保管庫」を意味し、データを保管するデータベースへのアクセスを一手に引き受ける部品です。

前章で作った User エンティティは、テーブルの「1行分の形」を表す Java クラスでした。しかし、エンティティ自体はデータを保存・取得する能力を持っていません。Repository を通じてはじめて、User をデータベースに保存したり、データベースから取り出したりできるようになります。

（usersテーブル）")] D["Entity: User
（テーブルの1行を表す）"] -.データの形.-> B

第1章で学んだ3層構造を思い出してください。Service が「何をするか」を決め、Repository が「データベースとやり取りする」役割を担います。Repository は SQL を自動で生成・実行 してくれるため、私たちは SQL を書かずに済みます。

ひとことでいうと

Repository は、データベースとやり取りするための入口です。エンティティが「データの形」なら、Repository は「データの出し入れをする窓口」と考えると分かりやすいです。

5.2 CrudRepository を継承する

Spring Data JDBC では、CrudRepository を継承したインターフェースを作るだけで、基本的な CRUD 操作が使えるようになります。自分でメソッドを実装（コードを書く）する必要はありません。

以下が、users テーブルを操作するための Repository です。シンプルですが、これだけで登録・取得・更新・削除のすべてが使えます。

UserRepository.java 
import org.springframework.data.repository.CrudRepository;

// User エンティティを Long 型の主キーで扱う Repository
// CrudRepository を継承するだけで CRUD メソッドが使える
public interface UserRepository extends CrudRepository<User, Long> {
    // メソッドを1つも書かなくてよい — Spring が自動で用意してくれる
}

このコードのポイントを整理しましょう。

  • UserRepository は、users テーブルを操作するための Repository です。
  • CrudRepository<User, Long> を継承 します。User は「扱うエンティティの型」、Long は「主キー id の型」を表します。
  • メソッドを1つも書かなくても、基本的な CRUD 操作が使えます。これが Spring Data JDBC の便利な点です。

なぜインターフェースなのに実装を書かなくていいの？

通常、Java のインターフェースは「設計図」なので、使うためには実装クラスが必要です。しかし Spring Data JDBC では、Spring が起動時に自動で実装クラスを作り、Bean（Springが管理するオブジェクト）として用意してくれます。そのため私たちはインターフェースを宣言するだけでよく、実装コードを1行も書く必要がありません。DI（依存性の注入）によって、この自動生成された実装が必要な場所へ渡されます。

5.3 CrudRepository で使える主なメソッド

CrudRepository を継承するだけで、以下のメソッドがすぐに使えるようになります。これらのメソッドを呼ぶと、Spring が裏側で適切な SQL を自動生成して実行してくれます。

メソッド 内容
findAll() 全件取得
findById(id) IDで1件取得
save(entity) 登録または更新
deleteById(id) IDで削除
count() 件数取得
existsById(id) IDの存在確認

save() は「登録」と「更新」の両方を兼ねる

save(entity) は、渡したエンティティに id が設定されていなければ新規登録（INSERT）、id がすでに存在する値であれば更新（UPDATE）を行います。1つのメソッドで登録と更新の両方を扱えるため、呼び出す側がシンプルになります。詳しい挙動は第6章・第8章で実際に試します。

これらのメソッドを実際に使うコードのイメージを見てみましょう。今はコードの詳細より「こんな風にシンプルに書ける」という感覚をつかんでください。

CrudRepository のメソッド使用例（イメージ） 
// 全件取得（SELECT * FROM users）
Iterable<User> users = userRepository.findAll();

// IDで1件取得（SELECT * FROM users WHERE id = 1）
Optional<User> user = userRepository.findById(1L);

// 新規登録（INSERT INTO users ...）
User newUser = new User(null, "田中太郎", "tanaka@example.com");
userRepository.save(newUser);

// IDで削除（DELETE FROM users WHERE id = 2）
userRepository.deleteById(2L);

5.4 継承関係を確認する

UserRepository と CrudRepository の関係を図で整理しましょう。UserRepository は CrudRepository を継承しており、CrudRepository が持つすべての CRUD メソッドをそのまま使えます。

flowchart TD A["CrudRepository
（UserとLong型を指定）
Spring Data が提供する基底インターフェース"] B["UserRepository
（私たちが作るインターフェース）"] B -->|継承 extends| A C["CRUD メソッド群
findAll / findById / save / deleteById / count / existsById"] A --- C

UserRepository は CrudRepository を継承（extends）するだけで、すべての CRUD メソッドを手に入れます。自分で SQL を書いたり、メソッドを実装したりする必要はありません。

重要ポイント

  • CrudRepository を継承するだけで CRUD メソッドが手に入る
  • 自分で SQL を書かなくてよい — Spring が自動で SQL を発行する
  • メソッドの実装コードも書かなくてよい — Spring が起動時に自動生成する

理解度確認クイズ

  1. Q1. Repository は何を担当しますか？

    回答を表示

    データベースアクセス（DBとのやり取り）を担当します。データの出し入れをする「入口（窓口）」の役割を果たします。

  2. Q2. CrudRepository を使うメリットは何ですか？

    回答を表示

    メソッドを自分で書かなくても基本的な CRUD 操作（findAll・findById・save・deleteById など）が使えます。SQL も接続管理も Spring が自動で行うため、コードがシンプルになります。

  3. Q3. CrudRepository<User, Long> の User と Long は何を表しますか？

    回答を表示

    User は「扱うエンティティの型」（このRepositoryが操作する対象のクラス）を表し、Long は「主キー id の型」を表します。
