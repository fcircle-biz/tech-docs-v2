Spring Data JDBCとは

Spring Data JDBC は、Spring Boot アプリケーションからデータベースを「Spring らしくシンプルに」操作するための仕組みです。この章では、そもそも Spring Data JDBC とは何か、通常の JDBC とどう違うのか、そして Entity・Repository・Service という登場人物がどう連携するのか、という全体像をつかみます。コードを書く前に、まずは「考え方」をしっかり理解しましょう。

この章で学ぶこと

  • Spring Data JDBC が「何のための仕組み」なのかを説明できる
  • 通常の JDBC と Spring Data JDBC の違いを理解する
  • CRUD という言葉の意味を理解する
  • Entity・Repository・Service の役割分担（全体像）をつかむ

1.1 Spring Data JDBC とは何か

Spring Data JDBC は、Java からデータベースを操作するためのコードを、Spring らしくシンプルに書けるようにする仕組みです。Spring Data JDBC は「Spring Data」という大きなファミリーの一員で、その中でも素直で分かりやすいのが特徴です。データベースを「Excel のような表（テーブル）」だとイメージすると、Spring Data JDBC は「その表に、Java のコードから簡単にデータを出し入れするための道具」だと考えてください。

前教材「Spring 基礎」では、DI（依存性の注入）や Bean、@Service・@Repository といった考え方を学びました。本教材では、その土台の上に「データベース操作」を乗せていきます。つまり、これまで「処理を整理する」ために使ってきた Service と Repository が、いよいよ本物のデータベースとつながります。

用語: データベースと SQL

データベースは、アプリが扱うデータ（会員情報・注文履歴など）を整理して保存しておく「データの保管庫」です。よく使われるのは表（テーブル）の形でデータを管理する RDB（リレーショナルデータベース）で、その表に対して「このデータが欲しい」「この値を更新して」と指示する専用の言語が SQL です。Spring Data JDBC は、基本的な操作であればこの SQL を私たちの代わりに用意してくれます。

1.2 通常の JDBC では何をしていたか

Spring Data JDBC のありがたみを理解するために、まずは「素の JDBC（通常の JDBC）」を思い出してみましょう。JDBC は Java からデータベースを操作するための標準の仕組みですが、自分で書かなければならないことがたくさんあります。たとえば次のような作業です。

  • 接続の管理 — データベースへ接続（Connection）し、使い終わったら閉じる。
  • SQL の記述と実行 — SELECT や INSERT の SQL 文を自分で書いて実行する。
  • 結果の取り出し — 検索結果（ResultSet）から1件ずつ値を取り出し、Java オブジェクトに詰め替える。

これらは1つ1つは難しくありませんが、データを扱うたびに同じような「決まりきったコード」を書くことになり、コードが長くなりがちです。下のコードは、素の JDBC で全件取得をする「雰囲気」です（今は詳しく読まなくて大丈夫です）。

素の JDBC（イメージ・簡略版） 
// 接続して、SQL を書いて、結果を1件ずつ取り出す…をすべて自分で書く
Connection conn = DriverManager.getConnection(url, user, pass);
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT * FROM users");
while (rs.next()) {
    // 取り出した値を User オブジェクトへ詰め替える
    User u = new User(rs.getLong("id"), rs.getString("name"), rs.getString("email"));
}
// 使い終わったら閉じる（省略すると問題が起きる）

Spring Data JDBC を使うと、この「決まりきったコード」の多くを Spring に任せられます。次の節で、実際にどれだけシンプルになるのかを見ていきましょう。

1.3 JDBC と Spring Data JDBC の違い

Spring Data JDBC を使うと、Repository インターフェースを通じて、基本的な CRUD 処理を簡単に実装できます。SQL や接続の管理を Spring が肩代わりしてくれるため、私たちは「何をしたいか」に集中できます。両者の違いを表で整理しましょう。

項目 JDBC Spring Data JDBC
SQL 実行 自分で書く 基本的な CRUD は Repository で簡単に扱える
接続管理 自分で意識することが多い Spring が管理する
コード量 多くなりやすい 少なくしやすい
クラス設計 自分で整理する Entity・Repository・Service に分けやすい

参考までに、1.2 で見た「全件取得」を Spring Data JDBC で書くと、たった1行のメソッド呼び出しで済みます。SQL も接続管理も書きません。

Spring Data JDBC（イメージ） 
// findAll() を呼ぶだけで全件取得できる（SQL も接続管理も書かない）
Iterable<User> users = userRepository.findAll();

重要ポイント

Spring Data JDBC は JDBC を「置き換える」ものではなく、JDBC の上に乗って面倒な部分を肩代わりしてくれる仕組みです。私たちは「メソッドを呼ぶ」だけで基本的なデータベース操作ができ、その分、本来やりたい処理に集中できます。

1.4 CRUD とは

データベース操作の話で必ず出てくるのが CRUD（クラッド）という言葉です。これは、データに対する4つの基本操作の頭文字をつなげたものです。アプリがデータを扱うとき、結局はこの4つの組み合わせでできています。

頭文字 英語 意味 対応する SQL
C Create 登録（作る） INSERT
R Read 参照（読む） SELECT
U Update 更新（直す） UPDATE
D Delete 削除（消す） DELETE

本教材のゴールは、この CRUD を Spring Data JDBC でシンプルに実装できるようになることです。難しい SQL やテーブルの結合（JOIN）には踏み込まず、まずは「1つのテーブルに対する CRUD」をしっかり身につけます。

1.5 登場人物 — Entity・Repository・Service

Spring Data JDBC でデータベースを操作するとき、主に3つの「登場人物（クラス）」が協力します。それぞれの役割を最初につかんでおくと、第4章以降がぐっと理解しやすくなります。

Entity（エンティティ）

テーブルの「1行分のデータ」を表す Java クラス。User など。（第4章）

Repository（リポジトリ）

データベースとやり取りする「入口」。CRUD メソッドを提供する。（第5章）

Service（サービス）

業務処理を担当し、Repository を呼び出してデータ操作を行う。（第6章）

この3つは、次のような「層（レイヤー）」になって連携します。上から下へ呼び出していき、一番下のデータベースに届きます。本教材では、Web 画面の代わりに CommandLineRunner という仕組みから Service を呼び出して動作を確認します（第7章）。

flowchart TD A["CommandLineRunner
(動作確認の入口・第7章)"] --> B["Service
(業務処理・第6章)"] B --> C["Repository
(データアクセスの入口・第5章)"] C --> D[("データベース
(users テーブル)")] E["Entity: User
(テーブルの1行を表す・第4章)"] -.データの形.-> C

ここで「Spring 基礎」で学んだ DI が再び登場します。Service は Repository を、CommandLineRunner は Service を、それぞれ自分で作らずに Spring から受け取ります（注入してもらう）。「必要な部品は Spring が用意して渡してくれる」——この DI の考え方が、層をつなぐ接着剤になります。

復習: DI（依存性の注入）

あるクラスが「必要とする別の部品（依存）」を、自分で new せず、外（Spring）から渡してもらう仕組みが DI です。@Service や @Repository を付けたクラスは Bean として Spring に管理され、必要な場所へ自動で注入されます。本教材では、この仕組みを使って Service に Repository を渡します（第6章）。

環境構築の前に、まずは「これから作るもの」のイメージを固めましょう。全体像を自分の言葉で説明できると、以降の章がスムーズに進みます。

手順

  2. 2 CRUD の4文字が、それぞれどんな操作・どの SQL に対応するかを声に出して言ってみる。
  3. 3 1.5 の図を見ながら、「CommandLineRunner → Service → Repository → DB」の流れと、Entity の役割を自分の言葉で説明してみる。

理解度確認クイズ

  1. Q1. Spring Data JDBC は何のための仕組みですか？

    回答を表示

    Java からデータベースを操作するためのコードを、Spring らしくシンプルに書けるようにする仕組みです。SQL や接続管理を Spring が肩代わりし、Repository を通じて基本的な CRUD 処理を簡単に実装できます。

  2. Q2. 通常の JDBC と比べて、Spring Data JDBC のメリットは何ですか？

    回答を表示

    SQL の記述・接続管理・結果の取り出しといった「決まりきったコード」を自分で書かずに済み、コード量が少なくなります。また、Entity・Repository・Service に分けやすく、処理を整理しやすくなります。

  3. Q3. CRUD とは何の略ですか？

    回答を表示

    Create（登録）・Read（参照）・Update（更新）・Delete（削除）の頭文字をつなげた言葉で、データに対する4つの基本操作を表します。それぞれ SQL の INSERT・SELECT・UPDATE・DELETE に対応します。
