CommandLineRunnerで動作確認する

第6章までに Entity・Repository・Service が揃いました。Web 画面（Controller）を作る前に、CommandLineRunner を使ってコンソールから CRUD の流れを実際に動かして確認しましょう。「動くコードを目で見る」体験が、次に学ぶ Spring MVC の理解をぐっと深めてくれます。

この章で学ぶこと

  • Spring Boot 起動時に処理を実行する方法を理解する
  • Service を使ってデータベース操作を試す
  • CRUD 処理の流れをコンソール出力で確認する

7.1 CommandLineRunner とは

CommandLineRunner は、Spring Boot アプリが起動した直後に処理を自動で実行してくれる仕組みです。通常、Web アプリが動くためには「ブラウザからアクセスする → Controller が受け取る → Service を呼ぶ」という流れが必要です。しかし CommandLineRunner を使えば、Web 画面（Controller）を作らなくても、起動するだけで処理が走ります。

本教材では、Spring MVC（Web 画面）の章に入る前の動作確認ステップとして CommandLineRunner を活用します。「Entity・Repository・Service が正しくつながっているか」をコンソールに出力して目で確認するのが、この章の目的です。

用語: CommandLineRunner

CommandLineRunner は Java のインターフェース（お約束ごとの定義）で、run というメソッドを1つだけ持ちます。@Component などを付けて Spring の Bean にしておくと、アプリ起動後に Spring が自動で run を1回呼び出してくれます。テストデータの投入や起動時チェックなど、「アプリが立ち上がったら最初に1回だけやりたいこと」を書く場所として活用されます。

CommandLineRunner はインターフェースなので、これを実装したクラスを自分で作ります。クラス名は何でも構いませんが、本教材では AppRunner という名前にします。次のセクションで実際に作っていきましょう。

7.2 AppRunner を作る

AppRunner クラスを作ります。ポイントは2つです。

  • @Component を付けて Bean にする — Spring に管理してもらうことで、起動時に自動で run が呼ばれます。
  • UserService をコンストラクタで受け取る（DI） — 自分で new せず、Spring に渡してもらいます。

まず、クラス定義とコンストラクタ（DI 部分）を見てみましょう。

AppRunner.java — クラス定義とコンストラクタ 
import org.springframework.boot.CommandLineRunner;
import org.springframework.stereotype.Component;

// @Component を付けることで Spring が Bean として管理し、起動時に run を呼ぶ
@Component
public class AppRunner implements CommandLineRunner {

    // UserService を保持するフィールド
    private final UserService userService;

    // UserService を DI で受け取るコンストラクタ（Spring が自動で渡してくれる）
    public AppRunner(UserService userService) {
        this.userService = userService;
    }
    // run メソッドは次のブロックで解説
}

次に、run メソッドの中身です。ここに「起動したらやりたいこと」を書きます。今回は全件取得・登録・登録後の全件取得という流れを実行します。

AppRunner.java — run メソッドの中身 
@Override
public void run(String... args) {
    // ① 既存の全件を取得してコンソールに表示する
    System.out.println("=== 全件取得 ===");
    for (User user : userService.findAll()) {
        System.out.println(user.getId() + ": " + user.getName() + " / " + user.getEmail());
    }

    // ② 新しいユーザーを登録する
    System.out.println("=== 登録 ===");
    User created = userService.create("鈴木一郎", "suzuki@example.com");
    System.out.println(created.getId() + ": " + created.getName());

    // ③ 登録後にもう一度全件取得して、追加されたことを確認する
    System.out.println("=== 登録後の全件取得 ===");
    for (User user : userService.findAll()) {
        System.out.println(user.getId() + ": " + user.getName() + " / " + user.getEmail());
    }
}

補足: String... args とは？

String... args は「0個以上の文字列引数」を受け取る書き方（可変長引数）です。コマンドライン（ターミナル）からアプリを起動するときに追加で渡せるオプション文字列が入ります。動作確認の用途では使わないことが多いので、今回は無視して構いません。

7.3 実行の流れ

アプリが起動してから、コンソール出力が表示されるまでの流れを図で確認しましょう。Spring が各 Bean を組み立てて run を呼び出し、その中でレイヤーを順番に通って DB に到達する様子が分かります。

flowchart TD A["Spring Boot 起動"] --> B["AppRunner.run 自動実行
（CommandLineRunner）"] B --> C["UserService のメソッドを呼ぶ
findAll / create など"] C --> D["UserRepository 経由で
SQL を発行する"] D --> E[("DB
users テーブル")] E --> D D --> C C --> B B --> F["コンソールに結果を出力"]

図の上から下へ呼び出しが進み、DB からの応答が戻ってくると上向きに結果が返ってきます。この「呼び出し→応答」の流れは、第8章以降で Controller を使う場合も同じです。CommandLineRunner が Controller に変わるだけで、Service → Repository → DB の部分はそのまま再利用されます。

7.4 期待する出力

アプリを起動すると、コンソールに以下のような出力が表示されます（あらかじめ DB に田中太郎・佐藤花子が登録されている想定です）。

動作確認: 期待するコンソール出力

コンソール出力例 
=== 全件取得 ===
1: 田中太郎 / tanaka@example.com
2: 佐藤花子 / sato@example.com
=== 登録 ===
3: 鈴木一郎
=== 登録後の全件取得 ===
1: 田中太郎 / tanaka@example.com
2: 佐藤花子 / sato@example.com
3: 鈴木一郎 / suzuki@example.com
  • CommandLineRunner は起動後に run メソッドを自動で実行する
  • UserService を DI でコンストラクタに受け取り、メソッドを呼ぶだけで DB 操作できる
  • Web 画面（Controller）がなくても、起動するだけで DB 操作の動作を確認できる
  • 次の Spring MVC では、CommandLineRunner の代わりに Controller から Service を呼び出す形に発展する（予告）

「登録後の全件取得」で 3 件目に鈴木一郎が追加されていれば、Repository → Service → CommandLineRunner のつながりが正しく動いている証拠です。もし出力が期待と異なる場合は、第4〜6章で作成した Entity・Repository・Service のコードを見直してみましょう。

重要ポイント

CommandLineRunner を使えば、Web 画面なしで CRUD の動作をコンソールで確認できます。Entity・Repository・Service が正しく組み上がっているかを、最もシンプルな方法で検証できる便利な仕組みです。次の章からは、この CommandLineRunner を Controller（Web 画面）に置き換えて、ブラウザからも同じ操作ができるようにしていきます。

理解度確認クイズ

  1. Q1. CommandLineRunner は何のために使いますか？

    回答を表示

    Spring Boot アプリの起動後に処理を自動実行するために使います。Web 画面（Controller）を作らなくても、起動するだけで Service 経由の DB 操作を動作確認できます。

  2. Q2. AppRunner は UserService をどのように受け取っていますか？

    回答を表示

    コンストラクタで受け取っています（DI）。new で自分では作らず、Spring が UserService を自動で用意して渡してくれます。これが依存性の注入（DI）の仕組みです。

  3. Q3. Web 画面がない状態でも DB 操作を確認できる理由は何ですか？

    回答を表示

    CommandLineRunner が起動時に run メソッドを自動実行し、その中で Service 経由で DB 操作を行うためです。ブラウザからのリクエストを待たなくてもよいため、Controller なしで動作確認できます。
