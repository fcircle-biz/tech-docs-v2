ServiceからRepositoryを使う

第5章で Repository の準備ができました。いよいよ、業務処理を担う Service から Repository を呼び出して CRUD を動かしましょう。この章では、Service と Repository の役割分担・DI で Repository を受け取る流れ・そして findAll / findById / save / deleteById の使い方を順番に身につけます。

この章で学ぶこと

  • Service と Repository の役割分担を理解する
  • DI で Repository を Service に渡す流れを理解する
  • findAll、findById、save、deleteById を使えるようにする

6.1 Service と Repository の役割分担

アプリケーションのコードを整理するとき、「何をするか」と「どうやってデータを取るか」を分けるのが Spring のお作法です。この2つを分けるのが Service と Repository の役割分担です。

Service（サービス）

業務処理を担当します。「登録するときにバリデーションをする」「取得した一覧を加工する」といった何をすべきかのロジックを書く場所です。

Repository（リポジトリ）

データベースアクセスを担当します。「findAll で全件取る」「save でデータを保存する」といったどうやってデータを操作するかを担う窓口です。

Service から Repository を呼び出すことで、業務処理とデータアクセス処理を分けられます（責務の分離）。なぜ分けるのでしょうか？

  • 変更に強くなる — DBの変更は Repository だけ直せばよく、Service のロジックには手を触れずに済みます。
  • テストしやすい — Repository を差し替えることで、DB なしでも Service の業務ロジックを検証できます。
  • 役割が明確 — コードを読んだときに「ここは業務処理」「ここはDB操作」とすぐに判断できます。

6.2 DI で Repository を受け取る（復習）

「Spring 基礎」で学んだ DI（依存性の注入）を思い出しましょう。UserService は UserRepository をコンストラクタで受け取ります。自分で new はしません。

@Service を付けると、そのクラスは Spring が管理する Bean になります。Spring は「UserService を作るとき、コンストラクタに UserRepository が必要だ」と判断して、自動で渡してくれます。

UserService.java — クラス定義とDIコンストラクタ 
import org.springframework.stereotype.Service;

// @Service を付けると Spring が管理する Bean になる
@Service
public class UserService {

    private final UserRepository userRepository;

    // Spring が UserRepository を自動で渡してくれる（DI）
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

復習: DI（依存性の注入）

必要な部品を自分で new せず、外（Spring）から渡してもらう＝ DI です。@Service や @Repository を付けたクラスは Bean として Spring に管理され、必要な場所へ自動で注入されます。自分で部品を作らないことで、テストや切り替えが楽になります。

6.3 CRUDメソッドを実装する

コンストラクタで受け取った userRepository を使って、CRUD の各操作を実装します。それぞれのメソッドを見ていきましょう。

UserService.java — 参照メソッド（全件取得・1件取得） 
// 全件取得: users テーブルの全レコードを返す
public Iterable<User> findAll() {
    return userRepository.findAll();
}

// IDで1件取得: 見つからなければ null を返す
public User findById(Long id) {
    // findById は Optional を返すため orElse(null) で中身を取り出す
    return userRepository.findById(id).orElse(null);
}
UserService.java — 登録・削除メソッド 
// 新規登録: id を null にすると自動採番される
public User create(String name, String email) {
    User user = new User(null, name, email);
    return userRepository.save(user); // 保存後のエンティティ（id 付き）を返す
}

// 削除: 指定した id のレコードを削除する
public void delete(Long id) {
    userRepository.deleteById(id);
}

各メソッドのポイントをまとめます。

  • findAll — 全件をまとめて取得します。戻り値は Iterable なので、for文でループできます。
  • findById — ID を指定して1件取得します。該当なしの場合は orElse(null) で null を返します。
  • save — エンティティの登録（id が null のとき）または更新（id があるとき）を行います。
  • deleteById — 指定した id のレコードを削除します。

用語: Optional と orElse

Optional は「値があるかもしれないし、ないかもしれない」を表す入れ物です。findById が返す Optional は、ID に一致するデータが存在すれば値あり、なければ空です。orElse(null) で「中身があればそれを、なければ null を返す」という処理になります。

重要ポイント

  • ・Service は業務処理、Repository はDBアクセスを担当する。
  • ・Service が Repository を呼ぶ形で役割を分担する（責務の分離）。
  • ・@Service を付けると Bean になり、DI で Repository を受け取れる。
  • ・save は id が null なら登録、id があれば更新として動く。

理解度確認クイズ

  1. Q1. Service は何を担当しますか？

    回答を表示

    業務処理を担当します。バリデーションや取得データの加工など「何をすべきか」のロジックを記述する場所です。

  2. Q2. Repository は何を担当しますか？

    回答を表示

    データベースアクセスを担当します。findAll・save・deleteById などのデータ操作メソッドを提供する窓口です。

  3. Q3. Service と Repository を分けるメリットは何ですか？

    回答を表示

    業務処理とデータアクセスを整理でき、変更に強くなる・テストしやすくなる・役割が明確になるという3つのメリットがあります。

  4. Q4. save メソッドは何を行いますか？

    回答を表示

    エンティティの登録または更新を行います。渡したエンティティの id が null の場合は新規登録（自動採番）、id がすでにある場合は既存レコードの更新として動作します。
