Entityの関連を扱う

第1章から予告してきた「部署と社員のつながり」をいよいよコードで表現します。データベースでは employees.department_id が departments.id を指す「外部キー」でテーブルをつないでいます。JPA では、この外部キーの関係を Java オブジェクトの参照として自然に扱うことができます。本章では「多数の社員が1つの部署に所属する」という多対1（ManyToOne）の関連を、@ManyToOne と @JoinColumn の2つのアノテーションで表す方法を丁寧に学んでいきます。

この章で学ぶこと

  • データベースの外部キーと JPA の関連（リレーション）の対応関係を説明できる
  • 「多対1」の意味を図で理解し、社員と部署の関係として説明できる
  • @ManyToOne と @JoinColumn を Employee エンティティに追加できる
  • employee.getDepartment().getName() で所属部署名を取得できる
  • 双方向関連（@OneToMany）は本章では扱わない理由を理解する

8.1 「関連」とは — テーブルのつながりをオブジェクトで表す

データベースでは、複数のテーブルを「外部キー」でつないでデータを管理します。本教材の例では、employees テーブルの department_id 列が departments テーブルの id を指しています。これが「外部キー」によるつながりです。

SQL でこのつながりを使う場合、JOIN（結合）を書く必要があります。しかし JPA では、このつながりを「Java オブジェクトの参照」として持つことができます。Employee オブジェクトの中に Department オブジェクトが丸ごと入っているイメージです。

departments テーブル

id name
1 営業部
2 開発部

employees テーブル（department_id が外部キー）

id name email department_id
1 山田太郎 yamada@example.com 1
2 鈴木花子 suzuki@example.com 1
3 田中次郎 tanaka@example.com 2

表を見ると、山田さんと鈴木さんの department_id は 1（営業部）、田中さんは 2（開発部）です。この「department_id が departments.id を指す」という外部キーの関係を、JPA では Employee クラスが Department オブジェクトをフィールドとして持つことで表現します。

用語: 関連（リレーション）

JPA の世界では、テーブル間の外部キーによるつながりを「関連（association / relationship）」と呼びます。関連を使うと、SQL の JOIN を自分で書かなくても、JPA が自動でテーブルをつないで関連先のデータを取得してくれます。

8.2 多対1（ManyToOne）とは

社員と部署の関係をよく考えてみましょう。1つの部署に複数の社員が所属できます。逆に言うと、複数の社員が1つの部署に所属します。この関係を「多対1（Many-to-One）」と呼びます。「社員（Many）」が「部署（One）」に対して多対1の関係にある、という意味です。

多対1のイメージ図

flowchart LR A["社員A: 山田太郎"] --> D["部署: 営業部"] B["社員B: 鈴木花子"] --> D C["社員C: 田中次郎"] --> E["部署: 開発部"]

図のように、「複数の Employee → 1つの Department」という矢印になります。多（社員）→ 1（部署）なので「多対1（ManyToOne）」です。JPA ではこの方向性を @ManyToOne というアノテーション1つで表すことができます。

多対1の覚え方

「@ManyToOne をどちらのクラスに付けるか」は迷いやすいポイントです。答えは「多」の側（Employee）です。「社員は1つの部署に属する（Employee → Department）」という方向で考えると覚えやすくなります。アノテーションを付けるのは、外部キー（department_id）を持つテーブルに対応するクラス、つまり Employee クラスです。

JPA で扱う主な関連の種類（本章では ManyToOne に集中）

関連の種類 アノテーション 例 本章
多対1 @ManyToOne 社員（多）→ 部署（1） ✓ 扱う
1対多 @OneToMany 部署（1）→ 社員リスト（多） 本章では扱わない
1対1 @OneToOne 社員（1）→ 社員証（1） 本章では扱わない
多対多 @ManyToMany 社員（多）→ プロジェクト（多） 本章では扱わない

8.3 Department エンティティを作る

まず、関連先になる Department（部署）エンティティを作ります。departments テーブルに対応するシンプルなクラスです。第3章で学んだ @Entity・@Id・@GeneratedValue の組み合わせです。

Department.java 
package com.example.jpalearning.entity;

import jakarta.persistence.*;

@Entity
@Table(name = "departments")  // departments テーブルに対応
public class Department {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;  // 部署名（departments.name 列）

    // getter / setter
    public Long getId() { return id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}

Department は第3章の Employee と同じパターン

Department は第3章で作った Employee と全く同じパターンです。@Entity でエンティティであることを宣言し、@Id ＋ @GeneratedValue で主キーを指定するだけです。新しいことは何もありません。

8.4 Employee に @ManyToOne で関連を追加する

次に、本章の核心です。第3章で作った Employee に「部署への関連」を追加します。追加するのは2つのアノテーションと1つのフィールドだけです。

@ManyToOne

「このフィールドは多対1の関連先を示す」と JPA に伝えるアノテーション。Employee（多）→ Department（1） の方向を表す。

@JoinColumn

「この関連は employees テーブルの department_id 列でつながっている」と外部キーの列名を指定する。

Employee.java（関連を追加した版） 
package com.example.jpalearning.entity;

import jakarta.persistence.*;

@Entity
@Table(name = "employees")
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;   // 社員名
    private String email;  // メールアドレス

    // 【新規追加】部署への多対1の関連
    @ManyToOne
    @JoinColumn(name = "department_id")  // 外部キー列
    private Department department;

    // getter / setter（フォームのバインドや画面表示で使うため両方そろえる）
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
    public Department getDepartment() { return department; }
    public void setDepartment(Department department) {
        this.department = department;
    }
}

変更点は @ManyToOne・@JoinColumn(name = "department_id")・private Department department の3行だけです。この3行を追加することで、Employee オブジェクトが Department オブジェクトを「内側に持つ」構造になります。

Employee クラスが department フィールドで Department を参照する構造

classDiagram class Employee { Long id String name String email Department department } class Department { Long id String name } Employee --> Department : "@ManyToOne
@JoinColumn（department_id）"

図のとおり、Employee クラスが Department クラスへの参照（矢印）を持ちます。この矢印の部分に @ManyToOne と @JoinColumn が付いています。

@JoinColumn(name = "department_id") の意味

name 属性に指定する文字列は、DB テーブルの外部キー列名です。employees テーブルには department_id という列があり、それが departments.id を参照している——この対応を JPA に教えるのが @JoinColumn の役割です。@JoinColumn を省略しても JPA は自動で列名を推測しますが、明示的に書くほうがコードの意図が伝わりやすいです。

8.5 関連を使ってデータを取得する

関連を追加した Employee を Repository で取得すると、JPA が標準設定では関連先の Department もあわせてデータベースから取得してくれます。取得後は getDepartment() でそのまま部署オブジェクトにアクセスできます（この「いつ取得するか」の設定については、このあとの補足で触れます）。

Service での使用例（社員から部署名を取得） 
// EmployeeService.java（抜粋）
@Service
public class EmployeeService {

    private final EmployeeRepository employeeRepository;

    public EmployeeService(EmployeeRepository employeeRepository) {
        this.employeeRepository = employeeRepository;
    }

    // 社員IDから所属部署名を取得する
    public String getDepartmentName(Long employeeId) {
        // findById で Employee を取得（標準設定では関連の Department も一緒に取得される）
        Employee e = employeeRepository.findById(employeeId).orElseThrow();
        // getDepartment() で Department オブジェクトにアクセス
        return e.getDepartment().getName(); // 例: "営業部"
    }
}

e.getDepartment().getName() とメソッドを2段チェーンするだけで所属部署名が取れます。SQL で言えば、employees と departments を JOIN して取得するのと同じ結果です。私たちが JOIN を直接書かなくても、JPA が裏側でテーブルの結合（JOIN）や追加の問い合わせを自動的に行ってくれます。

補足: 関連先を「いつ」取得するか（Fetch / EAGER / LAZY）

Fetch とは「関連先のデータをいつ読み込むか」の設定です。@ManyToOne は、fetch を指定しないと標準で EAGER（即時取得）になり、社員を取得する時点で部署も一緒に読み込まれます。本教材ではこの初期設定のまま進めるため、上のように getDepartment() でそのまま部署にアクセスできます。

ただし実務では、性能上の理由で @ManyToOne(fetch = FetchType.LAZY)（遅延取得＝必要になったときに読み込む）を使うことも多くあります。また、一覧で多数の社員を取得すると関連先の取得が件数分くり返される「N+1 問題」が起きることもあります。これらは発展的なテーマなので、本教材では初学者向けに初期設定のまま簡略化し、「標準では関連先も取得される」とだけ理解しておけば十分です。

社員を部署と一緒に登録する

新しい社員を登録するとき、department フィールドに Department オブジェクトをセットしてから save() します。

社員を部署に所属させて登録するイメージ 
// DepartmentRepository から部署を取得
Department dept = departmentRepository.findById(1L).orElseThrow();

// 新しい社員を作成して部署をセット
Employee newEmployee = new Employee();
newEmployee.setName("佐藤一郎");
newEmployee.setEmail("sato@example.com");
newEmployee.setDepartment(dept);  // 部署オブジェクトをセット

// 保存すると department_id = 1 が自動で書き込まれる
employeeRepository.save(newEmployee);

重要ポイント: ID ではなくオブジェクトをセットする

JPA では、外部キーの「値（数字の 1）」を直接セットするのではなく、Department オブジェクトそのものをセットします。JPA が自動的にそのオブジェクトの ID（1）を取り出して department_id 列に書き込んでくれます。これが「オブジェクト中心」で考える JPA の特徴です。

8.6 双方向関連は今は深入りしない

ここまで学んできた「Employee から Department を参照する」方向を 単方向関連と呼びます。逆に「Department（部署）が社員のリストを持つ」方向を追加した場合、それを双方向関連と呼びます。

双方向関連では Department クラスに List<Employee> フィールドを追加し、@OneToMany アノテーションを付けることになります。しかし、これには注意が必要な点がいくつかあります。

双方向関連には落とし穴がある — 今は深入りしない

  • mappedBy の指定: @OneToMany には mappedBy 属性が必要で、どちら側が外部キーを「所有」するか明示する必要がある。
  • JSON 循環参照: REST API で返す際に Employee → Department → Employee → ... と無限に参照が繰り返され、エラーになることがある。
  • 遅延読み込み（FetchType）: リストをいつ取得するかを FetchType.LAZY / EAGER で制御する必要があり、トランザクション外での参照でエラーが起きることがある。

これらは「次のステップ（発展学習）」で改めて学びます。本教材では社員から部署を参照する単方向に絞り、まず基礎を確実に身につけましょう。

本章で扱う範囲（単方向）と、発展で学ぶ範囲（双方向）

flowchart LR subgraph "本章の範囲（単方向）" E["Employee"] -->|"@ManyToOne"| D["Department"] end subgraph "発展学習（双方向）" D2["Department"] -->|"@OneToMany（mappedBy）"| E2["Employee リスト"] end

やってみよう 8-1: @ManyToOne で関連を追加する

第3章で作った Employee クラスに部署への関連を追加し、社員から所属部署名を取得するコードを書いてみましょう。

手順

  1. 1 Department.java を新規作成し、8.3 のコードをそのまま入力する。
  2. 2 既存の Employee.java を開き、@ManyToOne・@JoinColumn(name = "department_id")・private Department department の3行を追加する。
  3. 3 DepartmentRepository（JpaRepository を継承）を作成する。
  4. 4 H2 Console や data.sql で部署データと社員データを1件ずつ投入し、EmployeeService の getDepartmentName(1L) を呼び出して「営業部」などの部署名が取得できることを確認する。

動作確認のポイント

  • アプリ起動時にコンソールへ SELECT 文（JOIN または別クエリ）が出力される
  • e.getDepartment().getName() で期待どおりの部署名（例: "営業部"）が返る
  • e.getDepartment() が null にならない（社員データに department_id が正しく設定されている）

理解度確認クイズ

  1. Q1. @ManyToOne はどちらのクラス（Employee / Department）に付けますか？ その理由は？

    回答を表示

    Employee クラスに付けます。「多（Employee）→ 1（Department）」という多対1の方向性を考えると、「多」の側が「1」の側を参照するためです。また、データベース上では employees テーブルが外部キー（department_id）を持っており、外部キーを持つテーブルに対応するクラスが関連の起点になります。

  2. Q2. @JoinColumn(name = "department_id") の name に指定する文字列は何を表していますか？

    回答を表示

    データベーステーブルの外部キー列名を表します。ここでは employees テーブルの department_id 列が departments テーブルの id を参照している、という外部キーの対応を JPA に伝えるために指定します。

  3. Q3. 本章で双方向関連（@OneToMany）を扱わない理由を2つ挙げてください。

    回答を表示

    以下のいずれか2つが正解です。
    (1) mappedBy の指定が必要で、どちら側が外部キーを「所有」するかを明示する複雑さがある。
    (2) JSON 循環参照: REST API で返す際に Employee → Department → Employee … と無限ループになるエラーが発生しやすい。
    (3) 遅延読み込み（FetchType）: リストをいつ取得するかの制御が必要で、トランザクション外でのアクセスエラーが起きやすい。
