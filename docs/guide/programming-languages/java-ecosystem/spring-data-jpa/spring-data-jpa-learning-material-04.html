RepositoryによるCRUD操作

JpaRepository を継承したインターフェースを作るだけで、データの一覧取得・検索・登録・更新・削除といった基本的な CRUD 操作がすぐに使えます。この章では、Spring Data JPA が CRUD メソッドを自動で用意してくれる仕組みを理解し、各メソッドの使い方をコードで確認します。

この章で学ぶこと

  • JpaRepository<Employee, Long> の型引数（Entity型・主キー型）の意味を理解する
  • インターフェースを作るだけで CRUD メソッドが使える仕組みを説明できる
  • findAll・findById・save・deleteById・count の使い方を理解する
  • findById が返す Optional の基本的な扱い方を理解する

4.1 RepositoryとJpaRepository — 作るだけで使えるCRUD

第3章で Employee Entity を作りました。次は、この Entity のデータをデータベースに保存・取得・削除する「Repository」を用意します。Spring Data JPA では、インターフェースを1つ作って JpaRepository を継承するだけで、CRUD に必要なメソッドが自動的に使えるようになります。

「実装クラスが必要では？」と思うかもしれません。Spring Data JPA は、アプリケーション起動時にインターフェースを自動スキャンし、実装クラスを Spring が自動生成します。私たちは何も実装しなくてよいのです。

EmployeeRepository.java 
package com.example.jpalearning.repository;

import com.example.jpalearning.entity.Employee;
import org.springframework.data.jpa.repository.JpaRepository;

// インターフェースを宣言するだけ — 中身は空でOK
// Spring が起動時に実装クラスを自動生成する
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
}

型引数の意味: JpaRepository<Employee, Long>

JpaRepository には2つの型引数が必要です。

  • 第1引数 — Entity の型: このRepositoryが操作する Entity クラス。ここでは Employee。
  • 第2引数 — 主キーの型: @Id フィールドの型。Employee.id は Long なので Long を指定。

たったこれだけで、次のセクションで紹介する CRUD メソッドがすべて使えるようになります。「インターフェースを継承するだけで動く」——このSpring Data JPA の自動化の仕組みを体感することがこの章の最大のポイントです。

（インターフェース。中身は空）"] -->|継承| B["JpaRepository<Employee, Long>
（Spring Data JPA が用意）"] B -->|Spring が起動時に実装を自動生成| C["実装クラス（SimpleJpaRepository）

4.2 CRUDメソッド一覧 — 操作とメソッドの対応

JpaRepository を継承すると、以下の基本 CRUD メソッドがすぐに使えます。まずは対応表で全体像をつかみましょう。

操作 メソッド 戻り値 説明
全件検索 findAll() List<Employee> テーブルの全行を取得
ID検索 findById(id) Optional<Employee> 指定IDの1件を取得（存在しない場合は空の Optional）
登録・更新 save(employee) Employee idがなければINSERT、あればUPDATE
削除 deleteById(id) void 指定IDの行を削除
件数取得 count() long テーブルの総件数を取得

これらはすべて JpaRepository が提供するメソッドで、私たちは何も実装しなくてよいです。次のセクションから、各メソッドの使い方をコードで確認していきます。

4.3 findAll・findById — データを取得する

データを取得するメソッドは主に2つです。findAll() は全件取得、findById(id) はIDを指定して1件取得します。

findAll() — 全件取得の例 
// Repository を @Autowired または コンストラクタインジェクションで取得
EmployeeRepository employeeRepository; // 詳細は第5章で学ぶ

// 全社員を取得 — employees テーブルの全行が List で返る
List<Employee> employees = employeeRepository.findAll();

// 件数を確認
System.out.println("社員数: " + employees.size());
findById() — ID指定で1件取得の例 
// findById はデータが存在しない場合もあるため Optional で返る
Optional<Employee> result = employeeRepository.findById(1L);

// isPresent() でデータの有無を確認
if (result.isPresent()) {
    Employee emp = result.get(); // Optional から取り出す
    System.out.println("取得: " + emp.getName());
}

なぜ Optional で返るの？

IDを指定して検索したとき、そのIDのデータが存在しない場合があります。存在しないのに Employee をそのまま返すと null になり、うっかり使おうとすると NullPointerException が発生します。Optional は「値があるかもしれないし、ないかもしれない」を型で表現し、null チェックを忘れにくくする仕組みです。

Optionalの代表的な扱い方 
Optional<Employee> result = employeeRepository.findById(1L);

// 方法1: なければ null を返す（シンプルだが null チェックが必要）
Employee emp1 = result.orElse(null);

// 方法2: なければ例外を投げる（取得できないと処理を続けられない場面で）
Employee emp2 = result.orElseThrow();

// 方法3: isPresent() + get() でチェックしてから取り出す
if (result.isPresent()) {
    Employee emp3 = result.get();
}

4.4 save — 登録と更新を1つのメソッドで行う

save(entity) は、登録（INSERT）と更新（UPDATE）を1つのメソッドで行います。「idがないオブジェクト」を渡すと INSERT、「idがあるオブジェクト」を渡すと UPDATE になります。

save() — 登録（INSERT）の例 
// id を設定しない → INSERT（@GeneratedValue が id を自動採番）
Employee newEmp = new Employee();
newEmp.setName("田中 一郎");
newEmp.setEmail("tanaka@example.com");

// save の戻り値は保存後の Entity（自動採番された id が入っている）
Employee saved = employeeRepository.save(newEmp);
System.out.println("登録完了。id = " + saved.getId());
save() — 更新（UPDATE）の例 
// まず id=1 の社員を取得
Employee emp = employeeRepository.findById(1L).orElseThrow();

// フィールドを変更して save → UPDATE になる
emp.setEmail("tanaka-new@example.com");
employeeRepository.save(emp); // 同じ id があるので UPDATE が発行される

重要ポイント: save が INSERT か UPDATE かを決める仕組み

初級段階では、ID がない新規 Entity を保存すると INSERT、既存 ID を持つ Entity を保存すると UPDATE として扱われる、と理解しておきましょう。@GeneratedValue を付けていれば、新規登録時は id を設定しなくてよく、データベースが自動採番してくれます。なお、INSERT か UPDATE かはより厳密には Entity の状態や永続化コンテキストにも依存しますが、本教材ではこの理解で十分です。

4.5 deleteById・count — 削除と件数取得

削除は deleteById(id)、件数取得は count() を使います。どちらも引数や戻り値がシンプルで使いやすいメソッドです。

deleteById() / count() の例 
// 削除前の件数を確認
long before = employeeRepository.count(); // 例: 3
System.out.println("削除前の件数: " + before);

// id=2 の社員を削除
employeeRepository.deleteById(2L); // DELETE FROM employees WHERE id=2

// 削除後の件数を確認
long after = employeeRepository.count(); // 例: 2
System.out.println("削除後の件数: " + after);

注意: 存在しないIDを deleteById に渡すと例外

deleteById に存在しない ID を渡すと、Spring Data JPA は EmptyResultDataAccessException をスローします。削除前に findById で存在を確認するか、Service 層でハンドリングするのが安全です（第5章で学びます）。

ここまでの5メソッド（findAll・findById・save・deleteById・count）が、JPA を使ったデータ操作の基本セットです。実際の業務アプリでは Service 層からこれらを呼び出します（第5章）。

やってみよう 4-1: EmployeeRepository を作って動作確認する

第2・3章で作ったプロジェクトに EmployeeRepository を追加し、アプリ起動時に CRUD メソッドを呼び出して動作を確認してみましょう。

手順

  1. 1 com.example.jpalearning.repository パッケージに EmployeeRepository.java を作成し、JpaRepository<Employee, Long> を継承する（中身は空でよい）。
  2. 2 @SpringBootApplication クラスに CommandLineRunner を追加し、save で社員を2人登録後、findAll で一覧を取得してコンソールに出力する。
  3. 3 アプリを起動し、コンソールに社員情報が出力されること、count() が 2 と表示されることを確認する。
  4. 4 deleteById でどちらか1件を削除し、再度 count() が 1 になることを確認する。

期待するコンソール出力の例

アプリ起動後、以下のような出力があれば成功です（H2 の SQL ログも表示される場合があります）。

社員数（登録後）: 2
id=1, 田中 一郎, tanaka@example.com
id=2, 山田 花子, yamada@example.com
社員数（削除後）: 1

理解度確認クイズ

  1. Q1. JpaRepository<Employee, Long> の2つの型引数はそれぞれ何を表しますか？

    回答を表示

    第1引数（Employee）はこの Repository が操作する Entity の型、第2引数（Long）は Entity の主キー（@Id フィールド）の型を表します。

  2. Q2. save(employee) が INSERT になるか UPDATE になるかは何で決まりますか？

    回答を表示

    渡した Entity の id が null であれば INSERT（新規登録）、id に値が設定されていれば UPDATE（更新）になります。

  3. Q3. findById(id) の戻り値が Optional<Employee> である理由を説明してください。

    回答を表示

    指定した ID のデータが存在しない場合があるためです。null をそのまま返すと NullPointerException の原因になります。Optional にすることで「値があるかもしれない」という状態を型で表現し、安全に扱えるようにしています。
