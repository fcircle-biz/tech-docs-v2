セッション管理 - ユーザー状態の保持

この章では、HTTP の「ステートレス」という性質を理解し、HttpSession と Cookie を使ってユーザーの状態を保持する方法を学びます。ログイン・ログアウト機能の実装を通して、実際のWebアプリでなくてはならないセッション管理をマスターします。

この章で学ぶこと

  • HTTP のステートレス性とセッションが必要な理由を理解する
  • HttpSession を使ってユーザーデータを管理できる
  • Cookie の仕組みを理解し、作成・送信・読み取りができる
  • ログイン・ログアウト機能を実装できる

7.1 セッションとは

ネットショッピングでは、商品ページから移動してカートに進んでも「さっきカートに入れた商品」がちゃんと残っていますよね。でも、HTTP という通信の仕組みは、本来は前のやり取りを一切覚えていない設計になっています。この節では、その仕組みと対策を理解しましょう。

HTTPのステートレス性

HTTP は ステートレス（stateless）なプロトコルです。「ステートレス」とは「状態を持たない」という意味です。ブラウザがページを1回リクエストするごとに、サーバーはそのやり取りが終わると誰からのリクエストだったかを完全に忘れてしまいます。次のリクエストが来ても、「あなたさっきも来ましたよね？」とはわかりません。

ステートレスなHTTPの問題: 各リクエストは独立しており、サーバーはユーザーを覚えていない

flowchart LR U["ユーザー
（ブラウザ）"] U -->|"① ログインリクエスト"| S["サーバー"] S -->|"② ログイン成功レスポンス"| U U -->|"③ 次のページリクエスト"| S S -->|"④ 誰？知らない人だ..."| U

これでは「ログイン済みか否か」を判定したり、カートの中身を維持したりすることができません。そこで登場するのが セッションという仕組みです。

セッションの仕組み（セッションID + Cookie）

セッションとは、「同じユーザーの一連のやり取りをひとまとまりとして扱う」仕組みです。具体的には次の流れで機能します。

手順 サーバーの動作 ブラウザの動作
①初回アクセス ユニークなセッションIDを発行してサーバー側に保存 セッションIDをCookieに保存
②2回目以降 CookieのセッションIDを見て「あの人だ」と識別 自動でCookieをサーバーへ送信
③終了 セッションを無効化してデータを削除 CookieのセッションIDが無効になる

Jakarta Servlet では、このセッションIDは JSESSIONID という名前の Cookie として扱われます。ブラウザが次のリクエストを送る際に自動的に Cookie を添付するため、サーバーは「これはさっきの人だ」と判断できます。

セッションによる状態管理: JSESSIONIDでユーザーを追跡する

flowchart LR B["ブラウザ"] S["サーバー
（セッション記憶）"] B -->|"① リクエスト
（初回）"| S S -->|"② JSESSIONID=abc123
をCookieで返す"| B B -->|"③ 次のリクエスト
+ Cookie:JSESSIONID=abc123"| S S -->|"④ abc123のユーザー
と識別して応答"| B

セッションが必要な場面

現実のWebアプリでセッションが使われる代表的な場面をまとめました。

ログイン状態の維持

「ログイン済みか否か」を複数ページにわたって判定する。

ショッピングカート

商品を選んでいる間、ページを移動しても選択内容を保持する。

入力ウィザード

複数ステップの入力フォームで途中のデータを一時保存する。

セキュリティ考慮: セッションハイジャックとID再生成

セッションIDが第三者に盗まれると、なりすましログインが可能になります（セッションハイジャック）。これを防ぐため、ログイン成功後は必ずセッションIDを再生成します。Servletでは session.invalidate() で古いセッションを破棄し、新しいセッションを取得することが推奨されます。

7.2 HttpSessionの使い方

Jakarta Servlet では jakarta.servlet.http.HttpSession インタフェースを使ってセッションを操作します。HttpServletRequest からセッションを取得し、属性（データ）を読み書きするのが基本パターンです。

セッションの取得

request.getSession() を呼ぶと、既存のセッションがあればそれを返し、なければ新しく作って返します。

LoginServlet.java — セッションの取得 
// 既存セッションを取得、なければ新規作成
HttpSession session = request.getSession();

// false を渡すと「既存があれば返す・なければ null」になる
HttpSession session = request.getSession(false);

主要メソッド一覧

HttpSession が提供するメソッドを把握しておきましょう。

メソッド 役割 例
setAttribute(name, value) 属性を保存する session.setAttribute("user", userObj)
getAttribute(name) 属性を取得する（Object型） session.getAttribute("user")
removeAttribute(name) 属性を削除する session.removeAttribute("user")
invalidate() セッションを無効化（ログアウト） session.invalidate()
setMaxInactiveInterval(秒) タイムアウト秒数を設定 session.setMaxInactiveInterval(1800)
getId() セッションIDを取得 session.getId()

属性の設定と取得のコード例

SessionDemoServlet.java — 属性の操作 
HttpSession session = request.getSession();

// セッションにユーザー名を保存
session.setAttribute("loginUser", "山田太郎");

// セッションのタイムアウトを30分に設定
session.setMaxInactiveInterval(30 * 60); // 秒で指定

// 別のServlet/JSPで取得する（Object型を適切にキャスト）
String loginUser = (String) session.getAttribute("loginUser");

// null チェックは必ず行う
if (loginUser != null) {
    response.getWriter().println("ようこそ、" + loginUser);
}

タイムアウト設定の単位に注意

setMaxInactiveInterval() の引数は秒です。分ではありません。30分にしたいなら 30 * 60 = 1800 と指定します。デフォルトは Tomcat では30分（1800秒）が多いですが、web.xml の <session-timeout> で分単位で変更できます。

7.3 Cookieの基礎

Cookie（クッキー）は、サーバーがブラウザに「小さなデータを覚えておいて」と頼む仕組みです。ブラウザはそのデータを保存し、次のリクエスト時に自動でサーバーへ送り返します。セッションIDの保存にも使われていますが、独自データを直接保存する用途にも利用できます。

CookieとHttpSessionの違い

比較項目 Cookie HttpSession
データの保存場所 ブラウザ（クライアント側） サーバー側
有効期限 setMaxAge() で任意設定可能 タイムアウトまたはinvalidate()まで
適している用途 「次回からチェックを入れる」など長期的な設定 ログイン状態など機密性の高いデータ
ユーザーが操作可能か 削除・無効化できる ユーザーは直接操作できない

Cookieの作成と送信

jakarta.servlet.http.Cookie クラスでCookieオブジェクトを作り、response.addCookie() でブラウザへ送ります。

CookieDemoServlet.java — Cookieの作成と送信 
// 新しいCookieを作成（名前と値）
Cookie cookie = new Cookie("username", "yamada");

// 有効期限: 7日間（秒で指定）
cookie.setMaxAge(7 * 24 * 60 * 60);

// 有効なパス（"/" はサイト全体）
cookie.setPath("/");

// Cookieをレスポンスに追加してブラウザへ送信
response.addCookie(cookie);

Cookieの読み取り

次のリクエストでは request.getCookies() で全Cookieを配列として取得し、目的の名前のものを探します。

CookieDemoServlet.java — Cookieの読み取り 
Cookie[] cookies = request.getCookies();
String username = null;

// null チェック: Cookieが全くない場合は null
if (cookies != null) {
    for (Cookie c : cookies) {
        if ("username".equals(c.getName())) {
            username = c.getValue(); // 値を取得
            break;
        }
    }
}

セッションCookieと永続Cookie

セッションCookie

setMaxAge() を呼ばないか、または -1 を指定。

ブラウザを閉じると消える。JSESSIONIDはデフォルトでこちら。

永続Cookie

setMaxAge(秒) で正の値を指定。

ブラウザを閉じても指定期間は残る。「次回から自動入力」などに利用。

7.4 ログイン機能の実装

セッションを使ったログイン機能の実装パターンを見ていきます。ここでは理解のため、認証は固定のユーザー情報を使った簡易版で説明します。実際のアプリではデータベースと組み合わせます。

ログインフォーム（login.jsp）

login.jsp — ログインフォーム 
<!-- エラーメッセージ表示 -->
<% String error = (String) request.getAttribute("error"); %>
<% if (error != null) { %>
    <p style="color:red;"><%= error %></p>
<% } %>

<form method="post" action="LoginServlet">
    <label>ユーザーID: <input type="text" name="userId"></label>
    <label>パスワード: <input type="password" name="password"></label>
    <button type="submit">ログイン</button>
</form>

LoginServlet（認証とセッション保存）

LoginServlet.java — 認証処理 
protected void doPost(HttpServletRequest request,
        HttpServletResponse response) throws IOException, ServletException {
    String userId = request.getParameter("userId");
    String password = request.getParameter("password");

    // 簡易認証（実際はDBで照合）
    if ("admin".equals(userId) && "pass1234".equals(password)) {
        // セッションIDを再生成（セキュリティ対策）
        HttpSession oldSession = request.getSession(false);
        if (oldSession != null) oldSession.invalidate();
        HttpSession session = request.getSession(true);
        session.setAttribute("loginUser", userId); // ユーザー情報を保存
        response.sendRedirect("home.jsp");          // ログイン後ページへ
    } else {
        request.setAttribute("error", "IDまたはパスワードが違います");
        request.getRequestDispatcher("login.jsp").forward(request, response);
    }
}

ログインチェック（未ログイン時のリダイレクト）

ログインが必要なページでは、セッションにユーザー情報があるか確認し、なければログインページへリダイレクトします。

home.jsp — ログインチェック 
// JSP の先頭でログインチェック
HttpSession session = request.getSession(false);
String loginUser = (session != null)
        ? (String) session.getAttribute("loginUser")
        : null;

// 未ログインならログインページへ強制転送
if (loginUser == null) {
    response.sendRedirect("login.jsp");
    return; // 以降の処理をしない
}

LogoutServlet（セッション無効化）

LogoutServlet.java — ログアウト処理 
protected void doGet(HttpServletRequest request,
        HttpServletResponse response) throws IOException {
    // セッションを取得（なければ null）
    HttpSession session = request.getSession(false);
    if (session != null) {
        session.invalidate(); // セッションを無効化してデータを消去
    }
    response.sendRedirect("login.jsp"); // ログインページへ転送
}

ログイン実装の重要ポイント

  • ログイン成功後はセッションIDを再生成する（セッションハイジャック対策）
  • ログインチェックは保護したい全ページの先頭に記述する
  • ログアウトは invalidate() でセッション全体を無効化する

7.5 つまずきやすいポイント

セッション管理でよく発生するトラブルと対処法をまとめました。

セッションが維持されない

ブラウザのCookieが無効になっていると、JSESSIONIDが送受信されずセッションが維持されません。

対処法

  • ブラウザのCookie設定を確認する
  • Cookieが使えない環境向けには、URLにセッションIDを埋め込む response.encodeURL(url) を使う（URLリライティング）

getAttribute() で ClassCastException が起きる

getAttribute() の戻り値は Object 型です。保存したオブジェクトと異なる型にキャストすると例外が発生します。

対処法

保存時と取得時で同じ型を使うこと。取得時はキャスト前に instanceof チェックや、null チェックを忘れずに行う。

セッションタイムアウト後の操作でエラーになる

しばらく操作しないとセッションが失効し、再度アクセスしたときに getAttribute() が null を返します。そのまま処理を続けると NullPointerException になります。

対処法

ログインチェックで getSession(false) を使い、セッションが null だった場合は「セッションが切れました。再ログインしてください」などのメッセージとともにログインページへリダイレクトする。

理解度確認クイズ

  1. Q1. HTTPがステートレスとはどういう意味ですか？セッションが必要な理由と合わせて説明してください。

    回答を表示

    HTTPはリクエストとレスポンスが完結するたびに状態を忘れる「ステートレス」なプロトコルです。そのため「前回のリクエストでログインした」という情報をHTTP自体は保持できません。セッションはサーバー側でユーザーごとのデータを保持し、JSESSIONIDというCookieでユーザーを識別することで、この制限を補う仕組みです。

  2. Q2. session.getAttribute("user") を使う際に注意すべきことを2つ挙げてください。

    回答を表示

    ① 戻り値はObject型なので、保存時と同じ型へのキャストが必要です。型が一致しない場合はClassCastExceptionが発生します。② セッションが存在しないか属性が未設定の場合はnullが返るため、NullPointerExceptionを避けるためnullチェックが必要です。

  3. Q3. ログアウト処理で session.invalidate() を呼ぶ理由を説明してください。

    回答を表示

    invalidate() を呼ぶとセッション全体が無効化され、サーバー側に保存されていたユーザー情報などのデータもすべて消去されます。これにより、ログアウト後に同じセッションIDを持つリクエストが来ても認証済みとして扱われなくなり、セキュリティを確保できます。単に属性を削除するだけでは不十分で、セッション自体を無効にすることが重要です。
