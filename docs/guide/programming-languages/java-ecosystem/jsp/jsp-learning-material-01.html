Webアプリケーション開発の世界へ

この章では、Web アプリケーションがどのような仕組みで動いているのかを理解し、HTTP プロトコルの基礎を学びます。そして、Java で Web アプリを動かすための土台となる Apache Tomcat をインストールし、最初の Web ページを表示するところまでを体験します。

この章で学ぶこと

  • Web アプリケーションの基本的な仕組み（クライアント・サーバーモデル）を理解する
  • HTTP プロトコル（リクエスト・レスポンス・メソッド・ステータスコード）の基礎を理解する
  • Apache Tomcat をインストールし、起動・停止・動作確認ができる
  • Web アプリのディレクトリ構成（WEB-INF・classes・lib）と web.xml の役割を理解する

1.1 Webアプリケーションとは

私たちが普段ブラウザで使っている、ネットショッピング・SNS・予約サイトなどは、すべて Web アプリケーションです。Web アプリケーションとは、ブラウザを通して利用する、サーバー上で動くプログラムのことです。まずは、その土台となる「クライアント・サーバーモデル」から理解していきましょう。

クライアント・サーバーモデル

Web の世界では、サービスを「お願いする側」と「応える側」に役割が分かれています。お願いする側を クライアント（＝あなたのブラウザ）、応える側を サーバーと呼びます。ブラウザが「このページを見せて」と要求（リクエスト）を送り、サーバーがそれに応えて結果（レスポンス）を返す——この往復が Web の基本です。

flowchart LR A["クライアント
（ブラウザ）"] -->|"① リクエスト
（ページをください）"| B["サーバー
（Tomcatなど）"] B -->|"② レスポンス
（HTMLを返す）"| A

静的Webサイトと動的Webアプリの違い

Web サイトには大きく2種類あります。あらかじめ用意された同じファイルを返すだけの「静的」なものと、アクセスのたびに内容を作り出す「動的」なものです。この違いが、Web アプリ開発の出発点になります。

種類 内容 例
静的Webサイト 誰がいつ見ても同じHTMLを返す。ファイルをそのまま配信する。 会社の紹介ページ、固定の説明ページ
動的Webアプリ アクセスや入力に応じて内容を作って返す。プログラムが動く。 ログイン後の画面、検索結果、ショッピングカート

Java Webアプリケーションの特徴と利点

動的な Web アプリを作る言語はいろいろありますが、本ガイドでは Java を使います。Java は企業システムで長年使われてきた実績があり、次のような利点があります。

企業システムでの実績

銀行・公共システムなど大規模で堅牢なシステムに広く採用されています。

豊富な部品とAPI

標準仕様（Servlet/JSP）やライブラリが整備され、再利用しやすい。

発展の土台になる

ここで学ぶ基礎は Spring などのフレームワークの理解にも直結します。

ServletとJSPの役割と関係性

Java の Web アプリ開発では、Servlet（サーブレット）と JSP（ジェイエスピー）という2つの技術が中心になります。ざっくり言うと、Servlet は「処理（プログラム）担当」、JSP は「画面（見た目）担当」です。両者は協力して1つのアプリを作ります。

技術 得意なこと イメージ
Servlet Java のロジック（計算・判定・データ取得）を書く Java の中に HTML を書く
JSP HTML（画面）を書き、必要な部分に Java を埋め込む HTML の中に Java を書く

この先の学習の流れ

本ガイドではまず Servlet（第2〜3章）を学び、次に JSP（第4〜5章）を学びます。そして両者を組み合わせた MVC パターン（第8章）で、実用的なアプリの作り方へと進んでいきます。今は「処理は Servlet、画面は JSP」とだけ覚えておけば十分です。

1.2 HTTPプロトコルの基礎

クライアントとサーバーは、HTTP（エイチティーティーピー）という共通の約束ごと（プロトコル）を使って会話します。Web アプリを作るうえで、この HTTP の仕組みを理解しておくことはとても重要です。難しく考えず、「決まった書式の手紙のやり取り」とイメージしてください。

リクエストとレスポンスの構造

HTTP のやり取りは、クライアントが送る リクエストと、サーバーが返す レスポンスの2つで構成されます。それぞれ「ヘッダー（付随情報）」と「ボディ（本体）」を持ちます。実際のリクエストはこのような姿をしています。

HTTPリクエストの例 
GET /shop/items HTTP/1.1     ← リクエスト行（メソッド＋URL＋バージョン）
Host: example.com            ← ヘッダー（あて先などの付随情報）
User-Agent: Mozilla/5.0
Accept: text/html
                             ← 空行のあとにボディ（GETでは通常空）

これを受け取ったサーバーは、次のようなレスポンスを返します。

HTTPレスポンスの例 
HTTP/1.1 200 OK              ← ステータス行（成功を表す200）
Content-Type: text/html; charset=UTF-8   ← ヘッダー（中身の種類）

<html>...</html>            ← ボディ（実際のHTML本体）

HTTPメソッド

リクエストの先頭にある メソッドは「何をしたいか」を表します。Web アプリでよく使うのは GET と POST の2つです。

メソッド 意味 主な用途
GET 情報を取得する ページの表示、検索
POST データを送信する フォーム送信、ログイン、登録
PUT データを更新する 既存データの置き換え（API など）
DELETE データを削除する 既存データの削除（API など）

ステータスコード

レスポンスの先頭にある ステータスコードは「結果がどうだったか」を3桁の数字で表します。開発中によく出会うものを覚えておきましょう。

200

OK（成功）

正常に処理できた。一番うれしい結果。

404

Not Found

指定したURLが見つからない。配置やURLの確認を。

500

Server Error

サーバー側のプログラムでエラー発生。ログを確認。

ステータスコードの分類

先頭の数字でおおまかな意味がわかります。2xx＝成功、3xx＝リダイレクト（転送）、4xx＝クライアント側の誤り、5xx＝サーバー側の誤り。「4から始まったら自分のリクエストを疑う、5から始まったらサーバーのプログラムを疑う」と覚えると便利です。

1.3 開発環境のセットアップ

いよいよ実際に手を動かす準備をします。Java Web アプリの開発には、JDK（Java 本体）と Apache Tomcat（Web アプリを動かすサーバー）が必要です。順番にそろえていきましょう。

① JDKのインストール確認

まず Java が使えるか確認します。本ガイドは JDK 17 以降を前提とします。ターミナル（コマンドプロンプト）で次を実行してください。

terminal — Javaのバージョン確認 
# Javaのバージョンを確認する
java -version

# 例: このように表示されればOK（17以降であること）
# openjdk version "17.0.10" 2024-01-16

「コマンドが見つかりません」と出る場合は JDK が未インストールか、環境変数 JAVA_HOME / PATH の設定が必要です。JDK を導入し、JAVA_HOME に JDK のインストール先フォルダを設定します。

② Apache Tomcatのインストール

Tomcat は、Servlet や JSP を動かすための「サーブレットコンテナ」と呼ばれるソフトです。公式サイトからダウンロードして展開（解凍）するだけで使えます。本ガイドは Jakarta 名前空間に対応した Tomcat 10.1 以降を使います。

バージョンに注意（javax → jakarta）

Tomcat 9 以前と Tomcat 10 以降では、使うパッケージ名が異なります。古い教材の javax.servlet.* は、Tomcat 10 以降では jakarta.servlet.* に変わりました。本ガイドは新しい jakarta を前提に解説します。インストール時は必ず Tomcat 10.1 以降を選んでください。

③ Tomcatの起動・停止と動作確認

展開した Tomcat の bin フォルダに起動・停止用のスクリプトがあります。起動したらブラウザで http://localhost:8080/ を開いて、Tomcat のトップ画面が出れば成功です。

terminal — Tomcatの起動と停止 
# Tomcatを起動する（Windowsは startup.bat）
./bin/startup.sh

# ブラウザで http://localhost:8080/ を開いて確認

# Tomcatを停止する（Windowsは shutdown.bat）
./bin/shutdown.sh

④ IDEとTomcatの連携

実際の開発では Eclipse や IntelliJ IDEA などの IDE（統合開発環境）を使うと効率的です。IDE に Tomcat を登録（サーバー設定）しておくと、ボタン1つでアプリの配置・起動・再起動ができます。設定方法は IDE ごとに異なりますが、「Server（サーバー）」設定で Tomcat のインストール先フォルダを指定するのが基本です。

ここまでの動作確認

次の状態になっていれば、開発環境の準備は完了です。

  • java -version で 17 以降が表示される
  • Tomcat を起動して http://localhost:8080/ が表示される

1.4 Webアプリケーションの構造

Java の Web アプリには、決められたフォルダ構成があります。この「お作法」を守ると、Tomcat が正しくアプリを認識して動かしてくれます。最初に全体像をつかんでおきましょう。

ディレクトリ構成

Tomcat の webapps フォルダの中に、アプリごとのフォルダ（＝アプリの入れ物）を置きます。標準的な構成は次のとおりです。

Webアプリケーションの標準構成 

webapps/
└── myapp/                  ← アプリ1つ分のフォルダ（コンテキスト）
    ├── index.html          ← 静的リソース（誰でもアクセス可）
    ├── style.css           ← CSSなどもここに置ける
    └── WEB-INF/            ← 非公開フォルダ（直接アクセス不可）
        ├── web.xml         ← 設定ファイル（任意）
        ├── classes/        ← コンパイル済みServlet（.class）を置く
        └── lib/            ← 外部ライブラリ（.jar）を置く

WEB-INFという特別なフォルダ

WEB-INF はブラウザから直接アクセスできない特別なフォルダです。ここに Java のプログラム（classes）やライブラリ（lib）、設定ファイル（web.xml）を置くことで、外部から中身を覗かれないように保護します。

場所 役割
WEB-INF/classes/ 自分で書いた Servlet などのコンパイル済み .class ファイルを置く
WEB-INF/lib/ 外部のライブラリ（JDBCドライバや JSTL の .jar）を置く
WEB-INF/web.xml アプリの設定ファイル（デプロイメント記述子）

web.xmlの役割

web.xml は、アプリ全体の設定を書くファイルです。最近はアノテーション（第2章で学びます）で設定できるため必須ではなくなりましたが、基本構造は知っておきましょう。

WEB-INF/web.xml の基本構造 
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
         version="6.0">
    <!-- ここに設定（Servletの登録など）を書く -->
    <welcome-file-list>
        <welcome-file>index.html</welcome-file>
    </welcome-file-list>
</web-app>

コンテキストパスとURLの関係

webapps に置いたフォルダ名が、URL の一部（コンテキストパス）になります。たとえば webapps/myapp/ に index.html を置くと、次の URL で表示されます。

flowchart LR A["http://localhost:8080"] --> B["/myapp
（コンテキストパス＝フォルダ名）"] B --> C["/index.html
（フォルダ内のファイル）"]

つまり http://localhost:8080/myapp/index.html でアクセスできます。URL とファイルの置き場所が対応している——これが分かると、404 エラーの原因も突き止めやすくなります。

1.5 つまずきやすいポイント

環境構築は最初の関門です。よくあるトラブルと対処法をまとめました。困ったらここに戻ってきてください。

Tomcatが起動しない

よくある原因は2つです。

  • ポート競合（8080）: 他のアプリが 8080 番ポートを使っていると起動できません。そのアプリを止めるか、Tomcat の設定（server.xml）でポート番号を変更します。
  • JAVA_HOME 未設定: 環境変数 JAVA_HOME が JDK を指していないと起動に失敗します。設定を確認しましょう。

404エラーが出る

URL とファイルの置き場所が対応していないのが原因です。「コンテキストパス（フォルダ名）」と「ファイル名」が URL と一致しているか、配置場所を見直しましょう（1.4 参照）。

日本語が文字化けする

ファイルの保存エンコーディングが UTF-8 になっているか確認します。HTML には <meta charset="UTF-8"> を入れましょう。エンコーディングの詳しい扱いは第3章で学びます。

やってみよう 1-1: 最初のWebページを表示する

Tomcat を起動し、自分で作った HTML ファイルをブラウザに表示してみましょう。Web アプリ開発の第一歩です。

手順

  1. 1 Tomcat を起動し、ブラウザで http://localhost:8080/ にアクセスして管理トップ画面が出ることを確認する。
  2. 2 webapps の中に myapp フォルダを作り、index.html に「Hello, Web!」と書いて保存する。
  3. 3 http://localhost:8080/myapp/index.html にアクセスして「Hello, Web!」が表示されることを確認する。
  4. 4 ブラウザの開発者ツール（F12キー）を開き、「ネットワーク」タブでページを再読み込みして、HTTP リクエスト・レスポンス（ステータスコード 200 やヘッダー）を観察する。

理解度確認クイズ

  1. Q1. 静的Webサイトと動的Webアプリケーションの違いを説明してください。

    回答を表示

    静的Webサイトは、誰がいつ見ても同じHTMLファイルをそのまま返します。動的Webアプリは、アクセスや入力に応じてプログラムが内容を作り出して返します。ログイン後の画面や検索結果のように、状況によって表示が変わるのが動的です。

  2. Q2. ServletとJSPは、それぞれどんな役割が得意ですか？

    回答を表示

    Servlet は処理（プログラム・ロジック）担当で「Javaの中にHTMLを書く」イメージ、JSP は画面（見た目）担当で「HTMLの中にJavaを書く」イメージです。両者が協力して1つの Web アプリを作ります。

  3. Q3. ステータスコード 404 と 500 は、それぞれ何を表しますか？

    回答を表示

    404（Not Found）は、指定した URL が見つからないことを表し、URL やファイルの配置を見直します。500（Server Error）は、サーバー側のプログラムでエラーが起きたことを表し、サーバーのログを確認します。
