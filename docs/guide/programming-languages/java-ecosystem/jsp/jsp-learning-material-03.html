リクエストとレスポンス - データの受け渡し

この章では、ブラウザから送られてくるデータ（リクエストパラメータ）を Servlet でどう受け取るかを学びます。GET と POST の違い、HttpServletRequest / HttpServletResponse の使い方、そして日本語を正しく扱うための文字エンコーディング設定まで、Servlet 開発の核心をマスターします。

この章で学ぶこと

  • getParameter() などを使い、フォームのリクエストパラメータを取得できる
  • GET と POST の違いを理解し、用途に応じて使い分けられる
  • HttpServletResponse を使って適切なレスポンスを生成できる
  • 文字エンコーディングを正しく設定し、日本語の文字化けを防げる

3.1 HttpServletRequestの活用

ブラウザがフォームを送信すると、入力された値は HTTP リクエストに乗ってサーバーへ届きます。Servlet はその値を HttpServletRequest オブジェクト経由で取り出します。「パラメータ」という言葉をキーワードに整理していきましょう。

リクエストパラメータとは

ブラウザが送ってくる「名前と値のペア」のことを リクエストパラメータと呼びます。たとえば検索フォームで「Java」と入力して送信すると、keyword=Java という形でサーバーに届きます。Servlet はこれをキー（名前）を指定して取り出します。

flowchart LR A["ブラウザ
（フォーム入力）"] -->|"keyword=Java＆lang=ja"| B["Servlet
（HttpServletRequest）"] B -->|"getParameter（keyword）で取得"| C["取得した値「Java」"]

主要なパラメータ取得メソッド

HttpServletRequest には、パラメータを取り出すメソッドが3種類あります。状況に合わせて使い分けましょう。

メソッド 戻り値 使いどき
getParameter(String name) String（またはnull） 1つの値を取る（テキスト入力など）
getParameterValues(String name) String[]（またはnull） 複数値を取る（チェックボックスなど）
getParameterNames() Enumeration<String> 送られたパラメータ名を全列挙する
Java — パラメータ取得の基本パターン 
// 単一パラメータの取得（nullチェックが必須）
String name = request.getParameter("username");
if (name != null) {
    // 値を使う処理
}

// 複数値の取得（チェックボックスなど）
String[] hobbies = request.getParameterValues("hobby");
if (hobbies != null) {
    for (String hobby : hobbies) {
        // 各値を処理
    }
}

パラメータとリクエスト属性の違い

「パラメータ」とよく混同されるのが リクエスト属性です。両者は似ていますが、用途がまったく異なります。パラメータはブラウザから送られてくる外部の値、属性は Servlet が自分でセットするサーバー内部の値です。

種類 セットする側 取得メソッド 主な用途
リクエストパラメータ ブラウザ（外部） getParameter() フォーム入力値の受け取り
リクエスト属性 Servlet（内部） getAttribute() Servlet から JSP へのデータ渡し
Java — リクエスト属性のセットと取得 
// Servletで属性をセット（JSPへ渡すデータを準備）
request.setAttribute("message", "処理が完了しました");

// 後で getAttribute() で取得（JSP側で使う）
String msg = (String) request.getAttribute("message");

パラメータ vs 属性 — 覚え方

「パラメータはブラウザから届く郵便」、「属性はサーバー内のメモ用紙」とイメージすると覚えやすいです。属性は Servlet から JSP へデータを引き渡す際（第7章のMVCパターン）に欠かせない仕組みです。

3.2 GETリクエストの処理

GET はブラウザがページを取得するときに使うメソッドです。パラメータは クエリ文字列として URL の末尾に付加されます。「?」の後ろに続く 名前=値 の形式を見たことがあるでしょうか。これがクエリ文字列です。

クエリ文字列の構造

flowchart LR A["http://localhost:8080/myapp/search"] -->|"？マークの後"| B["name=山田"] B -->|"＆で連結"| C["age=25"]

複数のパラメータは & で繋ぎます。Servlet 側ではこれを意識せずに getParameter() で取り出せます。また、URL に使えない文字（日本語やスペースなど）は URLエンコーディング（%エンコード）に変換されて送信されます。Servlet は自動でデコードしてくれるので、通常は意識不要です。

GETの用途と制限

GETが向いている用途

  • ページの閲覧・検索
  • ブックマークしたいページ
  • URLを共有したい場合

GETの制限・注意点

  • URLの長さに制限あり（数KB程度）
  • パラメータがURLに露出する
  • ブラウザ履歴・サーバーログに残る

簡単な検索フォームの例

HTML フォームで method="get"（または省略）とすると、送信時にパラメータが URL に付いて Servlet へ届きます。

search.html — 検索フォーム（GET） 
<!-- GETで送信：URLにパラメータが付く -->
<form action="SearchServlet" method="get">
    <input type="text" name="keyword" placeholder="検索キーワード">
    <button type="submit">検索</button>
</form>
Java — SearchServlet.java（doGetでパラメータを受け取る） 
@WebServlet("/SearchServlet")
public class SearchServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // URLから ?keyword=Java のように届くパラメータを取得
        String keyword = request.getParameter("keyword");
        if (keyword == null || keyword.isEmpty()) {
            keyword = "（未入力）";
        }
        // ... keyword を使った処理を続ける
    }
}

3.3 POSTリクエストの処理

POST はデータをサーバーへ送信するときに使うメソッドです。パラメータは URL ではなく リクエストボディの中に入るため、URL には現れません。ログインフォームや登録フォームは必ず POST を使います。

GETとPOSTの比較

観点 GET POST
パラメータの場所 URL（クエリ文字列） リクエストボディ（URL に出ない）
データ量の制限 数KB程度（URL長制限） 実質制限なし（大量データも可）
ブックマーク 可能 不可
セキュリティ URLに露出（パスワード不可） ボディに隠れる（ログイン・登録向き）
主な用途 検索・閲覧・URL共有 ログイン・登録・更新・削除

getParameter() はGETでもPOSTでも同じ

パラメータの取得コードは GET と POST で同じです。request.getParameter("name") を呼ぶだけで、URL からでもボディからでも自動的に取り出してくれます。違いは doGet() か doPost() かというメソッドの選択だけです。

POSTフォームとdoPost()の実装例

login.html — ログインフォーム（POST） 
<!-- POSTで送信：パスワードがURLに出ない -->
<form action="LoginServlet" method="post">
    <input type="text"     name="username" placeholder="ユーザー名">
    <input type="password" name="password" placeholder="パスワード">
    <button type="submit">ログイン</button>
</form>
Java — LoginServlet.java（doPostでパラメータを受け取る） 
@WebServlet("/LoginServlet")
public class LoginServlet extends HttpServlet {
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        // GETと同じ getParameter() で取得できる
        String username = request.getParameter("username");
        String password = request.getParameter("password");
        // username, password を使ってログイン処理を行う
    }
}

3.4 HttpServletResponseの活用

リクエストを受け取ったら、次はブラウザへ レスポンスを返す番です。Servlet は HttpServletResponse オブジェクトを通じてレスポンスを組み立てます。

主要なResponseメソッド

メソッド 役割
setContentType(String type) レスポンスの種類を指定（例: "text/html; charset=UTF-8"）
setCharacterEncoding(String enc) 文字エンコーディングを指定（"UTF-8"）
getWriter() テキスト出力用の PrintWriter を取得する
getOutputStream() バイナリ出力用のストリームを取得する（画像など）
sendRedirect(String url) ブラウザを別URLへリダイレクトさせる
Java — レスポンスの基本パターン 
// コンテンツタイプ＋文字コードを設定してから出力する
response.setContentType("text/html; charset=UTF-8");

// PrintWriter でHTMLを書き出す
PrintWriter out = response.getWriter();
out.println("<html><body>");
out.println("<h1>Hello, Servlet!</h1>");
out.println("</body></html>");

リダイレクトの仕組み

sendRedirect() を使うと、処理後にブラウザを別の URL へ転送できます。ログイン後にトップページへ移動させる、登録完了後に一覧ページへ戻すといった場面で活躍します。

flowchart LR A["ブラウザ"] -->|"POSTリクエスト"| B["Servlet
（処理実行）"] B -->|"sendRedirect（302）"| A A -->|"GETリクエスト
（新URLへ）"| C["次のページ"]
Java — sendRedirectの例 
// 処理成功後、一覧ページへリダイレクト
// sendRedirectの後はreturnでServletの処理を終了する
response.sendRedirect(request.getContextPath() + "/list");
return;

getWriter() と getOutputStream() は排他

1つのレスポンスで getWriter() と getOutputStream() を同時に呼ぶと IllegalStateException が発生します。テキスト（HTML）は getWriter()、バイナリ（画像・PDF）は getOutputStream() と使い分けてください。

3.5 文字エンコーディング対策

日本語を含む Web アプリで最も多いトラブルが 文字化けです。Servlet で正しく日本語を扱うには、リクエスト・レスポンスそれぞれにエンコーディングを設定する必要があります。設定の順序が重要——このルールを守れば文字化けのほとんどは防げます。

文字化けが起きる仕組み

flowchart TD A["ブラウザ（UTF-8で送信）"] -->|"文字化け！"| B{"サーバーの
デフォルト文字コード
が違う"} B -->|"エンコード設定なし"| C["ISO-8859-1で読む
→ 文字化け発生"] B -->|"request.setCharacterEncoding
（UTF-8）を設定"| D["UTF-8で正しく読む
→ 正常"]

エンコーディング設定のルール

重要: エンコーディングはパラメータ取得より「前」に設定する

request.setCharacterEncoding() は getParameter() を呼ぶ前に設定しないと効果がありません。Servlet コンテナはパラメータの読み込みを一度しか行わないため、後から設定しても間に合いません。

Java — 文字化け対策の正しい順序 
protected void doPost(HttpServletRequest request,
                      HttpServletResponse response)
        throws ServletException, IOException {

    // ① 必ず最初にエンコーディングを設定する（getParameterの前！）
    request.setCharacterEncoding("UTF-8");

    // ② レスポンスの文字コードも設定する
    response.setContentType("text/html; charset=UTF-8");

    // ③ エンコーディング設定後にパラメータを取得する
    String name = request.getParameter("name");
    PrintWriter out = response.getWriter();
    out.println("<p>こんにちは、" + name + "さん！</p>");
}

GETとPOSTで異なる文字化け対策

POST のリクエストボディには request.setCharacterEncoding() が効きますが、GET のクエリ文字列は Tomcat サーバー側の設定（server.xml の URIEncoding）に依存します。Tomcat 8.5 以降はデフォルトが UTF-8 になっているため、通常は追加設定不要です。

リクエスト種別 文字化け対策 設定場所
POST request.setCharacterEncoding("UTF-8") doPost() の先頭（getParameterより前）
GET Tomcat 8.5+ はデフォルトUTF-8。古い場合は server.xml の URIEncoding を設定 server.xml（サーバー設定）
レスポンス response.setContentType("text/html; charset=UTF-8") getWriter() より前

Jakarta EE ではFilterで一元管理も可能

本番アプリでは、すべての Servlet に個別設定するのは大変です。jakarta.servlet.Filter を使って「全リクエストの前処理として UTF-8 を設定する」という共通フィルターを作ると管理が楽になります。Spring など主要フレームワークもこの仕組みを使っています。

3.6 つまずきやすいポイント

リクエスト・レスポンスの処理でよくはまるポイントをまとめました。困ったときはここを確認してください。

パラメータが null になる

原因は主に2つです。

  • パラメータ名の大文字小文字が一致していない: フォームの name="userName" に対して getParameter("username")（小文字）を呼ぶと null になります。名前を完全に一致させましょう。
  • nullチェックを忘れている: パラメータが送られなかった場合は null が返ります。if (value != null && !value.isEmpty()) で必ず確認しましょう。

日本語が文字化けする

エンコーディング設定の順序が間違っています。request.setCharacterEncoding("UTF-8") は getParameter() を呼ぶ前に書く必要があります（3.5 参照）。また、POST か GET かで対処法が異なる点も確認しましょう。

POSTでパラメータが取れない

HTMLフォームの method 属性が post になっているか確認してください。省略するとデフォルトの GET になり、doPost() が呼ばれません。また Servlet の @WebServlet アノテーションの URL パターンがフォームの action と一致しているかも確認しましょう。

やってみよう 3-1: 挨拶Servlet

名前を入力すると「こんにちは、〇〇さん！」と挨拶を返す Servlet を作ってみましょう。

手順

  1. 1 greeting.html に名前を入力するフォーム（method="post"）を作り、action="GreetingServlet" とする。
  2. 2 GreetingServlet の doPost() で エンコーディングを先に設定し、パラメータ name を取得する。
  3. 3 name が null または空の場合は「名前を入力してください」と表示するよう null チェックを入れる。
  4. 4 日本語の名前（例: 「山田太郎」）を入力して文字化けなく表示されることを確認する。

やってみよう 3-2: 簡易電卓Servlet

2つの数値と演算子を受け取り、計算結果を表示する電卓 Servlet を作ってみましょう。

  1. 1 フォームに数値2つ（num1/num2）と演算子の選択（operator: ＋・－・×・÷）を配置する。
  2. 2 Servlet で Integer.parseInt() を使って数値に変換し、演算子で場合分けして計算結果を出力する。
  3. 3 0で割ったときのエラー処理（ArithmeticException）を try-catch で実装する。

やってみよう 3-3: アンケートフォームServlet

複数のチェックボックスを含むアンケートを受け取り、選択内容を一覧表示する Servlet を作りましょう。

  1. 1 フォームに同じ name="hobby" を持つチェックボックスを複数作る（読書・音楽・スポーツなど）。
  2. 2 Servlet で getParameterValues("hobby") を使って全選択値を取得し、null チェックをした上で <ul> リストで表示する。
  3. 3 1つも選ばなかった場合は「選択なし」と表示する分岐を追加する。

理解度確認クイズ

  1. Q1. GETとPOSTの違いを、「パラメータの場所」と「主な用途」の2点から説明してください。

    回答を表示

    GETはパラメータがURLのクエリ文字列に付くため、検索やページ閲覧など「情報の取得」に向いています。POSTはパラメータがリクエストボディに入りURLに現れないため、ログインや登録など「データの送信」に使います。パスワードのような機密情報はGETのURLに露出するので必ずPOSTを使います。

  2. Q2. 次のコードで日本語が文字化けする原因と修正方法を答えてください。
    String name = request.getParameter("name"); request.setCharacterEncoding("UTF-8");

    回答を表示

    getParameter()を先に呼んでしまい、その後でsetCharacterEncoding()を設定しているため効果がありません。Servlet コンテナはパラメータを一度読んだらキャッシュするので、後から設定しても間に合いません。修正は順序を入れ替えて、request.setCharacterEncoding("UTF-8")を先に呼んでからgetParameter()を呼ぶようにします。

  3. Q3. リクエスト「パラメータ」と「属性」の違いを、誰がセットするかという観点で説明してください。

    回答を表示

    リクエストパラメータはブラウザ（外部）が ?name=値 のようにリクエストに付けて送ってくる値で、getParameter() で取得します。リクエスト属性はサーバー側の Servlet が setAttribute() で自ら設定するサーバー内部の値で、JSP などに処理結果を渡す用途に使います。
