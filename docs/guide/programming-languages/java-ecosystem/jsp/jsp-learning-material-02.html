はじめてのServlet - Javaでリクエストを処理する

この章では、Java の Web アプリ開発の核心である Servlet（サーブレット）とは何かを理解し、実際に最初の Servlet を作成します。Servlet のライフサイクルや URL マッピングの仕組みを学び、リクエストに応じたレスポンスを返せるようになることを目指します。

この章で学ぶこと

  • Servlet の概念と役割、サーブレットコンテナ（Tomcat）の働きを理解する
  • HttpServlet を継承した簡単な Servlet を作成できる
  • @WebServlet アノテーションと web.xml の両方で URL マッピングができる
  • Servlet のライフサイクル（init → service → destroy）を理解する

2.1 Servletとは何か

Servlet（サーブレット）とは、Java で書かれたサーバーサイドのプログラムで、ブラウザからの HTTP リクエストを受け取り、処理を行い、HTTP レスポンスを返す役割を担います。「Servlet」という名前は "Server + Applet" の造語です。

サーブレットコンテナ（Tomcat）の役割

Servlet 単体では動作しません。Servlet を動かすには、サーブレットコンテナ（別名: Web コンテナ）が必要です。第1章で使った Apache Tomcat がまさにこれです。コンテナは Servlet の生成・初期化・呼び出し・破棄まで一貫して管理します。私たちは「どんな処理をするか」を Servlet に書くだけで、HTTP 通信の細かい制御はコンテナに任せられます。

flowchart LR A["ブラウザ
（クライアント）"] -->|"① HTTPリクエスト"| B["Tomcat
（サーブレットコンテナ）"] B -->|"② URLに対応する
Servletを呼び出す"| C["HelloServlet
（あなたのコード）"] C -->|"③ 処理結果を返す"| B B -->|"④ HTTPレスポンス"| A

CGIとの違いとServletの利点

Servlet 以前、サーバーサイド処理の主役は CGI（Common Gateway Interface）でした。CGI はリクエストごとに新しいプロセス（OS のプログラム単位）を起動するため、大量アクセス時に重くなる欠点がありました。Servlet はリクエストを スレッド（軽量な処理単位）で処理するため、はるかに効率的です。

比較項目 CGI Servlet
リクエスト処理の単位 プロセス（重い） スレッド（軽い）
インスタンスの生成 リクエストごとに生成 起動時に1度だけ生成・使い回し
大量アクセスへの対応 苦手（負荷が高い） 得意（効率的）
型安全・再利用性 低い 高い（Java の型システムを活用）

Jakarta Servlet API（javaxからjakartaへ）

Servlet の仕様は Jakarta EE（旧 Java EE）として標準化されています。2020年、仕様の管理が Oracle から Eclipse Foundation に移管された際に、パッケージ名が javax.servlet.* から jakarta.servlet.* に変更されました。

名前空間の変更に注意

Tomcat 10 以降（Jakarta EE 9+）では jakarta.servlet.* を使います。Tomcat 9 以前の古い教材では javax.servlet.* が使われていますが、Tomcat 10 以降ではコンパイルエラーになります。本ガイドは jakarta 名前空間を前提に解説します。

2.2 最初のServlet作成

いよいよ Servlet を作ってみましょう。Servlet を作るには、HttpServlet クラスを継承して、必要なメソッドをオーバーライドするのが基本です。

HttpServletクラスの継承

jakarta.servlet.http.HttpServlet は、HTTP 通信の処理をまとめた便利なクラスです。これを継承することで、HTTP に関する多くの処理が自動で行われ、私たちは「GET リクエスト時の処理」や「POST リクエスト時の処理」だけを書けばよくなります。

doGet()メソッドのオーバーライド

ブラウザがURLを開くとき（GETリクエスト）に呼ばれるのが doGet() メソッドです。引数として受け取る2つのオブジェクトが重要です。

HttpServletRequest

ブラウザから届いたリクエスト情報を持つオブジェクト。パラメータ・ヘッダー・セッションなどが取れる。

HttpServletResponse

ブラウザへ返すレスポンス情報を操作するオブジェクト。コンテンツタイプの設定や出力ストリームの取得ができる。

HelloServlet の完全なコード例

以下が最初の Servlet の完全なコードです。response.getWriter().println(...) でレスポンスに文字列を書き出します。

HelloServlet.java 
import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import java.io.IOException;
import java.io.PrintWriter;

// このURLパターンにアクセスされたらこのクラスが動く
@WebServlet("/hello")
public class HelloServlet extends HttpServlet {

    // GETリクエスト時に呼ばれるメソッド
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // レスポンスの文字コードとコンテンツタイプを設定
        response.setContentType("text/html; charset=UTF-8");

        // 出力ストリームを取得して文字列を書き出す
        PrintWriter out = response.getWriter();
        out.println("<html><body>");
        out.println("<h1>Hello, Servlet!</h1>");
        out.println("</body></html>");
    }
}

throws ServletException, IOException について

doGet() メソッドのシグネチャには throws ServletException, IOException が必須です。これを省略するとオーバーライドではなく別メソッドの定義になり、Tomcat から呼ばれなくなります。第1章で学んだ「第3章で詳しく扱う」例外処理の基礎にもなります。

2.3 Servletのマッピング

Servlet を作っただけでは、どの URL でアクセスすればいいかが分かりません。「どの URL に来たリクエストをどの Servlet で処理するか」を定義するのが URLマッピングです。方法は2つあります。

①アノテーション方式（@WebServlet）

最も簡単な方法は、クラスに @WebServlet アノテーションを付けることです。urlPatterns 属性（または省略形の value 属性）でパターンを指定します。

@WebServlet アノテーションの書き方 
// シンプルな書き方（1つのURLを指定）
@WebServlet("/hello")

// urlPatterns 属性で明示的に指定する書き方
@WebServlet(urlPatterns = "/hello")

// 複数のURLパターンをまとめて指定する書き方
@WebServlet(urlPatterns = {"/hello", "/greeting"})

②web.xml方式（従来の設定ファイル）

アノテーション登場以前は WEB-INF/web.xml に記述していました。既存の大規模システムや、アノテーションでは難しい細かい設定が必要な場面で今も使われます。

WEB-INF/web.xml — Servletのマッピング設定 
<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee" version="6.0">

    <!-- Servletの定義（クラスに名前をつける） -->
    <servlet>
        <servlet-name>helloServlet</servlet-name>
        <servlet-class>HelloServlet</servlet-class>
    </servlet>

    <!-- URLとServletを対応づける -->
    <servlet-mapping>
        <servlet-name>helloServlet</servlet-name>
        <url-pattern>/hello</url-pattern>
    </servlet-mapping>

</web-app>

アノテーションとweb.xmlの対比

比較項目 @WebServlet アノテーション web.xml 設定
書く場所 Servlet クラスファイル内 WEB-INF/web.xml
設定の分かりやすさ コードとマッピングが一目でわかる ファイルが分かれており、全体像を把握しやすい
主な用途 新規開発・小〜中規模 既存システム・高度な設定が必要な場合
再ビルドが必要か URLを変えたらコンパイルが必要 XML を書き換えるだけで変更できる場合も

ワイルドカードパターン

URL パターンにはワイルドカードが使えます。複数の URL を一括で受け付けたいときに便利です。

パターン マッチする URL 例 用途
/hello /hello のみ 特定の1つのURL
/api/* /api/users, /api/items など パス前方一致
*.do login.do, search.do など 拡張子一致
/ 他のパターンにマッチしない全URL デフォルトサーブレット

2.4 Servletのライフサイクル

Servlet には「生まれてから死ぬまで」の決まった流れ（ライフサイクル）があります。コンテナ（Tomcat）がこの流れを管理し、私たちは各フェーズに対応するメソッドを実装するだけです。

flowchart TD A["初回リクエスト到着
（コンテナが検知）"] --> B["インスタンス生成
new HelloServlet（）"] B --> C["init（） — 初期化
1度だけ呼ばれる"] C --> D["service（） — 振り分け
リクエストごとに呼ばれる"] D -->|"GETリクエスト"| E["doGet（）
実装するメソッド"] D -->|"POSTリクエスト"| F["doPost（）
実装するメソッド"] E --> G["レスポンスを返す"] F --> G G -->|"次のリクエスト"| D G -->|"アプリ停止時"| H["destroy（） — 終了処理
1度だけ呼ばれる"]

各メソッドの役割

① init（） — 初期化

Servlet インスタンス生成直後に1度だけ呼ばれます。データベース接続の初期化など、起動時に1回だけやりたい処理をここに書きます。

public void init() throws ServletException { ... }

② service（） — 振り分け

リクエストのたびに呼ばれ、HTTP メソッド（GET/POST など）を判定して doGet() や doPost() に振り分けます。通常はオーバーライドしません。

protected void service（request, response） { ... }

③ doGet（）/ doPost（） — メインの処理

私たちが主に実装するメソッドです。doGet() は GET リクエスト、doPost() は POST リクエスト（フォーム送信など）を処理します。

protected void doGet（request, response） throws ... { ... }

④ destroy（） — 終了処理

アプリ停止・サーバーシャットダウン時に1度だけ呼ばれます。データベース接続のクローズなど、終了時のクリーンアップ処理をここに書きます。

public void destroy（） { ... }

Servletはシングルトン的に動く

非常に重要なポイントです。Tomcat は Servlet のインスタンスを初回リクエスト時に1つだけ生成し、それをすべてのリクエストで使い回します。これをシングルトン的な動作と呼びます。

インスタンス変数は使わないこと

Servlet のインスタンス変数（フィールド）は複数のリクエストで共有されてしまいます。リクエストごとに独立した値を持ちたい場合は、必ず doGet() などのメソッド内でローカル変数として宣言してください。意図せず値が混ざるバグの原因になります。

2.5 つまずきやすいポイント

Servlet 開発で初心者がよく詰まる場面と、その対処法を紹介します。困ったときのチェックリストとして活用してください。

クラスが見つからない（ClassNotFoundException）

Servlet のクラスが見つからないと Tomcat がエラーを出します。次を確認してください。

  • 配置場所: コンパイル済み .class ファイルは WEB-INF/classes/ 以下に置く。パッケージがある場合は WEB-INF/classes/com/example/HelloServlet.class のようにサブフォルダも対応させる。
  • コンパイル: IDE の「ビルド」を実行して最新の .class が生成されているか確認する。

404エラー（ページが見つからない）

Servlet は存在するのに 404 が出る場合、URLマッピングが原因のことがほとんどです。

  • @WebServlet("/hello") と指定したなら、ブラウザで開くのは http://localhost:8080/myapp/hello（コンテキストパス＋urlPatterns）。
  • web.xml の <url-pattern> と実際のアクセス URL が一致しているか確認する。

doGet/doPostが呼ばれない

メソッドをオーバーライドできていない可能性があります。次の3点を確認してください。

  • 引数の型: HttpServletRequest・HttpServletResponse が正確に指定されているか。
  • 例外宣言: throws ServletException, IOException が抜けていないか。
  • @Override アノテーション: @Override を付けるとシグネチャが間違っていてもコンパイルエラーになるため、ミスに気づきやすい。

やってみよう 2-1: Servletを作成して動かす

3つの Servlet を作り、Tomcat で動かしてみましょう。それぞれの課題を通じて、Servlet の基本を体験します。

課題①: "Hello, Servlet!" を表示する

  1. 1 2.2 節の HelloServlet.java をそのまま実装し、WEB-INF/classes/ にコンパイルする。
  2. 2 Tomcat を起動し、http://localhost:8080/myapp/hello にブラウザでアクセスして "Hello, Servlet!" が表示されることを確認する。

課題②: 現在の日時を表示する

  1. 1 @WebServlet("/datetime") をつけた DateTimeServlet を作成する。
  2. 2 doGet() 内で new java.util.Date() を取得し、out.println() で出力する。
  3. 3 ブラウザをリロードするたびに表示時刻が変わることを確認する（動的に生成されていることの確認）。

課題③: HTMLリストを動的に生成する

  1. 1 @WebServlet("/fruits") をつけた FruitListServlet を作成する。
  2. 2 String[] fruits = {"りんご", "みかん", "バナナ"}; を定義し、for ループで <ul><li>...</li></ul> を出力する。
  3. 3 ブラウザで確認し、配列の中身を変えて再コンパイル・再デプロイすると表示が変わることを確かめる。

理解度確認クイズ

  1. Q1. Servlet と CGI の最大の違いは何ですか？パフォーマンスの観点から説明してください。

    回答を表示

    CGI はリクエストのたびに新しいプロセス（重い処理単位）を起動するため、大量アクセス時に負荷が高くなります。Servlet はインスタンスを1つだけ生成して使い回し、リクエストはより軽い「スレッド」で処理するため、大量アクセスでも効率的に動作します。

  2. Q2. Servlet のライフサイクルで、init() と destroy() はそれぞれ何回呼ばれますか？またどんな処理を書きますか？

    回答を表示

    どちらも1度だけ呼ばれます。init() はインスタンス生成直後の初期化タイミングで呼ばれ、データベース接続の初期化など起動時に1回だけ実行したい処理を書きます。destroy() はアプリ停止時に呼ばれ、接続のクローズなど終了時のクリーンアップ処理を書きます。

  3. Q3. @WebServlet("/shop/*") とマッピングした Servlet に /shop/items や /shop/cart でアクセスできますか？また /products でのアクセスはどうなりますか？

    回答を表示

    /shop/items と /shop/cart はともに /shop/* パターンにマッチするため、この Servlet で処理されます。/products は /shop で始まらないためマッチせず、他のマッピングやデフォルトサーブレットに委ねられます（設定がなければ 404 になります）。
