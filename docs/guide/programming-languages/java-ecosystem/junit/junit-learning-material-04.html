ホーム JUnit 第4章
第4章 所要時間: 約45分

テストを整理整頓しよう テストクラスの構成

複数のテストをどう整理するかを学びます。テストクラスの命名規則、テストとソースコードの配置のしかた、テストメソッドの書き方、@Test アノテーションの意味を丁寧に解説します。本棚に本を整理するように、テストも整理することの大切さを、実際のコードを通して身につけましょう。

この章で学ぶこと

  • テストクラスの命名規則（「テスト対象クラス名 + Test」の慣習）を説明できる
  • テストクラスとテスト対象クラスを正しい場所に配置できる
  • @Test アノテーションの意味とテストメソッドの書き方を理解する
  • AAAパターン（Arrange-Act-Assert）でテストを読みやすく書ける

4.1 本棚に本を整理するように

第3章では、テストの「判定」に使う アサーション を学びました。でも、実際のプロジェクトではテストが1つだけということはありません。電卓アプリなら「足し算のテスト」「引き算のテスト」「ゼロ割りのテスト」……と、テストはどんどん増えていきます。

テストが増えたとき、すべてをひとつのファイルにごちゃ混ぜに書いてしまうと、あとから「どこに何のテストがあるのか」が分からなくなってしまいます。これは、本棚の本をジャンルに関係なく無造作に詰め込んでしまった状態と同じです。

たとえ話: 本棚の整理術

  • ・料理本は料理の棚へ、プログラミング本はプログラミングの棚へ分ける
  • ・同じジャンルの本は、ひとつの棚にまとめておくと探しやすい
  • ・本の背表紙（タイトル）を見れば、中身の見当がつく

テストも同じです。「テスト対象のクラス（コード）1つに対して、テストクラス1つ」を用意し、関連するテストをまとめてあげるのが基本の整理術です。

Java（JUnit）の世界では、テストは テストクラス という入れ物にまとめます。そして、そのクラスの中にテストメソッドをずらりと並べていきます。この章では、テストクラスをどう作り、どう整理するかを学んでいきましょう。

4.2 テストクラスの命名規則

JUnit では、テストのコードも Java のクラス として書きます。このクラスを テストクラス と呼びます。テストクラスには、長年 Java コミュニティに受け継がれてきたシンプルな命名ルールがあります。

命名規則: テスト対象のクラス名 + Test

テストしたいクラスの名前の末尾に Test を付けるだけです。

  • ・Calculator（電卓クラス）→ テストクラスは CalculatorTest
  • ・StringUtils（文字列ユーティリティ）→ テストクラスは StringUtilsTest
  • ・OrderService（注文サービス）→ テストクラスは OrderServiceTest

この「〜Test」という命名規則は、Java の世界でほぼ全員が守っている慣習です。ルールとして厳密に決まっているわけではありませんが、「このクラスはテストだ」と一目で分かるようにするために、チーム全体で守るのが一般的です。プロジェクトによっては「〜Tests」（複数形）や「Test〜」（前置き）の場合もありますが、「〜Test」が最もよく見かけるパターンです。

良い例

  • CalculatorTest
  • StringUtilsTest
  • UserServiceTest

名前を見ただけで「何のテストか」が分かる

避けたい例

  • TestClass1
  • MyTest
  • Hoge

何をテストしているのか名前から分からない

4.3 テストクラスとテスト対象の配置

テストクラスはどこに置けばよいでしょうか。Maven プロジェクト（第2章で作成）では、ソースコードとテストコードを別々のフォルダに分けるのが決まりになっています。

重要なのは、テスト対象クラスとテストクラスで、パッケージ（フォルダの階層）を揃えることです。たとえば、本番コードが com.example パッケージにあれば、テストコードも com.example パッケージに置きます。本のジャンルを棚で合わせるイメージです。

Maven プロジェクトのファイル構成 

my-project/
├── src/
│   ├── main/
│   │   └── java/
│   │       └── com/example/
│   │           └── Calculator.java       ← 本番コード（テスト対象）
│   └── test/
│       └── java/
│           └── com/example/
│               └── CalculatorTest.java   ← テストコード
└── pom.xml

src/main/java が本番コードの置き場所、src/test/java がテストコードの置き場所です。パッケージ（com/example/）は両方で同じにします。

なぜパッケージを揃えるの？

同じパッケージにすると、テストクラスからテスト対象クラスの package-private（パッケージプライベート） なメンバーにもアクセスできます。これにより、public にしなくてもテストが書けるので、不必要にクラスを公開する必要がありません。これは JUnit 5 からより意識されるようになったメリットです。

本番コードがビルドされると src/main/java の内容が成果物に含まれますが、src/test/java のテストコードは成果物に含まれません。ユーザーに渡すアプリにテストコードが混入しない仕組みになっています。

4.4 @Test アノテーションの意味

第1章で少しだけ登場した @Test。ここで改めてきちんと理解しましょう。

@Test は アノテーション（annotation） と呼ばれる、Java のコードに「メモ書き」を追加する仕組みです。コード自体の動きを変えるのではなく、「このメソッドは〇〇です」という目印・ラベルを付ける役割を果たします。

たとえ話: ラベルシールを貼る

書類の束に「重要」「要確認」「処理済み」とラベルシールを貼ると、分類や扱い方が一目で分かりますよね。@Test アノテーションも同じです。「このメソッドはテストです」というシールを JUnit に向けて貼っているイメージです。JUnit はこのシールを見て「実行すべきテストメソッドを発見した」と認識し、自動で実行してくれます。

@Test は org.junit.jupiter.api.Test パッケージにあります。使うときは import 文を書く必要があります（後のコード例で確認できます）。

重要ポイント: @Test を付けないとテストとして実行されない

テストクラスの中にメソッドを書いただけでは、JUnit はそれをテストと認識しません。必ず @Test を付けてください。逆に言えば、@Test を付けないメソッドをテストクラス内に書いてもエラーにはならず、「テストではないヘルパーメソッド」として利用できます。

4.5 テストメソッドの書き方

テストメソッドには、いくつかのルールがあります。JUnit 5 ではかなり書きやすくなりました。

ルール1: @Test を付ける

メソッドの直前に @Test アノテーションを書きます。これが必須です。

ルール2: 戻り値は void

テストメソッドは何も返しません。アサーションが成功すれば「テスト通過」、失敗すれば「テスト失敗」となるため、戻り値は不要です。

ルール3: アクセス修飾子は不要（JUnit 5 の特徴）

JUnit 4 では public が必要でしたが、JUnit 5 では package-private（何も書かない）でOK です。public と書いても動きますが、書かない方がすっきりします。

テストメソッドの命名

テストメソッドの名前は、「何を確認するか」が一目で分かる名前にしましょう。

JUnit 5 では 日本語のメソッド名も使えます。「足し算が正しく計算できる」「ゼロ割りで例外がスローされる」のように書くと、テストの意図がとても読みやすくなります。英語で書く場合は shouldReturnSumWhenAdded のようなスタイルがよく使われます。

良い名前の例

  • 足し算が正しく計算できる
  • 負の数を足した場合も正しい
  • add_twoPositive_returnsSum

何を確認するテストかが名前から分かる

避けたい名前の例

  • test1
  • testMethod
  • check

何を確認しているのか名前から分からない

4.6 AAAパターンで読みやすく書く

テストを書くときに広く使われているのが、AAAパターン（トリプルA） と呼ばれる書き方です。テストの本文を3つのブロックに分けることで、「何を準備して、何を実行して、何を確かめるのか」が一目で分かるようになります。

A

Arrange（準備）

テストに必要なオブジェクトや値を準備します。「材料をそろえる」工程です。

例: Calculator calc = new Calculator();

A

Act（実行）

テスト対象のメソッドを呼び出します。「実際に料理する」工程です。

例: int result = calc.add(2, 3);

A

Assert（検証）

結果が期待どおりかを確かめます。「味見して確認する」工程です。

例: assertEquals(5, result);

AAAパターンを使うと、テストを読んだ人が「準備・実行・確認」の3つのステップを即座に理解できます。コメントで // Arrange・// Act・// Assert と書いておくとさらに分かりやすくなります。

また、1つのテストメソッドでは1つのことだけを確認するのがベストプラクティスです。「足し算」のテストに「引き算」まで混ぜてしまうと、失敗したときに「どちらの計算が間違っているのか」が分かりにくくなってしまいます。

4.7 テストクラスの全体像

ここまでの知識をまとめて、テストクラスの完成形を見てみましょう。AAAパターンで書いた2つのテストメソッドを含む、シンプルな CalculatorTest です。import 文も含めて全体の骨組みを確認してください。

CalculatorTest.java 
package com.example; // 本番コードと同じパッケージ

import org.junit.jupiter.api.Test; // @Test の import
import static org.junit.jupiter.api.Assertions.*; // アサーションの import

// テストクラス名 = テスト対象クラス名 + "Test"
class CalculatorTest {

    @Test // 「これはテストです」という目印
    void 足し算が正しく計算できる() {
        // Arrange（準備）: テスト対象を用意する
        Calculator calc = new Calculator();

        // Act（実行）: テスト対象のメソッドを呼ぶ
        int result = calc.add(2, 3);

        // Assert（検証）: 結果が期待どおりか確かめる
        assertEquals(5, result); // 5 になるはず
    }

    @Test
    void 負の数を足した場合も正しく計算できる() {
        Calculator calc = new Calculator(); // 準備
        int result = calc.add(-1, 5);       // 実行
        assertEquals(4, result);            // 検証: -1 + 5 = 4
    }
}

コードのポイント解説

  • クラス宣言: public は付けていません。JUnit 5 では package-private（何も付けない）でOKです。
  • import 文: import org.junit.jupiter.api.Test で @Test を、import static ... Assertions.* でアサーション群を一括 import しています。
  • 日本語メソッド名: JUnit 5 では日本語が使えます。「何を確認するか」が伝わる名前にしましょう。

これが JUnit テストクラスの基本の骨組みです。新しいテストを追加したいときは、@Test を付けた void メソッドをこのクラスに追加していくだけです。

テストクラスの構成要素を整理すると、次のようになります。

flowchart TD A["CalculatorTest クラス
（テストクラス）"] --> B["@Test
足し算が正しく計算できる（）"] A --> C["@Test
負の数を足した場合も正しく計算できる（）"] A --> D["@Test
… さらにテストを追加できる …"] B --> B1["Arrange: 準備"] B --> B2["Act: 実行"] B --> B3["Assert: 検証"]

やってみよう 4-1: CalculatorTest を作ってみよう

第2章で作成した Maven プロジェクトを使って、テストクラスの骨組みを実際に作ってみましょう。

手順

  1. 1 src/main/java/com/example/Calculator.java に、足し算メソッド add(int a, int b) を持つ Calculator クラスを作る。
  2. 2 src/test/java/com/example/CalculatorTest.java を作成し、この章のコード例を参考にテストメソッドを2つ書く。
  3. 3 IDE のテスト実行ボタン（または mvn test）を使ってテストを実行し、2つのテストが「緑（成功）」になることを確認する。
  4. 4 試しに assertEquals(5, result) の 5 を 99 に変えて再実行し、「赤（失敗）」になることを確かめてみる。

確認ポイント: AAAパターンのコメント（// Arrange など）を入れると、テストの意図がさらに読みやすくなります。ぜひ試してみましょう。

理解度確認クイズ

  1. Q1. Calculator クラスをテストするテストクラスの名前は、慣習上どのようにすべきですか？

    回答を表示

    CalculatorTest です。JUnit の慣習では、テスト対象クラスの名前の末尾に Test を付けてテストクラスを命名します。

  2. Q2. Maven プロジェクトで、テストコードは src/main/java と src/test/java のどちらに置きますか？ また、パッケージはどうすべきですか？

    回答を表示

    テストコードは src/test/java に置きます。パッケージは、テスト対象クラスと同じパッケージにすることが推奨されています（例: テスト対象が com.example ならテストも com.example）。

  3. Q3. AAAパターンの「A」は、それぞれ何を意味しますか？

    回答を表示

    Arrange（準備）・Act（実行）・Assert（検証）の3つです。テストを「準備→実行→検証」の3ブロックに分けて書くことで、何をしているテストかが一目で分かるようになります。
