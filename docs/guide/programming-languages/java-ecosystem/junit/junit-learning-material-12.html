良いテストを書くコツ ベストプラクティス

いよいよ最終章です。ここまで学んできた JUnit の技術を活かして、「良いテストとは何か」を深掘りします。F.I.R.S.T 原則という5つの指針と、避けるべきアンチパターンを学び、保守しやすく読みやすいテストが書けるようになりましょう。このガイドの集大成として、プロのテストの書き方を身につけます。

この章で学ぶこと

  • F.I.R.S.T 原則（Fast・Independent・Repeatable・Self-validating・Timely）の5つを説明できる
  • 避けるべきアンチパターンを判別し、良いテストコードへ書き直せる
  • 「何を・どんな条件で・どうなるか」が伝わるテスト名を付けられる
  • JUnit 学習ガイド全体の内容を振り返り、次のステップへの道筋を描ける

12.1 F.I.R.S.T 原則とは

良いテストコードには、共通した特徴があります。それを5つの頭文字でまとめたものが F.I.R.S.T 原則です。「テストを書けること」と「良いテストを書けること」の差は、まさにこの5つを意識しているかどうかにあります。

F.I.R.S.T はそれぞれ Fast（速い）・Independent（独立している）・Repeatable（繰り返し可能）・Self-validating（自己検証）・Timely（適時） の頭文字を取ったものです。一つひとつをじっくり見ていきましょう。

F

Fast（速い）

テストはすぐに終わるべきです。1件が数秒かかるテストが100件あると、全体で何分もかかります。

なぜ重要か: 遅いテストは実行されなくなります。「いちいち待てないから後で」が積み重なり、テストが形骸化します。DB接続やネットワーク呼び出しはテストを遅くする代表例です。

I

Independent（独立している）

テスト同士は互いに依存してはいけません。実行する順番を変えても、結果が変わらないことが理想です。

なぜ重要か: 依存があると「テストAが先に動かないとテストBが失敗する」という壊れやすい構造になります。第5章の @BeforeEach で各テストの初期状態を整えることが助けになります。

R

Repeatable（繰り返し可能）

いつ・どの環境で実行しても、同じ結果になるべきです。「今日は通ったけど昨日は失敗した」はNGです。

NG例: 現在日時（LocalDate.now()）や乱数（Math.random()）に依存するテスト。外部APIの応答に依存するテスト。これらは結果がその時々で変わるため「壊れやすいテスト（Flaky Test）」と呼ばれます。

S

Self-validating（自己検証）

テストは成功か失敗かを自動で判定しなければなりません。人が目でログを読んで判断するのでは不十分です。

これがポイント: アサーション（assertEquals 等）が判定役です。アサーションのないテストは「いつも成功」になってしまい、バグを見逃します。第3章で学んだアサーションを必ず入れましょう。

T

Timely（適時）

テストは適切なタイミングで書きます。理想は実装と同時か、TDD ならば実装より先です。

なぜか: 実装後ずっと経ってからテストを書こうとすると、仕様が曖昧になっていたり、テストしにくい設計になっていたりします。第11章で学んだ TDD（Red-Green-Refactor）はこの "Timely" を最大限に体現したアプローチです。

原則 意味 違反した場合の問題
Fast（速い） 1テスト数ミリ秒〜数秒以内で終わる 実行が面倒になりテストを省略し始める
Independent（独立） テスト同士が影響し合わない 順番依存で「不思議な失敗」が起きる
Repeatable（再現性） 環境・日時に関係なく同じ結果 「昨日は通ったのに」が頻発する
Self-validating（自己検証） アサーションで合否を自動判定 バグがあっても「成功」と判定され見逃す
Timely（適時） 実装と同時か先にテストを書く 後から書けないテスト不能な設計になりやすい

12.2 避けるべきアンチパターン

「アンチパターン」とは、一見うまくいきそうに見えて、実は問題を引き起こす典型的な悪いやり方のことです。テストにもよくある落とし穴があります。具体的な Bad 例と Good 例を見ながら、避けるポイントを覚えましょう。

アンチパターン1: 1つのテストでアレコレ確認しすぎる

「どうせテストするなら、まとめてやろう」という発想は危険です。1つのテストメソッドでたくさんの確認を詰め込むと、どれが失敗したのか分かりにくくなります。

Bad: まとめすぎ
CalculatorTest.java（Bad例） 
@Test
void 電卓のすべての機能をテスト() {
    Calculator calc = new Calculator();
    // 足し算・引き算・掛け算・割り算を1つのメソッドで
    assertEquals(5,  calc.add(2, 3));
    assertEquals(1,  calc.subtract(3, 2));
    assertEquals(6,  calc.multiply(2, 3));
    assertEquals(2,  calc.divide(6, 3));
    // 失敗したとき「どれが原因か」が分かりにくい
}
Good: 1テスト1責務
CalculatorTest.java（Good例） 
@Test
void 足し算は正の整数を正しく計算できる() {
    Calculator calc = new Calculator();
    assertEquals(5, calc.add(2, 3)); // 1つのことだけ確認
}

@Test
void 割り算はゼロ除算で例外が発生する() {
    Calculator calc = new Calculator();
    assertThrows(ArithmeticException.class,
        () -> calc.divide(6, 0)); // 目的が明確
}

原則: 1テスト1アサーション（または1つの振る舞い）

1つのテストメソッドでは、1つの振る舞いだけを確かめましょう。失敗したとき「何が壊れているのか」が一目でわかり、デバッグが格段に楽になります。

アンチパターン2: アサーションが無い・弱い

アサーションのないテストは「プログラムが例外なく実行できた」ことしか確認していません。期待した結果になっているかどうかは全くチェックされていない状態です。

Bad: アサーションなし
UserServiceTest.java（Bad例） 
@Test
void ユーザー名を取得できる() {
    UserService service = new UserService();
    // ただ実行するだけ。何も確認していない！
    String name = service.getUserName(1);
    // assertEquals がない → 常に「成功」になる
}
Good: 具体的なアサーション
UserServiceTest.java（Good例） 
@Test
void IDが1のユーザー名はAliceである() {
    UserService service = new UserService();
    String name = service.getUserName(1);
    // 期待する値を具体的に指定する
    assertEquals("Alice", name,
        "ユーザーID=1の名前はAliceのはず");
}

アンチパターン3: テスト同士が依存している

あるテストが「前のテストで作ったデータ」を使う構造は、テストを実行する順番を変えると失敗するという壊れやすい状態です。F.I.R.S.T の Independent（独立性）に違反します。

Bad: テスト間で状態を共有
CartTest.java（Bad例） 
// クラス変数を使い回す（依存の原因）
static Cart cart = new Cart();

@Test
void 商品を追加できる() {
    cart.add("リンゴ"); // テスト1が状態を作る
    assertEquals(1, cart.size());
}

@Test
void 商品が2つある() {
    // テスト1の後でないと失敗する！
    assertEquals(1, cart.size());
}
Good: @BeforeEach で毎回初期化
CartTest.java（Good例） 
Cart cart;

@BeforeEach // 毎テストの前に新しいインスタンスを用意
void setUp() {
    cart = new Cart(); // 常にクリーンな状態から始まる
}

@Test
void 空のカートに商品を追加するとサイズが1になる() {
    cart.add("リンゴ");
    assertEquals(1, cart.size()); // 独立して動く
}

アンチパターンのまとめ

  • ・1テストでアレコレ確認しすぎ → 失敗の原因が分からなくなる
  • ・アサーションなし・弱い → バグを見逃し、常に成功と表示される
  • ・テスト同士が依存 → 実行順で結果が変わる壊れやすいテストになる
  • ・テスト名が不明瞭（例: test1()・テスト()）→ 何を確認しているか読めない
  • ・マジックナンバー（意味不明な数値の直書き）→ なぜその値なのかが伝わらない

12.3 良いテスト名の付け方

テストの名前は「仕様書」でもあります。テストが失敗したとき、名前を見ただけで「何が壊れているか」がわかる名前を目指しましょう。

テスト名の公式: 「対象 _ 条件 _ 期待する結果」

テスト名に以下の3つを盛り込むと、読む人が内容を瞬時に理解できます。

何を（対象）: テストするメソッドや機能（例: 割り算、ログイン）

どんな条件で（条件）: 入力やシナリオ（例: ゼロで割ったとき、パスワードが空のとき）

どうなるか（期待）: 結果や振る舞い（例: 例外が発生する、falseが返る）

悪い名前 問題 良い名前
test1() 何をテストしているか全くわからない 足し算に負の数を渡すと負の合計が返る()
テスト() 対象も条件も期待結果も不明 割り算でゼロ除算するとArithmeticException発生()
divideTest() 対象はわかるが条件と期待結果が不明 割り算に正の整数を渡すと正しい商が返る()
testUserNameIsAlice() 英語で読みにくい（日本語チームの場合） IDが1のユーザーの名前はAliceである()
BankAccountTest.java（良い命名の例） 
class BankAccountTest {

    @Test
    void 残高より多い金額を引き出すと残高不足例外が発生する() {
        // Arrange: 残高1000円の口座を用意
        BankAccount account = new BankAccount(1000);
        // Act + Assert: 2000円引き出しは失敗する
        assertThrows(InsufficientFundsException.class,
            () -> account.withdraw(2000));
    }

    @Test
    void 残高以内の金額を引き出すと残高が減る() {
        BankAccount account = new BankAccount(1000);
        account.withdraw(300); // 300円引き出す
        assertEquals(700, account.getBalance()); // 残高は700円
    }
}

重要ポイント: AAA パターン（Arrange-Act-Assert）

良いテストの本文は3段落で書けます。Arrange（準備）→ Act（実行）→ Assert（確認）。コメントでこの3つを区切るだけで、誰が読んでも意図が伝わる読みやすいテストになります。

12.5 次のステップへ

このガイドを通じて、JUnit 5 の基礎から実践まで習得しました。しかし、実際の開発現場でテストを書き続けると、次のような場面に出会います。「外部 API を呼ぶクラスをテストしたい」「Spring Boot アプリ全体が動くかテストしたい」「コードを変更するたびに自動でテストを実行したい」。それぞれに対応する次のステップを紹介します。

Mockito（モック）

外部サービスや DB に依存するクラスを「偽物（モック）」に差し替えてテストするライブラリです。Fast・Independent を実現するための強力な道具です。

キーワード: @Mock・when(...).thenReturn(...)・verify(...)

Spring Boot のテスト

Spring Boot アプリケーション全体を起動してテストする統合テストや、Web API のエンドポイントをテストする方法を学びます。

キーワード: @SpringBootTest・MockMvc・@WebMvcTest

CI/CD（GitHub Actions）

コードをプッシュするたびに自動でテストを実行し、失敗を即座に検知する仕組みを構築します。チーム開発での品質維持に欠かせません。

キーワード: GitHub Actions・mvn test・ワークフロー定義

やってみよう 12-1: 過去に書いたテストを F.I.R.S.T で評価しよう

第9章・第10章で書いた電卓アプリのテストコードを見直して、F.I.R.S.T 原則に照らして評価・改善してみましょう。

手順

  1. 1 第9章・第10章のテストコードを開き、F.I.R.S.T の各原則を1つずつ確認する（「このテストは Fast か？」「Independent か？」）。
  2. 2 テスト名を本章の公式「対象 _ 条件 _ 期待する結果」で評価し、改善できるものは書き直す。
  3. 3 改善したテストを実行して、すべてグリーン（成功）になることを確認する。

発展課題: Mockito を使って、電卓アプリに外部サービス呼び出し（例えば税率を外部 API から取得する機能）を追加し、モックを使ってテストしてみましょう。

理解度確認クイズ

  1. Q1. F.I.R.S.T 原則の「R」— Repeatable（繰り返し可能）に違反する典型例を1つ挙げてください。

    回答を表示

    テスト内で LocalDate.now()（現在日時）や Math.random()（乱数）を使うケース、また外部 API の応答に依存するケースなどが典型例です。これらは実行するたびに値が変わるため、同じ結果を保証できません。

  2. Q2. アサーションのないテストがなぜ危険なのか説明してください。

    回答を表示

    アサーションがないテストは「例外が発生しなかった」ことしか確認せず、期待した結果になっているかを検証していません。プログラムにバグがあっても「成功」と表示され続けるため、テストが全く意味をなさなくなります。F.I.R.S.T の Self-validating（自己検証）に違反します。

  3. Q3. テスト名「割り算でゼロ除算するとArithmeticException発生()」を、「対象・条件・期待結果」の3要素で分析してください。

    回答を表示

    対象（何を）:「割り算」メソッド。条件（どんな条件で）:「ゼロ除算する（ゼロで割る）とき」。期待結果（どうなるか）:「ArithmeticException（算術例外）が発生する」。3要素がすべて揃っており、テスト名を読むだけで何を確認しているかがわかる良い命名例です。
