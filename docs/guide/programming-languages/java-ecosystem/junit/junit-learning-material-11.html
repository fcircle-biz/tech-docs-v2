第11章 所要時間: 約55分

テスト駆動開発（TDD）入門 テストから始めるプログラミング

これまでは「実装してからテストを書く」という順番でした。テスト駆動開発（TDD）はその順番を逆にします。先にテストを書き、そのテストが通るように実装する——この「テストファースト」の考え方と、Red-Green-Refactor サイクルを、小さな例題を通じて体験しましょう。

この章で学ぶこと

  • TDD（テスト駆動開発）の基本的な考え方と、従来の開発手順との違いを説明できる
  • Red-Green-Refactor サイクルの各ステップが何を意味するか説明できる
  • 小さな題材（isPositive メソッド）を使って Red→Green→Refactor の1サイクルを実際に経験できる
  • テストファーストのメリット（作りすぎ防止・仕様明確化・安心感・小さく進める）を説明できる
  • TDD の注意点と、まず小さく試すことの大切さを理解する

11.1 TDD とは何か？ 順番を逆にするだけで何が変わる？

これまでの章では「まず実装を書き、あとでテストを書く」という流れで作業してきました。これは自然な順番のように感じられます。しかしTDD（テスト駆動開発 / Test-Driven Development）はこの順番をひっくり返します。

TDD の基本ルールはたった1つ。「実装コードを書く前に、そのコードをテストするテストコードを書く」ことです。

観点 従来の開発（テスト後書き） TDD（テストファースト）
作業の順番 実装 → テスト テスト → 実装 → リファクタリング
ゴールの決め方 実装しながら考える テストでゴールを先に決める
テストが書かれるタイミング 実装が終わってから（後回しになりがち） 常に先に書かれる（必ずテストがある）
設計への効果 後から気づくことも多い テストを書くことで仕様を整理できる

たとえ話: 目的地を決めてから道を作る

カーナビを使うとき、まず「目的地」を入力してからルートを探しますよね。TDD もそれと同じです。

  • ・テスト = 目的地の設定: 「こう動いてほしい」というゴールをテストで定義する
  • ・実装 = ルートを走る: テストが通るように最短ルートで実装する
  • ・リファクタリング = 道をきれいに整備する: ゴール（テスト合格）を保ちながらコードを磨く

目的地を先に決めるから、迷わず最短ルートで進める。これが TDD の強みです。

11.2 Red-Green-Refactor サイクル

TDD の心臓部は Red-Green-Refactor（赤・緑・リファクタリング）と呼ばれる3ステップのサイクルです。このサイクルを小さく何度も繰り返すことで、少しずつ機能を積み上げていきます。

Red-Green-Refactor サイクル

flowchart LR R["Red（赤）
失敗するテストを書く"] G["Green（緑）
テストを通す最小の実装"] RF["Refactor（整理）
テストが緑のままコードを磨く"] R -->|"実装開始"| G G -->|"テスト合格"| RF RF -->|"次の機能へ"| R

ステップ1: Red（赤）— 失敗するテストを書く

まだ実装が存在しないので、テストは当然失敗します（赤）。しかしこれは「失敗」ではなく「ゴールを決めた」というポジティブな状態です。

このステップのポイント:

  • ・テストを書くことで「何を作るか」を明確にする
  • ・テストが失敗することを確認する（失敗を確認しないと、実装なしでも通るテストを作ってしまう恐れがある）
  • ・小さなテストを1つだけ書く

ステップ2: Green（緑）— テストを通す最小の実装を書く

テストが通る（緑になる）最小限の実装を書きます。この段階での合言葉は 「とにかく緑にする」。きれいなコードにこだわる必要はありません。

このステップのポイント:

  • ・テストを通すためだけの最小実装でよい
  • ・「作りすぎない」—テストが要求する以上のものを実装しない
  • ・全てのテストが緑であることを確認する

ステップ3: Refactor（リファクタリング）— コードをきれいにする

テストが緑のまま、コードを整理します。リファクタリングとは「外から見た振る舞いを変えずに、コードの内部構造を改善すること」です。

このステップのポイント:

  • ・テストがセーフネット（安全網）になる。壊したらすぐテストが赤になるので安心して整理できる
  • ・重複の除去・命名の改善・処理の整理などを行う
  • ・整理が終わったら、またテストが緑であることを確認する

重要ポイント: 小さいサイクルを素早く回す

1サイクルは数分以内に終わるくらい小さく刻むのが理想です。大きな機能は「小さな機能の積み重ね」に分解して、一度に1つずつ Red→Green→Refactor します。小さければ小さいほど、どこが壊れたかがすぐわかります。

11.3 1サイクル体験: isPositive メソッドを TDD で作る

実際に Red→Green→Refactor の1サイクルを体験しましょう。お題は「正の数かどうかを判定する isPositive メソッド」です。引数が正の数（1以上）なら true、そうでなければ false を返します。

TDD では実装クラスは後から作ります。まず NumberChecker というクラスに isPositive メソッドがあると仮定してテストを書きます。

Red

Step 1: 失敗するテストを書く

最初のテストは「1を渡したら true を返す」という最も基本的なケースです。まだ NumberChecker クラスは存在しないので、コンパイルエラー（テスト失敗）になります。

NumberCheckerTest.java（Step 1: Red） 
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class NumberCheckerTest {

    @Test
    void 正の数を渡したらtrueを返す() {
        NumberChecker checker = new NumberChecker();
        // まだ NumberChecker クラスが存在しないので失敗（赤）
        assertTrue(checker.isPositive(1));
    }
}

実行結果: コンパイルエラー（NumberChecker が存在しない）→ テスト失敗（赤）。これが「ゴールを決めた」証拠です。

Green

Step 2: テストを通す最小の実装を書く

テストを通すために NumberChecker クラスを作ります。「最小の実装」という点に注目してください。今必要なのは「1が正の数かどうか」だけです。

NumberChecker.java（Step 2: Green — 最小実装） 
public class NumberChecker {

    // テストを通すための最小実装
    // 「1 より大きいか？」を判定する
    public boolean isPositive(int number) {
        return number > 0; // 0より大きければ正の数
    }
}

実行結果: テスト成功（緑）。目的地に到着しました。

テストが緑になったら、次の小さなテストを追加します。「0 を渡したら false を返す」「負の数を渡したら false を返す」を追加しましょう。これが次の Red の起点になります（今回は実装がすでにあるので、テストはすぐ緑になります）。

NumberCheckerTest.java（テストを追加） 
@Test
void ゼロを渡したらfalseを返す() {
    NumberChecker checker = new NumberChecker();
    assertFalse(checker.isPositive(0)); // 0 は正の数ではない
}

@Test
void 負の数を渡したらfalseを返す() {
    NumberChecker checker = new NumberChecker();
    assertFalse(checker.isPositive(-5)); // 負の数も正の数ではない
}
Refactor

Step 3: テストが緑のままコードを整理する

全テストが緑になったので、コードを整理します。今回のコードはすでシンプルですが、テストコードの重複（new NumberChecker() を毎回書いている部分）を @BeforeEach でまとめましょう。

NumberCheckerTest.java（Step 3: Refactor — 整理後） 
class NumberCheckerTest {

    private NumberChecker checker; // フィールドにまとめる

    @BeforeEach
    void setUp() {
        checker = new NumberChecker(); // 各テスト前に1回だけ生成
    }

    @Test
    void 正の数を渡したらtrueを返す() {
        assertTrue(checker.isPositive(1));
    }

    @Test
    void ゼロを渡したらfalseを返す() {
        assertFalse(checker.isPositive(0));
    }

    @Test
    void 負の数を渡したらfalseを返す() {
        assertFalse(checker.isPositive(-5));
    }
}

整理後も全テストが緑であることを確認します。テストがセーフネットとして機能しているため、安心してコードを整理できました。

今回の isPositive 開発で回したサイクル

flowchart TD A["Red: isPositive テストを書く
（NumberChecker はまだ存在しない）"] B["Green: NumberChecker と isPositive を実装
（最小限の return number ＞ 0）"] C["追加 Red: 0 と負の数のテストを書く"] D["Green: 既存実装でテスト通過"] E["Refactor: @BeforeEach でまとめる
（テストは引き続き全て緑）"] A --> B B --> C C --> D D --> E

11.4 テストファーストのメリット

TDD を実践することで得られるメリットは、ただ「テストが先にある」という以上のものです。先にテストを書く習慣がもたらす4つのメリットを見ていきましょう。

1. 作りすぎを防ぐ（YAGNI）

テストが要求することだけを実装するため、「いつか使うかも」という余計な機能を作らずに済みます。コードがシンプルに保たれます。

2. 仕様が明確になる

テストを書くためには「どんな入力でどんな出力になるか」を事前に考える必要があります。曖昧な仕様のまま実装に突進することを防ぎます。

3. 常にテストがある安心感

開発を進めるにつれてテストが積み上がります。変更を加えたとき、既存の動作を壊していないことをすぐ確認できます。

4. 小さく進める

1つのテスト → 最小実装 → 整理、というリズムで少しずつ進みます。大きな機能を前に途方に暮れることなく、着実に前進できます。

11.5 TDD の注意点: まず小さく試すことから

TDD には多くのメリットがある一方で、注意点や向き不向きもあります。最初からすべての開発に TDD を強制しようとすると、かえって苦しくなることがあります。

TDD は銀の弾丸ではない

  • ・慣れが必要: 「テストを先に書く」という思考の切り替えには練習が要ります。最初はぎこちなく感じて当然です。
  • ・向き不向きがある: UI の見た目の確認、探索的な試作、仕様がまったく固まっていない初期段階では、TDD が必ずしもベストとは限りません。
  • ・テストの品質も大事: テストを先に書いても、テスト自体が良くなければ恩恵は半減します。第12章で良いテストの書き方を学びます。

まず小さく試すのがコツ

TDD を始めるときの心構えは「すべてに適用する」ではなく「一部の機能で試してみる」です。たとえば:

  • ・バグ修正のときに「まずバグを再現するテストを書いてから直す」だけでも TDD の第一歩
  • ・純粋な計算ロジック（ビジネスルールの判定など）から試すと成功体験を積みやすい
  • ・1日1つの小さなメソッドを TDD で作ることから始めてみる

TDD はスキルです。練習を重ねるほどに自然なリズムが生まれ、コードの設計力も高まっていきます。今日体験した Red-Green-Refactor のサイクルを、自分でも1度手を動かして試してみましょう。

やってみよう 11-1: TDD で isEven メソッドを作ろう

「偶数かどうかを判定する isEven メソッド」を TDD で実装してみましょう。Red→Green→Refactor の3ステップを意識しながら進めてください。

手順

  1. 1 （Red） まず isEven(2) が true を返すことを確認するテストを書き、実行して失敗を確認する。
  2. 2 （Green） テストを通す最小の実装（return number % 2 == 0;）を書き、緑になることを確認する。
  3. 3 奇数のテスト（isEven(3) が false）、0 のテスト（isEven(0) が true）を追加し、全て緑になることを確認する。
  4. 4 （Refactor） テストコードに重複があれば @BeforeEach でまとめ、テストが引き続き緑であることを確認する。

確認ポイント

Step 1 でテストが赤（失敗）になることを必ず確認しましたか？ 最初から緑になるテストは、実装の検証になっていません。

理解度確認クイズ

  1. Q1. TDD の Red-Green-Refactor サイクルの「Red（赤）」は何を意味しますか？

    回答を表示

    まだ実装が存在しないため、テストが失敗している状態を指します。「失敗」という意味ではなく、「これから作るものへのゴールを、テストで先に定義した」というポジティブな出発点です。失敗することを確認することが重要で、これによりテストが実際に機能していることを証明します。

  2. Q2. Green（緑）のステップで「最小限の実装」にとどめる理由は何ですか？

    回答を表示

    テストが要求する以上のものを作らないことで「作りすぎ（YAGNI 違反）」を防ぐためです。今必要ではない機能を実装すると、コードが複雑になり、後から変更しにくくなります。最小実装でテストを通し、次のテストで次の機能を追加するという流れを繰り返すことで、コードをシンプルに保ちながら機能を積み上げられます。

  3. Q3. Refactor（リファクタリング）のステップで「テストがセーフネットになる」とはどういう意味ですか？

    回答を表示

    リファクタリングでコードを整理する際に、もし誤って動作を変えてしまった場合、テストがすぐに失敗（赤）になって気づかせてくれるということです。テストという安全網があるので、「壊してしまうのではないか」という恐怖なく安心してコードを整理できます。
