第8章 所要時間: 約45分

テストをもっと読みやすく DisplayNameとネストクラス

テスト結果を分かりやすくする @DisplayName と、テストをグループ化する @Nested を学びます。テストレポートを読む人のことを考えた「伝わるテスト」の書き方を、本棚の整理術にたとえながらやさしく解説します。

この章で学ぶこと

  • テストメソッド名だけだと結果一覧が読みづらい理由を説明できる
  • @DisplayName をクラスとメソッドに付けて、テスト結果に分かりやすい説明を表示できる
  • @Nested を使って関連するテストをグループ化できる
  • @DisplayName と @Nested を組み合わせて、仕様書のように読めるテストを書ける

8.1 「テスト名が読めない」問題

第4章では、テストクラスを本棚に整理するように構成する方法を学びました。でも、テストを書いて実行してみると、こんな結果が並ぶことがあります。

テスト実行結果（改善前）

CalculatorTest
  ✔ add_returns_correct_result
  ✔ add_with_negative_numbers
  ✔ divide_returns_correct_result
  ✗ divide_by_zero_throws_exception

英語のメソッド名だけでは「何を確かめているのか」が一目で分かりません。とくに テストを書いた人でない別のメンバーや、しばらく日が経ってから読み返したときに困ります。「これは何のテストだっけ？」と調べ直す手間が生じます。

本棚（第4章）の整理はできていても、棚にラベルが貼られていない状態です。これを解決するのが、この章で学ぶ @DisplayName と @Nested です。

たとえ話: 本棚の整理術

  • ・第4章: 本を本棚にきれいに並べる（テストをクラスにまとめる）
  • ・@DisplayName: 棚のタイトルを書いた分かりやすいラベルを貼る
  • ・@Nested: 「料理本コーナー」「小説コーナー」のようにジャンルで仕切りを入れる

この2つを組み合わせることで、テスト結果が「仕様書のように読める」状態になります。

8.2 @DisplayName — テスト結果に説明をつける

@DisplayName は、テストクラスやテストメソッドに日本語を含む自由な説明文を付けるアノテーションです。テスト実行ツール（IDE や Maven）の結果一覧に、その説明文が表示されるようになります。

付けられる場所

  • ・テストクラス（class）
  • ・テストメソッド（@Test が付くメソッド）
  • ・@Nested クラスにも付けられる

書ける文字

  • ・日本語 OK（例: 「足し算のテスト」）
  • ・記号・空白も使える（例: 「2 + 3 = 5 になること」）
  • ・メソッド名のルールに縛られない

早速、コードで使い方を確認してみましょう。

CalculatorTest.java（@DisplayName の使い方） 
// クラスにも @DisplayName を付けられる
@DisplayName("電卓クラスのテスト")
class CalculatorTest {

    @Test
    @DisplayName("2 と 3 を足すと 5 になること")
    void add_returns_correct_result() {
        Calculator calc = new Calculator();
        // 2 + 3 を計算して結果が 5 かを確認
        assertEquals(5, calc.add(2, 3));
    }

    @Test
    @DisplayName("0 で割ると例外が発生すること")
    void divide_by_zero_throws_exception() {
        Calculator calc = new Calculator();
        // 0 除算で例外が出ることを確認
        assertThrows(ArithmeticException.class,
                     () -> calc.divide(10, 0));
    }
}

@DisplayName を付ける前と後で、テスト結果の表示がどう変わるかを見てみましょう。

Before（@DisplayName なし）

# テスト結果一覧

CalculatorTest
  ✔ add_returns_correct_result
  ✗ divide_by_zero_throws_exception

何を確かめているか読み取りにくい

After（@DisplayName あり）

# テスト結果一覧

電卓クラスのテスト
  ✔ 2 と 3 を足すと 5 になること
  ✗ 0 で割ると例外が発生すること

一目で何を確かめているか分かる！

メソッド名はそのままでいいの？

@DisplayName を付けてもメソッド名は変わりません。メソッド名は Java のコードとして有効な識別子でなければなりませんが、@DisplayName に書く文字列には制約がないので、表示用の説明だけを自由に書けるのです。両方うまく使うのがポイントです。

8.3 @Nested — テストをグループ化する

テストが増えてくると、同じテストクラスの中に「足し算のテスト」「割り算のテスト」「掛け算のテスト」が混在して、どれがどれか分からなくなることがあります。@Nested を使うと、テストクラスの中に内部クラスを作って、関連するテストをグループ化できます。

flowchart TD A["CalculatorTest（外側のクラス）"] A --> B["@Nested
足し算のテスト"] A --> C["@Nested
割り算のテスト"] B --> D["2と3を足すと5になること"] B --> E["負の数を足せること"] C --> F["10を2で割ると5になること"] C --> G["0で割ると例外が発生すること"]

@Nested クラスには大切なルールが1つあります。それは static（静的）ではない内部クラスとして作る、という点です。

重要: @Nested クラスは非 static の内部クラス

@Nested を付けるクラスは static を付けない内部クラスにします。static を付けると JUnit に認識されず、テストが実行されません。外側クラスのフィールドやメソッドにもアクセスできます。

実際のコードで確認してみましょう。

CalculatorTest.java（@Nested の使い方） 
@DisplayName("電卓クラスのテスト")
class CalculatorTest {

    Calculator calc = new Calculator(); // 外側クラスで共有

    // ---- 足し算グループ ----
    @Nested
    @DisplayName("足し算のテスト")
    class AddTests {
        @Test
        @DisplayName("2 と 3 を足すと 5 になること")
        void add_two_positive_numbers() {
            assertEquals(5, calc.add(2, 3)); // 正の数の足し算
        }
    }

    // ---- 割り算グループ ----
    @Nested
    @DisplayName("割り算のテスト")
    class DivideTests {
        @Test
        @DisplayName("0 で割ると例外が発生すること")
        void divide_by_zero() {
            // 異常系: 0 除算で例外が出ることを確認
            assertThrows(ArithmeticException.class,
                         () -> calc.divide(10, 0));
        }
    }
}

よくある間違い: static を付けてしまう

static class AddTests と書いてしまうと、JUnit がこの内部クラスのテストを実行してくれません。@Nested の直下は必ず static なしの内部クラスにしましょう。

8.4 組み合わせると「仕様書」になる

@DisplayName と @Nested を組み合わせると、テスト結果が入れ子の見出し付きで表示されます。これは、まるでドキュメントや仕様書を読んでいるような読みやすさになります。

テスト実行結果（@DisplayName + @Nested を使った後）

電卓クラスのテスト
  足し算のテスト
    ✔ 2 と 3 を足すと 5 になること
    ✔ 負の数を足しても正しく計算できること
  割り算のテスト
    ✔ 10 を 2 で割ると 5 になること
    ✔ 0 で割ると例外が発生すること

グループ名と個別の説明が階層で表示されるため、どの機能について・どんな状況を・どう確かめているかが一目瞭然です。テストを書いた人だけでなく、レビューする人や将来読み返す人も迷わず読めます。

テストは「生きた仕様書」になる

第1章で「よく書かれたテストは仕様の説明書になる」と学びました。@DisplayName + @Nested を使うと、テスト結果の一覧が「このクラスはこういう条件でこう動く」という仕様書として読めます。コードと別にドキュメントを書かなくても、テストそのものが説明になります。

やってみよう 8-1: 電卓テストに @DisplayName と @Nested を追加しよう

第7章で作成した電卓テストクラスを改造して、@DisplayName と @Nested を追加してみましょう。テスト結果の表示がどう変わるかを確かめます。

手順

  1. 1 既存の CalculatorTest クラスを開き、クラス宣言の直上に @DisplayName("電卓クラスのテスト") を追加する。
  2. 2 足し算のテストメソッドを @Nested @DisplayName("足し算のテスト") class AddTests でグループ化し、各メソッドにも @DisplayName を付ける。
  3. 3 同様に割り算のテストを DivideTests クラスとしてグループ化する（static は付けない！）。
  4. 4 テストを実行して、結果が入れ子の見出し付きで表示されることを確認する。

確認ポイント

  • ・テスト結果のツリー表示に「電卓クラスのテスト」が表示される
  • ・「足し算のテスト」と「割り算のテスト」がグループとして展開される
  • ・各テストに日本語の説明文が表示される

理解度確認クイズ

  1. Q1. @DisplayName("2 と 3 を足すと 5 になること") をテストメソッドに付けると、何が変わりますか？

    回答を表示

    テスト実行ツールの結果一覧に、メソッド名のかわりに「2 と 3 を足すと 5 になること」という説明文が表示されるようになります。テストの内容が一目で分かりやすくなります。メソッド名自体は変わりません。

  2. Q2. @Nested クラスに static を付けてしまうと、どうなりますか？

    回答を表示

    JUnit がその内部クラスのテストを実行してくれません。@Nested クラスは非 static（static なし）の内部クラスとして作る必要があります。

  3. Q3. @DisplayName と @Nested を組み合わせて使うと、テスト結果はどのような見た目になりますか？

    回答を表示

    グループ名（@Nested クラスの @DisplayName）を親見出しとし、個々のテスト説明（各メソッドの @DisplayName）を子見出しとした、入れ子のツリー表示になります。仕様書のように階層で読める形になるため、何をどう確かめているかが一目で分かります。
