実践！電卓アプリをテストしよう 総合演習（後編）とカバレッジ

前章に引き続き電卓アプリのテストを完成させます。割り算の正常系・異常系テストを追加して全操作を網羅し、@Nested と @DisplayName で読みやすい仕様書のようなテストクラスに仕上げます。さらに「テストカバレッジ」の考え方を学び、どれだけのコードがテストで動いたかを可視化する方法を身につけましょう。

この章で学ぶこと

  • divide メソッドの正常系テストと、assertThrows を使った 0 除算の異常系テストを書ける

  • ・CalculatorTest.java（add・subtract・multiply の正常系テスト）

10.2 divide の正常系テスト

まず、割り算が正しく計算できることを確認する正常系テストを書きましょう。たとえば「10 ÷ 2 = 5」が期待値です。足し算テストと同じ構造で、assertEquals を使います。

CalculatorTest.java（divideの正常系テスト） 
@Test
@DisplayName("割り算が正しく計算できる")
void 割り算が正しく計算できる() {
    // 10 ÷ 2 の結果が 5 になることを確かめる
    double result = calc.divide(10, 2);
    assertEquals(5.0, result, "10 ÷ 2 は 5 のはず");
}

ヒント: 割り算は double 型で扱う

整数同士の割り算でも、結果が小数になる場合があります（例: 10 ÷ 3 = 3.333...）。そのため divide は double 型を返します。assertEquals(5.0, result) のように期待値も double で書きましょう。

10.3 0 除算の異常系テスト（assertThrows の復習）

電卓テストの山場が、0 で割ったときの異常系テストです。「0 で割ることはできない」という数学のルールをプログラムでも守るために、divide(10, 0) が呼ばれたとき、IllegalArgumentException（不正な引数の例外）が投げられることを確かめます。

第7章で学んだ assertThrows を使います。「期待した例外が出ること自体がテストの成功条件」、という異常系テストのポイントです。

Calculator.java（divideメソッド再掲） 
// 割り算: 0 除算のときは例外を投げる
public double divide(int a, int b) {
    if (b == 0) {
        // 不正な引数（0 で割ることはできない）
        throw new IllegalArgumentException("0 で割ることはできません");
    }
    return (double) a / b;
}
CalculatorTest.java（0除算の異常系テスト） 
@Test
@DisplayName("0 で割るとエラーになる")
void ゼロ除算で例外が投げられる() {
    // assertThrows: 「この処理で指定の例外が出ること」を確かめる
    assertThrows(
        IllegalArgumentException.class, // 期待する例外の型
        () -> calc.divide(10, 0)        // 例外が出るはずの操作
    );
}

重要ポイント: 「エラーが出ること」もテストできる

assertThrows は「例外が投げられなかったらテスト失敗」という判定をします。「正しくエラーになること」を確かめることが、異常系テストの目的です。エラーを無視するのではなく、意図した例外が出ること自体を仕様として検証します。

10.4 @Nested と @DisplayName で仕様書のように整理する

ここまでのテストをバラバラに書くと、どのメソッドのどのテストか一目でわかりません。第8章で学んだ @Nested（グループ化）と @DisplayName（日本語の説明ラベル）を組み合わせて、電卓テスト全体を見通しよく整理しましょう。

完成形の CalculatorTest.java 全体を確認してください。4つの演算がそれぞれ @Nested クラスに整理され、読んだだけで仕様がわかる状態になっています。

CalculatorTest.java（完成形 — クラス骨格） 
@DisplayName("電卓アプリのテスト仕様")
class CalculatorTest {

    private Calculator calc;

    @BeforeEach
    void setUp() { calc = new Calculator(); }

    @Nested
    @DisplayName("足し算のテスト")
    class AddTests {
        // 第9章で作成済み
    }

    @Nested
    @DisplayName("引き算のテスト")
    class SubtractTests {
        // 第9章で作成済み
    }

    @Nested
    @DisplayName("掛け算のテスト")
    class MultiplyTests {
        // 第9章で作成済み
    }

    @Nested
    @DisplayName("割り算のテスト")
    class DivideTests {
        // 第10章で追加
    }
}
CalculatorTest.java（DivideTests 内部） 
@Nested
@DisplayName("割り算のテスト")
class DivideTests {

    @Test
    @DisplayName("割り算が正しく計算できる")
    void 割り算が正しく計算できる() {
        assertEquals(5.0, calc.divide(10, 2));
    }

    @Test
    @DisplayName("0 で割るとエラーになる")
    void ゼロ除算で例外が投げられる() {
        assertThrows(IllegalArgumentException.class,
                     () -> calc.divide(10, 0));
    }
}

これでテストを実行すると、IDE のテスト結果ツリーが「電卓アプリのテスト仕様 → 割り算のテスト → 割り算が正しく計算できる」のように表示されます。コードを読まなくても、テスト結果の画面を見るだけで仕様がわかるのが、@Nested + @DisplayName の大きなメリットです。

10.5 テストカバレッジとは何か

テストをたくさん書いたとき、「どのくらいのコードをテストできているか」が気になります。それを数値で表したものがテストカバレッジ（coverage、網羅率）です。「テスト対象のコードのうち、実際にテストで実行された行の割合」を表します。

たとえ話: 教科書の練習問題

100ページある教科書の練習問題のうち、あなたが解いたのが80問だったとします。カバレッジは 80% です。残りの 20 問（20%）は、まだ練習していない「テストしていない部分」です。カバレッジが低い部分には、まだバグが潜んでいるかもしれません。

行カバレッジ（Line Coverage）

ソースコードの各行が実行されたかどうかを測る最も基本的なカバレッジ。「何行のうち何行を通ったか」で計算します。

例: 20行中16行を実行 → 80%

分岐カバレッジ（Branch Coverage）

if 文の両方の分岐（true 側・false 側）を通ったかを測ります。行カバレッジより厳密で、条件分岐のテスト漏れを見つけやすいです。

例: if文が4つ → 8分岐中7を通過 → 87.5%

行カバレッジと分岐カバレッジの違いを具体例で見てみましょう。次の divide メソッドには、if (b == 0) という分岐があります。

分岐カバレッジのイメージ 
public double divide(int a, int b) {
    if (b == 0) {                        // ← 分岐①: true の場合（0除算）
        throw new IllegalArgumentException("...");
    }
    return (double) a / b;               // ← 分岐②: false の場合（通常の割り算）
}
// 正常系テストだけ書いた場合: 行カバレッジは高いが、分岐①を通っていない

正常系テスト（divide(10, 2)）だけ書くと、throw の行には到達しないため、行カバレッジは高くても分岐カバレッジは低い状態になります。これが、異常系テストが重要な理由です。

10.6 JaCoCo と IDE でカバレッジを可視化する

カバレッジを自動で計測してくれるツールが JaCoCo（Java Code Coverage、ジャココ）です。Maven プロジェクトでは pom.xml に設定を追加するだけでカバレッジレポートを生成できます（設定の詳細はこの章では扱いません）。

もっと手軽なのが、IDE のカバレッジ実行機能です。IntelliJ IDEA や Eclipse には、テストを「カバレッジ付きで実行」するボタンがあります。これを押すだけで、ソースコードに色がつきます。

緑 = テスト済み

テストで実行された行。「ここはちゃんとテストされている！」という安心の色。

赤 = テスト未通過

テストで一度も実行されなかった行。「まだここのテストが足りていない」という警告の色。

黄 = 分岐の一方のみ

if 文など、片方の分岐しかテストされていない行。分岐カバレッジの警告。

IntelliJ IDEA でのカバレッジ実行手順

  1. テストクラスを右クリックする
  2. 「Run 'CalculatorTest' with Coverage」を選ぶ
  3. テスト実行後、エディタの左端に緑・赤のマーカーが表示される
  4. 右側の「Coverage」ウィンドウでパーセンテージを確認する

10.7 カバレッジ 100% の落とし穴

カバレッジを学ぶと「100% を目指さなければ！」という気持ちになりますが、ここで重要な注意点があります。

重要な注意: カバレッジ 100% ＝ バグゼロ ではない！

すべての行を「実行した」としても、正しい値で確認したか（アサーションの妥当性）は別の話です。次の例を見てください。

// NG例: 行は通るが、アサーションが何も確かめていない
@Test
void 割り算テスト() {
    double result = calc.divide(10, 2);
    // assertEquals を書き忘れた → 常に「成功」するが意味がない
}

このテストを実行すれば行カバレッジは上がりますが、バグが見つかりません。カバレッジは「どこをテストしていないか」を教えてくれる道具であって、テストの品質そのものを保証するものではありません。

また、すべてのコードに対して 100% のカバレッジを盲目的に追求すると、意味のないテストを量産してしまうリスクがあります。テストを書く時間と労力には限りがあります。では、何を優先してテストすべきでしょうか？

優先①: 重要なビジネスロジック

計算・ルール判定・料金計算など、間違えると困るコアな処理。ここのテストは手厚く書く。

優先②: 分岐・条件分岐

if 文や switch 文の両方（全）の分岐を通るテストを書く。条件の境界値（0, -1, 最大値など）は特に重要。

優先③: 異常系・エラーケース

0 除算・null 入力・範囲外の値など、正常以外の入力に対してどう振る舞うかは必ずテストする。

低優先: 単純な getter/setter、定数定義

ロジックがほぼない単純なコードは、カバレッジのためだけにテストを書く必要は薄い。

カバレッジとの向き合い方

現場では「行カバレッジ 80% 以上を目標にする」プロジェクトが多いです。100% を目指さなくていい理由は、すべてのコードが同じリスクを持つわけではないからです。カバレッジは「まだテストしていない部分を発見するための地図」として活用しましょう。

10.8 総合演習のまとめ — 電卓テストで学んだこと

第9章・第10章の総合演習を通じて、これまでの章で学んだ JUnit の機能を一つのまとまったテストクラスに統合する経験ができました。改めて全体を振り返りましょう。

やってみよう 10-1: 電卓テストを完成させよう

第9章で作った CalculatorTest.java に、割り算テストを追加して完成させます。さらに IDE のカバレッジ実行機能を使って、緑・赤の色分けを確認しましょう。

手順

  1. 1 第9章の CalculatorTest.java に DivideTests クラスを追加し、正常系と異常系（0 除算）の2つのテストを書く。
  2. 2 全テストを実行し（mvn test）、すべてグリーン（PASS）になることを確認する。
  3. 3 IDE の「カバレッジ付きで実行」を試し、Calculator.java のどの行が緑になるかを確認する。

確認ポイント: 正常系・異常系が両方揃ったとき、Calculator.java 全体のカバレッジが高くなることを体感してみましょう。

理解度確認クイズ

  1. Q1. テストカバレッジとは何ですか？ 一言で説明してください。

    回答を表示

    テスト対象のコードのうち、テストで実際に実行された行（またはパス）の割合です。「テストの網羅率」とも言います。カバレッジが低い部分は、まだテストされていないコードが多いことを示します。

  2. Q2. 行カバレッジと分岐カバレッジの違いを説明してください。

    回答を表示

    行カバレッジは「ソースコードの各行が実行されたか」を測ります。分岐カバレッジは「if 文などの条件分岐で、true 側と false 側の両方を通ったか」を測ります。分岐カバレッジのほうが厳密で、条件によって動きが変わるコードのテスト漏れを発見しやすいです。

  3. Q3. 「カバレッジ 100% を達成すればバグゼロ」は正しいですか？ 理由とともに答えてください。

    回答を表示

    正しくありません。カバレッジはあくまで「コードが実行されたか」の割合であり、アサーション（確認）が正しいかどうかは別の問題です。実行はされていても、期待値の確認が甘ければバグを見逃します。カバレッジは「テストしていない部分を教えてくれる道具」であり、品質の万能指標ではありません。
