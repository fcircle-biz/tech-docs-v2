ホーム Docker 第8章
第8章 所要時間: 約35分

よくあるエラーと確認手順

Dockerを使っていると、思いがけないエラーに出くわすことがあります。この章では「困ったときに何を確認すればよいか」という確認フローを身につけます。よくあるエラーとその対処法を把握しておけば、慌てずに自分で解決できるようになります。これが本教材の最終章です。

この章で学ぶこと

  • 困ったときの確認フローを身につけて、上から順に切り分けられるようになる
  • docker logs でコンテナのログを確認できる
  • よくあるエラーメッセージの意味と対処法を知る
  • 自力で切り分けてコンテナを削除・作り直せるようになる

実行前チェック（この章のコマンドを試す前に確認）

  • Rancher Desktop が起動しているか（タスクトレイ／メニューバーのアイコンを確認）
  • docker version が成功するか（Client と Server の両方が表示される）
  • 前の章で起動したコンテナ（例: my-nginx）が残っていないか（docker ps -a で確認）
  • 8080番ポートを他のコンテナが使っていないか（docker ps で確認）

8.1 困ったら「確認フロー」で切り分ける

エラーが起きたとき、いきなり検索したり、コンテナをやみくもに操作するのは避けましょう。決まった順番で確認することで、問題の原因をすばやく特定できます。下のフローを上から順に確認していくのが基本です。

図1: 初心者向け Dockerトラブル確認フロー

flowchart TD A["① Docker ／ Rancher Desktop は起動しているか？"] --> B{"起動している？"} B -->|いいえ| B1["アプリを起動して待つ"] B -->|はい| C["② docker version は成功するか？"] C --> D{"成功する？"} D -->|いいえ| D1["インストールを確認（第2章参照）"] D -->|はい| E["③ コンテナは起動しているか？（docker ps）"] E --> F{"起動している？"} F -->|いいえ| F1["docker run または docker compose up を実行"] F -->|はい| G["④ ポート番号は正しいか？"] G --> H{"正しい？"} H -->|いいえ| H1["ポート番号を確認・修正する"] H -->|はい| I["⑤ 同じ名前のコンテナが既に存在しないか？"] I --> J{"重複している？"} J -->|はい| J1["docker rm で削除してから作り直す"] J -->|いいえ| K["⑥ ログにエラーが出ていないか？（docker logs）"] K --> L{"エラーあり？"} L -->|はい| L1["エラーメッセージを読んで対処する（8.3節参照）"] L -->|いいえ| M["⑦ コンテナを削除して作り直す"]

このフローを一度頭に入れておくだけで、「何を確認すればよいか分からない」状態から抜け出せます。特にログの確認（手順⑥）は、エラーの直接的な手がかりが見つかりやすく、問題解決の近道になります。

フローを使うコツ

「もう確認した」と思っても、もう一度上から順に確認してみると見落としに気づくことがあります。焦らず順番通りに進めるのがポイントです。

8.2 確認に使うコマンド

確認フローで使うコマンドを整理します。それぞれ「何を確認するためのコマンドか」をセットで覚えておきましょう。

コマンド 何を確認するか
docker version Dockerが使える状態かを確認する。表示されればOK。
docker ps 現在動作中のコンテナを一覧で確認する。
docker ps -a 停止済みも含めた全コンテナを確認する。名前の重複チェックに便利。
docker logs <名前/ID> コンテナが出力したログ（エラーの手がかり）を表示する。
docker stop <名前/ID> コンテナを停止する。削除の前に必要。
docker rm <名前/ID> コンテナを削除する。停止してから実行する。
docker images 手元のイメージ一覧を確認する。
docker compose down Composeで起動した環境を停止・片付けする。

それぞれのコマンドを実際に使って確認する手順を見てみましょう。まずDockerが起動しているかを確認します。

ターミナル 
# Dockerが使える状態かを確認する
docker version

docker version を実行してバージョン情報が表示されれば、Dockerは正常に動いています。次にコンテナの状態を確認します。

ターミナル 
# 動作中のコンテナを確認する
docker ps

# 停止済みも含めた全コンテナを確認する（名前の重複チェックに使う）
docker ps -a

コンテナが起動しているのにアプリが動かない場合は、ログを確認します。ログはエラーの原因が書かれていることが多く、問題解決の重要な手がかりになります。

ターミナル 
# コンテナ名が「my-nginx」の場合のログ確認
docker logs my-nginx

# ログをリアルタイムで追い続ける（Ctrl+C で終了）
docker logs -f my-nginx

docker logs の後にコンテナ名またはIDを指定します。エラーメッセージがそのまま表示されるので、それを手がかりに原因を探ります。

8.3 よくあるエラーと対処【重要】

Dockerを使い始めた段階でよく出会うエラーを6つ紹介します。エラーメッセージ・原因・対処の3点をセットで確認しておきましょう。よくあるエラーは原因がほぼ決まっています。表示されたメッセージをよく読めば、対処法に自然とたどり着けます。

エラー①: Cannot connect to the Docker daemon

Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?

原因: Rancher Desktop（またはDockerデーモン）が起動していない、または起動途中。

対処: Rancher Desktopアプリを起動して、完全に起動し終わるまで待ちます。タスクトレイのアイコンが安定するまで数十秒かかることがあります。

エラー②: docker: command not found

docker: command not found

原因: Dockerがインストールされていない、またはインストール後にターミナルやPCの再起動が必要な状態。

対処: 第2章の手順でRancher Desktopをインストールしてください。インストール済みの場合は、ターミナルを閉じて開き直す、またはPCを再起動してから再度試します。

エラー③: port is already allocated

Error response from daemon: driver failed programming external connectivity: Bind for 0.0.0.0:8080 failed: port is already allocated

原因: 指定したポート番号（例: 8080）がすでに別のコンテナまたはアプリに使用されている。

対処: 使用中のコンテナを docker stop で停止するか、別のポート番号（例: 8081）に変えて実行します。

ターミナル 
# ポートが使われているときは別ポートに変える例
# 8080 の代わりに 8081 を使う（-p ホスト側:コンテナ側）
docker run -d -p 8081:80 --name my-nginx nginx

エラー④: Conflict. The container name is already in use

docker: Error response from daemon: Conflict. The container name "/my-nginx" is already in use by container "...". You have to remove (or rename) that container to be able to reuse that name.

原因: 同じ名前のコンテナがすでに存在している（停止中の場合も含む）。

対処: 対象のコンテナが起動中か停止中かで手順が変わります。まず docker ps -a で状態（Up＝起動中／Exited＝停止中）を確認しましょう。

  • 停止中（Exited）の場合： docker rm コンテナ名 で削除する。
  • 起動中（Up）の場合： 先に docker stop コンテナ名 で停止してから、docker rm コンテナ名 で削除する。
ターミナル 
# まず docker ps -a で状態を確認する（Up＝起動中／Exited＝停止中）
docker ps -a

# 【停止中（Exited）の場合】そのまま削除する
docker rm my-nginx

# 【起動中（Up）の場合】先に停止してから削除する
docker stop my-nginx
docker rm my-nginx

# 削除できたら、もう一度作り直す
docker run -d -p 8080:80 --name my-nginx nginx

補足：docker rm -f（起動中でも一気に削除）

起動中のコンテナでも、docker rm -f コンテナ名 と書くと、停止と削除を1コマンドでまとめて行えます（-f は force＝強制の意味）。

ただし強制削除なので、確認なしにコンテナを消します。慣れるまでは docker stop → docker rm の2段階で確認しながら操作し、-f は慣れてから使うのがおすすめです。

エラー⑤: no configuration file provided

no configuration file provided: not found

原因: compose.yml ファイルが存在しないフォルダで docker compose コマンドを実行している。

対処: compose.yml があるフォルダに移動してから、コマンドを再実行します。

ターミナル 
# 現在のフォルダを確認する
pwd

# compose.yml があるフォルダへ移動する（例: my-project）
cd ~/my-project

# 移動後に再実行する
docker compose up -d

エラー⑥: permission denied

Got permission denied while trying to connect to the Docker daemon socket

原因: Dockerデーモンへのアクセス権がない。Rancher Desktopが正しく動いていない場合や、Macでフォルダのアクセス許可が足りない場合がある。

対処: Rancher Desktopを再起動して正常に動いているか確認します。MacではシステムのプライバシーとセキュリティでRancher Desktopへのフォルダアクセス許可を見直してください。

よくあるエラー一覧まとめ

エラーメッセージ（抜粋） 主な原因 対処
Cannot connect to the Docker daemon Rancher Desktop 未起動 アプリを起動して待つ
docker: command not found 未インストール／再起動が必要 第2章を確認・再起動
port is already allocated ポートが使用中 コンテナを停止 or 別ポートに変更
container name is already in use 同じ名前のコンテナが存在 停止中は docker rm、起動中は docker stop 後に docker rm
no configuration file provided compose.yml がないフォルダ 正しいフォルダに移動する
permission denied 権限不足 Rancher Desktop再起動・権限確認

8.4 それでも解決しないとき — コンテナを作り直す

ログを確認しても原因がわからないとき、最もシンプルで効果的な方法はコンテナを一度削除して作り直すことです。コンテナはイメージから何度でも作れるので、気軽に削除してかまいません。

docker run で起動したコンテナの場合

まずコンテナを停止し、次に削除してから、もう一度 docker run で作り直します。

ターミナル 
# ①コンテナを停止する
docker stop my-nginx

# ②コンテナを削除する
docker rm my-nginx

# ③もう一度作り直す
docker run -d -p 8080:80 --name my-nginx nginx

Docker Compose の場合

Compose環境は docker compose down でまとめて停止・削除できます。その後 docker compose up で再起動します。

ターミナル（compose.yml があるフォルダで実行） 
# Compose環境を停止してコンテナを削除する
docker compose down

# 改めて起動し直す
docker compose up -d

コンテナは気軽に作り直せる

コンテナを削除してもイメージは残ります。イメージさえあれば、同じコンテナを何度でも作り直せます。行き詰まったら「down して up」が最強のリセット手段です。ただし、コンテナ内に保存したデータ（ボリューム未使用の場合）は失われるので注意してください。

  • 困ったときは確認フローで上から順に切り分ける。いきなり焦らず、1ステップずつ進める。
  • docker logs でログを確認するのが解決の近道。エラーの手がかりが直接見つかることが多い。
  • よくあるエラー（デーモン未起動・ポート競合・名前重複・フォルダ違い）は原因がほぼ決まっている。メッセージを読めば対処できる。
  • 行き詰まったらコンテナを削除して作り直すのが最も確実。Composeなら docker compose down → up で環境をリセットできる。

確認問題

  1. Q1.「Cannot connect to the Docker daemon」というエラーが出たとき、最初に確認することは何ですか？

    回答を表示

    Rancher Desktop（またはDockerデーモン）が起動しているかどうかを確認します。起動していない場合はアプリを起動して、完全に立ち上がるまで待ちます。

  2. Q2.「port is already allocated」（ポート競合）が起きたとき、対処法を2つ挙げてください。

    回答を表示

    ①そのポートを使っているコンテナを docker stop で停止する。②-p オプションで別のポート番号（例: 8081）に変えてコンテナを起動し直す。

  3. Q3. コンテナ名が重複して起動できない（「container name is already in use」）ときの対処手順を答えてください。

    回答を表示

    まず docker ps -a で状態を確認します。停止中（Exited）なら docker rm <コンテナ名> で削除します。起動中（Up）なら docker stop <コンテナ名> で停止してから docker rm <コンテナ名> で削除します。その後、改めて docker run で作り直します。

  • Dev Containers — VS Codeとコンテナを組み合わせた開発環境の構築
  • Kubernetes（k8s） — 多数のコンテナを大規模に管理するオーケストレーションツール
  • クラウドへのデプロイ — AWS ECS / Google Cloud Run / Azure Container Apps などへの公開
  • CI/CDでのDocker利用 — GitHub Actionsなどでイメージのビルド・テスト・デプロイを自動化する
