ServiceからRepositoryを利用する

前章では Repository の CRUD メソッドを直接試しました。しかし実際のアプリでは、Controller が Repository を直接呼ぶのではなく、Service を経由させるのが基本です。この章では、なぜ Service 層が必要なのかを理解したうえで、コンストラクタインジェクションで Repository を注入し、一覧取得・登録・更新・削除を Service のメソッドとしてまとめる方法を学びます。

この章で学ぶこと

  • Service 層の役割と、Controller から Repository を直接呼ばない設計の理由を説明できる
  • コンストラクタインジェクションで Repository を Service に注入できる
  • 一覧取得・登録・更新・削除を Service のメソッドとして整理できる
  • 「Controller → Service → Repository → Database」という層の流れを図で理解する

5.1 Service 層の役割 — 業務処理をまとめる場所

Spring アプリには Controller・Service・Repository という3つの層があります。それぞれが担当を分けています。

層 クラス例 役割
Controller EmployeeController 画面・HTTPリクエストの受け付けと振り分け
Service EmployeeService 業務処理（ビジネスロジック）のまとめ
Repository EmployeeRepository データベースへのアクセス（CRUD）

Controller は「画面からのリクエストを受け取り、どの処理を呼ぶか振り分ける」担当です。一方 Repository は「データベースに対して読み書きする」専門の担当です。Service はその中間に位置し、業務処理のロジックをまとめる役割を持ちます。

「Controller から Repository を直接呼べばよいのでは？」と思うかもしれません。しかし、業務処理が増えると Controller が肥大化し、テストも難しくなります。Service に業務処理を集めることで、画面処理（Controller）と業務処理（Service）を明確に分離できます。

flowchart TD A["Controller
（画面・リクエスト）"] --> B["Service
（業務処理）"] B --> C["Repository
（データアクセス）"] C --> D[("Database")]

重要ポイント

Controller は Service だけを呼び、Repository は直接呼びません。Repository を直接呼ぶのは Service のみです。この原則を守ることで、それぞれの層が自分の役割に集中でき、コードの見通しが良くなります。

5.2 Repository の DI — コンストラクタインジェクション

Service が Repository を使うには、Spring の DI（依存性の注入）で Repository のインスタンスを受け取ります。注入方法はいくつかありますが、現在の Spring ではコンストラクタインジェクションが推奨されています。

Spring 基礎の復習: DI（コンストラクタインジェクション）

DI とは、クラスが必要とする別のクラス（依存オブジェクト）を、自分で new するのではなく Spring が外部から注入してくれる仕組みです。コンストラクタインジェクションでは、コンストラクタの引数として依存オブジェクトを受け取ります。Spring Boot では、コンストラクタが1つだけの場合は @Autowired を省略できます。

@Service を付けると Spring が自動でこのクラスを Bean（管理対象）として登録します。コンストラクタで EmployeeRepository を受け取ることで、Spring が Repository を注入してくれます。

EmployeeService.java（基本構造） 
package com.example.jpalearning.service;

import com.example.jpalearning.repository.EmployeeRepository;
import org.springframework.stereotype.Service;

// @Service を付けると Spring が Bean として管理する
@Service
public class EmployeeService {

    // Repository を final で保持（変更不可）
    private final EmployeeRepository employeeRepository;

    // コンストラクタで Repository を受け取る（コンストラクタインジェクション）
    public EmployeeService(EmployeeRepository employeeRepository) {
        this.employeeRepository = employeeRepository;
    }
}

private final にすることで、注入後に Repository が書き換えられるのを防ぎます。これはコンストラクタインジェクションを使う際の定型パターンです。

5.3 一覧取得・登録・更新・削除を Service にまとめる

Repository のメソッド（findAll・findById・save・deleteById）をそのまま呼ぶだけでなく、Service のメソッドとしてわかりやすい名前でラップします。業務処理が増えても Service の中で完結します。

一覧取得: getAll()

全社員を取得して返します。Repository の findAll() をそのままラップするシンプルなメソッドです。

EmployeeService.java（一覧取得） 
import java.util.List;

// 全社員を取得する
public List<Employee> getAll() {
    // Repository の findAll() で全件取得
    return employeeRepository.findAll();
}

1件取得: getById(Long id)

ID を指定して1件取得します。findById() は Optional<Employee> を返します。見つからない場合は例外を投げます。

EmployeeService.java（1件取得） 
// ID で1件取得（見つからなければ例外）
public Employee getById(Long id) {
    // orElseThrow で見つからない場合に例外を投げる
    return employeeRepository.findById(id)
            .orElseThrow(() -> new RuntimeException("社員が見つかりません: " + id));
}

登録処理: register(Employee employee)

新規社員を登録します。save() は INSERT を行い、保存後の Entity（ID が設定済み）を返します。

EmployeeService.java（登録） 
// 社員を新規登録して保存済み Entity を返す
public Employee register(Employee employee) {
    // save() が INSERT を行い ID が設定された Entity を返す
    return employeeRepository.save(employee);
}

更新処理: update(Employee employee)

既存社員の情報を更新します。save() に ID あり Entity を渡すと UPDATE が行われます。

EmployeeService.java（更新） 
// 社員情報を更新する（ID ありの場合 save() は UPDATE になる）
public Employee update(Employee employee) {
    return employeeRepository.save(employee);
}

削除処理: delete(Long id)

ID を指定して社員を削除します。deleteById() は対象が存在しない場合に例外を投げます。

EmployeeService.java（削除） 
// ID を指定して社員を削除する
public void delete(Long id) {
    // deleteById() は ID に該当する行を削除する
    employeeRepository.deleteById(id);
}

5.4 EmployeeService の全体像

ここまで作ったメソッドをまとめると、EmployeeService の全体はこのようになります。

EmployeeService.java（全体） 
package com.example.jpalearning.service;

import com.example.jpalearning.entity.Employee;
import com.example.jpalearning.repository.EmployeeRepository;
import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class EmployeeService {

    private final EmployeeRepository employeeRepository;

    // コンストラクタインジェクション
    public EmployeeService(EmployeeRepository employeeRepository) {
        this.employeeRepository = employeeRepository;
    }

    public List<Employee> getAll()               { return employeeRepository.findAll(); }
    public Employee getById(Long id)             { return employeeRepository.findById(id).orElseThrow(); }
    public Employee register(Employee employee)  { return employeeRepository.save(employee); }
    public Employee update(Employee employee)    { return employeeRepository.save(employee); }
    public void delete(Long id)                  { employeeRepository.deleteById(id); }
}

Controller

画面からのリクエストを受け取り、Service のメソッドを呼ぶだけ。Repository は知らない。

Service

業務処理をまとめる。 Repository を持ち、CRUD をメソッドとして整理する。

Repository

データベースの読み書きに専念。Service からだけ呼ばれる。

よくある誤解

「登録も更新も save() を使うなら Service メソッドを分ける必要はない」と思われがちです。しかし、register() と update() を分けることで、将来「登録時だけ入力チェックをする」「更新時だけメール通知をする」といった業務処理の追加が Service に閉じてできます。Service メソッドを意味ある単位で分けておくことが重要です。

理解度確認クイズ

  1. Q1. Controller から Repository を直接呼ばず、Service を経由させる理由は何ですか？

    回答を表示

    画面処理（Controller）と業務処理（Service）を明確に分離するためです。Service に業務処理を集めることで、Controller が肥大化せず、将来の処理追加もテストも容易になります。

  2. Q2. コンストラクタインジェクションで Repository を受け取るコードを書いてください。

    回答を表示
    private final EmployeeRepository employeeRepository; public EmployeeService(EmployeeRepository employeeRepository) { this.employeeRepository = employeeRepository; }

    コンストラクタが1つだけの場合、Spring Boot では @Autowired を省略できます。

  3. Q3. register() と update() はどちらも内部で save() を呼びますが、なぜメソッドを分けるのですか？

    回答を表示

    将来「登録時だけ入力チェック」「更新時だけ通知処理」といった業務処理が追加されたとき、それぞれのメソッド内で対応できるためです。意味のある単位でメソッドを分けておくことで、Service 層が業務処理の追加に柔軟に対応できます。
