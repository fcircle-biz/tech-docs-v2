Spring Data JPAとは

Spring Data JPA は、Java の「オブジェクト」とデータベースの「テーブル」を対応させながら、データをより扱いやすくするための仕組みです。この章では、よく一緒に出てくる JPA・Hibernate・Spring Data JPA の3つがそれぞれ何を担当しているのか、すでに学んだ Spring Data JDBC とどう違うのか、そして本教材で作る「部署・社員管理アプリ」の全体像をつかみます。まずはコードを書く前に、考え方をしっかり理解しましょう。

この章で学ぶこと

  • JPA が「何のための仕様」なのか、O/R マッピングとは何かを説明できる
  • JPA・Hibernate・Spring Data JPA の3つの役割の違いを理解する
  • Spring Data JDBC と Spring Data JPA の違いを説明できる
  • 本教材で作る「部署・社員管理アプリ」の全体像をつかむ

1.1 JPA とは — オブジェクトとテーブルを対応させる仕様

JPA（Jakarta Persistence API）は、Java のオブジェクトとデータベースのテーブルを対応させるための「仕様（ルール）」です。Java の世界では、データを Employee（社員）のようなクラスのオブジェクトとして扱います。一方、データベースの世界では、データは employees のようなテーブル（表）で管理されます。この2つの世界をつなぎ、「Java オブジェクト ⇔ テーブルの行」を自動で行き来させる技術を O/R マッピング（ORM）と呼びます。

前教材「Spring Data JDBC」では、テーブルや SQL に近い感覚でデータベースを操作しました。本教材の JPA では、もう一歩「Java オブジェクト中心」に近づきます。テーブルの1行を Java オブジェクトとして受け取り、オブジェクトを保存すればテーブルに書き込まれる——そんな「オブジェクトのまま扱える」世界を体験していきます。

用語: O/R マッピング（ORM）

O/R マッピングとは、Object（Java のオブジェクト）と Relational database（リレーショナルデータベースの表）の間を「対応づけ（マッピング）」する技術のことです。たとえば「Employee オブジェクト1つ＝employees テーブルの1行」というルールを決めておくと、オブジェクトの保存・取得がそのままテーブルの読み書きになります。JPA はこの O/R マッピングのやり方を定めた仕様です。

JPA を使うと、たとえば社員クラスはこんなイメージになります（詳しくは第3章で作ります）。クラスにいくつかの目印（アノテーション）を付けるだけで、「このクラスはこのテーブルに対応する」と JPA に伝えられます。

Entity のイメージ（第3章で詳しく学びます） 
// @Entity を付けると「このクラスはテーブルに対応する」と JPA に伝わる
@Entity
@Table(name = "employees")   // employees テーブルに対応
public class Employee {
    @Id                       // 主キー（id 列）
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;      // name 列に対応
    private String email;     // email 列に対応
    // ... 続きは第3章で
}

1.2 JPA・Hibernate・Spring Data JPA の関係

Spring Data JPA を学び始めると、JPA・Hibernate・Spring Data JPA という似た言葉が次々に出てきて混乱しがちです。まずはこの3つの「役割分担」をはっきりさせておきましょう。結論から言うと、3つはライバルではなく、役割の違う仲間です。

JPA

対応づけの「仕様（ルール）」。どう書けばよいかを決めるが、それ自体は動かない。

Hibernate

JPA という仕様を実際に動かす実装。SQL を生成し DB を操作する代表的な製品。

Spring Data JPA

JPA・Hibernate を Spring で使いやすくする仕組み。Repository を作るだけで CRUD が使える。

身近なものに例えると、JPA は「規格・設計図」、Hibernate はその規格に従って作られた「エンジン（実際に動く製品）」、Spring Data JPA は、そのエンジンを楽に操作するための「ハンドルやリモコン」のような関係です。私たちが普段書くのは一番外側の Spring Data JPA で、その内側で Hibernate が JPA の仕様どおりに SQL を作って動いてくれます。

flowchart TD A["あなたのコード
(Repository・Service)"] --> B["Spring Data JPA
(使いやすくする仕組み)"] B --> C["JPA
(対応づけの仕様・ルール)"] C --> D["Hibernate
(JPAの実装・実際に動く)"] D --> E[("データベース
(H2 など)")]

重要ポイント

私たちが直接書くのは Spring Data JPA の部分だけです。その裏で Hibernate が JPA の仕様に従って SQL を自動生成し、データベースを操作してくれます。3つの違いを完璧に覚える必要はありませんが、「仕様 = JPA、実装 = Hibernate、楽にする道具 = Spring Data JPA」という関係だけは押さえておきましょう。

1.3 Spring Data JDBC と Spring Data JPA の違い

前教材で学んだ Spring Data JDBC と、本教材の Spring Data JPA は名前がよく似ていて混同しやすいので、序盤でしっかり違いを整理しておきます。どちらも「Repository を使ってデータベースを操作する」点は同じですが、考え方の重心が異なります。

観点 Spring Data JDBC Spring Data JPA
考え方 テーブル・SQL に近い Java オブジェクト中心
関連（テーブル同士のつながり） シンプルに扱う Entity 同士の関連を扱いやすい
自動処理 少なめ 多め
学習難易度 低め やや高め
向いている学習段階 初めての DB 操作 関連テーブルの学習

いちばん大きな違いは、「関連（テーブル同士のつながり）」の扱いやすさです。本教材では「部署」と「社員」のようにつながりを持ったデータを扱います。こうした関連を、Java オブジェクトの参照として自然に書けるのが JPA の強みです。

よくある誤解

「JPA を使えば SQL を一切書かなくてよい」と思われがちですが、それは正しくありません。Spring Data JPA は基本的な操作の SQL を自動で用意してくれるだけで、複雑な検索では自分で問い合わせを書くこともあります（第7章の @Query）。JPA は「SQL を完全に隠すもの」ではなく、「SQL を意識しながら、Java オブジェクトとしてデータを扱いやすくするもの」だと理解しておきましょう。実際、自動生成された SQL は第2章で設定する spring.jpa.show-sql でログに表示して確認できます。

1.4 この教材で作るもの — 部署・社員管理アプリ

本教材では、ゴールとして部署・社員管理アプリケーションを作ります。会社にはいくつかの「部署」があり、それぞれの部署に「社員」が所属している——という、身近でイメージしやすい題材です。扱うテーブルは departments（部署）と employees（社員）の2つだけです。

departments テーブル（部署）

カラム 型 内容
idBIGINT部署ID
nameVARCHAR部署名

employees テーブル（社員）

カラム 型 内容
idBIGINT社員ID
nameVARCHAR社員名
emailVARCHARメールアドレス
department_idBIGINT所属部署ID

この2つのテーブルを、Java の世界では Department クラスと Employee クラスとして表します。テーブルとクラスがきれいに1対1で対応している点に注目してください。これが JPA の「オブジェクトとテーブルの対応」のイメージです。

flowchart LR A["Department クラス"] -.対応.-> B[("departments テーブル")] C["Employee クラス"] -.対応.-> D[("employees テーブル")] C -->|所属する部署を参照| A

さらに、社員は必ずどこかの部署に所属します。つまり Employee は Department を「参照する」関係になります。この「多数の社員が1つの部署に所属する（多対1）」関連を、JPA では @ManyToOne という目印で表現します（第8章で詳しく学びます）。本教材では混乱を避けるため、社員から部署を参照する単方向の関連に絞って学習を進めます。

理解度確認クイズ

  1. Q1. JPA とは何のための仕様ですか？

    回答を表示

    Java のオブジェクトとデータベースのテーブルを対応させるための仕様（ルール）です。この対応づけの技術を O/R マッピング（ORM）と呼び、JPA はそのやり方を定めています。

  2. Q2. JPA・Hibernate・Spring Data JPA の役割の違いを説明してください。

    回答を表示

    JPA は「対応づけの仕様（ルール）」、Hibernate は「その仕様を実際に動かす実装」、Spring Data JPA は「JPA・Hibernate を Spring で使いやすくする仕組み」です。私たちが書くのは Spring Data JPA の部分で、裏では Hibernate が JPA の仕様どおりに SQL を生成します。

  3. Q3. Spring Data JDBC と比べたとき、Spring Data JPA の特徴は何ですか？

    回答を表示

    テーブル・SQL に近い Spring Data JDBC に対し、Spring Data JPA は Java オブジェクト中心で、Entity 同士の関連（テーブルのつながり）を扱いやすいのが特徴です。自動処理が多めで、関連テーブルの学習に向いています。
