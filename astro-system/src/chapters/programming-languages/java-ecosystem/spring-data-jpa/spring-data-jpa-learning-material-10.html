Spring MVCと連携する

Spring MVC で学んだ Controller・Model・View の流れに、JPA による DB 操作を組み込みます。社員一覧・詳細・登録・編集・削除・検索の各画面と処理を作り、「ブラウザ → Controller → Service → Repository → DB → View → ブラウザ」という一連の流れを通して理解します。JPA の Serviceを Spring MVC から呼び出すことで、これまでの学習の集大成となる Web アプリケーションの骨格が完成します。

この章で学ぶこと

  • 「ブラウザ → Controller → Service → Repository → DB → View」の流れを説明できる
  • @Controller の EmployeeController を実装し、一覧・詳細・登録・編集・削除・検索の処理を書ける
  • Thymeleaf の th:each で社員一覧を表示し、employee.department.name で所属部署名を表示できる
  • 登録フォームの部署プルダウンを th:each で Department リストから動的生成できる

10.1 Spring MVC と JPA を組み合わせる — 全体の流れ

この章では、これまで別々に学んできた Spring MVC（Controller・Model・View・Thymeleaf）と Spring Data JPA（Entity・Repository・Service・関連・トランザクション）を1つのアプリケーションにつなげます。

前提として、Spring MVC の Controller・Model・View の役割や Thymeleaf の基本構文はすでに学習済みという位置づけです。この章で特に注目するのは、「Controller から JPA の Service を呼び出す」という接続部分です。JPA を使うことで、Controller が SQL を直接書くことなく、Service を通してデータベースとやりとりできるようになります。

Spring MVC の復習ポイント

@Controller のメソッドで Model にデータを入れると、Thymeleaf のテンプレートから th:* 属性でアクセスできます。この章では、Model に渡すデータを JPA の Service から取得するようにします。

10.2 EmployeeController を実装する

EmployeeController は EmployeeService と DepartmentService の2つを注入します。EmployeeService は社員の CRUD・検索操作に使い、DepartmentService は部署一覧を取得して登録フォームのプルダウンに使います。

EmployeeController.java（一覧・詳細・検索） 
@Controller
public class EmployeeController {

    // コンストラクタインジェクション
    private final EmployeeService employeeService;
    private final DepartmentService departmentService;

    public EmployeeController(EmployeeService es, DepartmentService ds) {
        this.employeeService = es;
        this.departmentService = ds;
    }

    // 社員一覧: 全社員を取得してModelに渡す
    @GetMapping("/employees")
    public String list(Model model) {
        model.addAttribute("employees", employeeService.findAll());
        return "employees/list"; // templates/employees/list.html
    }

    // 社員詳細: URLパスからIDを受け取り1件取得
    @GetMapping("/employees/{id}")
    public String detail(@PathVariable Long id, Model model) {
        model.addAttribute("employee", employeeService.findById(id));
        return "employees/detail";
    }
}
EmployeeController.java（登録・削除・検索） 
    // 登録フォーム表示: 空社員＋部署一覧をModelに渡す
    @GetMapping("/employees/new")
    public String newForm(Model model) {
        model.addAttribute("employee", new Employee());
        model.addAttribute("departments", departmentService.findAll());
        return "employees/form";
    }

    // 登録処理: フォームの値を受け取りDB保存
    @PostMapping("/employees")
    public String create(@ModelAttribute Employee employee) {
        employeeService.save(employee);
        return "redirect:/employees"; // 一覧へリダイレクト
    }

    // 検索: クエリパラメータ ?name=... で部分一致検索
    @GetMapping("/employees/search")
    public String search(@RequestParam String name, Model model) {
        model.addAttribute("employees", employeeService.findByNameContaining(name));
        model.addAttribute("keyword", name);
        return "employees/list";
    }

Controller の役割は「つなぐだけ」

Controller は Service を呼ぶ・Model にデータを入れる・テンプレート名を返す、という3つだけを担当します。DB の操作ロジックは Service に、データの永続化は Repository に、それぞれ委ねます。Controller に SQL やビジネスロジックを書かないことが、保守しやすいコードの基本です。

削除処理は、HTML フォームが DELETE メソッドを直接送れないため、POST で受け取るのが一般的です。Spring MVC では HiddenHttpMethodFilter を使う方法もありますが、ここではシンプルに POST で実装します。

EmployeeController.java（編集・削除） 
    // 編集フォーム表示: 既存データ＋部署一覧を渡す
    @GetMapping("/employees/{id}/edit")
    public String editForm(@PathVariable Long id, Model model) {
        model.addAttribute("employee", employeeService.findById(id));
        model.addAttribute("departments", departmentService.findAll());
        return "employees/form"; // 登録と同じフォームを再利用
    }

    // 削除処理: 指定IDの社員を削除して一覧へ
    @PostMapping("/employees/{id}/delete")
    public String delete(@PathVariable Long id) {
        employeeService.deleteById(id);
        return "redirect:/employees";
    }

登録フォームと編集フォームを共通化する

登録（/employees/new）と編集（/employees/{id}/edit）は同じ employees/form.html を使います。employee.id が null なら新規登録、値があれば更新、という Thymeleaf の条件分岐で1つのテンプレートを使い回せます（第11章の演習で実践します）。

10.3 Thymeleaf テンプレートを実装する

Controller が Model にデータを渡したら、Thymeleaf のテンプレートでそれを表示します。ここでは、社員一覧テンプレートと登録フォームの2つの重要なポイントを見ていきます。

社員一覧テンプレート（th:each で繰り返し表示）

th:each を使って社員リストを1件ずつ展開します。注目ポイントは employee.department.name で所属部署名を表示できる点です。これは第8章の @ManyToOne による関連の成果です。

templates/employees/list.html（抜粋） 
<table>
  <thead>
    <tr>
      <th>社員名</th><th>メール</th><th>所属部署</th>
    </tr>
  </thead>
  <tbody>
    <!-- th:each で employees リストを繰り返す -->
    <tr th:each="employee : ${employees}">
      <td th:text="${employee.name}"></td>
      <td th:text="${employee.email}"></td>
      <!-- @ManyToOne の成果: 部署名がそのまま取れる -->
      <td th:text="${employee.department.name}"></td>
    </tr>
  </tbody>
</table>

employee.department.name で部署名を取れる理由

employee.getDepartment().getName() という Java のメソッド呼び出しに相当します。第8章で Employee に @ManyToOne で Department を関連付けたため、employee オブジェクトが department を参照として持ち、そのまま .name でアクセスできます。SQL の JOIN を自分で書く必要がありません。

部署プルダウン（th:each で動的生成）

登録フォームの部署選択は、Controller が渡した departments（全部署のリスト）を th:each で繰り返し、<option> タグを動的に生成します。

templates/employees/form.html（部署プルダウン抜粋） 
<form th:action="@{/employees}" th:object="${employee}" method="post">
  <!-- 社員名 -->
  <input type="text" th:field="*{name}" placeholder="社員名" />
  <!-- メールアドレス -->
  <input type="email" th:field="*{email}" placeholder="メールアドレス" />
  <!-- 部署プルダウン: departments リストから動的生成 -->
  <select th:field="*{department}">
    <option th:each="dept : ${departments}"
            th:value="${dept}"
            th:text="${dept.name}"></option>
  </select>
  <button type="submit">登録</button>
</form>

th:field と @ModelAttribute の連携

th:field="*{name}" は th:object で指定したオブジェクト（ここでは employee）の name フィールドに対応する id・name・value を自動で補ってくれます。POST メソッドで送信された値は @ModelAttribute Employee employee のパラメータに自動でバインドされます。

検索フォームとクエリパラメータ

検索は GET メソッドで ?name=キーワード というクエリパラメータを送ります。Controller で @RequestParam String name を受け取り、EmployeeService の findByNameContaining（第6章で実装）を呼び出して結果を一覧表示します。

10.4 テンプレートのファイル構成

Spring MVC + Thymeleaf では、テンプレートは src/main/resources/templates/ 配下に置きます。Controller が返す文字列（例: "employees/list"）が、テンプレートのパス（templates/employees/list.html）に対応します。

テンプレートのファイル構成 

src/main/resources/templates/
└── employees/
    ├── list.html     ← 社員一覧（検索結果も兼用）
    ├── detail.html   ← 社員詳細
    └── form.html     ← 登録・編集フォーム（共通）

Controller が返す文字列と、テンプレートのパスの対応を整理しておきましょう。

Controller の戻り値 使われるテンプレート 用途
"employees/list" templates/employees/list.html 一覧・検索結果表示
"employees/detail" templates/employees/detail.html 社員詳細表示
"employees/form" templates/employees/form.html 登録・編集フォーム
"redirect:/employees" （テンプレートなし） 一覧画面へリダイレクト

POST の後は redirect で一覧へ戻る

登録・削除などの POST 処理の後に "employees/list" を返すと、ブラウザの再読み込みで同じ POST が繰り返される（二重送信）問題が起きます。"redirect:/employees" を返して一覧画面に GET でアクセスし直す（PRGパターン）のが正しい実装です。

やってみよう 10-1: EmployeeController を実装して動かす

第5章で作成した EmployeeService と DepartmentService を利用して、EmployeeController を実装し、社員一覧と登録フォームを動かしてみましょう。

手順

  1. 1 controller/ パッケージに EmployeeController.java を作成し、10.2 のコードを参考に list()・newForm()・create() メソッドを実装する。
  2. 2 templates/employees/list.html を作成し、th:each で社員名・メール・所属部署名（employee.department.name）を表示するテーブルを作る。
  3. 3 templates/employees/form.html を作成し、部署プルダウンを th:each で動的生成する。
  4. 4 アプリを起動し、http://localhost:8080/employees で社員一覧が表示されること、登録フォームで社員を追加できることを確認する。

確認ポイント

  • 一覧画面で「所属部署」列に部署名が表示される（null や @12ab34 ではなく部署名が出る）
  • 登録フォームの部署プルダウンに、DBにある部署が選択肢として並ぶ
  • 登録後に一覧画面へリダイレクトされ、新しい社員が表示される

理解度確認クイズ

  1. Q1. EmployeeController が DepartmentService を注入する理由は何ですか？

    回答を表示

    社員の登録・編集フォームで「部署のプルダウン」を表示するためです。全部署リストを DepartmentService.findAll() で取得し、Model に渡して Thymeleaf の th:each で <option> を動的生成します。

  2. Q2. Thymeleaf のテンプレートで employee.department.name と書くだけで部署名が表示できる理由を説明してください。

    回答を表示

    第8章で Employee クラスに @ManyToOne で Department のフィールドを定義したため、employee オブジェクトが Department を参照として持っています。employee.getDepartment().getName() に相当するプロパティアクセスを JPA が自動で解決するため、SQL の JOIN を自分で書かなくても部署名を取得できます。

  3. Q3. 社員登録 POST 処理の後に "employees/list" ではなく "redirect:/employees" を返す理由は何ですか？

    回答を表示

    POST 処理の後にそのままテンプレートを表示すると、ブラウザを再読み込みしたときに同じ POST リクエストが再送信され、社員が二重登録される問題（二重送信）が発生します。"redirect:/employees" を返すことで、ブラウザを一覧画面への GET リクエストに切り替えます（PRGパターン）。再読み込みしても GET リクエストが繰り返されるだけになり、二重送信を防げます。
