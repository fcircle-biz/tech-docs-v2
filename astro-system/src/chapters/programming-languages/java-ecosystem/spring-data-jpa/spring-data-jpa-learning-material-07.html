@Queryで検索処理を書く

第6章ではメソッド名から検索を自動生成する「クエリメソッド」を学びました。この章では、メソッド名だけでは表現しづらい複雑な検索条件を、@Query アノテーションを使って自分で書く方法を学びます。JPA が使う問い合わせ言語「JPQL」の考え方を理解し、クエリメソッドと @Query を使い分けられるようになりましょう。

この章で学ぶこと

  • JPQL がテーブルではなく Entity クラスを対象にした問い合わせ言語であることを理解する
  • @Query を使って Repository メソッドに JPQL を直接書く方法を理解する
  • @Param で名前付きパラメータに値を渡す方法を理解する
  • クエリメソッド（第6章）と @Query の使い分けを説明できる
  • ネイティブ SQL が必要な場面と nativeQuery = true の使い方を知る

7.1 JPQL とは — テーブルではなく Entity で問い合わせる言語

データベースにはデータを取り出すための言語として SQL（Structured Query Language） があります。SQL では select * from employees のようにテーブル名・カラム名を使って書きます。

JPA にはこれに相当する言語として JPQL（Jakarta Persistence Query Language） があります。JPQL は SQL と非常に似た文法ですが、テーブル名やカラム名ではなく Entity クラス名・フィールド名で書くのが最大の特徴です。

項目 SQL JPQL
対象 テーブル・カラム Entity クラス・フィールド
全件取得 select * from employees select e from Employee e
名前で絞り込み where name = 'Alice' where e.name = 'Alice'
大文字・小文字 テーブル名は大文字・小文字どちらも可 Entity クラス名・フィールド名は大文字・小文字を区別する

重要ポイント: JPQL は Entity 名・フィールド名で書く

JPQL で書く文字列はテーブルの物理名（employees、name）ではなく、Java の Entity クラス名（Employee）とフィールド名（e.name） です。クラス名は大文字・小文字も正確に書く必要があります。

7.2 @Query の基本 — メソッドに JPQL を書く

@Query は、Repository インターフェースのメソッドに JPQL（または SQL）を直接書くためのアノテーションです。第6章の「クエリメソッド」がメソッド名から検索を自動生成したのに対し、@Query は自分で検索文を書く方法です。

クエリメソッド vs @Query の使い分け

  • クエリメソッド（第6章）: 1〜2 条件程度のシンプルな検索に向く。メソッド名が長くなりすぎる場合は @Query へ切り替える。
  • @Query: 複数条件・集計・サブクエリなど、メソッド名では表現しづらい複雑な検索に使う。JPQL が読みやすく保守しやすい。

まず最もシンプルな例として、名前が一致する社員を検索する @Query を見てみましょう。

EmployeeRepository.java — @Query の基本 
package com.example.jpalearning.repository;

import com.example.jpalearning.entity.Employee;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;
import java.util.List;

public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    // JPQL で name が一致する社員を検索（:name が @Param で渡す値）
    @Query("select e from Employee e where e.name = :name")
    List<Employee> searchByName(@Param("name") String name);

}

ポイントをまとめると次のとおりです。

@Query

メソッドの上に付け、引数の文字列に JPQL を書く。Employee・e.name のように Entity 名・フィールド名を使う。

:name（名前付きパラメータ）

JPQL 内の :name はプレースホルダー（値を差し込む場所）。コロン + 名前で定義する。

@Param

メソッド引数に付け、JPQL の :name と引数を結びつける。名前は一致させる必要がある。

7.3 部分一致検索 — LIKE を使った @Query

第6章では findByNameContaining でキーワードが名前に含まれる社員を検索しました。これと同じことを @Query で書いてみましょう。SQL の LIKE に相当する文法が JPQL にもあります。

EmployeeRepository.java — 部分一致検索 
// 名前にキーワードが含まれる社員を検索
// 基本形: 標準 JPQL の concat 関数で前後に % を付ける（多くの JPA 実装で動作する）
@Query("select e from Employee e where e.name like concat('%', :keyword, '%')")
List<Employee> searchByNameKeyword(@Param("keyword") String keyword);

Tips: 本教材では concat に統一します

本教材では基本形として concat('%', :keyword, '%') を使います。標準 JPQL の関数なので、ほとんどの JPA 実装でそのまま動作するためです。なお、Spring Data JPA には ... like %:keyword% という短い書き方（参考）もありますが、書き方が混在すると初心者は迷いやすいため、本教材では concat に統一します。

クエリメソッドと @Query では同じ検索が実現できますが、条件が複雑になるほど @Query の JPQL で書くほうが意図が明確になります。下の表を参考に使い分けてください。

検索内容 クエリメソッド（第6章） @Query（この章）
名前が一致 findByName(name) @Query("...where e.name = :name")
名前に含まれる findByNameContaining(kw) @Query("...like concat('%', :keyword, '%')")
複合条件（例: 名前 OR メール） メソッド名が長くなりがち @Query が読みやすい
集計・並び替えの組み合わせ 難しい・限界がある @Query が向く

7.4 ネイティブ SQL — テーブル名・カラム名で書きたいとき

通常は JPQL（Entity 名・フィールド名）を使いますが、データベース固有の関数や構文を使いたい場面では JPQL で表現できないこともあります。そのような場合に限り、ネイティブ SQL（通常の SQL）を書く方法があります。

@Query に nativeQuery = true を付けると、テーブル名・カラム名で SQL を書けます。

EmployeeRepository.java — ネイティブSQL 
// nativeQuery = true でテーブル名・カラム名の SQL を書く
@Query(value = "select * from employees where name like concat('%', :keyword, '%')",
       nativeQuery = true)
List<Employee> searchByNameNative(@Param("keyword") String keyword);

注意: ネイティブ SQL は必要な場合のみ

ネイティブ SQL はデータベース製品固有の構文を使えますが、DB を変えると動かなくなるリスクがあります。また、テーブル名・カラム名を直接書くため、Entity のフィールド名変更が SQL に反映されない点も注意が必要です。学習・業務ともに、まず JPQL で書けないか試し、どうしても必要な場合だけネイティブ SQL を使うのが基本方針です。

7.5 @Query を使った Repository の全体像

これまで学んだ内容を組み合わせた EmployeeRepository の全体像を確認しましょう。クエリメソッドと @Query を混在させることもできます。

EmployeeRepository.java — クエリメソッド＋@Query の組み合わせ 
package com.example.jpalearning.repository;

import com.example.jpalearning.entity.Employee;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;
import java.util.List;

public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    // --- クエリメソッド（シンプルな条件）---
    List<Employee> findByName(String name);          // 名前が一致
    List<Employee> findByNameContaining(String kw); // 名前の部分一致

    // --- @Query（JPQL）---
    // 名前またはメールが一致する社員を検索（OR 条件）
    @Query("select e from Employee e where e.name = :q or e.email = :q")
    List<Employee> searchByNameOrEmail(@Param("q") String query);

}

重要ポイント: @Param の名前は JPQL のパラメータ名と一致させる

JPQL に :q と書いたら、引数の @Param("q") の文字列も q に揃えます。名前が一致しないと実行時エラーになります。わかりやすい名前（keyword、name など）を付けると読みやすくなります。

やってみよう 7-1: @Query でメール部分一致検索を実装する

メールアドレスにキーワードが含まれる社員を検索する @Query メソッドを EmployeeRepository に追加し、実際に動作を確認しましょう。

手順

  1. 1 EmployeeRepository に次のメソッドを追加する。
    @Query("select e from Employee e where e.email like concat('%', :keyword, '%')")
    List<Employee> searchByEmail(@Param("keyword") String keyword);
  2. 2 EmployeeService に searchByEmail(String keyword) メソッドを追加し、Repository の searchByEmail を呼ぶ。
  3. 3 アプリを起動し、H2 コンソール（/h2-console）でテストデータを数件 INSERT してから、Service のメソッドを呼び出し（またはテストクラスで）動作を確認する。
  4. 4 コンソールに出力される SQL を確認し、Hibernate が生成した SQL がどうなっているかを観察する（application.properties に spring.jpa.show-sql=true が必要）。

動作確認のポイント

  • キーワードを含むメールの社員だけが返ること。
  • コンソールに like ? 形式の SQL が出力されること（Hibernate が JPQL を SQL に変換した結果）。
  • キーワードを空文字にすると全件返ること（like %% は全一致）。

理解度確認クイズ

  1. Q1. JPQL と SQL の最大の違いは何ですか？

    回答を表示

    SQL がデータベースのテーブル名・カラム名を使うのに対し、JPQL は Java の Entity クラス名・フィールド名を使います。たとえば全件取得は SQL が select * from employees、JPQL では select e from Employee e と書きます。

  2. Q2. @Query と @Param はそれぞれ何をするためのアノテーションですか？

    回答を表示

    @Query は Repository のメソッドに JPQL（または SQL）を直接書くためのアノテーションです。@Param はメソッドの引数と JPQL 内の名前付きパラメータ（:name の形式）を結びつけるアノテーションです。たとえば JPQL に :keyword と書いた場合、引数に @Param("keyword") を付けて渡します。

  3. Q3. ネイティブ SQL を使うのはどのような場合ですか？また、使いすぎると何が問題になりますか？

    回答を表示

    ネイティブ SQL（nativeQuery = true）は、JPQL では表現できないデータベース固有の関数や構文を使いたい場合に限り使います。ただし、テーブル名・カラム名で書くため、データベース製品を変えると動作しなくなる可能性があります。また、Entity のフィールド名を変更しても SQL には自動で反映されないため、メンテナンスコストが上がります。基本は JPQL を優先し、どうしても必要な場合のみネイティブ SQL を使うのが正しい使い方です。
