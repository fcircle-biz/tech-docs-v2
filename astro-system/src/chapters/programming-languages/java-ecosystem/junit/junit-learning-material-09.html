9.1 今回のテスト対象：Calculator クラスを知ろう

前編・後編の2章にわたって取り組む総合演習では、シンプルな電卓アプリ Calculator クラスをテスト対象として使います。まずはクラスの仕様を確認しましょう。

Calculator クラスの仕様

  • add(int a, int b) ：a と b の足し算結果を返す
  • subtract(int a, int b) ：a から b を引いた結果を返す
  • multiply(int a, int b) ：a と b の掛け算結果を返す
  • divide(int a, int b) ：a を b で割った結果を返す。b が 0 のとき IllegalArgumentException を投げる

実装コードを確認しましょう。コードは理解しやすいように2ブロックに分けて示します。

Calculator.java（前半：add・subtract） 
// 電卓クラス：4つの四則演算メソッドを持つ
public class Calculator {

    // 足し算：a + b を返す
    public int add(int a, int b) {
        return a + b;
    }

    // 引き算：a - b を返す
    public int subtract(int a, int b) {
        return a - b;
    }
}
Calculator.java（後半：multiply・divide） 
    // 掛け算：a × b を返す
    public int multiply(int a, int b) {
        return a * b;
    }

    // 割り算：a ÷ b を返す（b が 0 なら例外を投げる）
    public int divide(int a, int b) {
        if (b == 0) {
            throw new IllegalArgumentException("0 で割ることはできません");
        }
        return a / b;
    }

divide メソッドだけ少し特別です。0 で割ろうとすると IllegalArgumentException（不正な引数の例外）を投げます。この異常系のテストは後編（第10章）で扱います。前編の今章では add・subtract・multiply の正常系をカバーすることに集中します。

前編・後編の役割分担

前編（この章）：add・subtract・multiply の正常系テストを書く。AAAパターンと @BeforeEach の活用方法を体験する。
後編（次章）：divide の異常系（0除算例外）テストを書き、テストカバレッジの考え方を学ぶ。

9.2 テストの設計図：AAAパターンを意識しよう

テストコードを書く前に、AAAパターンという考え方を整理します。第3章のアサーション学習でも出てきた概念ですが、今回の総合演習では特に意識して使います。

A — Arrange（準備）

テストに必要なオブジェクトやデータを用意します。電卓なら Calculator のインスタンスを作る工程です。

A — Act（実行）

テスト対象のメソッドを実際に呼び出します。「2 + 3 を計算させる」などの操作がここに入ります。

A — Assert（検証）

実行結果が期待どおりかを assertEquals などで確認します。「5 になるはず」と主張する部分です。

図にするとこのような流れです。

flowchart LR A["Arrange（準備）
Calculator を new する"] --> B["Act（実行）
add など呼び出す"] B --> C["Assert（検証）
assertEquals で確かめる"] C --> D{"一致？"} D -->|"はい"| E["テスト成功"] D -->|"いいえ"| F["テスト失敗
＝バグ発見"]

コメントでAAAを区切ると読みやすい

テストコードの中に // Arrange・// Act・// Assert というコメントを書くと、どこで何をしているかが一目で分かります。今回の演習では積極的に書いてみましょう。

9.3 足し算のテストを書こう（@BeforeEach で準備を共通化）

まずは add メソッドのテストから始めます。毎回 Calculator を new するのは手間なので、第5章で学んだ @BeforeEach を使って共通セットアップにまとめましょう。

CalculatorTest.java（クラス定義と @BeforeEach） 
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {

    // テスト対象のインスタンスをフィールドに宣言
    private Calculator calc;

    @BeforeEach
    void setUp() {
        // 各テストの前に新しい Calculator を生成（Arrange の共通部分）
        calc = new Calculator();
    }
}

@BeforeEach に setUp メソッドを書いたことで、すべてのテストメソッドは「すでに新鮮な Calculator が calc に入っている」状態からスタートできます。次は add の正常系テストを追加します。

CalculatorTest.java（addメソッドのテスト） 
    @Test
    void 正の数同士を足し算できる() {
        // Act：2 + 3 を計算
        int result = calc.add(2, 3);
        // Assert：結果は 5 であるべき
        assertEquals(5, result);
    }

    @Test
    void 負の数を含む足し算ができる() {
        // Act：-4 + 10 を計算
        int result = calc.add(-4, 10);
        // Assert：結果は 6 であるべき
        assertEquals(6, result);
    }

AAAパターンの「Arrange（準備）」は @BeforeEach の setUp が担っているため、各テストメソッドには Act と Assert だけ書けばよいのがポイントです。テストが増えるほどこの恩恵が大きくなります。

テストメソッド名は「日本語 OK」

JUnit 5 ではテストメソッド名に日本語が使えます。正の数同士を足し算できる のように書くと、テスト結果レポートがそのまま仕様書のように読めます。英語名でも問題ありませんが、読みやすさを意識して命名することが大切です。

9.4 引き算のテストを書こう（パラメータ化テストで複数ケースをまとめる）

次は subtract（引き算）のテストです。「正の数」「負の数」「ゼロ」など複数のパターンを確認したいとき、一つひとつ @Test を書くと重複が多くなります。第6章で学んだパラメータ化テスト（@ParameterizedTest + @CsvSource）を活用しましょう。

CalculatorTest.java（subtractのパラメータ化テスト） 
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.CsvSource;

    @ParameterizedTest
    @CsvSource({
        "10, 3,  7",  // 正の数同士：10 - 3 = 7
        " 5, 5,  0",  // 同じ数：差はゼロ
        " 0, 5, -5",  // ゼロから引く：負の数になる
        "-3, 2, -5"   // 負の数から引く
    })
    void 引き算の各パターンを検証できる(int a, int b, int expected) {
        // Act + Assert をまとめて書く
        assertEquals(expected, calc.subtract(a, b));
    }

@CsvSource の各行が1テストケース分の入力（a, b）と期待値（expected）になります。4行書くだけで「正の数」「ゼロ」「負の数」「負の数から引く」の4パターンを一度に検証できます。

flowchart TD A["@CsvSource の各行"] --> B["1行目: a=10, b=3, expected=7"] A --> C["2行目: a=5, b=5, expected=0"] A --> D["3行目: a=0, b=5, expected=-5"] A --> E["4行目: a=-3, b=2, expected=-5"] B --> F["subtract を呼び出して assertEquals"] C --> F D --> F E --> F

9.5 掛け算のテストを書こう（ゼロや負の数も忘れずに）

multiply（掛け算）のテストも同様にパラメータ化テストで書きます。掛け算で注意したいのは「ゼロを掛けると必ずゼロになる」「負の数と正の数を掛けると負になる」といったケースです。

CalculatorTest.java（multiplyのパラメータ化テスト） 
    @ParameterizedTest
    @CsvSource({
        "3,  4,  12",   // 正の数同士：3 × 4 = 12
        "0,  5,   0",   // ゼロを掛けると結果はゼロ
        "-2, 3,  -6",   // 負×正 = 負
        "-2, -3,  6"    // 負×負 = 正
    })
    void 掛け算の各パターンを検証できる(int a, int b, int expected) {
        assertEquals(expected, calc.multiply(a, b));
    }

「ゼロを掛ける」「負の数同士を掛ける」といった境界値を意識してテストケースを選ぶことが、品質の高いテストを書くコツです。「正の数同士だけ動けばいい」では不十分で、数学的に少し変わったケースも必ず試しましょう。

正常系だけでは不十分なこともある

今回の multiply は単純な掛け算なので境界値でも壊れませんが、複雑なロジックほど「ゼロ」「負の数」「最大値・最小値」などの境界値でバグが潜みやすくなります。「どんなデータでテストするか」を考えながらケースを選ぶ習慣をつけましょう。

9.6 ここまでのテスト全体像を確認しよう

前編で書いてきたテストを一枚にまとめると、クラス全体はこのような構成になります。

CalculatorTest.java（前編の全体像） 
// 前編でここまで書いた CalculatorTest の構成
class CalculatorTest {
    private Calculator calc;

    @BeforeEach void setUp() { calc = new Calculator(); }

    // add のテスト（通常の @Test）
    @Test void 正の数同士を足し算できる() { ... }
    @Test void 負の数を含む足し算ができる() { ... }

    // subtract のパラメータ化テスト（4ケース）
    @ParameterizedTest @CsvSource({...})
    void 引き算の各パターンを検証できる(...) { ... }

    // multiply のパラメータ化テスト（4ケース）
    @ParameterizedTest @CsvSource({...})
    void 掛け算の各パターンを検証できる(...) { ... }
}

前編のまとめ：使った技術

  • @BeforeEach（第5章）：共通の準備を一箇所にまとめて重複を減らした
  • @Test + assertEquals（第2〜3章）：add の正常系を個別テストで確認した
  • @ParameterizedTest + @CsvSource（第6章）：subtract と multiply で複数パターンをまとめて検証した
  • AAAパターン：Arrange・Act・Assert を意識してテストを整理した

9.7 後編（第10章）への橋渡し

前編ではCalculatorの正常系テストをひととおり書きました。しかしまだテストには「穴」があります。divide メソッドを0で割ったときの動作（例外が投げられることの確認）がテストされていません。

次章（第10章）で扱う内容

  • divide の正常系テスト：普通の割り算が正しく動くことを確認する
  • divide の異常系テスト：0除算で IllegalArgumentException が投げられることを assertThrows で検証する（第7章の応用）
  • テストカバレッジ：「どのくらいのコードをテストできているか」を測定する方法と、その読み方を学ぶ

前編・後編の2章を通じて完成したテストスイートは、Calculator クラスの全メソッドの正常系・異常系をカバーする本格的なテストになります。後編でしっかり仕上げましょう。

やってみよう 9-1: 掛け算のテストケースを1つ追加しよう

9.5節で書いた @CsvSource に、あなたが「これも確かめたい」と思うテストケースをもう1行追加してみましょう。

手順

  1. 1 今の4ケースを見て、「まだ試していない状況」を一つ考える（例：1 × 1 = 1 や -5 × 0 = 0 など）。
  2. 2 @CsvSource の中に "a, b, expected" の形式で1行追加する。
  3. 3 テストを実行して、追加したケースも含め全件グリーン（成功）になることを確認する。

追加のチャレンジ

余裕があれば、add のテストもパラメータ化テストに書き換えてみよう。現在の2つの @Test を1つの @ParameterizedTest にまとめるとどうなるか試してみてください。

理解度確認クイズ

  1. Q1. AAAパターンの「A・A・A」はそれぞれ何の頭文字ですか？また各ステップで何をしますか？

    回答を表示

    Arrange（準備）・Act（実行）・Assert（検証）の頭文字です。「Arrange でテストに必要なオブジェクトやデータを用意し、Act でテスト対象のメソッドを呼び出し、Assert で結果が期待値どおりかを assertEquals などで確認する」という3ステップです。

  2. Q2. @BeforeEach を使ってCalculatorのインスタンス生成を共通化すると、どんなメリットがありますか？

    回答を表示

    各テストメソッドに new Calculator() を書く必要がなくなり、コードの重複を減らせます。またテストごとに新鮮なインスタンスが用意されるので、前のテストの影響を受けない独立した実行環境が保証されます。

  3. Q3. 今回の前編では Calculator の divide メソッドをテストしませんでした。その理由と、後編で何をテストするかを説明してください。

    回答を表示

    前編は「正常系をひととおりカバーする」ことを目標にしているため、異常系（0除算で例外が投げられること）が含まれる divide は後編に分けました。後編（第10章）では assertThrows を使った0除算テストと、テストカバレッジ（どの程度コードをテストできているか）の考え方を学びます。
