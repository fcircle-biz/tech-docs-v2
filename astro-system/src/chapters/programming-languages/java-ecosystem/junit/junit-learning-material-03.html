期待通りか確かめよう アサーションの基本

テストで「結果が正しいかどうか」を判定するアサーション（assertion）を学びます。assertEquals・assertTrue・assertFalse など基本のアサーションを、電卓の計算結果を確認するたとえで丁寧に理解していきましょう。アサーションこそが「テストをテストたらしめる」最重要の仕組みです。

この章で学ぶこと

  • アサーションとは何か——「結果はこうであるはずだ」という断言の意味を説明できる
  • assertEquals(期待値, 実際の値) の引数の順序を正しく書ける
  • assertTrue / assertFalse / assertNull / assertNotNull を使い分けられる
  • 失敗時のメッセージ付きアサーションと static import の書き方を理解する

3.1 アサーションとは——「主張・断言」の仕組み

アサーション（assertion）とは、英語で「主張・断言」を意味する言葉です。テストコードの世界では、「この結果は、こうであるはずだ」と断言するコードのことを指します。アサーションがテストに書かれていることで、JUnit は「実際の結果が断言どおりかどうか」を自動で判定できます。

たとえ話: 電卓の答えを確認する

あなたが電卓で「3 × 7」を計算したとき、答えが「21」になるはずだと分かっているとします。画面に「21」と表示されていれば「正しい！」、「22」と表示されていれば「おかしい！壊れてる？」と気づきますよね。

アサーションは、まさにこの「21 になるはずだ、と確認する」行為をコードで書いたものです。テストの中に「こうなるはずだ」という断言を入れておくと、JUnit が自動で「実際にそうなっているか」を確かめてくれます。

アサーションがないテストコードは、いわば「電卓のボタンを押すだけで、答えを確認しないまま終わる」ようなものです。JUnit でテストを書くとき、アサーションは必ず入れるものだと覚えておきましょう。

3.2 static import の準備

JUnit 5 のアサーションはすべて org.junit.jupiter.api.Assertions クラスに入っています。通常は Assertions.assertEquals(...) のように クラス名を付けて呼びますが、毎回書くのは大変です。そこで static import（静的インポート）を使うと、クラス名なしで assertEquals(...) と短く書けるようになります。

テストクラスの冒頭に書く import 
// すべてのアサーションメソッドを短く使えるようにする
import static org.junit.jupiter.api.Assertions.*;

// これにより、以下のように書けるようになる
// assertEquals(5, result);  ← Assertions. が不要になる

用語: static import とは？

通常の import は「クラスを短く書けるようにする」仕組みですが、static import は「クラスのメソッドや定数を、クラス名なしで呼べるようにする」仕組みです。import static クラス名.*; と書くと、そのクラスのすべての static メンバーが直接使えます。JUnit のアサーションメソッドはすべて static なので、この書き方が定番です。

3.3 assertEquals — 値が等しいことを断言する

最もよく使うアサーションが assertEquals です。「期待する値と実際の値が等しい」ことを断言します。引数の順序は必ず 「期待値（expected）が先、実際の値（actual）が後」という決まりがあります。

よくある間違い: 引数の順序

assertEquals(期待値, 実際の値) の順序を逆にしてしまうミスがとても多いです。逆にしてもテストの成功・失敗の判定は変わりませんが、テストが失敗したときのエラーメッセージが逆になってしまい、「期待値が実際の値、実際の値が期待値」のように読めてしまいます。混乱のもとになるので、必ず「期待値を先に」書くクセをつけましょう。

✔ 正しい: assertEquals(5, result); // 期待値=5、実際=result

✘ 逆順: assertEquals(result, 5); // メッセージが混乱する

CalculatorTest.java 
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class CalculatorTest {
    @Test
    void 掛け算が正しく計算できる() {
        Calculator calc = new Calculator();
        int result = calc.multiply(3, 7);  // 3 × 7 を計算
        // 期待値=21 を先、実際の結果=result を後に書く
        assertEquals(21, result);
    }
}

テストが失敗すると、JUnit は次のようなメッセージを表示します。「expected（期待値）は何で、actual（実際の値）は何だったか」が一目でわかります。

アサーション失敗時の出力例

org.opentest4j.AssertionFailedError:

expected: <21>

but was: <22>

「期待は 21 だったのに、実際は 22 でした」と教えてくれます。これを手がかりにバグを修正します。

3.4 assertTrue / assertFalse — 条件を断言する

assertTrue(条件) は「その条件が true（正しい） であることを断言」します。逆に assertFalse(条件) は「その条件が false（正しくない） であることを断言」します。計算の結果が「0 より大きい」といった true/false で答えが出る判定に向いています。

assertTrue と assertFalse の例 
@Test
void 割り算の結果が正の数かどうかを確認する() {
    Calculator calc = new Calculator();
    int result = calc.divide(10, 2); // 10 ÷ 2 = 5

    // 結果が 0 より大きい（正の数）であることを断言
    assertTrue(result > 0);

    // 結果が 0 未満（負の数）ではないことを断言
    assertFalse(result < 0);
}

Tips: assertTrue と assertEquals どちらを使う？

assertTrue(result == 5) と assertEquals(5, result) は同じ意味ですが、assertEquals の方がおすすめです。理由は、失敗時に「expected: 5 but was: 4」のように具体的な値を表示してくれるからです。assertTrue が失敗すると「expected: true but was: false」としか表示されず、原因が分かりにくくなります。具体的な値の一致を確認するときは assertEquals を使いましょう。

3.5 assertNull / assertNotNull — null かどうかを断言する

null とは「何も入っていない・存在しない」を表す特別な値です。assertNull(値) は「その値が null であること」を、assertNotNull(値) は「その値が null ではないこと（何か入っている）」を断言します。メソッドが正しくオブジェクトを返すかどうか確認するときによく使います。

assertNull と assertNotNull の例 
@Test
void ユーザー検索が正しく動くか確認する() {
    UserRepository repo = new UserRepository();

    // 存在するIDで検索→ユーザーが取得できる（null ではない）
    User found = repo.findById(1);
    assertNotNull(found); // null でないことを断言

    // 存在しないIDで検索→取得できない（null になる）
    User notFound = repo.findById(999);
    assertNull(notFound); // null であることを断言
}

3.6 失敗時メッセージ付きアサーション

アサーションメソッドの第3引数（または第2引数の最後）にメッセージを渡すと、テストが失敗したときにそのメッセージを表示してくれます。テストが多くなってくると「どのアサーションで失敗したのか」が分かりにくくなるので、メッセージを付けておくと原因の特定がぐっと楽になります。

メッセージ付きアサーションの例 
@Test
void 足し算と引き算をメッセージ付きで確認する() {
    Calculator calc = new Calculator();

    // 第3引数: 失敗したときに表示するメッセージ
    assertEquals(5, calc.add(2, 3), "2 + 3 は 5 になるはず");
    assertEquals(7, calc.subtract(10, 3), "10 - 3 は 7 になるはず");

    // assertTrue にもメッセージを付けられる（第2引数）
    assertTrue(calc.add(1, 1) > 0, "1+1 は正の数のはず");
}

重要ポイント: メッセージは「なぜそうなるべきか」を書く

失敗時メッセージには「2 + 3 は 5 になるはず」のように、なぜその値を期待しているのかが伝わるメッセージを書きましょう。「アサーション失敗」のような機械的なメッセージより、「3 ヶ月後の自分や他のメンバー」が読んでも理由が分かるメッセージが理想です。

3.7 assertNotEquals / 主要アサーション一覧

assertNotEquals(期待しない値, 実際の値) は、assertEquals の逆で「その値ではないはずだ」を断言します。たとえば「ランダムな数値が毎回 0 にはならないはず」というときに使えます。使う頻度は assertEquals より低いですが、覚えておくと便利です。

メソッド 断言の内容 使いどころ
assertEquals(期待値, 実際値) 2つの値が等しい 計算結果・文字列の比較など最頻出
assertNotEquals(値, 実際値) 2つの値が等しくない 特定の値にならないことを確認
assertTrue(条件) 条件が true true/false で表現できる判定
assertFalse(条件) 条件が false 「そうでないこと」を確認
assertNull(値) 値が null である 存在しない場合に null を返すメソッド
assertNotNull(値) 値が null でない オブジェクトが正しく生成・取得されたか

Tips: 浮動小数点の比較には delta（誤差許容値）が必要

double や float（小数点を含む数値）は、コンピューター内部での表現に微小な誤差が生じることがあります。たとえば 0.1 + 0.2 は数学的には 0.3 ですが、Javaでは 0.30000000000000004 のような値になることがあります。

そのため浮動小数点の比較には、許容誤差（delta）を第3引数に渡します。assertEquals(0.3, result, 0.0001) と書くと、「0.3 との差が 0.0001 以内ならOK」と判定されます。

3.8 アサーションをまとめて書いてみよう

ここまで学んだアサーションを組み合わせた、電卓テストの完成例を見てみましょう。static import・assertEquals・assertNotNull・assertTrue がすべて登場します。

CalculatorAssertionsTest.java（完成例） 
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class CalculatorAssertionsTest {
    @Test
    void 電卓の各アサーションを確認する() {
        Calculator calc = new Calculator();

        // assertEquals: 値が一致するか（期待値が先！）
        assertEquals(10, calc.add(3, 7), "3+7=10 のはず");

        // assertNotNull: null ではないか
        assertNotNull(calc, "電卓オブジェクトは null ではないはず");

        // assertTrue: 条件が true か
        assertTrue(calc.add(1, 1) > 0, "1+1 は正の数のはず");
    }
}

やってみよう 3-1: アサーションを書いてみよう

電卓クラスに subtract（引き算） メソッドがあるとします。以下の手順で、このメソッドをテストするアサーションを書いてみましょう。

手順

  1. 1 テストクラスの冒頭に import static org.junit.jupiter.api.Assertions.*; を書く。
  2. 2 subtract(10, 4) の結果が 6 であることを assertEquals で断言する。
  3. 3 テストを実行して「成功（緑のチェック）」が表示されることを確認する。次に、期待値をわざと間違えて（たとえば assertEquals(999, ...)）、失敗時のメッセージ（expected/actual）を観察する。
  4. 4 さらに assertFalse(subtract(10, 4) == 100) も追加してみよう。

模範解答（参考）

assertEquals(6, calc.subtract(10, 4), "10-4=6 のはず");

assertFalse(calc.subtract(10, 4) == 100, "10-4 は 100 ではない");

理解度確認クイズ

  1. Q1. assertEquals の引数の正しい順序はどちらですか？　(A) assertEquals(実際の値, 期待値)　(B) assertEquals(期待値, 実際の値)

    回答を表示

    正解は (B) assertEquals(期待値, 実際の値) です。「期待値を先に、実際の値を後に」が JUnit の決まりです。逆にすると、失敗時のエラーメッセージで expected と actual の表示が逆転してしまい、デバッグが混乱します。

  2. Q2. import static org.junit.jupiter.api.Assertions.*; を書く目的は何ですか？

    回答を表示

    Assertions クラスのメソッドを Assertions. というクラス名なしで直接呼べるようにするためです（static import）。これにより assertEquals(5, result) のように短く書けます。

  3. Q3. double 型の計算結果を比較するとき、なぜ assertEquals(0.3, result) ではなく assertEquals(0.3, result, 0.0001) と書く方がよいのですか？

    回答を表示

    浮動小数点（double・float）はコンピューター内部の表現上、微小な誤差が生じることがあるためです（例: 0.1 + 0.2 が 0.30000000000000004 になる）。第3引数の delta（誤差許容値）を指定することで、「その差が delta 以内ならOK」と柔軟に比較できます。
