様々な状況を確かめよう パラメータ化テスト入門

「入力が違うだけで、ほぼ同じテストを何個も書く」のは大変です。この章では、同じテストロジックをさまざまなデータで繰り返し実行できる パラメータ化テスト を学びます。九九を1マスずつ手で確認するのではなく、一覧表を使って一気に確かめるイメージです。@ParameterizedTest・@ValueSource・@CsvSource を中心に、コピペ地獄を卒業しましょう。

この章で学ぶこと

  • 「コピペテスト」の問題点を理解し、パラメータ化テストが解決する理由を説明できる
  • @ParameterizedTest と @ValueSource を使って、1引数のテストを複数データで実行できる
  • @CsvSource で入力と期待値をまとめてテストできる
  • @MethodSource の役割をおおまかに把握できる
  • 実行結果に「5件中5件成功」のようにデータ単位で結果が出ることを確認できる

6.1 コピペテストの「コピペ地獄」

たとえば、あなたが「数が偶数かどうか判定する」メソッドを書いたとします。0・2・4・100 が偶数であることを確かめたくなりますよね。素直に書くと、こんなテストになりがちです。

EvenNumberTest.java（コピペ版・よくある失敗例） 
@Test
void 0は偶数() { assertTrue(isEven(0)); }   // ほぼ同じ

@Test
void 2は偶数() { assertTrue(isEven(2)); }   // ほぼ同じ

@Test
void 4は偶数() { assertTrue(isEven(4)); }   // ほぼ同じ

@Test
void 100は偶数() { assertTrue(isEven(100)); } // ほぼ同じ

4つのメソッドが並んでいますが、やっていることはまったく同じです。テストしたい数が変わるたびに、コピーして貼り付け、数字だけ変える——まるでコピペ地獄です。こんな問題があります。

コードが増える

テストケースが増えるほど、ほぼ同じメソッドが増え続けます。ファイルが長くなり、全体が把握しにくくなります。

修正が大変

テストのロジック（assertTrue の部分）を変えたくなったとき、すべてのメソッドを1つずつ直す必要があります。

意図が伝わりにくい

「どんな値で確かめているか」の一覧が一目で見えないため、テストの網羅性を把握しにくくなります。

この問題をすっきり解決してくれるのが、パラメータ化テスト（Parameterized Test）です。「テストのロジックは1つ、データだけ複数」という形でまとめることができます。

たとえ話: 九九の確認表

九九を確認するとき、「2×1=2」「2×2=4」「2×3=6」……と1マスずつ手で紙に書いて確かめるのは大変ですよね。でも、九九の一覧表があれば、同じ「掛け算ルール」で一気に81マスを確かめられます。パラメータ化テストは、まさにその「一覧表でまとめて確認」に相当します。

6.2 @ParameterizedTest — パラメータ化テストの目印

パラメータ化テストを書くには、@Test の代わりに @ParameterizedTest を使います。「このメソッドはデータを受け取って繰り返し実行されるテストだ」という目印です。

@ParameterizedTest だけではデータが何かわかりません。そこで「どこからデータを受け取るか」を別のアノテーション（@ValueSource など）で一緒に指定します。次のセクションで順番に見ていきましょう。

重要ポイント: @Test との使い分け

1種類のデータだけで確認するテストには @Test を使います。複数のデータパターンで同じロジックを確認したいときに @ParameterizedTest を使います。両者は排他で、1つのメソッドに両方つけることはできません。

ヒント: junit-jupiter-params が必要

@ParameterizedTest は junit-jupiter-params というモジュールに含まれています。Maven の pom.xml で junit-jupiter（集約依存）を指定していれば自動的に含まれます。個別に junit-jupiter-api のみを追加した場合は、junit-jupiter-params を別途追加する必要があります。

6.3 @ValueSource — 1引数のテストに使う

@ValueSource は、テストメソッドに渡したい値を リストとして直接書く 最もシンプルな方法です。テストメソッドは引数を1つ受け取り、その引数に順番に値が渡されて繰り返し実行されます。

対応している型のプロパティは ints・strings・longs・doubles などです。先ほどの「偶数判定」を書き直してみましょう。

EvenNumberTest.java（@ValueSource 版） 
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.ValueSource;
import static org.junit.jupiter.api.Assertions.assertTrue;

// @ParameterizedTest + @ValueSource で4パターンをまとめて確認
@ParameterizedTest
@ValueSource(ints = {0, 2, 4, 100}) // テストに渡す整数値のリスト
void 偶数はisEvenがtrueを返す(int number) {
    assertTrue(isEven(number)); // numberに0→2→4→100が順に渡される
}

たった1つのメソッドで、4パターンを確認できるようになりました。ints の部分を strings に変えれば文字列も扱えます。次の例は文字列が空かどうかをチェックしています。

StringTest.java（strings の例） 
@ParameterizedTest
@ValueSource(strings = {"hello", "world", "JUnit"}) // 文字列のリスト
void 空でない文字列はisBlankがfalseを返す(String text) {
    assertFalse(text.isBlank()); // "hello"→"world"→"JUnit"の順に渡される
}

実行すると — データ単位で結果が出る

✓ 偶数はisEvenがtrueを返す(int) [1] 0

✓ 偶数はisEvenがtrueを返す(int) [2] 2

✓ 偶数はisEvenがtrueを返す(int) [3] 4

✓ 偶数はisEvenがtrueを返す(int) [4] 100

→ Tests run: 4, Failures: 0

データごとに番号が付いて結果が表示されます。もし特定の値だけ失敗しても、どのデータで失敗したか一目でわかります。

6.4 @CsvSource — 入力と期待値をまとめて渡す

@ValueSource は引数1つしか渡せません。「入力値と期待値の両方をまとめてテストしたい」ときに使うのが @CsvSource です。CSV（カンマ区切り）の文字列で複数の引数を一度に渡せます。

MultiplyTest.java（@CsvSource で掛け算テスト） 
import org.junit.jupiter.params.provider.CsvSource;

// "a, b, 期待値" をカンマ区切りで列挙する
@ParameterizedTest
@CsvSource({
    "2, 3, 6",    // 2×3=6
    "3, 3, 9",    // 3×3=9
    "5, 4, 20",   // 5×4=20
    "0, 7, 0"     // 0×7=0
})
void 掛け算が正しく計算できる(int a, int b, int expected) {
    assertEquals(expected, multiply(a, b)); // 期待値と実際の計算結果を比較
}

各行が1つのテストケースです。行を追加するだけで新しいパターンを増やせます。"2, 3, 6" のように1つの文字列でカンマ区切りにし、メソッドの引数（int a, int b, int expected）と左から対応します。

改善前・改善後を比べると、効果がよく分かります。

文字列を含む場合の書き方

文字列に空白やカンマを含む場合は、シングルクォートで囲みます。例: "'hello world', 11" のように書くと、第1引数は "hello world"（スペース込み）として渡されます。

6.5 @MethodSource — メソッドからデータを供給する

@CsvSource は手軽ですが、テストデータが複雑になったり量が多くなると読みにくくなります。そんなときに使えるのが @MethodSource です。データを供給するメソッドを別に定義して、そのメソッド名を参照する形式です。

AddTest.java（@MethodSource の例） 
import java.util.stream.Stream;
import org.junit.jupiter.params.provider.Arguments;
import org.junit.jupiter.params.provider.MethodSource;

// データ供給メソッド（static メソッドで Stream を返す）
static Stream<Arguments> 足し算データ() {
    return Stream.of(
        Arguments.of(1, 2, 3),   // 1+2=3
        Arguments.of(0, 5, 5)    // 0+5=5
    );
}

@ParameterizedTest
@MethodSource("足し算データ") // 上のメソッド名を文字列で指定
void 足し算が正しく計算できる(int a, int b, int expected) {
    assertEquals(expected, add(a, b));
}

@MethodSource はデータ量が多い場合や、テストデータを外部ファイルから読み込みたい場合などに向いています。初心者のうちは @ValueSource と @CsvSource で十分です。必要になったときに @MethodSource を思い出してください。

アノテーション 渡せる引数 向いている場面
@ValueSource 1つ（同じ型の値リスト） シンプルな1引数テスト
@CsvSource 複数（カンマ区切り） 入力＋期待値をまとめてテスト（最も多用）
@MethodSource 複数（Stream<Arguments>） データが多い・複雑な場合

6.6 実行結果の見え方 — 5件中5件成功

パラメータ化テストを実行すると、データ1件ごとに独立したテストとして結果が表示されます。IDE（IntelliJ IDEA や Eclipse）では次のようにツリー状に展開して確認できます。

IDE での実行結果（イメージ）

▼ 掛け算が正しく計算できる

✓ [1] 2, 3, 6

✓ [2] 3, 3, 9

✓ [3] 5, 4, 20

✓ [4] 0, 7, 0

→ 4 tests found, 4 passed

もし [3] 5, 4, 20 だけ失敗した場合、その行だけ赤くなり、どの入力で失敗したかがすぐに分かります。

5件のテストが全部成功すれば「5 tests found, 5 passed」と表示されます。逆に、テストデータの中に失敗するものがあれば「5 tests found, 4 passed, 1 failed」と出て、どのデータで失敗したかを教えてくれます。これが普通の @Test との大きな違いです。

重要ポイント: 失敗データが一目でわかる

コピペテストでは「どの入力で落ちたか」をメソッド名から読み解く必要がありました。パラメータ化テストなら、実行結果に入力値が表示されるので、「[3] 5, 4, 20 が失敗」と即座に分かります。デバッグ効率が大幅に上がります。

やってみよう 6-1: 九九の一部をパラメータ化テストで確かめよう

九九の「3の段」を @CsvSource を使ってパラメータ化テストで確かめましょう。コピペ地獄からどれだけすっきりするかを体験してください。

手順

  1. 1 以下の multiply メソッドをテスト対象のクラスに追加する。
Calculator.java（テスト対象メソッド） 
// 掛け算メソッド
public int multiply(int a, int b) {
    return a * b; // a と b を掛けて返す
}
  1. 2 @ParameterizedTest と @CsvSource を使い、「3×1=3」「3×3=9」「3×5=15」「3×9=27」の4パターンをテストする1つのメソッドを書く。
  2. 3 テストを実行し、「4 tests found, 4 passed」となることを確認する。
  3. 4 試しに期待値を1つ間違えて（例: "3, 3, 99"）実行し、失敗時のメッセージを確認してみる。

理解度確認クイズ

  1. Q1. パラメータ化テストを書くとき、@Test の代わりに使うアノテーションは何ですか？

    回答を表示

    @ParameterizedTest です。「このメソッドはデータを受け取って繰り返し実行されるテストだ」という目印です。@Test と @ParameterizedTest は同時に使えません。

  2. Q2. @ValueSource と @CsvSource の使い分けを説明してください。

    回答を表示

    @ValueSource は引数1つ（同じ型の値リスト）のシンプルなテストに使います。@CsvSource は「入力値と期待値」のように複数の引数をまとめてカンマ区切りで渡したいときに使います。

  3. Q3. @CsvSource で5パターン書いたテストを実行すると、テスト件数はいくつになりますか？

    回答を表示

    5件になります。パラメータ化テストではデータ1件ごとに独立したテストとして扱われるため、5パターン書けば「5 tests found」と表示されます。全パターン成功なら「5 passed」となります。
