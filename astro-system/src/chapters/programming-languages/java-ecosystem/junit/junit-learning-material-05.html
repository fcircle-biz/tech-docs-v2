ホーム JUnit 第5章
第5章 所要時間: 約50分

テストの準備と後片付け ライフサイクルを理解しよう

テストを実行する「前後」に自動で動く仕組みを学びます。料理の「材料の下ごしらえ」と「片付け」になぞらえながら、@BeforeEach・@AfterEach・@BeforeAll・@AfterAll の役割と実行順序を理解しましょう。テストが互いに影響し合わない「クリーンな状態」を保つ方法を身につけます。

この章で学ぶこと

  • なぜテストの「準備」と「後片付け」が必要なのかを説明できる
  • @BeforeEach と @AfterEach の役割と使い方を理解する
  • @BeforeAll と @AfterAll の役割と、static が必要な理由を理解する
  • テストライフサイクルの実行順序を図で説明できる

5.1 なぜ「準備」と「後片付け」が必要なの？

前章までで、テストクラスに複数のテストメソッドを書く方法を学びました。ここで一つ問題が出てきます。それは「テスト同士が影響し合う」可能性です。

料理で考えてみましょう。あなたがカレーを作って、次にシチューを作るとします。カレーで使ったお鍋をそのまま使ったら、シチューにカレーの風味が残ってしまいます。もしくは、すでに使った計量スプーンがそのままテーブルに出ていて、次の人が間違って使ってしまうかもしれません。これと同じことが、テストでも起こりえます。

たとえ話: 料理の下ごしらえと片付け

  • ・下ごしらえ（@BeforeEach） = 各料理の前に、まな板をきれいに洗って材料を用意する
  • ・片付け（@AfterEach） = 各料理の後に、使った道具を洗って元の場所に戻す
  • ・キッチン全体の準備（@BeforeAll） = 料理を始める前に、ガスを点けてオーブンを予熱する（1回だけ）
  • ・キッチン全体の片付け（@AfterAll） = 全料理が終わったらガスを消してキッチン全体を掃除する（1回だけ）

各テストが「きれいなまな板」から始まることで、前の料理の影響を受けずに正しい味が確認できます。

プログラムのテストでも、各テストを毎回まっさらな状態から始めることが重要です。あるテストが途中でオブジェクトの状態を書き換えてしまったとき、その影響が次のテストに残っていたら、テストの結果が信用できなくなってしまいます。JUnit にはこの問題を解決するための、4つの「ライフサイクルアノテーション」が用意されています。

@BeforeEach

各テストメソッドが実行される直前に毎回呼ばれる。テスト対象オブジェクトの初期化など「下ごしらえ」に使う。

@AfterEach

各テストメソッドが実行された直後に毎回呼ばれる。リソースの解放や後処理など「片付け」に使う。

@BeforeAll

クラス内の全テストが始まる前に1回だけ呼ばれる。DB接続など重い初期化処理を一度だけ行う「キッチン全体の準備」。

@AfterAll

クラス内の全テストが終わった後に1回だけ呼ばれる。DB接続のクローズなど「キッチン全体の片付け」に使う。

テストの実行順序について

JUnit 5 では、テストメソッドの実行順序はデフォルトでは保証されていません。つまり「テスト1が必ずテスト2より先に実行される」とは限りません。だからこそ、各テストが独立して動くよう @BeforeEach でリセットすることが大切なのです。

5.3 @BeforeEach と @AfterEach を使ってみよう

実際にコードで見てみましょう。電卓クラス Calculator のテストを例に、@BeforeEach の使い方を確認します。

まず、@BeforeEach を使わない場合を見てみます。各テストメソッドの中で new Calculator() を繰り返し書いています。

CalculatorTest.java（Before を使わない場合） 
class CalculatorTest {
    @Test
    void 足し算ができる() {
        Calculator calc = new Calculator(); // 毎回書くのが面倒…
        assertEquals(5, calc.add(2, 3));
    }

    @Test
    void 引き算ができる() {
        Calculator calc = new Calculator(); // また書く…
        assertEquals(1, calc.subtract(3, 2));
    }
}

new Calculator() を毎回書くのは繰り返しで、コードが増えるほど冗長になります。@BeforeEach を使うと、この初期化処理を一箇所にまとめられます。

CalculatorTest.java（@BeforeEach を使う場合） 
class CalculatorTest {
    private Calculator calc; // フィールドとして宣言

    @BeforeEach               // 各テストの直前に毎回実行される
    void setUp() {
        calc = new Calculator(); // 毎回フレッシュなインスタンスを用意
    }

    @Test
    void 足し算ができる() {
        assertEquals(5, calc.add(2, 3)); // setUp() 後のきれいな calc を使う
    }

    @Test
    void 引き算ができる() {
        assertEquals(1, calc.subtract(3, 2)); // こちらも毎回新鮮な calc
    }
}

重要ポイント: @BeforeEach のメリット

  • ・初期化コードが1箇所にまとまり、テストが読みやすくなる
  • ・各テストが必ず新しいインスタンスを受け取るので、テスト同士が干渉しない
  • ・初期化ロジックを変更するときは setUp() の1箇所だけ直せばよい

次に、@AfterEach の例を見てみましょう。テスト後に何かリソースを解放したい場合に使います。

CalculatorTest.java（@AfterEach を追加） 
    @AfterEach                // 各テストの直後に毎回実行される
    void tearDown() {
        calc = null;          // 参照を解放して後片付け（GCに任せる）
        // ファイルやDB接続の後片付けもここで行う
    }

シンプルなケースでは @AfterEach は省略できます。ファイルや外部接続など「使ったら必ず閉じなければならないリソース」がある場合に書くと覚えておきましょう。

5.4 @BeforeAll と @AfterAll — クラス全体で1回だけ

@BeforeAll と @AfterAll は、クラス内のすべてのテストを通じて1回だけ実行されます。立ち上げに時間がかかる処理（データベース接続の確立など）に向いています。

DatabaseTest.java（@BeforeAll / @AfterAll の例） 
class DatabaseTest {
    static Connection conn; // static フィールドで共有する

    @BeforeAll              // クラス全体で1回だけ・static メソッド必須
    static void setUpAll() {
        conn = Database.connect(); // 接続は1回だけ確立
    }

    @AfterAll               // クラス全体で1回だけ・static メソッド必須
    static void tearDownAll() {
        conn.close();        // 全テスト終了後に接続を閉じる
    }
}

なぜ @BeforeAll / @AfterAll のメソッドは static でなければならないの？

JUnit 5 は、テストメソッドを実行するたびにテストクラスの新しいインスタンスを生成します。しかし @BeforeAll は「インスタンスが存在する前」から呼ばれる必要があります。そのため、インスタンスなしで呼び出せる static メソッドである必要があるのです。static を付け忘れると JUnit がエラーを報告してくれます。

@BeforeEach vs @BeforeAll の使い分け

観点 @BeforeEach @BeforeAll
実行タイミング 各テストの直前（毎回） クラスの全テスト開始前（1回）
static 修飾子 不要 必須
向いている処理 オブジェクト初期化・テストデータの準備 DB接続・重いリソースの確立
テストの独立性 高い（毎回フレッシュな状態） 中（テスト間で状態が共有される可能性）

迷ったときは @BeforeEach を使うのが基本です。各テストがきれいな状態から始まるので、テスト同士の干渉を防ぎやすくなります。@BeforeAll は「初期化コストが高くてどうしても1回にしたい」場面で検討しましょう。

5.5 4つのアノテーションをまとめて使う

ここまでの内容を1つのテストクラスにまとめた、実行の流れを確認できる例を見てみましょう。

LifecycleDemoTest.java（実行順序を確認する例） 
import org.junit.jupiter.api.*;

class LifecycleDemoTest {
    private Calculator calc;

    @BeforeAll
    static void キッチン準備() { System.out.println("=== BeforeAll: 全体の準備 ==="); }

    @BeforeEach
    void 下ごしらえ() {
        calc = new Calculator();         // 毎回新しい Calculator を用意
        System.out.println("--- BeforeEach: テスト前の準備 ---");
    }

    @Test
    void 足し算テスト() { assertEquals(5, calc.add(2, 3)); }

    @Test
    void 引き算テスト() { assertEquals(1, calc.subtract(3, 2)); }

    @AfterEach
    void 片付け() { System.out.println("--- AfterEach: テスト後の後片付け ---"); }

    @AfterAll
    static void キッチン片付け() { System.out.println("=== AfterAll: 全体の片付け ==="); }
}

このコードを実行すると、コンソールには次のような出力が表示されます。

実行結果（コンソール出力）

=== BeforeAll: 全体の準備 ===
--- BeforeEach: テスト前の準備 ---
--- AfterEach: テスト後の後片付け ---
--- BeforeEach: テスト前の準備 ---
--- AfterEach: テスト後の後片付け ---
=== AfterAll: 全体の片付け ===

BeforeAll は1回だけ、BeforeEach と AfterEach はテスト数（2回）だけ繰り返されていることが確認できます。

やってみよう 5-1: @BeforeEach を使ってテストをリファクタリングしよう

第4章で書いたテストクラスを改修し、各テストメソッドで重複していた初期化コードを @BeforeEach にまとめます。コードをすっきりさせながら、各テストが独立したきれいな状態で始まることを確認しましょう。

手順

  1. 1 既存のテストクラスで、各テストメソッド内の Calculator calc = new Calculator(); を1行ずつ確認する。
  2. 2 クラスのフィールドとして private Calculator calc; を宣言し、@BeforeEach メソッド（void setUp()）の中で calc = new Calculator(); と初期化する。
  3. 3 各テストメソッドから Calculator calc = new Calculator(); を削除し、テストが問題なくすべてパスすることを IDE で確認する。
  4. 4 （発展）@BeforeAll と static キーワードを組み合わせた例を書き、static を外すとどんなエラーが出るかを確かめてみる。

理解度確認クイズ

  1. Q1. @BeforeEach と @BeforeAll の最も大きな違いは何ですか？

    回答を表示

    実行される回数が異なります。@BeforeEach は各テストメソッドの直前に「毎回」実行されますが、@BeforeAll はクラス内の全テストが始まる前に「1回だけ」実行されます。また @BeforeAll のメソッドには static 修飾子が必要です。

  2. Q2. テストが3つあるクラスで @BeforeEach は合計何回実行されますか？

    回答を表示

    3回です。@BeforeEach はテストメソッドの数だけ実行されます。テストメソッドが3つあれば、テストごとに1回ずつ、合計3回実行されます。

  3. Q3. なぜ各テストで @BeforeEach を使ってオブジェクトを作り直すと良いのですか？

    回答を表示

    前のテストがオブジェクトの状態を変えていても、毎回新しいインスタンスを作り直すことで「テスト同士が影響し合わない」独立した状態を保てるからです。テストの実行順序が変わっても結果が変わらない、信頼性の高いテストになります。
