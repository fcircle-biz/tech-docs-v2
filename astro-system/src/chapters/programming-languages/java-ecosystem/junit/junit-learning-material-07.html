ホーム JUnit 第7章
第7章 所要時間: 約50分

エラーもテストしよう 例外テストの書き方

「正しくエラーになるか」を確かめる異常系テストを学びます。0 で割ったときに例外が発生することを確認する例を軸に、assertThrows の使い方から例外メッセージの検証まで、一歩ずつ丁寧に解説します。

この章で学ぶこと

  • 正常系と異常系の違いを説明でき、なぜ異常系テストが必要かを理解する
  • Java の例外（Exception）の基本的な仕組みを知る
  • assertThrows を使って「期待した例外が発生すること」を確かめられる
  • 投げられた例外のメッセージを取り出して追加検証できる
  • assertDoesNotThrow の使い方と使い所を知る

7.1 正常系と異常系——なぜエラーもテストするの？

これまで書いてきたテストは、すべて「うまくいくケース」——つまり正常系（happy path）のテストでした。足し算に正しい数を入れて、正しい答えが返ってくるかを確かめてきましたね。

しかし現実のプログラムには、うまくいかない入力も飛んできます。0 で割ろうとする、空欄に文字を入れる、存在しないファイルを開こうとする——こうした「わざと失敗・エラーになるケース」を異常系（error path）と呼びます。

たとえ話: 自動販売機にニセ硬貨を入れてみる

自動販売機が正しく動くかテストするとき、正しい硬貨を入れて飲み物が出てくるかを確かめるのは当然です（正常系）。でもそれだけでは不十分です。

  • ・ニセ硬貨を入れたとき、ちゃんと弾いてくれるか（異常系テスト）
  • ・お金が足りないとき、エラー表示になるか（異常系テスト）

異常な入力に対して「安全に止まる（エラーを出す）」ことを確かめるのが、異常系テストの目的です。

正常系テスト

正しい入力を与えたとき、期待どおりの結果が返ってくることを確かめる。

例: 10 ÷ 2 = 5 になるか確認する

異常系テスト

不正な入力を与えたとき、期待どおりのエラー（例外）が発生することを確かめる。

例: 10 ÷ 0 のとき ArithmeticException が出るか確認する

「エラーが出るはずなのにエラーが出ない」——これは実は深刻なバグです。たとえば、0 で割ったのに何事もなかったかのように計算が続くと、その後の処理がおかしくなり、気づかないうちに誤った結果がユーザーに届くことになります。「正しくエラーになること」を保証するのが異常系テストの使命です。

7.2 Java の例外（Exception）超入門

異常系テストを学ぶ前に、Java の例外（Exception）の基本を一段落だけ押さえておきましょう。Java では、何かおかしなことが起きたとき、プログラムが「throw（スロー）」と呼ばれる形でエラーを「投げる」仕組みがあります。投げられたエラーのことを例外（Exception）といいます。

例外にはたくさんの種類があります。0 で割ったときに Java が自動で投げる ArithmeticException、メソッドに不正な引数を渡したときに投げる IllegalArgumentException、null のオブジェクトを操作したときに投げる NullPointerException などが代表例です。本章ではこれらの詳細には深入りせず、「例外＝プログラムがエラーを知らせる仕組み」と覚えておけば十分です。

例外が発生するイメージ

flowchart LR A["メソッド呼び出し
（例: 10 ÷ 0）"] --> B{"おかしな状況？"} B -->|正常| C["結果を返す"] B -->|異常| D["例外を throw
（投げる）"] D --> E["ArithmeticException
など"]

よく出てくる例外の種類

  • ArithmeticException: 0 で割るなど算術計算のエラー（Java が自動で投げる）
  • IllegalArgumentException: 不正な引数が渡されたとき（自分でチェックして投げることも多い）
  • NullPointerException: null のオブジェクトを操作しようとしたとき
  • IllegalStateException: オブジェクトが正しくない状態のとき

7.3 assertThrows の使い方

JUnit 5 では、「この処理を実行したら、この例外が投げられるはず」を確かめるために assertThrows を使います。書き方は次のとおりです。

assertThrows の基本構文 
assertThrows(期待する例外クラス, () -> 実行する処理);

ポイントは2つです。第1引数に「どの例外クラスが投げられるはずか」を書きます。第2引数には () -> から始まるラムダ式で「実際に実行する処理」を渡します。ラムダ式とは「小さな処理のかたまりを後で渡せる形にしたもの」で、今は「矢印の後ろに実行したいコードを書く」とだけ覚えておけば大丈夫です。

実際の例を見てみましょう。0 で割ると ArithmeticException が発生することを確かめるテストです。

CalculatorTest.java 
@Test
void ゼロ除算のとき例外が発生する() {
    Calculator calc = new Calculator();

    // 0 で割ったら ArithmeticException が出るはず
    assertThrows(ArithmeticException.class, () -> {
        calc.divide(10, 0); // ← この処理が例外を投げるはず
    });
}

() -> { ... } の中に「例外が投げられるはずの処理」を書きます。JUnit はこの処理を実行し、ArithmeticException が投げられればテスト成功、何も投げられなければテスト失敗と判定します。

assertThrows の判定の仕組み

  • 期待した例外が投げられた → テスト成功（PASS）
  • 例外が投げられなかった → テスト失敗（FAIL）。「例外が出るはずなのに出なかった」＝バグ
  • 別の種類の例外が投げられた → テスト失敗（FAIL）。違う例外クラスは期待と不一致

「例外が出るはずなのに出ない」＝バグ

0 で割っても例外が出ず、プログラムが動き続けてしまったとします。すると、その後の計算で Infinity（無限大）や NaN（非数）が混入し、予測不能な結果が広がります。「正しくエラーで止まること」を保証する異常系テストは、こうした連鎖的な不具合を防ぐ最初の砦です。

7.4 例外メッセージも検証する

「例外の種類（クラス）が合っているか」だけでなく、「例外のメッセージが正しいか」も確かめたい場面があります。たとえば IllegalArgumentException を投げる処理なら、「なぜエラーになるか」を説明するメッセージを含めることが多いです。

assertThrows は、投げられた例外オブジェクトを戻り値として返してくれます。その戻り値を変数に受け取り、getMessage() でメッセージを取り出して検証できます。

CalculatorTest.java — 例外メッセージを検証 
@Test
void ゼロ除算のエラーメッセージを検証する() {
    Calculator calc = new Calculator();

    // 例外オブジェクトを受け取る
    ArithmeticException e = assertThrows(
        ArithmeticException.class,
        () -> calc.divide(10, 0) // 例外が出るはずの処理
    );

    // 例外のメッセージが正しいか確かめる
    assertEquals("0 で割ることはできません", e.getMessage());
}

このように、assertThrows の結果を変数 e に受け取ることで、例外クラスの確認（assertThrows が担当）と例外メッセージの確認（assertEquals が担当）を両方できます。

例外メッセージ検証の流れ

flowchart TD A["assertThrows を呼ぶ"] --> B["ラムダ式の処理を実行"] B --> C["ArithmeticException が throw される"] C --> D["assertThrows がキャッチ
→ 戻り値として返す"] D --> E["変数 e に受け取る"] E --> F["e.getMessage（）を assertEquals で検証"] F --> G["テスト成功"]

メッセージ検証のポイント

例外メッセージはユーザーや開発者へのヒントになるため、正確であることが大切です。「どんな理由でエラーになるかが、メッセージを見ればわかる」状態を保つために、メッセージの内容もテストで固めておくと安心です。

7.5 assertDoesNotThrow — 例外が出ないことを確かめる

assertThrows とは逆に、「この処理では例外が投げられないはず」を確かめたいときは assertDoesNotThrow を使います。

assertDoesNotThrow の使用例 
@Test
void 正常な割り算では例外が出ない() {
    Calculator calc = new Calculator();

    // 正常な入力では例外が投げられないことを確かめる
    assertDoesNotThrow(() -> calc.divide(10, 2));
}

assertDoesNotThrow は、渡した処理が例外なしに完了すれば成功、何らかの例外が投げられれば失敗となります。「正常系でも安全に動くこと」を明示的に示したいときに使います。

メソッド テスト成功の条件 使い所
assertThrows 指定した例外が投げられた 異常系: 「エラーになるべきとき、ちゃんとエラーになるか」を確かめる
assertDoesNotThrow 例外が一切投げられなかった 正常系: 「正常な入力でエラーが出ないこと」を明示したいとき

まとめ: 異常系テストの考え方

  • ・正常系テスト: 正しい入力 → 正しい結果が返る
  • ・異常系テスト: 不正な入力 → 期待した例外が投げられる
  • ・assertThrows（例外クラス, ラムダ式） で例外の種類を確かめる
  • ・戻り値の例外オブジェクトに getMessage() でメッセージを検証できる
  • ・「例外が出るはずなのに出ない」状態はバグ。異常系テストがそれを検知する

やってみよう 7-1: 例外テストを書いて動かす

電卓クラスに「0 で割ったら ArithmeticException を投げる」機能を実装し、その動作を assertThrows で確認するテストを書いてみましょう。

手順

  1. 1 Calculator クラスに divide(int a, int b) メソッドを追加する。b == 0 のとき ArithmeticException を投げ、そうでなければ a / b を返す。
  2. 2 assertThrows(ArithmeticException.class, () -> calc.divide(10, 0)) を使ったテストを書く。
  3. 3 テストを実行して PASS することを確認する。次に divide メソッドから例外処理を削除して再実行し、今度は FAIL になることを観察する（「例外が出るはずなのに出ない」＝テスト失敗が確認できる）。
  4. 4 例外メッセージの検証も追加する。e.getMessage() で取り出したメッセージを assertEquals で確認してみよう。

参考: Calculator.java の divide メソッド

Calculator.java（抜粋） 
public int divide(int a, int b) {
    if (b == 0) {
        // 0 除算なら例外を投げる
        throw new ArithmeticException("0 で割ることはできません");
    }
    return a / b; // 正常な場合は割り算の結果を返す
}

理解度確認クイズ

  1. Q1. 「異常系テスト」とはどのようなテストですか？ 「正常系テスト」との違いも含めて答えてください。

    回答を表示

    正常系テストは「正しい入力を与えたとき、期待した結果が返ってくるか」を確かめるテストです。それに対して異常系テストは「不正な入力や問題のある状況で、期待した例外（エラー）が投げられるか」を確かめるテストです。プログラムが不正な入力に対して安全に止まることを保証します。

  2. Q2. 次のコードは何をテストしているか説明してください。
    assertThrows(IllegalArgumentException.class, () -> service.save(null));

    回答を表示

    service.save(null)（null を引数に渡した場合）を実行したとき、IllegalArgumentException が投げられることを確かめています。null を渡すという不正な入力に対して、サービスが正しくエラーを出すかを検証する異常系テストです。

  3. Q3. assertThrows の戻り値を変数に受け取ると、どんな追加検証ができますか？

    回答を表示

    assertThrows は投げられた例外オブジェクトを返します。その戻り値を変数に受け取ることで、getMessage() を呼び出して例外メッセージが期待どおりかを assertEquals で追加検証できます。例外の種類だけでなく、「なぜエラーになったか」のメッセージも正確かどうかを確かめられます。
