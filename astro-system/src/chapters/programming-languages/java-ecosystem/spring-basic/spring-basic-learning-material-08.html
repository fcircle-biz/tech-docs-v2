総合演習：DIとAOPを使った簡単な処理

DI を使って3つのクラスを連携させ、AOP でログ出力を本来の処理から分離する総合演習に取り組みます。Spring基礎で学んできた DI・Bean・アノテーション・AOP の考え方を一つにまとめる章です。最後に教材全体のまとめと、次の教材（Spring Data JDBC）への橋渡しも確認します。

この章で学ぶこと

  • DI を使ってクラスを連携させる
  • AOP を使ってログ出力を分離する
  • Spring 基礎で学んだ内容を整理する

8.1 演習の概要

この総合演習では、3つのクラスを DI で連携させます。それぞれの役割は次のとおりです。

UserController

入力（ユーザー登録要求）を受け取り、Service を呼び出す役割。@Component を付ける。

UserService

ユーザー登録の業務処理を担当し、Repository を呼び出す役割。@Service を付ける。

UserRepository

データの保存処理を担当。実際の保存は今回はコンソール出力で代用。@Repository を付ける。

3つのクラスの処理の流れを図で確認しましょう。

flowchart LR UC["UserController"] -->|"register（）を呼ぶ"| US["UserService"] US -->|"save（）を呼ぶ"| UR["UserRepository"] UR -->|"コンソールに出力"| OUT["「田中を保存しました」"]

さらに、AOP（アスペクト指向プログラミング）を使って、UserService のメソッド実行前後に自動でログを出力する LoggingAspect クラスも追加します。

8.2 演習の要件

以下の要件に沿って、4つのクラスを作成してください。

要件

  1. 1 UserController からユーザー登録処理（register メソッド）を呼び出す
  2. 2 UserService でユーザー登録の業務処理を行い、UserRepository を呼び出す
  3. 3 UserRepository で「〇〇を保存しました」とコンソールへ表示する
  4. 4 AOP を使って、UserService のメソッド実行の前後に「メソッド開始」「メソッド終了」をログ出力する

8.3 コード例（4クラス）

これまでの章（第3章・第5章・第7章）で学んだ内容を統合したコードです。各クラスを順番に確認しましょう。

① UserController — 入力を受け取り、Service を呼び出す

UserController.java 
package com.example.demo.controller;

import com.example.demo.service.UserService;
import org.springframework.stereotype.Component;

// Spring に「このクラスを管理してね」と伝えるアノテーション
@Component
public class UserController {

    private final UserService userService;

    // コンストラクタで UserService を受け取る（DI）
    public UserController(UserService userService) {
        this.userService = userService;
    }

    // ユーザー登録処理を Service に委譲する
    public void register() {
        userService.register("田中");
    }
}

② UserService — 業務処理を担当し、Repository を呼び出す

UserService.java 
package com.example.demo.service;

import com.example.demo.repository.UserRepository;
import org.springframework.stereotype.Service;

// 業務処理クラスに付けるアノテーション
@Service
public class UserService {

    private final UserRepository userRepository;

    // コンストラクタで UserRepository を受け取る（DI）
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    // 登録処理を Repository に委譲する
    public void register(String name) {
        userRepository.save(name);
    }
}

③ UserRepository — データ保存を担当（今回はコンソール出力で代用）

UserRepository.java 
package com.example.demo.repository;

import org.springframework.stereotype.Repository;

// データアクセス処理クラスに付けるアノテーション
@Repository
public class UserRepository {

    // 保存処理（今回はコンソールへの出力で代用）
    public void save(String name) {
        System.out.println(name + "を保存しました");
    }
}

④ LoggingAspect — AOP でログを分離する

LoggingAspect.java 
package com.example.demo.aspect;

import org.aspectj.lang.annotation.After;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.annotation.Before;
import org.springframework.stereotype.Component;

// アスペクトクラスに付けるアノテーション（@Component も必要）
@Aspect
@Component
public class LoggingAspect {

    // service パッケージ内の全メソッド実行前に動く
    @Before("execution(* com.example.demo.service.*.*(..))")
    public void logBefore() {
        System.out.println("メソッド開始");
    }

    // service パッケージ内の全メソッド実行後に動く
    @After("execution(* com.example.demo.service.*.*(..))")
    public void logAfter() {
        System.out.println("メソッド終了");
    }
}

ポイント: execution 式の読み方

execution(* com.example.demo.service.*.*(..)) は「com.example.demo.service パッケージ内の、どのクラスの、どのメソッドでも、どんな引数でも」という意味です。* がワイルドカード、(..) が「引数は何でもよい」を表します。

8.4 期待する出力

アプリを起動して UserController#register() を呼ぶと、コンソールに次の3行が出力されます。

コンソール出力の確認

コンソール出力 
メソッド開始
田中を保存しました
メソッド終了

この順番になる理由

  1. @Before アドバイスが先に動き「メソッド開始」を出力
  2. 本来の save() 処理が動き「田中を保存しました」を出力
  3. @After アドバイスが最後に動き「メソッド終了」を出力

ポイント: UserService や UserRepository のコードには、ログ出力の記述が一切ありません。にもかかわらず「メソッド開始」「メソッド終了」が自動で出力されるのは、AOP（LoggingAspect）が割り込んでいるからです。これが AOP の「処理を分離する」メリットそのものです。

8.5 まとめ — Spring基礎で学んだこと

この教材「Spring基礎」で学んだ内容を振り返りましょう。

Spring基礎 重要ポイントまとめ

  • Spring はオブジェクトを管理してくれる — 部品の生成・受け渡しを Spring が肩代わりする
  • DI により、必要な部品を Spring から受け取れる — クラス自身が new する必要がなくなる
  • Bean は Spring が管理するオブジェクト — Spring Container が生成・保持・注入する
  • アノテーションでクラスの役割を Spring に伝える — @Component・@Service・@Repository など
  • AOP により、ログなどの共通処理を分離できる — 本来の処理がすっきりし、共通処理の重複を防げる
  • 次の教材では Spring Data JDBC を学ぶ — 実際のデータベース操作へと進む

総合確認問題

  1. Q1. Spring がオブジェクトを管理するメリットは何ですか？

    回答を表示

    部品の生成・受け渡しを Spring が肩代わりするため、開発者は本来の処理（業務ロジック）に集中できます。クラス間の結合が緩くなり、変更やテストもしやすくなります。

  2. Q2. DI を使うと、なぜテストや変更がしやすくなりますか？

    回答を表示

    クラスが相手を自分で new せず、外部から受け取るため、テスト用の部品（モック）や別の実装への差し替えが簡単です。クラス間の結合が緩いため、一方を変更しても他方に影響しにくくなります。

  3. Q3. Bean とは何ですか？

    回答を表示

    Spring が管理するオブジェクトです。@Component や @Service などのアノテーションを付けると、Spring Container が生成・管理し、必要な場所へ自動で注入してくれます。

  4. Q4. Controller・Service・Repository はそれぞれ何を担当しますか？

    回答を表示

    Controller（@Component）= 入力を受け取り Service を呼び出す役割。Service（@Service）= 業務処理を担当。Repository（@Repository）= データアクセス処理を担当。

  5. Q5. AOP を使うメリットは何ですか？

    回答を表示

    ログ出力などの共通処理を本来の処理から分離できます。各クラスに同じログコードを書く必要がなくなり重複が減ります。また、本来の処理（業務ロジック）が読みやすく、保守しやすくなります。

  6. Q6. Spring Data JDBC に進む前に、なぜ DI を理解しておく必要がありますか？

    回答を表示

    Spring Data JDBC でも Repository などの部品を DI で受け取って使うためです。DI の仕組みを理解していないと、Spring がどのように部品を連携させているか読めず、設定や構成の意味が分からなくなります。
