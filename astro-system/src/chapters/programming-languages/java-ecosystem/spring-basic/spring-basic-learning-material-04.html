Spring Beanとは

Spring が管理するオブジェクトを Bean（ビーン） と呼びます。この章では、Bean がどのようなものかを理解し、@Component・@Service・@Repository などのアノテーションをクラスに付けることで「Spring に管理してほしい」と伝える仕組みを、初心者にもわかりやすく学びます。第3章で学んだ DI とも深く結びついた、Spring の中心概念です。

この章で学ぶこと

  • Spring Bean の意味を理解する
  • Spring が管理するオブジェクトの考え方を理解する
  • 代表的なアノテーションの役割を理解する

4.1 Bean とは

Spring アプリケーションの中で、Spring が作成し管理するオブジェクトのことを「Bean（ビーン）」と呼びます。

この章でいちばん大切な一文

Bean は Spring が管理するオブジェクトである。

第3章で学んだ DI（Dependency Injection）では、「必要なオブジェクトを Spring が用意して渡してくれる」という仕組みを学びました。このとき Spring が用意して渡してくれていたオブジェクトこそが、まさに Bean です。

第3章で学んだこと（DI）

Spring がオブジェクトを用意して、必要なクラスへ注入（渡）してくれる仕組み。

「何を」注入してくれるのか？

この章で学ぶこと（Bean）

DI で注入されるオブジェクトが「Bean」。Spring が管理するオブジェクトに名前が付いたイメージ。

DI で渡されていたのが Bean！

普通の Java では、オブジェクトを使いたいときは自分で new して作ります。Spring を使う場合、あらかじめ「このクラスを Bean にする」と伝えておくと、Spring がそのオブジェクトを作成・管理し、必要な場所へ自動で渡してくれます。開発者が new を書かなくてよくなるのが、Bean の大きなメリットです。

4.2 アノテーションで役割を伝える

では、どうやって Spring に「このクラスを Bean にしてほしい」と伝えるのでしょうか？　そのための方法が アノテーション（annotation）です。

クラスの上に @Component や @Service などのアノテーションを付けるだけで、「Spring よ、このクラスを管理してください（Bean にしてください）」と伝えることができます。

GreetingService.java 
import org.springframework.stereotype.Service;

// @Service を付けるだけで Spring の管理対象（Bean）になる
@Service
public class GreetingService {

    // このメソッドはどこからでも DI で受け取って使える
    public String greet(String name) {
        return "こんにちは、" + name + "さん！";
    }
}

アノテーションとは？

アノテーションとは、クラスやメソッドに付ける「メモ書き（注釈）」のようなものです。@ で始まる記号として書きます。Spring はこのメモを見て、「このクラスは Bean にしよう」などの処理を自動で行ってくれます。

Spring には、Bean を宣言するためのアノテーションがいくつか用意されています。代表的なものを次の表にまとめます。

アノテーション 用途
@Component 汎用的な部品
@Service 業務ロジックを担当するクラス
@Repository データベースアクセスを担当するクラス
@Controller 画面遷移を扱うクラス
@RestController REST API を扱うクラス

4.3 役割ごとのアノテーションの使い分け

アノテーションの使い分けについて、もう少し詳しく見ていきましょう。重要なポイントを以下にまとめます。

  • Bean は Spring が管理するオブジェクト
  • アノテーションによってクラスの役割を示す
  • @Service は業務処理を担当するクラスに使う
  • @Repository はデータアクセスを担当するクラスに使う
  • @Controller や @RestController は Spring MVC で詳しく扱う（本教材では深入りしない）

覚えておこう

@Component が基本で、@Service・@Repository・@Controller はどれも「@Component に役割名という意味を加えた仲間」です。動作はほぼ同じですが、役割が名前でわかるようにアノテーションを使い分けることで、コードが読みやすくなります。

図にすると、次のような関係になります。

flowchart TD A["@Component（汎用部品・基本）"] A --> B["@Service
（業務ロジック担当）"] A --> C["@Repository
（データアクセス担当）"] A --> D["@Controller
（画面遷移担当）"]

どのアノテーションを選べばいい？

迷ったときは 役割で選ぶ のが基本です。業務処理なら @Service、データベースアクセスなら @Repository。どれにも当てはまらない汎用的な部品なら @Component を使いましょう。

理解度確認クイズ

  1. Q1. Spring Bean とは何ですか？

    回答を表示

    Spring が管理するオブジェクトのことです。Spring がそのオブジェクトを作成し、必要な場所へ自動で渡してくれます。第3章で学んだ DI によって注入される「部品」がまさに Bean です。

  2. Q2. @Component は何を表しますか？

    回答を表示

    そのクラスを Spring の管理対象（Bean）にすることを表す汎用アノテーションです。@Service・@Repository・@Controller はすべて @Component に役割名を加えた仲間です。

  3. Q3. @Service と @Repository の違いは何ですか？

    回答を表示

    どちらもクラスを Bean にする点は同じです。ただし、@Service は業務ロジックを担当するクラスに使い、@Repository はデータベースアクセスを担当するクラスに使います。アノテーションの名前によって、クラスの役割が明確になります。
