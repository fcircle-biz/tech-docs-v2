テーブル作成と初期データ

前章では application.properties でデータベースへの接続設定を行いました。接続できる状態になったら、次は「どんなテーブル（表）を用意するか」を定義し、そこへ最初のデータを入れる必要があります。この章では schema.sql と data.sql の2つのファイルを使って、テーブルの作成と初期データの登録を行う方法を学びます。

この章で学ぶこと

  • schema.sql の役割を理解する
  • data.sql の役割を理解する
  • users テーブルの構成を理解する

3.1 schema.sql と data.sql の役割

Spring Boot には、アプリ起動時に SQL を自動実行してくれる便利な仕組みがあります。src/main/resources フォルダに決まった名前のファイルを置くだけで、Spring Boot が起動のたびにその SQL を実行してくれます。

使うファイルは2つです。それぞれ役割がはっきり分かれています。

ファイル名 役割 書く SQL の例
schema.sql テーブルの構造を定義する（箱を作る） CREATE TABLE など
data.sql テーブルへ初期データを登録する（箱の中身を入れる） INSERT INTO など

前章で設定した spring.sql.init.mode=always が、この自動実行を有効にしています。この設定がないと起動時に SQL が実行されないため、前章での設定が活きてきます。

重要ポイント

schema.sql ＝ 構造（テーブル定義）、data.sql ＝ 中身（初期データ）。2つのファイルの役割を明確に分けて覚えましょう。「まず箱を作り、それから中身を入れる」という順番でSpring Bootが実行します。

3.2 ファイル構成

この章で扱う2つのファイルは、前章の application.properties と同じ場所（src/main/resources）に置きます。

ファイル構成 

src
 └─ main
     └─ resources
         ├─ application.properties   ← 前章で作成（接続設定）
         ├─ schema.sql               ← この章で作成（テーブル定義）
         └─ data.sql                 ← この章で作成（初期データ）

src/main/resources は、Java のソースコードとは別に、設定ファイルやリソースファイルを置くための専用フォルダです。Spring Boot はここを自動的に探索して schema.sql や data.sql を見つけてくれます。

3.3 schema.sql でテーブルを作る

schema.sql には、テーブルを作るための SQL を書きます。今回は users（ユーザー情報）テーブルを1つ作ります。

src/main/resources/schema.sql 
-- users テーブルを作成する
CREATE TABLE users (
    id    BIGINT AUTO_INCREMENT PRIMARY KEY, -- 主キー（自動採番）
    name  VARCHAR(100) NOT NULL,             -- ユーザー名（最大100文字、必須）
    email VARCHAR(255) NOT NULL              -- メールアドレス（最大255文字、必須）
);

各列（カラム）の意味を順番に見ていきましょう。

  • id — 各ユーザーを識別するための番号です。AUTO_INCREMENT により、データを登録するたびに自動で 1, 2, 3… と採番されます。PRIMARY KEY（主キー）を付けることで、重複のない唯一の値になります。
  • name — ユーザー名を格納する列です。VARCHAR(100) は「最大100文字の文字列」という型を意味します。NOT NULL を付けることで、空のまま（NULL）にはできません（必須項目）。
  • email — メールアドレスを格納する列です。メールアドレスは長くなる場合があるため、VARCHAR(255) と少し大きめに確保しています。こちらも NOT NULL で必須項目です。

用語: PRIMARY KEY（主キー）

PRIMARY KEY（主キー）とは、テーブルの各行を1つに識別できる列のことです。主キーには同じ値が2つ入ることができず（重複禁止）、空（NULL）にすることもできません。Excelの表で「行番号」があるように、データベースの各行には主キーで一意の番号が振られます。id 列を主キーにするのは、この教材でよく使うパターンです。

3.4 data.sql で初期データを入れる

テーブルの構造（箱）ができたら、次は初期データ（最初から入っているデータ）を登録します。data.sql に INSERT INTO 文を書くと、アプリ起動時に自動でデータが追加されます。

src/main/resources/data.sql 
-- 初期ユーザーデータを2件登録する
-- ※ id は AUTO_INCREMENT なので指定しない（自動で 1, 2 と振られる）
INSERT INTO users (name, email) VALUES ('田中太郎', 'tanaka@example.com');
INSERT INTO users (name, email) VALUES ('佐藤花子', 'sato@example.com');

ポイントを整理します。

  • INSERT INTO users (name, email) VALUES (...) は「users テーブルの name と email 列に値を登録する」という意味です。
  • id は書かないのがポイントです。AUTO_INCREMENT を設定しているため、データベースが自動で採番します。最初の行は id = 1、次は id = 2 になります。
  • 2つの INSERT 文で、田中太郎さんと佐藤花子さんの2件のユーザーが登録されます。

3.5 起動時の流れ

schema.sql と data.sql がどのように実行されるか、起動時の流れを図で確認しましょう。Spring Boot はこの順番を自動的に守ってくれるため、「テーブルが存在しないのにデータを入れようとして失敗する」といった問題は起きません。

flowchart TD A["アプリ起動"] --> B["schema.sql 実行
（usersテーブル作成）"] B --> C["data.sql 実行
（初期データ登録）"] C --> D["DB準備完了
（2件のユーザーが入った状態）"]

起動が完了した時点で、データベースには users テーブルが存在し、2件のユーザーデータが入った状態になります。この状態から、第5章以降で Repository を使ってデータの取得・追加・更新・削除を行っていきます。

3.6 users テーブルの構成まとめ

この章で作成する users テーブルの列構成を表にまとめます。第4章でエンティティクラスを作るときにもこの構成を参照します。

列名 データ型 内容・特徴
id BIGINT 主キー・AUTO_INCREMENT（自動採番）
name VARCHAR(100) ユーザー名（最大100文字、必須）
email VARCHAR(255) メールアドレス（最大255文字、必須）

用語: データ型（BIGINT / VARCHAR）

BIGINT は整数を格納するための型で、非常に大きな数まで扱えます。ID のような連番の数値によく使われます。
VARCHAR(n) は「最大 n 文字の可変長文字列」を格納するための型です。実際に入力された文字数分だけ容量を使うため、無駄なくデータを保存できます。

理解度確認クイズ

  1. Q1. schema.sql には何を書きますか？

    回答を表示

    テーブル作成用のSQL（CREATE TABLE 等）を書きます。テーブルの構造（定義）を記述するファイルです。どんな列を持つか、各列のデータ型や制約は何か、といった「箱の設計」を書きます。

  2. Q2. data.sql には何を書きますか？

    回答を表示

    初期データ登録用のSQL（INSERT INTO 等）を書きます。アプリ起動時に最初から入っているデータ（初期データ）を定義するファイルです。

  3. Q3. PRIMARY KEY とは何ですか？

    回答を表示

    各行を一意に識別する主キーのことです。同じ値を2つ入れることができず（重複禁止）、空（NULL）にすることもできません。テーブルの各行を区別するための「番号」の役割を果たします。
