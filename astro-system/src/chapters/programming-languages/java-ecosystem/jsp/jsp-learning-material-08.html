ServletとJSPの連携 - MVCパターン入門

この章では、Servlet と JSP がどのように協力して1つの画面を作るかを学びます。「フォワード」と「リダイレクト」という2つの転送方式を理解し、役割分担の設計思想である MVC パターンを習得することで、保守しやすく拡張しやすいアプリの作り方が身につきます。

この章で学ぶこと

  • フォワード（サーバー内転送）とリダイレクト（クライアント再リクエスト）を使い分けられる
  • MVC パターン（Model／View／Controller）の概念と役割分担を理解する
  • Servlet（Controller）と JSP（View）を連携させた実装ができる
  • RequestDispatcher の forward() と include() を使いこなせる

8.1 フォワードとリダイレクト

Servlet がリクエストを処理した後、次の画面へ移動するには2つの方法があります。フォワードとリダイレクトです。どちらも「次のページへ移る」操作ですが、仕組みがまったく異なります。この違いを正確に理解することが、本章の第一歩です。

フォワードの仕組み

フォワードは「サーバーの中だけで処理を引き渡す」方式です。ブラウザはリクエストを1回送るだけで、サーバーが内部でServletからJSPへ処理を渡し、最終的な画面をブラウザへ返します。ブラウザは途中でServletが動いたことを知りません。そのためURLはServletのURL（転送前のURL）のままです。また、リクエスト属性（setAttributeで設定した値）をJSPへ引き継ぐことができます。

【フォワード】ブラウザから見るとリクエストは1往復

sequenceDiagram participant B as ブラウザ participant S as Servlet participant J as JSP B->>S: ① リクエスト Note over S: リクエスト属性をsetAttribute S->>J: ② forward（サーバー内転送） J-->>B: ③ レスポンス（HTML） Note over B: URLはServletのまま変わらない

リダイレクトの仕組み

リダイレクトは「ブラウザに別のURLへアクセスし直すよう指示する」方式です。サーバーはまず「302 Found」というレスポンスを返し、Location ヘッダーに移動先URLを教えます。ブラウザはそのURLへ自動的に再度リクエストを送ります。このためURLが新しいURL（リダイレクト先）に変わります。また、最初のリクエストの属性は引き継げません（ブラウザが別のリクエストとして送るため）。

【リダイレクト】ブラウザとサーバーが2往復する

sequenceDiagram participant B as ブラウザ participant S as Servlet participant T as 遷移先ページ B->>S: ① リクエスト S-->>B: ② 302 Found（Location: /next） B->>T: ③ 新しいURLへ再リクエスト T-->>B: ④ レスポンス（HTML） Note over B: URLが遷移先のURLに変わる

フォワードとリダイレクトの比較

比較項目 フォワード リダイレクト
URLの変化 変わらない（転送前のURL） 変わる（新しいURL）
リクエスト属性 JSPへ引き継げる 引き継げない（別リクエスト）
通信回数 1往復（サーバー内で完結） 2往復（ブラウザが再リクエスト）
主な用途 結果を表示するJSPへの転送 POST後の画面遷移（PRGパターン）

PRGパターン（Post-Redirect-Get）

フォームのPOST送信後にそのままフォワードすると、ブラウザの「更新ボタン」を押したとき同じPOSTが再送されてしまい、二重送信が起きることがあります。これを防ぐ定番パターンが PRG（Post-Redirect-Get）パターンです。

POST処理が終わったら、リダイレクトで結果表示ページ（GETリクエスト）へ誘導します。こうすると更新ボタンを押してもGETリクエストが再送されるだけなので、二重送信が起きません。

使い分けの基準

「データを渡してJSPで表示させたいだけ」→ フォワード。「POSTの後で画面遷移させたい・URLを変えたい・ブックマークさせたい」→ リダイレクト。迷ったらこの基準で判断してください。

8.2 RequestDispatcher

フォワードを実現するのが RequestDispatcher（リクエストディスパッチャー）です。Servlet の中で JSP や他の Servlet へ処理を引き渡す役割を担います。request.getRequestDispatcher("パス") で取得し、forward() または include() を呼び出します。

forward() メソッド

forward() は「処理を完全に JSP へ引き渡す」メソッドです。呼び出した後は Servlet では何も出力してはいけません。forward 後に response.getWriter().println() などを呼ぶと、後述の IllegalStateException が発生します。

SearchServlet.java — forward の基本 
// 1. リクエスト属性にデータをセット
request.setAttribute("resultList", list);

// 2. RequestDispatcherを取得してforwardする
RequestDispatcher dispatcher =
    request.getRequestDispatcher("/WEB-INF/views/result.jsp");
dispatcher.forward(request, response); // JSPへ処理を引き渡す

// ※ forward()の後には何も書かない！

include() メソッド

include() は「他のリソースの出力を現在のレスポンスに取り込む」メソッドです。ヘッダーやフッターなど、複数ページで共通する部品をJSPに埋め込むときに使います。forward と違い、include の後も Servlet（または呼び出し元のJSP）で処理を続けられます。

page.jsp — include の例（共通ヘッダー埋め込み） 
// ヘッダー部品を取り込む（include後も処理は続く）
RequestDispatcher header =
    request.getRequestDispatcher("/WEB-INF/parts/header.jsp");
header.include(request, response);

// この後も出力できる
out.println("<main>メインコンテンツ</main>");

パスの指定方法

パスの種類 書き方 基点
絶対パス（推奨） /WEB-INF/views/result.jsp コンテキストルート（アプリのルート）から
相対パス result.jsp 現在のServletのURLを基点とする（混乱しやすい）

JSPは WEB-INF/views/ に置こう

JSP ファイルを WEB-INF/views/ の下に置くと、ブラウザから直接アクセスできなくなります。必ず Servlet（Controller）を経由させる設計を強制できるため、MVCパターンを徹底できます。

8.3 MVCパターンの基礎

Web アプリが複雑になると、「ロジック」「データ」「画面」が入り混じったコードは読みにくく、修正が怖くなります。そこで登場するのが MVC パターンです。MVC は Model（モデル）・View（ビュー）・Controller（コントローラー）の3つの役割に処理を分ける設計思想です。

3つの役割

Model（モデル）

ビジネスロジック・データ処理を担当。データベースアクセスや計算処理など、アプリの「頭脳」部分。

例: 検索ロジック、DBアクセス

View（ビュー）

ユーザーへの表示を担当。HTML を生成してブラウザへ返す「顔」の部分。ロジックは書かない。

例: JSPファイル

Controller（コントローラー）

リクエストを受け取り、Model に処理を指示し、View へ転送する「指揮者」部分。

例: Servletクラス

MVCの処理フロー

リクエストからレスポンスまでの処理の流れ

（ユーザー）"] -->|"① リクエスト"| C["Controller
（Servlet）"] C -->|"② 処理依頼"| M["Model
（ビジネスロジック）"] M -->|"③ データ返却"| C C -->|"④ setAttribute
＋ forward"| V["View
（JSP）"] V -->|"⑤ HTML レスポンス"| B

なぜMVCが重要なのか

MVC パターンを守ることで、次のようなメリットが得られます。

メリット 理由
保守性の向上 「画面だけ直したい」「ロジックだけ変えたい」が独立してできる
テストのしやすさ Model のロジックを画面なしで単体テストできる
役割分担の明確化 「どこに何を書くか」が決まり、複数人での開発もしやすい

8.4 Servlet→JSPの連携実装

実際に Servlet（Controller）と JSP（View）を連携させる実装を見ていきましょう。典型的なパターンは「リクエスト受信 → Controller が Model を呼ぶ → 結果を属性にセット → JSP へフォワード → JSP が属性を取り出して表示」という流れです。

① Controller（Servlet）の実装

Servlet ではビジネスロジック（ここでは簡単な検索処理）を行い、結果を request.setAttribute() でリクエスト属性にセットしてから JSP へフォワードします。

SearchController.java 
@WebServlet("/search")
public class SearchController extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {
        // ① パラメータ取得
        String keyword = request.getParameter("keyword");
        // ② Modelに処理を委譲（例: 商品リストを取得）
        List<String> results = ItemModel.search(keyword);
        // ③ 結果をリクエスト属性にセット
        request.setAttribute("results", results);
        request.setAttribute("keyword", keyword);
        // ④ ViewのJSPへフォワード
        request.getRequestDispatcher(
            "/WEB-INF/views/searchResult.jsp")
            .forward(request, response);
    }
}

② View（JSP）の実装

JSP では request.getAttribute() でリクエスト属性を取り出して表示します。JSP に複雑なロジックを書かないのが MVC の鉄則です。

WEB-INF/views/searchResult.jsp（抜粋） 
<%-- リクエスト属性を取り出す --%>
<%
    String keyword = (String) request.getAttribute("keyword");
    List<String> results =
        (List<String>) request.getAttribute("results");
%>
<h1><%= keyword %> の検索結果</h1>
<ul>
<% for (String item : results) { %>
    <li><%= item %></li>
<% } %>
</ul>

処理の全体フロー（まとめ）

flowchart TD A["① ブラウザが /search?keyword=Java にリクエスト"] --> B B["② SearchController.doGet が実行される"] --> C C["③ ItemModel.search で検索処理"] --> D D["④ request.setAttribute で結果をセット"] --> E E["⑤ RequestDispatcher.forward で searchResult.jsp へ転送"] --> F F["⑥ JSP が request.getAttribute で結果を取り出して HTML 生成"] --> G G["⑦ ブラウザへ HTML レスポンス（URLは /search のまま）"]

setAttribute と getAttribute の対応

Servlet で request.setAttribute("results", list) と書いたら、JSP では同じキー名 request.getAttribute("results") で取り出します。キー名のスペルミスが最初のつまずきポイントです。

8.5 つまずきやすいポイント

Servlet と JSP を連携させるとき、初学者がよく遭遇するエラーと対処法をまとめました。

JSPでデータが表示されない（null になる）

原因は Servlet 側の request.setAttribute() の呼び出し漏れ、またはキー名のスペルミスです。

  • Servlet に setAttribute を書いたか確認する
  • JSP の getAttribute のキー名が Servlet 側と完全に一致しているか確認する

IllegalStateException が発生する

原因は forward() または sendRedirect() の後にレスポンスへ書き込もうとしたことです。一度レスポンスが確定した後に再度出力することは許可されていません。

  • forward() の後にコードを書かない（return で終了させる）
  • 条件分岐内で forward と別の出力が競合していないか確認する

URLが変わらない / 変わって困る

フォワードとリダイレクトの混同です。

  • 「URLを変えたくない・属性を引き継ぎたい」→ フォワードを使う
  • 「URLを変えたい・POST後に安全に画面遷移したい」→ リダイレクトを使う

やってみよう 8-1: 検索処理をMVCパターンで実装する

キーワードを受け取って結果を表示する検索処理を、MVC パターンで実装します。

手順

  1. 1 SearchController Servlet を作成し、@WebServlet("/search") を付ける。doGet でキーワードを受け取り、リスト（仮のデータで可）を setAttribute でセットする。
  2. 2 WEB-INF/views/searchResult.jsp を作成し、getAttribute でリストを取り出して <ul> で一覧表示する。
  3. 3 ブラウザで http://localhost:8080/アプリ名/search?keyword=Java にアクセスし、結果が表示されることを確認する。

やってみよう 8-2: ログイン処理のMVC化

第7章で実装したログイン処理を、MVC パターンへリファクタリングします。

  1. 1 ログインのバリデーションロジックを UserModel クラスに分離し、Servlet から呼び出す構造にする。
  2. 2 ログイン成功時はリダイレクト（PRGパターン）でトップページへ遷移し、失敗時はエラーメッセージを setAttribute でセットしてフォワードする。
  3. 3 ログインフォームの JSP はエラーメッセージ属性を取り出し、メッセージがあれば赤字で表示する。

やってみよう 8-3: 一覧→詳細画面の遷移をMVCで実装する

商品一覧から商品IDを受け取り、詳細画面を表示する遷移をMVC構造で実装します。

  1. 1 一覧JSPに各商品へのリンク（/detail?id=1 など）を設ける。
  2. 2 DetailController Servlet で ID を受け取り、Model から詳細データを取得して setAttribute でセット後、詳細JSPへフォワードする。
  3. 3 詳細JSPで商品の名前・価格・説明を表示する。一覧画面へ戻るリンクも設ける。

理解度確認クイズ

  1. Q1. フォワードとリダイレクトの違いを、「URLの変化」「リクエスト属性の引き継ぎ」の2点から説明してください。

    回答を表示

    フォワードはサーバー内で処理を転送するためURLは変わらず、同じリクエストオブジェクトが引き継がれるためリクエスト属性（setAttribute の値）もJSPへ渡せます。リダイレクトはブラウザへ別URLへの再アクセスを指示するためURLが変わり、新しいリクエストとなるため属性は引き継げません。

  2. Q2. MVCパターンにおけるModel・View・Controllerの役割を、Servlet/JSP開発の文脈で説明してください。

    回答を表示

    Model はビジネスロジックやデータアクセスを担うJavaクラス（DBへの検索など）、View は画面を生成するJSPファイル（HTMLの出力）、Controller はリクエストを受け取りModelに処理を指示してViewへフォワードするServletクラス、がそれぞれの役割です。

  3. Q3. フォーム送信（POST）後にリダイレクトを使う「PRGパターン」は、なぜ必要ですか？

    回答を表示

    POST後にそのままフォワードすると、ブラウザの更新ボタンを押したとき同じPOSTリクエストが再送され、二重送信が起きることがあります。POST後にリダイレクトでGETリクエストの結果表示ページへ誘導することで、更新しても安全なGETリクエストが再送されるだけになり、二重送信を防止できます。
