この章で学ぶこと

  • JDBCを使ってデータベースに接続し、try-with-resourcesで安全にリソースを管理できる

11.1 JDBCの基礎

JDBC（Java Database Connectivity）は、Javaプログラムからデータベースに接続するための標準APIです。「JDBC＝JavaとDBをつなぐ橋」と覚えてください。MySQL・PostgreSQL・H2などデータベースの種類を問わず、同じJDBC APIでアクセスできるのが大きな特徴です。

JDBCドライバの設定

JDBCを使うには、JDBCドライバ（.jarファイル）を用意する必要があります。JDBCドライバとは、各データベースメーカーが提供する「DBとJavaをつなぐ変換機」です。Mavenを使う場合はpom.xmlに依存を追加、使わない場合はWEB-INF/lib/ に .jar を配置します。

JDBCドライバの配置場所 

myapp/
└── WEB-INF/
    └── lib/
        └── mysql-connector-j-8.x.x.jar  ← ここに配置するだけ

JDBCは java.sql.* — 名前空間は変わらない

Jakarta EE では多くのパッケージが javax.* から jakarta.* に変わりましたが、JDBCは java.sql.* のままです。JDK 標準ライブラリに含まれており、名前空間は変更されていません。import 文は import java.sql.*; と書きます。

データベース接続とtry-with-resources

JDBCでDBに接続するには DriverManager.getConnection(url, user, password) を呼び出します。Connection（接続）・Statement（命令）・ResultSet（結果）の3つは使い終わったら必ずクローズしなければなりません。try-with-resourcesを使うと自動でクローズされるので、リソース漏れを防げます。

Java — try-with-resourcesによる接続とクローズ 
import java.sql.*;

// 接続情報（実際は設定ファイルや環境変数で管理する）
String url  = "jdbc:mysql://localhost:3306/mydb?useSSL=false";
String user = "root";
String pass = "password";

// try-with-resources: ブロック終了時に自動でクローズされる
try (Connection conn = DriverManager.getConnection(url, user, pass)) {
    // ここでSQLを実行する（次節で詳しく説明）
    System.out.println("接続成功！");
} catch (SQLException e) {
    // 接続失敗時の処理
    e.printStackTrace();
}
クラス 役割 クローズが必要
Connection DBとの接続そのもの 必須（重要）
PreparedStatement SQL文の準備と実行 必須
ResultSet SELECT結果の読み取り 必須

接続URLの書き方

接続URLの形式は jdbc:DBの種類://ホスト:ポート/DB名?オプション です。MySQL なら jdbc:mysql://localhost:3306/mydb、H2（インメモリ）なら jdbc:h2:mem:testdb のように書きます。接続エラーのほとんどはURLの書き間違いです。

11.2 SQLの実行とSQLインジェクション対策

SQLを実行するには Statement と PreparedStatement の2種類があります。PreparedStatement を使うことが大原則です。理由は後述しますが、セキュリティ（SQLインジェクション対策）とパフォーマンスの両面で優れています。

SELECT文の実行とResultSet処理

SELECT文の結果は ResultSet として返されます。while(rs.next()) でループしながら1行ずつ値を取り出します。

Java — SELECT文とResultSetのループ 
String sql = "SELECT id, name, price FROM products WHERE category = ?";

try (Connection conn = DriverManager.getConnection(url, user, pass);
     PreparedStatement ps = conn.prepareStatement(sql)) {

    ps.setString(1, "food"); // ? に値をバインドする（1始まり）

    try (ResultSet rs = ps.executeQuery()) { // SELECT はexecuteQuery()
        while (rs.next()) { // 1行ずつ取り出す
            int    id    = rs.getInt("id");
            String name  = rs.getString("name");
            int    price = rs.getInt("price");
            System.out.println(id + ": " + name + " (" + price + "円)");
        }
    }
} catch (SQLException e) {
    e.printStackTrace();
}

INSERT/UPDATE/DELETE文の実行

データを変更するSQL（INSERT・UPDATE・DELETE）は executeUpdate() を使います。戻り値は影響を受けた行数（int）です。

Java — INSERT文（executeUpdate） 
String sql = "INSERT INTO products (name, price, category) VALUES (?, ?, ?)";

try (Connection conn = DriverManager.getConnection(url, user, pass);
     PreparedStatement ps = conn.prepareStatement(sql)) {

    ps.setString(1, "りんご");  // 1つ目の ? に名前を設定
    ps.setInt(2, 150);          // 2つ目の ? に価格を設定
    ps.setString(3, "food");   // 3つ目の ? にカテゴリを設定

    int rows = ps.executeUpdate(); // INSERT/UPDATE/DELETE はこちら
    System.out.println(rows + "件登録しました");
} catch (SQLException e) {
    e.printStackTrace();
}

SQLインジェクション対策（最重要）

SQLインジェクションとは、悪意のあるユーザーが入力値に特殊なSQL文を埋め込み、DBを不正操作する攻撃です。文字列連結でSQLを組み立てると、この攻撃に対して無防備になります。必ず PreparedStatement のプレースホルダ（?）を使いましょう。

危険: 文字列連結でSQL組み立て

絶対にやってはいけない例 
// ユーザー入力をそのまま連結 ← 危険！
String name = request.getParameter("name");
String sql = "SELECT * FROM users"
           + " WHERE name = '" + name + "'";
// name に " ' OR '1'='1 " が入ると
// WHERE name = '' OR '1'='1'
// → 全件取得されてしまう！

安全: PreparedStatementを使う

正しい書き方 
// ? プレースホルダでSQL構造を固定する
String name = request.getParameter("name");
String sql = "SELECT * FROM users"
           + " WHERE name = ?"; // 構造はここで確定
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, name); // 値は後からバインド
// 特殊文字は自動でエスケープされるため安全

PreparedStatementが安全な理由

PreparedStatementではSQLの構造が先に確定し、値は後からバインドされます。DBはSQL構造を解析済みなので、後から渡された値に特殊文字が含まれていても「データ」として扱うだけでSQL命令として解釈しません。これがSQLインジェクション対策の仕組みです。

11.3 DAOパターン

アプリが大きくなると「DBアクセスのコードがServletや JSP に散らばってしまう」問題が起きます。これを解決する設計パターンが DAO（Data Access Object）パターンです。「データベースへのアクセスを1つのクラスに集約する」考え方です。

DTO・DAOとは何か

DTO（Data Transfer Object）

データの入れ物クラス。第9章で学んだJavaBeansと同じ形（フィールド＋getter/setter）です。DBの1行 ＝ DTOの1インスタンスと対応します。

例: Product, User, Order

DAO（Data Access Object）

DBアクセス専門のクラス。CRUD操作をメソッドとして提供します。ServletやJSPからはDAOを呼ぶだけでよく、SQLを直接書く必要がありません。

例: ProductDao, UserDao

Web層 → DAO → DBの流れ

MVC構成にDAOを組み込むと、役割がさらに明確になります。Servlet（コントローラ）はDAOを呼び、DAOがDBと話します。JSP（ビュー）はDAOの存在を知る必要もありません。

DTOクラスとDAOクラスの実装例

まずデータの入れ物（DTO）を作り、次にDBアクセス専用クラス（DAO）を実装します。DAOはインターフェースと実装クラスに分けると、将来の差し替えが容易になります。

Java — Product DTO（データの入れ物） 
// DBのproductsテーブル1行分を表すDTO
public class Product {
    private int    id;
    private String name;
    private int    price;
    private String category;

    // getter/setter (省略: IDEで自動生成)
    public int    getId()       { return id; }
    public void   setId(int id) { this.id = id; }
    public String getName()     { return name; }
    public void   setName(String name) { this.name = name; }
    public int    getPrice()    { return price; }
    public void   setPrice(int price)  { this.price = price; }
    public String getCategory() { return category; }
    public void   setCategory(String category) { this.category = category; }
}
Java — ProductDao（DAOインターフェース） 
import java.util.List;

// DAOのインターフェース: 提供する操作を定義
public interface ProductDao {
    List<Product> findAll();              // 全件取得
    Product        findById(int id);       // 1件取得
    void           insert(Product p);      // 登録
    void           update(Product p);      // 更新
    void           delete(int id);         // 削除
}
Java — ProductDaoImpl（findAll の実装例） 
public class ProductDaoImpl implements ProductDao {
    private static final String URL  = "jdbc:mysql://localhost:3306/mydb";
    private static final String USER = "root";
    private static final String PASS = "password";

    @Override
    public List<Product> findAll() {
        List<Product> list = new java.util.ArrayList<>();
        String sql = "SELECT id, name, price, category FROM products";
        try (Connection conn = DriverManager.getConnection(URL, USER, PASS);
             PreparedStatement ps = conn.prepareStatement(sql);
             ResultSet rs = ps.executeQuery()) {
            while (rs.next()) {
                Product p = new Product();
                p.setId(rs.getInt("id"));
                p.setName(rs.getString("name"));
                p.setPrice(rs.getInt("price"));
                p.setCategory(rs.getString("category"));
                list.add(p);
            }
        } catch (SQLException e) {
            throw new RuntimeException("DB取得エラー", e);
        }
        return list;
    }
    // findById, insert, update, delete も同様に実装する
}

分離のメリット

DAOパターンで分離することで、Servletは「何をするか」だけを書け、「どうやってDBにアクセスするか」はDAOが担当します。DBの種類が変わっても、DAO実装クラスだけを差し替えれば他に影響がありません。これは保守性・テストしやすさにもつながる重要な設計原則です。

11.4 Webアプリケーションでの実装

JDBC・DAOの基礎が分かったら、実際のWebアプリとして組み合わせましょう。コネクションプール・トランザクション・例外処理・CRUDの全体フローを理解することで、実務に近い実装ができます。

コネクションプール

Webアプリでは多くのユーザーが同時にアクセスします。リクエストのたびに DriverManager.getConnection() で新規接続するのはコストが高く、パフォーマンスが落ちます。コネクションプールは、あらかじめ複数の接続を用意しておき、必要なときに貸し出す仕組みです。

flowchart LR subgraph pool["コネクションプール（接続を使い回す）"] C1["接続1"] C2["接続2"] C3["接続3（待機中）"] end R1["リクエストA"] -->|"借りる"| C1 R2["リクエストB"] -->|"借りる"| C2 C1 -->|"返却"| pool C2 -->|"返却"| pool

Tomcat では JNDI DataSource としてコネクションプールを設定できます。アプリ側は DataSource.getConnection() で接続を取得するだけです。設定は context.xml に記述します。

META-INF/context.xml — コネクションプールの設定 
<Context>
    <Resource name="jdbc/mydb"
              type="javax.sql.DataSource"
              driverClassName="com.mysql.cj.jdbc.Driver"
              url="jdbc:mysql://localhost:3306/mydb"
              username="root" password="password"
              maxTotal="10" maxIdle="5" />
</Context>

トランザクション管理の基礎

「注文と在庫を同時に更新する」など、複数のSQL操作をまとめて成功・失敗させたい場合はトランザクションを使います。setAutoCommit(false) で自動コミットを無効にし、処理が成功したら commit()、失敗したら rollback() します。

Java — トランザクション管理のパターン 
try (Connection conn = DriverManager.getConnection(url, user, pass)) {
    conn.setAutoCommit(false); // 自動コミットをOFF
    try {
        // 複数のSQL操作（例: 注文登録 ＋ 在庫更新）
        orderDao.insert(conn, order);   // 注文を登録
        stockDao.update(conn, itemId);  // 在庫を減らす
        conn.commit();  // 全部成功したらコミット
    } catch (Exception e) {
        conn.rollback(); // 1つでも失敗したら巻き戻し
        throw e;
    }
}

ここまで学んだMVC（第8章）＋DAOを組み合わせると、CRUD機能の全体像は次のようになります。

PRGパターン（Post/Redirect/Get）

登録・更新・削除のPOST処理後は、JSPへforwardせずsendRedirect() で一覧画面（GETリクエスト）へリダイレクトします。これをPRGパターンと呼び、ブラウザの「再読み込みによる二重送信」を防ぐ重要なテクニックです（第8章でも触れました）。

11.5 つまずきやすいポイント

DBアクセスはエラーの種類が多く、最初はデバッグに戸惑いがちです。よくある詰まりどころを整理しました。

接続エラー（Communications link failure 等）

原因は大きく3つです。

  • JDBCドライバ（.jar）の配置忘れ: WEB-INF/lib/ に正しく置かれているか確認。
  • 接続URLの誤り: ホスト・ポート番号・DB名が正しいか確認。ポート番号が違うだけで接続できません。
  • DBサーバーが起動していない: MySQLなどが起動しているか確認。

SQLインジェクション対策の忘れ

ユーザーから受け取った値を文字列連結でSQLに組み込むのは絶対に禁止です。必ず PreparedStatement の ? プレースホルダを使用し、setString() / setInt() でバインドしてください。これはセキュリティの基本ルールです。

コネクションリーク（接続の解放忘れ）

Connectionをクローズし忘れると、接続が溢れてアプリがフリーズします。必ず try-with-resources を使うことで自動クローズを保証しましょう。Connection・PreparedStatement・ResultSet の3つはすべてクローズが必要です。

理解度確認クイズ

  1. Q1. PreparedStatementを使うべき理由を、SQLインジェクションと絡めて説明してください。

    回答を表示

    文字列連結でSQLを組み立てると、悪意のあるユーザーが入力に特殊なSQL文を埋め込む「SQLインジェクション攻撃」が成立します。PreparedStatementは先にSQL構造を確定させ、値を後からバインドするため、ユーザー入力が混入してもSQL命令として解釈されずデータとして扱われます。その結果、SQLインジェクションを根本的に防止できます。

  2. Q2. DAOパターンにおけるDTOとDAOの役割の違いを説明してください。

    回答を表示

    DTO（Data Transfer Object）はデータの入れ物で、DBの1行に対応するフィールド・getter/setterを持つクラスです。DAO（Data Access Object）はDBアクセス専用のクラスで、CRUD操作のメソッドを提供します。Servletは「何をするか」だけを書き、「どうやってDBにアクセスするか」はDAOが担当することで、ビジネスロジックとデータアクセスが分離されます。

  3. Q3. トランザクションが必要な場面はどのような場合ですか？また、Javaでどのように実装しますか？

    回答を表示

    注文登録と在庫減算のように「複数のSQL操作がすべて成功または全部失敗にならなければいけない」場面でトランザクションが必要です。実装は conn.setAutoCommit(false) で自動コミットを無効にし、全SQL成功後に conn.commit()、途中で例外が発生したら conn.rollback() で巻き戻します。
