ホーム Git/GitHub 第6章
第6章 所要時間: 約50分

Pull Requestで変更を取り込む

この章では、GitHubを使ったチーム開発の中心にある Pull Request（PR） を体験します。ブランチで作業した変更をどうやってmainブランチに取り込むのか、「レビュー依頼 → 確認 → マージ」の一連の流れを実際に操作しながら学びます。

この章で学ぶこと

  • Pull Request（PR）が何のために使うものかを説明できる
  • 作業ブランチをGitHubにpushして、PRを作成できる
  • PRのタイトルと説明を適切に書ける
  • レビューコメントの確認方法を理解する
  • PRをマージし、ローカルのmainを最新に更新できる

6.1 Pull Request とは何か

Pull Request（PR・プルリクエスト） とは、ひとことで言うと「自分の変更をmainに取り込んでよいか確認してもらう依頼」です。

もう少し詳しく言うと、「この作業ブランチの変更を、mainブランチに取り込んでください」というレビュー依頼の仕組みです。第5章でブランチに変更をコミットしましたが、そのままではmainに反映されていません。PRを使うことで、差分を見ながら議論・確認してから取り込めます。

個人作業でも、チーム開発でも、PRを通すことで「何が・なぜ変わったのか」をGitHub上で明確に記録できます。これがチーム開発の標準的な進め方です。

差分が見える

変更前後のファイルを並べて確認でき、どこが変わったか一目でわかります。

議論できる

コードの特定の行にコメントを付けて、疑問点や改善案を話し合えます。

記録が残る

「なぜこの変更をしたか」の経緯がGitHubに永続的に残ります。

この章で体験する全体の流れを図にするとこうなります。

flowchart TD A["作業ブランチで
コミット済み"] --> B["git push でブランチを
GitHubに送る"] B --> C["GitHubで PR を作成
（Compare ＆ pull request）"] C --> D["タイトル・説明を書いて
Create pull request"] D --> E["レビュー・コメント確認
（Conversation / Files changed）"] E --> F["Merge pull request
でmainに取り込む"] F --> G["ローカルで git switch main
→ git pull して最新化"]

6.2 ブランチをGitHubにpushする

PRを作成する前に、作業ブランチをGitHubに送る必要があります。第5章でローカルにコミットした変更を、リモート（GitHub）にも反映させましょう。

ターミナル 
# 現在のブランチを確認（feature/add-greeting などの作業ブランチにいること）
git branch

# ブランチをGitHubにpushする（初回は -u でリモート追跡ブランチを設定）
git push -u origin feature/add-greeting
実行結果の例 
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Writing objects: 100% (3/3), 320 bytes | 320.00 KiB/s, done.
To github.com:あなたのID/myproject.git
 * [new branch]      feature/add-greeting -> feature/add-greeting
Branch 'feature/add-greeting' set up to track remote branch.

-u の意味

-u（--set-upstream）は「このローカルブランチのpush先をoriginのfeature/add-greetingに紐付ける」という設定です。2回目以降は git push だけで送れます。

pushが完了すると、GitHubのリポジトリページを開いたとき「Compare & pull request」という黄色いバナーが表示されます。これが次のステップへの入り口です。

6.3 Pull Requestを作成する

GitHubのリポジトリページで「Compare & pull request」ボタンをクリックするとPR作成画面が開きます。ここで base と compare の2つのブランチを選ぶ必要があります。

項目 意味 通常の設定
base 取り込み先のブランチ main
compare 取り込みたい変更のあるブランチ feature/add-greeting（作業ブランチ）

「base ← compare」の矢印の向きで読みます。「compareの変更をbaseに取り込む」という意味です。大抵はデフォルトで正しく設定されているので、確認するだけで大丈夫です。

「Pull Request」という名前の由来

「私のブランチをpull（引っ張って取り込む）してほしい」というお願いがPull Requestの語源です。変更を送りつけるのではなく、受け取る側が「いつ取り込むか」をコントロールできる仕組みです。

6.4 タイトルと説明を書く

PR作成画面には タイトル と 説明（Description） の入力欄があります。ここに書く内容は「何を・なぜ変えたか」を伝えることが目的です。

良い例

タイトル:

ユーザー名を受け取るあいさつ機能を追加

説明:

## 変更内容
・greeting.py に挨拶関数を追加

## 変更理由
ユーザーが名前を入力すると個別に返事できる機能が必要なため

避けたい例

タイトル:

修正

説明:

変更しました。

タイトルは「何をしたか」が一目でわかる動詞始まりの短文が理想です。説明には「なぜこの変更が必要か」「どう動作するか」を書くと、レビュアーが確認しやすくなります。

入力が終わったら「Create pull request」ボタンをクリックしてPRを作成します。

6.5 レビューコメントの見方

PRが作成されると、レビューのページが開きます。主に2つのタブを使います。

Conversation タブ （コメントや会話を見る場所）

PRへの全体コメントや、コードの行へのコメントが時系列で表示されます。「LGTM（Looks Good To Me）＝よさそうです」などの承認コメントもここに流れます。レビュアーとのやり取りのメインの場所です。

Files changed タブ （変更されたファイルと差分を見る場所）

変更されたファイルの差分（diff）が表示されます。緑の行が追加、赤の行が削除です。行番号の左をクリックすると、その行に直接コメントを付けられます。

レビュアーは確認が終わると次のいずれかのアクションを行います。

Comment

承認でも否認でもなく、意見やメモだけ残す。

Approve （承認）

「問題ありません。マージしてOK」という承認。

Request changes （修正依頼）

「この点を直してください」という修正依頼。

練習中は自分で自分のPRをレビューしてマージして大丈夫です。チーム開発では、他のメンバーがレビューするのが一般的です。

6.6 Pull Requestをマージする

レビューが完了したら、PRページの下部にある「Merge pull request」ボタンをクリックして変更をmainに取り込みます。確認ダイアログで「Confirm merge」を押すとマージ完了です。

マージ後の画面

マージが成功すると「Pull request successfully merged and closed」という紫のバナーが表示されます。同時に「Delete branch」ボタンも出るので、不要になった作業ブランチをGitHub上で削除できます。「Delete branch」はGitHub上のリモートブランチを削除する操作で、手元のローカルブランチの削除（git branch -d）とは別物です。 リモートを削除してもローカルのブランチはそのまま残ります。

マージ方法について

GitHubには「Merge commit」「Squash and merge」「Rebase and merge」の3種類があります。今は基本の Merge commit（デフォルト）だけ覚えておけば十分です。他の方式はチームのルールに応じて後から学びましょう。

  • Squash（Squash and merge）＝複数のコミットを1つにまとめてからマージする方法
  • Rebase（Rebase and merge）＝履歴の土台を付け替えてからマージする方法。初級では詳細理解は不要

6.7 マージ後のローカルを最新に更新する

GitHub上でマージされましたが、手元のローカルのmainブランチにはまだその変更が反映されていません。ローカルのmainを最新の状態に合わせる手順を実行しましょう。

GitHub上でマージしても、手元のmainは古いままです。次の作業を最新の状態から始めるために、ローカルのmainを更新します。

まず今どのブランチにいるかを確認してから、mainに切り替えます。

ターミナル（ブランチ確認） 
# 現在のブランチを確認する（* が付いているブランチが今いる場所）
git branch

mainブランチに切り替えてから、リモートの最新状態を取り込みます。

ターミナル 
# mainブランチに切り替える
git switch main

# 切り替え後にも確認する習慣をつけておくと安心（main に * が付いていればOK）
git branch

# リモートのmainの変更をローカルに取り込む
git pull
実行結果の例 
Switched to branch 'main'
Your branch is behind 'origin/main' by 2 commits, and can be fast-forwarded.
Updating abc1234..def5678
Fast-forward
 greeting.py | 5 +++++
 1 file changed, 5 insertions（+）

git pull が成功すると、ローカルのmainにマージ済みの変更が反映されます。これで手元の状態とGitHub上の状態が一致しました。実行結果に「Fast-forward」と表示された場合は、fast-forward＝履歴が一直線に進む取り込み方（余分なマージコミットを作らずにポインタをすすめる方法）が行われたことを意味します。

不要になったブランチを削除する

マージが完了した作業ブランチは、ローカルからも削除しておくとすっきりします。必須ではありませんが、ブランチが増えすぎると管理が大変になります。

ターミナル（ローカルブランチの削除） 
# マージ済みのブランチをローカルから削除する（-d は安全な削除）
git branch -d feature/add-greeting

やってみよう 6-1: PRを作ってマージするまでを体験する

第5章で作業したブランチを使って、実際にPRを作成・マージし、ローカルのmainを最新にする一連の流れを体験してみましょう。

手順

  1. 1 第5章の作業ブランチ（例: feature/add-greeting）に変更がコミットされていることを確認する。
    git log --oneline でコミットが1件以上あればOK。
  2. 2 git push -u origin ブランチ名 でGitHubにpushする。
  3. 3 GitHubのリポジトリページで「Compare & pull request」をクリックし、タイトルと説明を書いて「Create pull request」を押す。
  4. 4 Files changed タブで差分を確認し、「Merge pull request」→「Confirm merge」でマージする。
  5. 5 ターミナルで git switch main → git pull を実行し、git log --oneline でマージコミットが反映されているか確認する。

確認ポイント: git log --oneline の最上部に「Merge pull request #1」のようなコミットが表示されていればマージ成功です。

重要ポイント: PRはコミュニケーションの場

Pull Requestは単に「変更を取り込むボタン」ではありません。「なぜこの変更をするのか」「この設計で問題ないか」をチームで確認し、知識を共有する場です。コメントの履歴はそのまま設計判断の記録になります。初めのうちは内容が少なくても、PR経由で変更を取り込む習慣をつけることが大切です。

理解度確認クイズ

  1. Q1. Pull Requestを作る前に、まずやることは何ですか？

    回答を表示

    作業ブランチをGitHubにpushすることです。git push -u origin ブランチ名 を実行して、ローカルの変更をリモートに送ります。

  2. Q2. PRの「base」と「compare」はそれぞれ何を指しますか？

    回答を表示

    base は変更を取り込みたい先のブランチ（通常はmain）、compare は取り込みたい変更が入っている作業ブランチです。「compareからbaseへ取り込む」という向きになります。

  3. Q3. GitHub上でマージした後、ローカルのmainに変更を反映させるにはどのコマンドを使いますか？

    回答を表示

    まず git switch main でmainブランチに切り替え、次に git pull を実行します。これでリモート（GitHub）のmainの最新状態がローカルに取り込まれます。
