ホーム Git/GitHub 第3章
第3章 所要時間: 約60分

Gitの基本操作：変更を記録する

この章では、実際にコマンドを使って「ファイルの変更をGitに記録する」一連の流れを体験します。git initでリポジトリを作ることから始めて、git add・git commitで履歴を積み上げるまでの操作を、順を追って学んでいきます。

この章で学ぶこと

  • git init でリポジトリを初期化し、Git管理を開始できる
  • git status で作業フォルダの状態（変更・未追跡ファイル）を確認できる
  • git add でステージングエリアに変更を追加できる
  • git commit で変更を1件の履歴として記録できる
  • 「作業ツリー → ステージングエリア → リポジトリ」の3つの場所の流れを説明できる
  • git log・git diff でコミット履歴と差分を確認できる

3.1 Gitの「3つの場所」を理解する

実際に操作を始める前に、Gitがどのように変更を管理するかを頭に入れておきましょう。Gitには「3つの場所」があり、変更は必ずこの3か所を順に移動して記録されます。

flowchart LR A["作業ツリー
（ファイルを編集する場所）"] -->|"git add"| B["ステージングエリア
（コミット予約置き場）"] B -->|"git commit"| C["リポジトリ
（履歴が積み重なる場所）"]

作業ツリー（Working Tree）

普段ファイルを作ったり編集したりするフォルダそのもの。ここでの変更はまだGitには記録されておらず、「下書き」の状態です。

ステージングエリア（Staging Area / Index）

「次のコミットに含める変更」を一時的に置いておく場所。git add を実行すると、変更がここに移ります。「コミット予約票」のイメージです。

リポジトリ（Repository / .git）

コミットが積み重なっていく「変更履歴の倉庫」。git commit を実行すると、ステージングエリアの内容が1件のコミットとして永久保存されます。フォルダ内の .git/ フォルダがこの倉庫の実体です。

なぜ「ステージングエリア」が必要なの？

「編集したらすぐコミット」でもよいのでは？ と思うかもしれません。ステージングエリアを挟む理由は「コミットに含める変更を選べる」からです。例えば、10ファイルを変更していても「関連する3ファイルだけ今回のコミットにまとめたい」という場合に、addで3ファイルだけ選んでコミットできます。1コミット＝1つのまとまった変更、という整理が可能になります。

3.2 リポジトリを作る：git init

まず、Gitで管理したいフォルダを作り、そこで git init を実行します。これで「Gitリポジトリ」が誕生し、そのフォルダの変更履歴をGitが追跡できるようになります。

ターミナル（コマンドプロンプト） 
# 作業フォルダを作って移動する
mkdir my-first-repo
cd my-first-repo

# Git リポジトリを初期化する
git init
実行結果の例 
Initialized empty Git repository in /Users/yourname/my-first-repo/.git/

「Initialized empty Git repository」と表示されれば成功です。フォルダの中に .git/ という隠しフォルダが作成されました。これが「リポジトリの実体」で、この中にすべての履歴が保存されます。

フォルダの中身（.git が追加された） 

my-first-repo/
└── .git/          ← Gitが自動生成した隠しフォルダ（触らなくてOK）
    ├── config
    ├── HEAD
    ├── objects/
    └── refs/

重要ポイント: .git フォルダを削除しないこと

.git/ の中にすべてのコミット履歴が入っています。誤って削除すると、積み上げてきた履歴がすべて失われます。通常は中を触る必要はまったくないので、「存在するな」くらいの認識でOKです。

3.3 状態を確認する：git status

まずファイルを作ってみましょう。そのあと git status を実行すると、「今どんな状態か」が分かります。

ターミナル 
# README.md ファイルを作成する（Macの場合）
echo "# はじめてのリポジトリ" > README.md

# Gitの状態を確認する
git status
git status の表示例 
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md

nothing added to commit but untracked files present (use "git add" to track)

表示のポイントを読み解いてみましょう。

Untracked files

「Gitにまだ追跡されていないファイル」の一覧。新しく作ったファイルはここに表示されます。git add するまでGitは変更を追いかけません。

On branch main

現在いるブランチ（枝）の名前。最初は main ブランチにいます。ブランチは第5章でくわしく学びます。

git status は何度でも実行してOK

git status は「現状を確認するだけ」のコマンドなので、何回実行しても状態は変わりません。操作の合間に迷ったら、まず git status を打つ癖をつけましょう。

3.4 ステージに乗せる：git add

git add は、「次のコミットに含める変更をステージングエリアに移す」操作です。ファイルを指定して追加することも、フォルダ全体を一度に追加することもできます。

ターミナル 
# 特定のファイルだけをステージに追加する
git add README.md

# すべての変更ファイルをまとめてステージに追加する
git add .
コマンド 意味 使いどき
git add README.md 指定したファイルだけを追加 一部のファイルだけをコミットしたいとき
git add . 変更された全ファイルを追加 まとめて全変更をコミットしたいとき
git add src/ フォルダ以下を全部追加 特定フォルダの変更をまとめたいとき

git add README.md を実行したあとに git status を確認すると、ファイルが「Untracked」から「Changes to be committed」に移動しているのが分かります。

git add 後の git status 
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   README.md

git add . の「.」（ドット）の意味

ターミナルでは「.」は「カレントディレクトリ（今いるフォルダ）」を表す記号です。つまり git add . は「今いるフォルダ以下の変更ファイルをすべてステージに追加する」という意味になります。ファイルが多いときに便利ですが、含めたくないファイルまで追加してしまわないよう git status で確認してから使う習慣をつけましょう。

3.5 変更を記録する：git commit

ステージングエリアに乗せた変更を、1件の履歴として確定するのが git commit です。-m オプションで「何をしたか」を説明するコミットメッセージを添付します。

ターミナル 
# ステージした変更を「はじめてのコミット」というメッセージで記録する
git commit -m "はじめてのコミット：README を追加"
実行結果の例 
[main (root-commit) a1b2c3d] はじめてのコミット：README を追加
 1 file changed, 1 insertion(+)
 create mode 100644 README.md

a1b2c3d の部分は コミットID（ハッシュ） で、このコミットに自動的に割り振られる一意の識別子です。「何件変更されたか」も表示されています。コミットに成功すると、Gitの履歴に1件積み上がった状態になります。

コミットメッセージは「短く具体的に」

良いコミットメッセージは、あとで履歴を見たときに「何をしたか」が一目で分かります。「何のためにこの変更をしたか」を短い1文で書くのが基本です。

○良い例: ログイン画面のバリデーションを追加
✗ 悪い例: 修正 / 更新 / fix（何をしたか分からない）

3.6 履歴を見る：git log

コミットを重ねると履歴が積み上がります。git log でこれまでのコミット一覧を確認できます。

ターミナル 
# コミット履歴を表示する
git log

# 1行でコンパクトに表示する（見やすい！）
git log --oneline

git log の表示（詳細）

出力例 
commit a1b2c3d4e5f6...
Author: あなたの名前 <you@example.com>
Date:   Tue Jun 24 10:00:00 2026

    はじめてのコミット：README を追加

git log --oneline の表示（コンパクト）

出力例 
a1b2c3d はじめてのコミット：README を追加

git log の画面から抜けるには？

git log は履歴が多い場合、画面をスクロールしながら表示します。q キーを押すと元の画面に戻れます（「quit＝終了」のq）。スペースキーで次のページ、b で前のページに移動できます。

3.7 変更内容を確認する：git diff

ファイルを編集したあと、「どこを変えたんだっけ？」と確認したいときは git diff を使います。まだ git add していない変更の差分を表示します。

ターミナル 
# 作業ツリーの変更差分を表示する（add前の変更）
git diff

# ステージ済みの変更差分を表示する（add後の変更）
git diff --staged
git diff の出力例（README.md に1行追加した場合） 
diff --git a/README.md b/README.md
index abc1234..def5678 100644
--- a/README.md
+++ b/README.md
@@ -1 +1,2 @@
 # はじめてのリポジトリ
+このリポジトリは学習用です。

差分の読み方: + で始まる行が「追加された行」、- で始まる行が「削除された行」です。コミット前にこの差分を確認しておくと、意図しない変更を防げます。

コマンド 表示する差分
git diff add していない変更（作業ツリー vs ステージ）
git diff --staged add 済みの変更（ステージ vs 最新コミット）

やってみよう 3-1: init → 作成 → add → commit → log を通してやる

ここまでの操作をまとめて1から実行し、「変更を記録する」流れを体験しましょう。コマンドを1行ずつ打って、git status で状態がどう変わるかを観察してください。

手順

  1. 1 作業フォルダを作り、移動する: mkdir my-first-repo && cd my-first-repo
  2. 2 リポジトリを初期化する: git init → 「Initialized empty Git repository」が出ればOK
  3. 3 README.md を作成する（テキストエディタで「# はじめてのリポジトリ」と書いて保存、またはコマンドで）。
  4. 4 状態を確認する: git status → 「Untracked files: README.md」が表示されることを確認する。
  5. 5 ステージに追加する: git add README.md
  6. 6 再度状態を確認する: git status → 「Changes to be committed」に変わったことを確認する。
  7. 7 コミットする: git commit -m "はじめてのコミット：README を追加"
  8. 8 履歴を確認する: git log --oneline → コミットが1件表示されることを確認する。
  9. 9 （発展）README.md に1行追加して保存し、git diff で差分を確認してから2回目のコミットをしてみましょう。

各ステップで git status を実行して、「作業ツリー → ステージ → リポジトリ」の変化を目で確認するのがコツです。

3.8 この章のコマンドまとめ

この章で使ったコマンドを整理しておきます。次章以降でも繰り返し使うので、ざっと見渡しておきましょう。

コマンド 役割 3つの場所での位置づけ
git init リポジトリを初期化する リポジトリを作成（.git を生成）
git status 状態を確認する 作業ツリー・ステージの状態を表示
git add <ファイル> 変更をステージに追加する 作業ツリー → ステージングエリア
git commit -m "..." 変更を履歴に記録する ステージングエリア → リポジトリ
git log コミット履歴を表示する リポジトリの履歴を参照
git diff add前の変更差分を表示する 作業ツリー vs ステージの比較

理解度確認クイズ

  1. Q1. git add と git commit の違いを説明してください。

    回答を表示

    git add は「次のコミットに含める変更をステージングエリアに移す」操作で、まだ履歴には記録されません。git commit はステージングエリアの内容を「1件の変更履歴」としてリポジトリに永続保存する操作です。add で準備し、commit で確定するという2段階の仕組みになっています。

  2. Q2. 「作業ツリー → ステージングエリア → リポジトリ」の各段階を進めるコマンドは何ですか？

    回答を表示

    作業ツリー → ステージングエリアは git add、ステージングエリア → リポジトリは git commit です。また今どの状態かは git status で確認できます。

  3. Q3. git diff と git diff --staged の違いは何ですか？

    回答を表示

    git diff は「まだ git add していない変更（作業ツリーとステージの差分）」を表示します。git diff --staged は「git add 済みでまだコミットしていない変更（ステージと最新コミットの差分）」を表示します。
