ホーム Git/GitHub 第7章
第7章 所要時間: 約40分

よくあるトラブルと最低限の対処

最終章では、Gitを使っていると必ず一度はぶつかる「困った！」を解決できるようにします。コマンドを覚えるより大切なのは「まず何を確認するか」という習慣です。一つひとつ落ち着いて対処できるようになりましょう。

この章で学ぶこと

  • 困ったときにまず git status で現在地を確認する習慣を身につける
  • コミット前の変更取り消し・addの取り消し・コミットメッセージ修正の方法を理解する
  • pull時のよくあるエラーの概要を把握する
  • Permission denied (publickey) など、SSH接続エラーの確認手順をたどれる
  • コンフリクト（衝突）マーカーを読み取り、自力で解消できる
  • 「困ったときの確認順」を手順として実行できる

7.1 困ったらまず git status

Gitで何かうまくいかないとき、まず最初にやることは決まっています。git status を実行することです。このコマンドは「今どういう状態か」を教えてくれる万能の現在地確認コマンドです。

どのブランチにいるか、変更したファイルはどれか、addされているファイルはどれか——これらすべてを一度に表示してくれます。エラーが出た時も、操作の途中でも、迷ったらまず git status を叩く癖をつけましょう。

ターミナル 
# まず現在の状態を確認する
git status
出力例（変更あり・未addの状態） 
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   README.md

no changes added to commit (use "git add" and/or "git commit -a")

今いるブランチ

On branch main の部分で現在のブランチ名が分かります。

変更されたファイル

modified: でaddされていない変更ファイルを確認できます。

ステージ済みのファイル

Changes to be committed: でcommit待ちのファイルが確認できます。

「最初の一歩」として覚えておく

エラーメッセージが出た時も、操作に迷った時も、何かが起きていると感じた時も——まず git status。状況が見えれば、次に何をすればよいかが分かります。エラーが出たときは、あわてて追加のコマンドを次々に試さないこと。まずはエラーメッセージと git status の結果をよく読み、状況を確認してから動きましょう。

7.2 コミット前の変更を取り消す（作業ツリーを元に戻す）

ファイルを編集したものの「やっぱり元に戻したい」という場合は、git restore <ファイル名> を使います。指定したファイルの変更を取り消し、直前のコミット時点の状態に戻します。

注意: 実行する前に必ず読んでください

git restore（特に git restore .）は、まだコミットしていない変更をまとめて消してしまう操作です。この操作で消えた未コミットの変更は、基本的に元に戻せません。実行する前に、まず次の git diff で「これから消える内容」を必ず確認しましょう。

ターミナル（取り消す前に消える内容を確認） 
# 取り消す前に、変更内容（消える内容）を確認する
git diff
ターミナル（内容を確認したうえで取り消す） 
# README.md の変更を取り消して直前のコミット時点に戻す
git restore README.md

# 全ファイルの変更を取り消す場合（消える範囲が広いので特に注意）
git restore .

注意: 実行前に内容を確認すること

git restore を実行すると、その編集内容は元に戻せません。コミットされていない変更はGitの履歴に残っていないため、実行後は削除されたのと同じ状態になります。「本当に要らない変更か」を確認してから実行しましょう。

状況 コマンド 変更は残るか
作業ツリーの変更を取り消す git restore <file> 消える（元に戻せない）
addを取り消す（次節） git restore --staged <file> 残る

7.3 addを取り消す（ステージから外す）

git add でステージに追加したものの「やっぱりこのファイルはコミットに含めたくない」という場合は、git restore --staged <ファイル名> を使います。

このコマンドはステージから外すだけで、ファイルへの変更内容はそのまま残ります。前節の git restore（--staged なし）とは異なり、編集内容が消えることはありません。

ターミナル 
# README.md をステージから外す（変更内容は残る）
git restore --staged README.md

# 状態を確認する
git status

--staged の有無で動作が変わる

git restore --staged（ステージから外す・変更は残る）と git restore（変更ごと取り消す）は動作が全く異なります。実行前にどちらのオプションを付けているかを確認しましょう。

7.4 直前のコミットメッセージを修正する

コミット直後に「メッセージに誤字があった」「もっと分かりやすい説明にしたい」と気づいた場合は、git commit --amend で直前のコミットメッセージを書き直せます。

ターミナル 
# 直前のコミットメッセージを新しいメッセージで上書きする
git commit --amend -m "新しいコミットメッセージ"

push前のコミットのみ修正できる

git commit --amend はGitHubへpushする前のコミットにのみ使用してください。すでにpushしたコミットに対して実行すると、リモートとローカルの履歴がズレて、チームメンバーに影響が出ます。pushしたコミットのメッセージ修正は、チームに相談してから行いましょう。

7.5 pull時のよくあるエラー

git pull を実行したときに発生しやすいエラーは、大きく2つあります。焦らずメッセージを読むことが大切です。

ローカルに未コミットの変更がある場合

pullしようとすると「error: Your local changes would be overwritten」のようなメッセージが出ます。Gitは手元の変更を上書きしないよう止めてくれています。

対処: ①変更をcommitしてからpull ②または変更を捨てて（git restore）からpullする

リモートと履歴がズレている場合

「CONFLICT」や「Automatic merge failed」と表示される場合があります。これはコンフリクト（衝突）が発生した状態で、次節で解決方法を説明します。

対処: 次節の「コンフリクト解決」の手順を参照する

エラーメッセージはヒントの宝庫

英語で長く見えても、Gitのエラーメッセージには「何が起きているか」と「どうすればよいか」が書かれています。日本語に翻訳しながら一行ずつ読む習慣をつけると、問題解決がぐっと速くなります。

7.6 簡単なコンフリクト（衝突）を解決する

コンフリクト（conflict: 衝突） とは、同じ場所（同じファイルの同じ行）を、複数人や複数のブランチでそれぞれ変更したために、Gitがどちらを採用すればよいか自動で判断できなくなった状態です。Gitはどちらを正しいとするか決められないため、人間に「どちらを使うか決めてください」とサインを出します。

flowchart LR A["同じ箇所を
それぞれ変更"] --> B{"pullすると
コンフリクト発生"} B --> C["マーカーを確認"] C --> D["どちらを残すか
決めて編集"] D --> E["git add"] E --> F["git commit
で解決完了"]

コンフリクトマーカーの読み方

コンフリクトが発生すると、対象ファイルの中に次のようなコンフリクトマーカーが挿入されます。

コンフリクト時のファイルの中身（例） 
<<<<<<< HEAD
# プロジェクト概要（ローカルの変更）
=======
# プロジェクト説明（リモートの変更）
>>>>>>> origin/main

<<<<<<< HEAD

ここからが自分（ローカル）の変更内容の開始（HEAD＝いま自分が見ているブランチ・コミットの位置）

=======

ローカルとリモートの境界線

>>>>>>> origin/main

ここまでがリモートの変更内容の終わり（origin/main＝GitHub上のmainブランチを指す名前）

コンフリクトの解消手順

やってみよう 7-1: コンフリクトを解消する

  1. 1 コンフリクトが発生したファイルをテキストエディタで開く。コンフリクトマーカー（<<<<<<<、=======、>>>>>>>）を探す。
  2. 2 ローカルとリモートの内容を見比べ、「どちらを残すか（または両方を合わせた内容にするか）」を決めて編集する。マーカー行（3つ全て）も必ず削除すること。
  3. 3 編集が終わったら git add <ファイル名> でステージに追加する。
  4. 4 git commit を実行してコンフリクト解消を記録する。このときエディタ（メッセージ入力画面）が開くことがあります。メッセージはあらかじめ用意されているので、そのまま保存して閉じればOKです。
ターミナル（手順3・4） 
# コンフリクトを解消したファイルをステージに追加
git add README.md

# コンフリクト解消をコミット
git commit

エディタの操作が難しいときは、次のようにメッセージを付けてコミットしても大丈夫です（この書き方ならエディタは開きません）。

ターミナル（メッセージを付ける場合） 
# メッセージを付けてコンフリクト解消をコミットする
git commit -m "コンフリクトを解消"

VS Codeならボタン一つで解決できる

VS Code はコンフリクト箇所を色分け表示し、「Accept Current Change（ローカルを使う）」「Accept Incoming Change（リモートを使う）」「Accept Both Changes（両方残す）」のボタンを表示してくれます。テキスト編集に慣れないうちはVS Codeの画面から操作するのが安全です。

注意: 分からなくなったら、まず手を止める

コンフリクトの最中に分からなくなったら、むやみに追加のコマンドを打たないでください。まず git status で今の状態を確認し、必要なら講師やチームのメンバーに相談しましょう。あわてて操作するより、状況を確認してから動くほうが安全です。

7.7 SSHで接続できないとき（Permission denied）

push / clone / pull のときに Permission denied (publickey). と表示された場合は、GitHubへのSSH接続がうまくいっていません。次の順番に確認していきましょう。これらのコマンドは状態を確認するだけなので、落ち着いて1つずつ実行して大丈夫です。

① SSH鍵が作成されているか確認する

次のコマンドで、SSH鍵を保存する ~/.ssh フォルダの中身を表示します。

ターミナル（Git Bash / macOS / Linux 共通） 
# ~/.ssh フォルダの中身を表示する
ls ~/.ssh

id_ed25519（秘密鍵）と id_ed25519.pub（公開鍵）の2つがあるか確認します。見当たらない場合は、第2章の手順でSSH鍵を作成してください。

② GitHubに公開鍵を登録したか確認する

ブラウザでGitHubを開き、Settings → SSH and GPG keys に、自分の公開鍵が登録されているか確認します。登録されていなければ、第2章の手順で id_ed25519.pub の内容を登録してください。

③ SSH接続を確認する

次のコマンドで、GitHubにSSHで接続できるかを試します。

ターミナル（Git Bash / macOS / Linux 共通） 
# GitHubにSSHで接続できるか確認する
ssh -T git@github.com

「Hi ユーザー名! You've successfully authenticated...」と表示されれば、SSHの設定は正しくできています。

④ リモートURLがSSH形式になっているか確認する

次のコマンドで、登録済みのリモートURLを表示します。

ターミナル 
# 登録済みのリモートURLを確認する
git remote -v

正しいSSH形式の例

git@github.com:ユーザー名/my-first-repo.git

HTTPS形式になっている例

https://github.com/ユーザー名/my-first-repo.git

⑤ HTTPS形式になっていたらSSH形式に変更する

https://… の形になっていた場合は、次のコマンドでSSH形式に変更します。ユーザー名 とリポジトリ名は自分のものに置き換えてください。

ターミナル 
# リモートURLをSSH形式に変更する
git remote set-url origin git@github.com:ユーザー名/my-first-repo.git

補足: 覚えておきたいポイント

  • Permission denied (publickey). は、SSH鍵がGitHubに正しく登録されていない、またはリモートURLがSSH形式になっていない場合によく出ます。
  • 秘密鍵 id_ed25519 は絶対に共有しないでください。
  • GitHubに登録するのは id_ed25519.pub（公開鍵）の内容だけです。

7.8 困ったときの確認順（チェックリスト）

Gitで困ったときの「確認する順番」を覚えておくと、焦らず一つひとつ解決できます。次のフローを習慣にしましょう。

flowchart TD A["困った！"] --> B["① git status を実行
現在地を確認する"] B --> C["② 今どのブランチにいるか確認
（On branch の行）"] C --> D["③ 変更は add 済みか
commit 済みか確認"] D --> E["④ エラーメッセージを
よく読む"] E --> F["⑤ 落ち着いて
1つずつ対処する"] F --> G["解決！"]
1

まず git status

何が起きているかを把握する。「現在地」を確認せずに操作しないこと。

2

今どのブランチか確認する

On branch ○○ の部分を見る。意図しないブランチで作業していることが意外と多い。

3

変更は add・commit 済みか確認する

「変更したのにpushできない」の多くはaddし忘れ・commitし忘れが原因。

4

エラーメッセージをよく読む

英語でも、Gitのメッセージには原因と解決策のヒントが書かれている。翻訳しながら一文ずつ読む。

5

落ち着いて1つずつ対処する

一度にたくさんのコマンドを試さない。1つ実行するたびに git status で状態を確認しながら進む。

理解度確認クイズ

  1. Q1. 作業ツリーの変更を取り消すコマンドと、addを取り消すコマンドをそれぞれ答えてください。

    回答を表示

    作業ツリーの変更取り消し: git restore <file>（変更は消える）。addの取り消し: git restore --staged <file>（変更は残る）。

  2. Q2. コンフリクトマーカー ======= は何を表していますか？

    回答を表示

    ローカル（HEAD）の変更内容とリモート（origin/main）の変更内容の境界線です。上側がローカルの変更、下側がリモートの変更を示します。

  3. Q3. コンフリクトを解消した後に実行する2つのコマンドを順番に答えてください。

    回答を表示

    ①git add <ファイル名> でステージに追加し、② git commit でコンフリクト解消をコミットします。

  4. Q4. push しようとして Permission denied (publickey). と表示されました。確認すべきことを2つ挙げてください。

    回答を表示

    ①SSHの公開鍵がGitHubに登録されているか（Settings → SSH and GPG keys）、②リモートURLがSSH形式（git@github.com:…）になっているか（git remote -v で確認）です。HTTPS形式になっていた場合は git remote set-url origin git@github.com:ユーザー名/リポジトリ.git でSSH形式に変更します。

学習の総まとめ・次のステップへ

全7章を通じて、Git/GitHubの基本的な流れ「記録（commit）→ 共有（push）→ レビュー（Pull Request）→ 取り込み（merge）」と、トラブル時の対処方法を学びました。実際の開発現場で役立つ次のテーマとして、コミットの粒度を小さく保つ習慣（1つの目的で1コミット）や、不要なファイルをGit管理から除外する .gitignore の活用があります。チーム開発ではブランチ戦略やコードレビューの進め方がさらに重要になります。もっと深く学ぶには、本サイトの Claude Code 入門学習ガイド や Codex 入門学習ガイド も参照してください。AIを使った開発補助とGit/GitHubを組み合わせることで、より効率的な開発スタイルへ発展できます。

  • コミットの粒度を意識する（1つの目的で1コミット）
  • .gitignore でパスワードファイルや中間ファイルを管理外に除外する
  • チーム開発のブランチ戦略（GitHub Flow など）を学ぶ
  • Claude Code / Codex 教材でAIを使った開発補助を学ぶ
