Git・GitHub連携

Codex はコードを書くだけでなく、Git での変更管理や GitHub への PR 作成まで代わりに行えます。この章では、Codex を活用したバージョン管理ワークフローを実務の流れに沿って学びます。

この章で学ぶこと

  • Codex に変更をコミットさせる方法と、わかりやすいコミットメッセージを作らせるコツを理解する
  • ブランチを作成・切り替えして、安全な変更管理ができる
  • gh CLI を使ったプルリクエスト作成・Issue 対応の流れを身につける
  • Codex が行った Git 操作を自分で確認・制御するポイントを押さえる

10.1 Codex と Git の関係

Codex はターミナル上で動作するエージェントです。あなたが使うターミナルと同じ環境で git コマンドを実行できます。つまり、「このファイルを修正してコミットして」と一言伝えるだけで、コードを書いてそのままコミットまで完了させることができます。

ただし、Codex がコマンドを実行する際は承認ポリシーとサンドボックス（第7章で学習済み）のルールが適用されます。Read Only ではファイル変更やコマンド実行の前に承認が必要です。Auto では作業フォルダ内の操作を進めつつ、必要に応じて承認を求めます。Full Access では確認なしに実行します。Git 操作を Codex に任せるときは、これらのモードを意識することが重要です。

前提: Git が使える状態にする

Codex が git を使うには、作業ディレクトリが Git リポジトリである必要があります。git init 済み、または git clone 済みのフォルダで Codex を起動してください。また、git config user.name と git config user.email の設定も事前に済ませておきましょう。

Codex を使った Git ワークフローの全体像

flowchart TD A["あなたの指示"] --> B["Codex がコードを編集"] B --> C["差分を確認する
git diff"] C --> D{"承認する？"} D -->|はい| E["ステージングに追加
git add"] D -->|いいえ| B E --> F["コミット
git commit -m 'メッセージ'"] F --> G["リモートへ Push
git push"] G --> H["PR を作成
gh pr create"]

10.2 変更のコミット

Codex にコミットを依頼する最もシンプルな方法は、「変更をコミットして」と伝えることです。Codex は変更済みのファイルを確認し、内容に合ったコミットメッセージを考えてから git commit を実行します。

terminal — コミット依頼の例 
$ codex
> 今の変更内容をコミットしてください

● git diff でファイルを確認しています…
● 変更: src/auth/login.js（トークン有効期限チェックを追加）
● コミットメッセージ案: "fix: トークン有効期限チェックをloginに追加"
● git add src/auth/login.js && git commit を実行します。承認しますか？

Codex は変更の内容を読んだうえで、なぜその変更をしたのかが伝わるコミットメッセージを提案します。「こういうメッセージにして」と具体的に伝えることもできます。

terminal — メッセージを指定してコミット 
> 変更をコミットしてください。
  メッセージは「feat: ログインにリトライ制限を追加」にしてください

● git add -A && git commit -m "feat: ログインにリトライ制限を追加"
  を実行します。承認しますか？

良いコミットメッセージのポイント

Codex が提案するメッセージは Conventional Commits 形式（fix:、feat:、docs: など）に沿っていることが多いです。「変更の理由」が伝わるメッセージになっているか確認してから承認しましょう。

10.3 ブランチ運用

実務では、main ブランチへ直接コミットするのではなく、作業ブランチを作って変更をまとめ、後から main にマージする流れが一般的です。Codex に「ブランチを作って作業して」と伝えれば、ブランチの作成から切り替えまでをまとめて行えます。

ブランチを使った作業フロー

flowchart LR A[main ブランチ] -->|ブランチ作成| B["feature/login-retry"] B --> C["コード変更＋コミット"] C --> D["リモートへ Push"] D --> E["PR 作成・レビュー"] E -->|マージ| A
terminal — ブランチ作成から Push まで 
> feature/login-retry というブランチを作成して切り替え、
  ログインのリトライ制限を実装してコミットしてください

● git checkout -b feature/login-retry を実行します。承認しますか？
● src/auth/login.js にリトライ制限のコードを追加しました。
● git commit -m "feat: ログインリトライ上限を5回に設定" を実行します。

推奨: 作業ブランチで進める

機能・バグ修正ごとにブランチを切ると、レビューや rollback が容易です。

注意: main 直コミットは避ける

AGENTS.md に「main への直接コミット禁止」と書いておくと Codex も守ります。

10.4 gh CLI を使ったプルリクエスト作成

gh CLI（GitHub 公式のコマンドラインツール）を使うと、ターミナルから直接 GitHub 上にプルリクエスト（PR）を作成できます。Codex は gh コマンドも実行できるため、「PR を作って」と伝えるだけで、タイトル・説明文の作成から PR 発行まで一括して行えます。

gh CLI のインストール

macOS は brew install gh、Ubuntu は sudo apt install gh、Windows は winget install GitHub.cli でインストールできます。インストール後は gh auth login で認証してください。

terminal — Codex に PR を作成させる 
> このブランチの変更内容をもとに、main へのプルリクエストを gh CLI で作成してください

● ブランチ: feature/login-retry → main
● タイトル案: "feat: ログインリトライ上限を5回に制限"
● 説明文（日本語）を生成しました。
● gh pr create --base main --title "feat: ..." --body "..." を実行します。承認しますか？
  PR URL: https://github.com/your-org/your-repo/pull/42

Codex はコミット履歴や変更差分を読んで、PR の説明文を自動で作成します。「レビュー観点・テスト方法・影響範囲を含めて」と追加で指示すると、より詳細な説明文を生成してくれます。

terminal — 詳細な PR 説明文を生成 
> PR の説明文に「変更の概要」「テスト方法」「影響範囲」を含めてください

● 説明文を更新しました。
  - 変更の概要: リトライ上限を 5 回に制限しブルートフォースを防止
  - テスト方法: npm test auth/ を実行して全テストが通ることを確認
  - 影響範囲: src/auth/login.js のみ。他モジュールへの影響なし

10.5 Issue 対応ワークフロー

GitHub の Issue（バグ報告・機能要望）をもとに開発を進めるとき、Codex に「この Issue を修正して PR を出して」と伝えることで、Issue の確認 → ブランチ作成 → コード修正 → コミット → PR 作成という一連の流れをほぼ自動化できます。

Issue 対応の自動化フロー

flowchart TD A["Issue 番号を Codex に伝える"] --> B["gh issue view で Issue を確認"] B --> C["対応ブランチを作成"] C --> D["コードを修正・テスト"] D --> E["コミット"] E --> F["gh pr create で PR 作成"] F --> G["Issue に PR リンクを追加"]
terminal — Issue を指定して PR まで一括処理 
> GitHub Issue #35 を確認して修正し、PR を作成してください

● gh issue view 35 — 内容: "ログインが5回以上失敗しても制限されない"
● ブランチ fix/issue-35-login-retry を作成しました。
● src/auth/login.js を修正しました。
● テストを実行: 全10件 PASS
● コミット: "fix: #35 ログインリトライ制限を追加"
● PR を作成しました: https://github.com/your-org/repo/pull/43

注意: 変更内容は必ず自分で確認する

Codex が生成したコードやコミット内容は、Push 前に必ず git diff や git log で確認してください。予期しないファイルが含まれていたり、テストが通っていなかったりする場合があります。確認してから承認する習慣をつけましょう。

10.6 AGENTS.md で Git ルールを定める

第8章で学んだ AGENTS.md に Git・GitHub のルールを書いておくと、Codex は毎回の指示なしにそのルールに従って動きます。チームの規約を AGENTS.md に落とし込むことで、プロジェクト全体での一貫した運用が可能になります。

AGENTS.md — Git ルールの記述例 
## Git ルール

- main ブランチへの直接コミットは禁止。必ず作業ブランチを切ること。
- ブランチ名: feature/XXX（新機能）または fix/XXX（バグ修正）。
- コミットメッセージは Conventional Commits 形式（feat/fix/docs: ...）。
- PR 説明文には「概要」「テスト方法」「影響範囲」を含める。
- テストが全件 PASS していることを確認してからコミットすること。

ブランチ戦略

命名規則をルール化すると Codex が自動で正しいブランチ名を付けます。

コミットメッセージ

Conventional Commits 形式を指定すると一貫性のある履歴になります。

PR テンプレート

説明文のフォーマットを指定すると Codex が毎回同じ形式で生成します。

やってみよう 10-1: Codex でブランチ作成からコミットまで

手元の Git リポジトリで、Codex にブランチ作成・修正・コミットを一括で任せてみましょう。

手順

  1. 1 Git 管理済みのプロジェクトフォルダでターミナルを開き、codex を起動する。
  2. 2 「feature/try-codex-git というブランチを作成して、README.md の末尾に1行追記してコミットしてください」と指示する。
  3. 3 Codex が提案するコマンドを確認し、git log --oneline でコミット履歴を確認する。
  4. 4 （オプション）gh CLI が使える場合は「PR を作成してください」と続けて依頼する。

確認ポイント

  • Codex がどんなコミットメッセージを提案しましたか？
  • 承認前に差分（git diff）を確認しましたか？

理解度確認クイズ

  1. Q1. Codex が Git コマンドを実行できるようにするために、作業前に必要な確認事項は何ですか？

    回答を表示

    作業ディレクトリが Git リポジトリ（git init または git clone 済み）であること、git config でユーザー名・メールアドレスが設定済みであること、そして承認モードが Read Only 以外に設定されていることを確認します。

  2. Q2. Codex に PR を作成させるために必要なコマンドラインツールは何ですか？また、使う前にどんな準備が必要ですか？

    回答を表示

    GitHub 公式の gh CLI が必要です。インストール後に gh auth login で GitHub アカウントへの認証を済ませておく必要があります。

  3. Q3. AGENTS.md に Git のルールを記述することで、どのような効果が得られますか？

    回答を表示

    Codex がセッションをまたいで一貫してプロジェクトの Git 規約（ブランチ命名、コミットメッセージ形式、PR 説明文フォーマットなど）に従って動くようになります。毎回指示しなくてもチームルールを自動的に守ってもらえるため、作業の一貫性と効率が上がります。
